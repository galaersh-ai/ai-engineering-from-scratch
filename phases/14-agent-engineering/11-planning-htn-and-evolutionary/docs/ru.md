# Планирование с помощью HTN и эволюционного поиска

> Символическое планирование (Symbolic Planning) подходит для случаев, когда план доказуемо корректен. Эволюционный поиск кода (Evolutionary Code Search) подходит для случаев, когда функция пригодности (fitness function) может быть проверена автоматически. ChatHTN (2025) и AlphaEvolve (2025) демонстрируют, что открывает каждое из этих направлений в связке с LLM.

**Тип:** Практическая работа
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Фаза 14 · 02 (ReWOO и Plan-and-Execute)
**Время:** ~75 минут

## Цели обучения

- Объяснить иерархические сети задач (Hierarchical Task Networks): задачи, методы, операторы, предусловия, эффекты.
- Описать гибридный цикл ChatHTN — символьный поиск с резервной декомпозицией (fallback) через LLM.
- Объяснить эволюционный цикл AlphaEvolve и почему он работает только с программным evaluator.
- Реализовать упрощённый планировщик HTN и упрощённый эволюционный поиск на стандартной библиотеке.

## Проблема

ReWOO (Урок 02), Plan-and-Execute и ReAct покрывают большинство задач планирования агентов. Два случая, которые они плохо решают:

1. **Планы с доказуемой корректностью.** Расписание, прокладка маршрутов полётов, бизнес-процессы compliance — план должен быть корректен по построению. Свободно составленный LLM-план, который иногда галлюцинирует (hallucination) шаг, неприемлем.
2. **Оптимизации с автоматически проверяемой функцией пригодности.** Умножение матриц, эвристики расписаний, проходы компилятора — цель не «корректный план», а «лучший план».

Планирование HTN и AlphaEvolve решают две разные задачи. Оба используют LLM как усилители, а не замены.

## Концепция

### Иерархические сети задач

HTN (Hierarchical Task Network) состоит из:

- **Задач (Tasks)** — составные (подлежащие декомпозиции) и примитивные (непосредственно выполняемые).
- **Методов (Methods)** — способов декомпозиции составной задачи на подзадачи с предусловиями.
- **Операторов (Operators)** — примитивных действий с предусловиями и эффектами.
- **Состояния (State)** — множества фактов.

Планирование: заданы целевая задача и начальное состояние — найдите декомпозицию на примитивные операторы, предусловия которых выполняются последовательно.

HTN старше LLM и по-прежнему является эталоном для доказуемо корректных планов.

### ChatHTN (Gopalakrishnan et al., 2025)

ChatHTN (arXiv:2505.11814) чередует символьный HTN с запросами к LLM:

1. Попытка декомпозировать текущую составную задачу с помощью существующих методов.
2. Если ни один метод не применим, запрос к LLM: «как бы вы декомпозировали задачу `task` в состоянии `s`?»
3. Преобразование ответа LLM в кандидатные подзадачи.
4. Проверка по схеме операторов; отклонение некорректных декомпозиций.
5. Рекурсия.

Центральное утверждение статьи: каждый созданный план доказуемо корректен, поскольку предложения LLM поступают только как кандидатные декомпозиции, но никогда как прямые правки плана. Символьный уровень отвечает за корректность; LLM расширяет библиотеку методов.

Онлайн-обучение методов (OpenReview `gwYEDY9j2x`, продолжение 2025) добавляет модуль обучения, который обобщает декомпозиции, произведённые LLM, методом регрессии — сокращая частоту запросов к LLM до 75%.

### AlphaEvolve (Novikov et al., 2025)

AlphaEvolve (arXiv:2506.13131, DeepMind, июнь 2025) — это нечто иное: эволюционный поиск кода, управляемый ансамблем Gemini 2.0 Flash/Pro.

Цикл (Loop):

1. Начать с начальной программы + программного evaluator (возвращает оценку пригодности).
2. Ансамбль LLM предлагает мутации.
3. Запуск мутаций через evaluator.
4. Сохранение лучших; повторная мутация.

Опубликованные результаты:

- Первое улучшение метода Штрассена (Strassen) для умножения комплексных матриц 4×4 за 56 лет (48 скалярных умножений).
- Восстановление 0,7% вычислительных ресурсов Google за счёт эвристики планирования в Borg.
- Ускорение FlashAttention на 32% на актуальной нагрузке.

Строгое ограничение: функция пригодности должна быть автоматически проверяемой. Эволюционный поиск по текстовым ответам не сходится.

### Когда что использовать

| Класс задач | Инструмент | Почему |
|---------------|-----|-----|
| Расписание с жёсткими ограничениями | HTN + ChatHTN | Доказуемая корректность |
| Оптимизация компилятора | AlphaEvolve | Автоматически проверяемая функция пригодности |
| Выполнение многошаговых задач | ReAct / ReWOO | LLM в цикле, без формальных гарантий |
| Улучшение кода с тестами | AlphaEvolve | Тесты выступают evaluator |
| Автоматизация с привязкой к политике | HTN | Предусловия кодируют политику |

### Типичные ошибки

- **HTN без операторов.** Без схем предусловий/эффектов утверждение о корректности рушится. «LLM предлагает декомпозицию» в ChatHTN требует схемы для отклонения некорректных ходов.
- **AlphaEvolve без реального evaluator.** «Спросить LLM, стал ли код лучше» — это не функция пригодности. Evaluator должен быть детерминированным и быстрым.
- **Избыточная инженерия.** Большинству задач агентов не нужно ни то, ни другое. Начните с ReAct или ReWOO.

## Реализация

`code/main.py` реализует два упрощённых примера:

- Планировщик HTN на стандартной библиотеке с операторами, методами, предусловиями, эффектами и механизмом `LLMFallback`, который включается, когда ни один метод не подходит для составной задачи. «LLM» здесь — запрограммированный декомпозитор, чтобы планировщик работал автономно.
- Эволюционный поиск на стандартной библиотеке для арифметических программ: построение выражений, минимизирующих `|f(x) - target|` по тестовому набору. Evaluator детерминирован.

Запуск:

```
python3 code/main.py
```

Трассировка показывает, как планировщик HTN декомпозирует составную задачу (с промежуточным резервным вызовом LLM) и как эволюционный цикл сходится к целевому выражению.

## Применение

- **Планировщики HTN** — `pyhop`, `SHOP3` или собственная реализация для целенаправленного обеспечения политик.
- **ChatHTN** — исследовательский код; паттерн (символизация + резервный LLM) легко переносится на любой планировщик HTN.
- **AlphaEvolve** — статья DeepMind; паттерн (ансамбль + evaluator) воспроизводим. Появляются проекты OpenEvolve и другие open-source форки.
- **Фреймворки агентов** — ни один пока не предоставляет HTN или AlphaEvolve первого класса. Реализуйте как подагента или фонового worker.

## Готовые продукты

`outputs/skill-hybrid-planner.md` генерирует каркас гибридного планировщика (HTN или эволюционный) с явно ограниченной ролью LLM.

## Упражнения

1. Добавьте планировщику HTN механизм отката (backtracking): при невыполнении постусловия оператора во время выполнения откатывайтесь и пробуйте следующий метод.
2. Добавьте кэш LLM-методов в ChatHTN: при декомпозиции LLM задачи `T` в паттерне состояний `P` сохраняйте результат. При следующем вызове сначала проверяйте библиотеку методов.
3. Замените evaluator эволюционного поиска на реальный набор тестов. Эволюционно создайте функцию сортировки, проходящую 20 тестов; сообщите количество поколений до сходимости.
4. Изучите описание проектирования evaluator в AlphaEvolve. Спроектируйте evaluator для предметной области, которая вам интересна (оптимизация SQL-запросов, минимизация набора тестов, YAML развёртывания).
5. Комбинация: используйте HTN для декомпозиции составной задачи на подзадачи, затем эволюционный поиск для примитивного оператора каждой подзадачи. Где это эффективно, а где — избыточная инженерия?

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|------|----------------|------------------------|
| HTN | «Иерархический планировщик» | Декомпозиция задач с операторами, предусловиями, эффектами |
| Метод | «Правило декомпозиции» | Способ разбиения составной задачи на подзадачи |
| Оператор | «Примитивное действие» | Конкретный шаг с предусловием и эффектом |
| ChatHTN | «LLM + HTN» | Символьный планировщик запрашивает LLM, когда ни один метод не подходит |
| AlphaEvolve | «Эволюционный поиск кода» | Ансамбль LLM мутирует код; детерминированный evaluator отбирает |
| Функция пригодности | «Evaluатор» | Детерминированная, автоматически проверяемая оценка выходов |
| Онлайн-обучение методов | «Кэшированная декомпозиция LLM» | Сохранение и обобщение планов LLM для снижения стоимости запросов |
