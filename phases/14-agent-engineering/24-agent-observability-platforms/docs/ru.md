# Наблюдаемость за агентами (Agent Observability): Langfuse, Phoenix, Opik

> Три открытые платформы наблюдаемости за агентами доминируют в 2026 году. Langfuse (MIT) — более 6 млн установок/мес, трассировка + управление промптами (prompt management) + оценки (evals) + воспроизведение сессий (session replay). Arize Phoenix (Elastic 2.0) — углублённые агентные оценки, релевантность RAG, автоматическая инструментация OpenInference. Comet Opik (Apache 2.0) — автоматическая оптимизация промптов, защитные ограничения (guardrails), обнаружение галлюцинаций (hallucination) через LLM-судью.

**Тип:** Учебный материал
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Фаза 14 · 23 (OTel GenAI)
**Время:** ~45 минут

## Цели обучения

- Назвать три ведущие открытые платформы наблюдаемости за агентами и их лицензии.
- Различать сильные стороны каждой из них: Langfuse (управление промптами + сессии), Phoenix (RAG + автоматическая инструментация), Opik (оптимизация + защитные ограничения).
- Объяснить, почему 89% организаций сообщают о наличии наблюдаемости за агентами к 2026 году.
- Реализовать конвейер (pipeline) «трассировка → панель мониторинга» на стандартной библиотеке с оценкой через LLM-судью.

## Проблема

OTel GenAI (Урок 23) предоставляет схему данных. Вам всё ещё нужна платформа, которая принимает спаны (spans), выполняет оценки, хранит версии промптов и выявляет регрессии. Каждая из трёх платформ акцентирует внимание на разных этапах жизненного цикла.

## Концепция

### Langfuse (MIT)

- Более 6 млн установок SDK/мес, более 19 тыс. звёзд на GitHub.
- Возможности: трассировка (tracing), управление промптами с версионированием и песочницей (playground), оценки (LLM-в-качестве-судьи, обратная связь пользователей, пользовательские оценки), воспроизведение сессий.
- Июнь 2025: ранее коммерческие модули (LLM-в-качестве-судьи, очереди аннотаций, эксперименты с промптами, Playground) выпущены под лицензией MIT.
- Максимально эффективен для: сквозной наблюдаемости с тесной обратной связью по управлению промптами.

### Arize Phoenix (Elastic License 2.0)

- Углублённые агентные оценки: кластеризация трассировок (trace clustering), обнаружение аномалий, релевантность извлечения для RAG.
- Нативная автоматическая инструментация OpenInference.
- Интегрируется с управляемым Arize AX для продакшена.
- Без версионирования промптов — позиционируется как инструмент обнаружения дрейфа (drift) и поведенческих регрессий наряду с более широкими платформами.
- Максимально эффективен для: релевантности RAG, поведенческого дрейфа, обнаружения аномалий.

### Comet Opik (Apache 2.0)

- Автоматическая оптимизация промптов через A/B-эксперименты.
- Защитные ограничения (guardrails): замаскировывание PII, тематические ограничения.
- Обнаружение галлюцинаций через LLM-судью.
- Бенчмарк от самого Comet: логирование + оценки Opik за 23,44 с против 327,15 с для Langfuse (~14-кратный разрыв) — воспринимайте вендорские бенчмарки как ориентировочные.
- Максимально эффективен для: цикла оптимизации, автоматизированных экспериментов, обеспечения защитных ограничений.

### Отраслевые данные

По данным Maxim (анализ 2026 года): 89% организаций имеют наблюдаемость за агентами; проблемы качества являются основным барьером для продакшена (32% респондентов указывают на них).

### Как выбрать

| Потребность | Выбор |
|------|------|
| Универсальное решение с управлением промптами | Langfuse |
| Глубокая оценка RAG + дрейф | Phoenix |
| Автоматическая оптимизация + защитные ограничения | Opik |
| Открытая лицензия, без ELv2 | Langfuse (MIT) или Opik (Apache 2.0) |
| Интеграция с Datadog / New Relic | Любая — все экспортируют OTel |

### Типичные ошибки

- **Отсутствие стратегии оценки.** Трассировка без оценки — это просто дорогое логирование.
- **Самодельная LLM-судья без привязки к фактам.** Применяется шаблон CRITIC (Урок 05) — судье необходимы внешние инструменты для проверки фактов.
- **Версии промптов не привязаны к трассировкам.** Когда в продакшене происходит регрессия, невозможно выделить промпт, который её вызвал.

## Реализация

`code/main.py` реализует сборщик трассировок на стандартной библиотеке и оценщик через LLM-судью:

- Приём спанов в формате GenAI.
- Группировка по сессиям, пометка неудачных запусков (сработавшие защитные ограничения, оценки с низкой достоверностью).
- Скриптованная LLM-судья, оценивающая ответы агента по рубрике.
- Сводка в формате панели мониторинга: частота отказов, основные причины отказов, распределение оценок.

Запуск:

```
python3 code/main.py
```

Вывод: оценки по сессиям и категоризация отказов, аналогичные тем, которые показывают Langfuse/Phoenix/Opik.

## Применение

- **Langfuse** — самостоятельный хостинг или облако; подключение через OTel или их SDK.
- **Arize Phoenix** — самостоятельный хостинг; автоматическая инструментация OpenInference.
- **Comet Opik** — самостоянный хостинг или облако; автоматический цикл оптимизации.
- **Datadog LLM Observability** — для смешанных команд операционистов и ML-специалистов, уже использующих Datadог.

## Деплой

`outputs/skill-obs-platform-wiring.md` выбирает платформу и подключает трассировки + оценки + версии промптов к существующему агенту.

## Упражнения

1. Экспортируйте неделю OTel-трассировок в Langfuse cloud (бесплатный тариф). Какие сессии завершились неудачно? Почему?
2. Напишите рубрику LLM-судьи для вашей предметной области (фактическая корректность, тон, соблюдение.scope). Протестируйте на 50 трассировках.
3. Сравните версионирование промптов в Langfuse с кластеризацией трассировок в Phoenix. Какой инструмент быстрее покажет, что сломалось?
4. Прочитайте документацию по защитным ограничениям Opik. Подключите защитное ограничение маскирования PII к одному из запусков вашего агента.
5. Проведите бенчмарк трёх платформ на вашем корпусе. Игнорируйте опубликованные вендорские цифры; измеряйте собственные.

## Ключевые термины

| Термин | Как обычно говорят | Что это на самом деле |
|------|----------------|------------------------|
| Трассировка (Tracing) | «Сборник спанов» | Приём OTel / SDK-спанов; индексация по сессиям |
| Управление промптами (Prompt management) | «CMS для промптов» | Версионированные промпты, привязанные к трассировкам |
| LLM-в-качестве-судьи (LLM-as-judge) | «Автоматизированная оценка» | Отдельная LLM оценивает выход агента по рубрике |
| Воспроизведение сессий (Session replay) | «Воспроизведение трассировки» | Пошаговый проход по прошлым запускам для отладки |
| Релевантность RAG (RAG relevancy) | «Качество извлечения» | Соответствует ли извлечённый контекст запросу |
| Кластеризация трассировок (Trace clustering) | «Группировка по поведению» | Группировка похожих запусков для обнаружения дрейфа |
| Защитные ограничения (Guardrail enforcement) | «Политика при логировании» | Проверки PII/токсичности/области на записываемом контенте |
