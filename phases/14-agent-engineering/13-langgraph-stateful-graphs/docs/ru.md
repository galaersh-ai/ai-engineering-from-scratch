# LangGraph: Stateful Graphs (сохраняющие состояние графы) и Durable Execution (устойчивое выполнение)

> LangGraph — эталонный инструмент 2026 года для низкоуровневой stateful-оркестрации. Агент (Agent) — это конечный автомат (state machine); узлы (nodes) — функции; рёбра (edges) — переходы; состояние неизменяемое и чекпоинтится после каждого шага. Возобновляйте выполнение из любой точки отказа точно с того места, где оно прервалось.

**Тип:** Изучение + Сборка
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 01 (Цикл агента), Фаза 14 · 14 (Шаблоны воркфлоу)
**Время:** ~75 минут

## Цели обучения

- Описать базовую модель LangGraph: конечный автомат с неизменяемым состоянием (state), функциями-узлами (nodes), условными рёбрами (conditional edges) и чекпоинтами (checkpoints) после каждого шага.
- Назвать четыре ключевые возможности, выделяемые документацией: durable execution (устойчивое выполнение), streaming (стриминг), human-in-the-loop (человек в цикле), comprehensive memory (полноценная память).
- Объяснить три топологии оркестрации (orchestration), поддерживаемые LangGraph: supervisor (супервайзер), peer-to-peer / swarm (роевой), hierarchical (иерархическая — вложенные подграфы).
- Реализовать stdlib state graph с неизменяемым состоянием, условными рёбрами и циклом чекпоинт / возобновление.

## Проблема

Агенты (agents) и воркфлоу (workflows) разделяют одну проблему: когда 40-шаговое выполнение падает на шаге 38, вы хотите возобновить с шага 38, а не начинать заново. Упрощённые модели состояния заставляют операторов городить повторные попытки вокруг библиотеки, которая рассчитывает на запуск с нуля.

Проектированный ответ LangGraph: состояние — это объект первого класса с типизацией, мутации — явные, а чекпоинты — персистентные после каждого узла. Возобновление — это вызов `load_state(session_id)`.

## Концепция

### Граф (graph)

Граф определяется:

- **Типом состояния (State).** Типизированный словарь (dict) или модель Pydantic, который читает и изменяет каждый узел.
- **Узлами (Nodes).** Чистые функции `(state) -> state_update`. Обновления сливаются (merge) с состоянием после возврата.
- **Рёбрами (Edges).** Условные (conditional) или прямые переходы между узлами.
- **Входом и выходом.** Специальные узлы `START` и `END` обозначают границы.

Пример: агент с узлами `classify`, `refund`, `bug`, `sales`, `done` — маршрутизирующий воркфлоу в виде графа.

### Durable Execution (устойчивое выполнение)

После каждого узла среда выполнения сериализует состояние и записывает его в чекпоинтер (checkpoint): SQLite, Postgres, Redis или собственная реализация. При отказе на шаге N среда выполнения может выполнить `resume(session_id)` и продолжить с шага N+1 с точным состоянием.

Документация LangGraph явно выделяет продакшн-пользователей, для которых это важно: Klarna, Uber, J.P. Morgan. Утверждение касается не формы графа; форма графа в сочетании с чекпоинтингом делает восстановление дешевым.

### Streaming (стриминг)

Каждый узел может генерировать частичный вывод. Граф передаёт вызывающему коду события дельт-на-узел (per-node-delta), чтобы интерфейс обновлялся по мере работы графа.

### Human-in-the-Loop (человек в цикле)

Просмотр и модификация состояния между узлами. Реализация: пауза перед критическим узлом, вывод состояния человеку, приём модификаций, возобновление. Чекпоинтер упрощает это, поскольку состояние уже сериализовано.

### Memory (память)

Краткосрочная (внутри запуска — история разговоров в состоянии) и долгосрочная (между запусками — персистентная через чекпоинтер плюс отдельное долгосрочное хранилище). LangGraph интегрируется с внешними системами памяти (Mem0, собственные реализации) через инструменты.

### Три топологии

1. **Supervisor (супервайзер).** Центральный маршрутизирующий LLM диспатчит (dispatch) задачи специализированным подагентам. `create_supervisor()` в `langgraph-supervisor` (хотя команда LangChain в 2026 году рекомендует делать это напрямую через вызовы инструментов для более точного контроля контекста).
2. **Swarm / peer-to-peer (роевой / равный-равному).** Агенты передают (handoff) друг другу задачи через общий набор инструментов. Без центрального маршрутизатора.
3. **Hierarchical (иерархическая).** Супервайзеры управляют подсупервайзерами, реализуемыми как вложенные подграфы.

### Где эта модель может привести к ошибкам

- **Слишком мелкие чекпоинты.** Чекпоинтинг только истории разговоров оставляет состояние инструментов и записи в памяти невосстанавливаемыми. Сериализоваться должно полное состояние.
- **Недетерминированные узлы.** Возобновление предполагает, что одни и те же входы узла производят одно и то же обновление состояния. Случайные засеивания (random seeds), системное время, внешние API должны фиксироваться.
- **Чрезмерное использование условных рёбер.** Граф, где каждое ребро условное, — это конечный автомат, который невозможно осмыслить. Предпочтительнее линейные цепочки с редкими ветвлениями.

## Реализация

`code/main.py` реализует stdlib stateful graph (граф, сохраняющий состояние):

- `State` — типизированный словарь с полями `messages`, `step`, `route`, `output`, `human_approval`.
- `Node` — вызываемый объект (callable), принимающий состояние и возвращающий словарь обновления.
- `StateGraph` — узлы + рёбра + условные рёбра + запуск + возобновление.
- `SQLiteCheckpointer` (фейковая in-memory реализация) — сериализует состояние после каждого узла; `load(session_id)` восстанавливает его.
- Демонстрационный граф: classify -> branch (refund / bug / sales) -> human gate -> send.

Запуск:

```
python3 code/main.py
```

Трассировка покажет, что первый запуск падает на human gate, происходит персистентность, затем возобновление генерирует финальный вывод.

## Упражнения

1. Добавьте условное ребро из `classify` в `end`, когда уверенность классификации ниже порога. Возобновите запуск после того, как человек вручную установит `route`.
2. Замените фейковую SQLite-реализацию на настоящий чекпоинтер SQLite. Измерьте накладные расходы сериализации на каждом шаге.
3. Реализуйте параллельные рёбра: два узла выполняются параллельно, результаты объединяются пользовательской функцией-редьюсером (reducer). Какую выгоду даёт неизменяемое состояние?
4. Изучите справочную документацию `langgraph-supervisor`. Перенесите игрушечный пример на `create_supervisor`. Сравните формы трассировок.
5. Добавьте стриминг (streaming): каждый узел генерирует частичное состояние по мере работы. Выводите дельты по мере поступления.

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|--------|------------|----------------------|
| State graph (граф состояний) | «Агент как конечный автомат» | Типизированное состояние + узлы + рёбра + редьюсеры |
| Checkpointer (чекпоинтер) | «Бэкенд персистентности» | Сериализует состояние после каждого узла; обеспечивает возобновление |
| Reducer (редьюсер) | «Мержер состояний» | Функция, объединяющая текущее состояние с обновлением узла |
| Conditional edge (условное ребро) | «Ветвление» | Ребро, выбираемое функцией состояния |
| Subgraph (подграф) | «Вложенный граф» | Граф, используемый как узел внутри другого графа |
| Durable execution (устойчивое выполнение) | «Возобновление после отказа» | Перезапуск с последнего успешного узла с точным состоянием |
| Supervisor (супервайзер) | «Маршрутизирующий LLM» | Центральный диспетчер для специализированных подагентов |
| Swarm (рой) | «P2P-агенты» | Агенты передают друг другу задачи через общие инструменты; без центрального маршрутизатора |
