# Гибридная память (Hybrid Memory): вектор + граф + KV (Mem0)

> Mem0 (Chhikara и др., 2025) рассматривает память как три параллельных хранилища — векторное для семантического сходства, KV для быстрого поиска фактов, графовое для рассуждений о связях между сущностями. Слой оценки (scoring layer) объединяет все три при извлечении (retrieval). Это стандарт для внешней памяти в продакшне на 2026 год.

**Тип:** Практическая работа
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 07 (MemGPT), Фаза 14 · 08 (Letta Blocks)
**Время:** ~75 минут

## Цели обучения

- Объяснить, почему одного хранилища (только векторного, только графового, только KV) недостаточно для памяти агента (agent).
- Назвать три параллельных хранилища Mem0 и указать, для чего каждое оптимизировано.
- Описать оценку слияния (Fusion Scoring) в Mem0 — релевантность, важность, актуальность — и объяснить, почему она представлена в виде взвешенной суммы, а не иерархии.
- Реализовать упрощённую трёххранилищную память на stdlib с методом `add()`, записывающим во все три хранилища, и методом `search()`, объединяющим результаты.

## Проблема

Одного хранилища недостаточно для одного из трёх классов запросов:

- **Семантическое сходство (Semantic Similarity)** — «что мы обсуждали на прошлой неделе о дрейфе агентов?» Выигрывает векторное хранилище; KV и граф не справляются.
- **Поиск фактов (Fact Lookup)** — «какой номер телефона у пользователя?» Выигрывает KV-хранилище; векторное расточительно, графовое — избыточно.
- **Рассуждения о связях (Relationship Reasoning)** — «какие клиенты используют одну и ту же учётную запись для оплаты?» Выигрывает графовое хранилище; векторное и KV не могут ответить.

Агенты в продакшне задают все три типа запросов в рамках одной сессии. Хранилище одного типа всегда непригодно для двух из них. Вклад Mem0 заключается в объединении всех трёх за единой поверхностью `add`/`search` с функцией оценки (scoring function), которая их объединяет.

## Концепция

### Три хранилища параллельно

Mem0 (arXiv:2504.19413, апрель 2025) при вызове `add(text, user_id, metadata)`:

1. Извлекает кандидатные факты из текста (этап, управляемый LLM).
2. Записывает каждый факт в векторное хранилище (эмбеддинг — embedding) для семантического поиска.
3. Записывает каждый факт в KV-хранилище с ключом (user_id, fact_type, entity) для поиска за O(1).
4. Записывает каждый факт в графовое хранилище (Mem0g) в виде типизированных рёбер для запросов по связям.

При вызове `search(query, user_id)`:

1. Векторное хранилище возвращает top-k по косинусному сходству эмбеддингов.
2. KV-хранилище возвращает прямые совпадения по ключу (user_id, type, entity), полученному из запроса.
3. Графовое хранилище возвращает подграф (subgraph), достижимый из сущностей запроса.
4. Слой оценки объединяет все три результата.

### Оценка слияния (Fusion Scoring)

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **Релевантность (Relevance)** — косинусное сходство векторов, точное совпадение в KV, вес пути в графе.
- **Важность (Importance)** — определяется при записи или обучается (некоторые факты имеют бо́льшую значимость: имена, идентификаторы, политики).
- **Актуальность (Recency)** — экспоненциальное затухание по времени с момента последней записи или чтения.

Веса настраиваются для каждого продукта. Для чат-агентов выше `w_recency`; для агентов комплаенса выше `w_importance`; для агентов извлечения выше `w_relevance`.

### Mem0g и временные рассуждения (Temporal Reasoning)

Mem0g добавляет детектор конфликтов. Когда новый факт противоречит существующему ребру, существующее ребро помечается как недействительное, но не удаляется. Временные запросы («каким был город пользователя в марте?») обходят подграф, действительный на заданный момент времени.

Это поведение уровня комплаенса, которое является обобщением паттерна инвалидации из Letta.

### Результаты бенчмарков

В работе Mem0 приводятся следующие результаты (2025):

- **LoCoMo** (память для длинных разговоров): 91,6
- **LongMemEval** (долгосрочная эпизодическая память): 93,4
- **BEAM 1M** (бенчмарк памяти на 1 млн токенов): 64,1

Эталонные модели (полный контекст 128k, плоское векторное хранилище, плоское KV) уступают на 10 и более баллов. Бенчмарки сами по себе не оправдывают выбор — определяет операционная модель. Однако цифры показывают, что архитектура слияния не является случайной ошибкой.

### Таксономия областей (Scope Taxonomy)

Mem0 разделяет память по областям:

- **Память пользователя (User Memory)** — сохраняется между сессиями, ключ — `user_id`.
- **Память сессии (Session Memory)** — сохраняется в пределах одного потока.
- **Память агента (Agent Memory)** — состояние конкретного экземпляра агента (instance).

При каждой записи выбирается одна область. При извлечении (retrieval) можно запрашивать данные из нескольких областей с весами для каждой области. Бессмысленное смешивание областей приводит к инцидентам вроде «ассистент рассказал Алисе о проекте Боба».

### Типичные ошибки при реализации этого паттерна

- **Дрейф эмбеддингов (Embedding Drift).** Векторные результаты, выглядящие верными при первых сотнях запросов, деградируют по мере роста корпуса. Добавьте периодическую переэмбеддингацию наиболее часто используемых записей.
- **Расползание схемы KV (KV Schema Creep).** Схема `(user_id, type, entity)` кажется простой, пока каждая команда не начинает добавлять собственные значения `type`. Проводите аудит набора типов ежеквартально.
- **Взрыв графа (Graph Explosion).** Один шумный экстрактор добавляет 50 рёбер на сообщение. Ограничивайте количество записей в граф за один вызов `add`; отбрасывайте рёбра с низкой достоверностью.

## Реализация

`code/main.py` реализует паттерн трёх хранилищ на stdlib:

- `VectorStore` — наивное сходство по пересечению токенов в качестве замены эмбеддингам.
- `KVStore` — словарь (dict) с ключом `(user_id, fact_type, entity)`.
- `GraphStore` — типизированные рёбра (предикат, отношение, объект, действительность).
- `Mem0` — верхнеуровневый фасад с методами `add()`, `search()`, оценкой слияния и извлечением с учётом областей.
- Пример трассировки для разговора несколькими пользователями в нескольких сессиях.

Запуск:

```
python3 code/main.py
```

Вывод демонстрирует три отдельных пути вспоминания, а также объединённый top-k. Измените веса оценки в начале функции `main()` и наблюдайте за изменением ранжирования.

## Применение

- **Mem0 (Apache 2.0)** — готов к продакшну. Самостоятельное развёртывание (deploy) с Postgres + Qdrant + Neo4j или использование облачного сервиса.
- **Letta** — трёхуровневая архитектура core/recall/archival; собственные векторные и графовые бэкенды.
- **Zep** — коммерциональная альтернатива с временным графом знаний (KG) и извлечением фактов.
- **Собственные решения** — когда требуется точный контроль над экстрактором (комплаенс) или весами слияния (голосовые агенты, где доминирует актуальность).

## Ключевые термины

| Термин | Общее название | Точное значение |
|--------|----------------|-----------------|
| Гибридная память (Hybrid Memory) | «Вектор плюс граф плюс KV» | Три хранилища, записываемых параллельно и объединяемых при извлечении |
| Извлекание фактов (Fact Extraction) | «Загрузка памяти» | Этап LLM, разбивающий текст на тройки (сущность, отношение, факт) |
| Оценка слияния (Fusion Scoring) | «Ранжирование по релевантности» | Взвешенная сумма релевантности, важности, актуальности |
| Область (Scope) | «Пространство имён памяти» | user / session / agent — определяет, кто видит какие данные |
| Mem0g | «Граф памяти» | Типизированные рёбра с временной действительностью для запросов по связям |
| Временная инвалидация (Temporal Invalidation) | «Мягкое удаление» | Помечает противоречащие рёбра как недействительные; никогда не удаляет |
| Дрейф эмбеддингов (Embedding Degradation) | «Ухудшение извлечения» | Качество векторов деградирует по мере роста корпуса; требуется переэмбеддингация |
