# Self-Refine и CRITIC: итеративное улучшение выходных данных

> Self-Refine (Madaan et al., 2023) использует одну LLM в трёх ролях — генерация (generate), обратная связь (feedback), доработка (refine) — в цикле. Средний прирост: +20 абсолютных пунктов на 7 задачах. CRITIC (Gou et al., 2023) укрепляет шаг обратной связи, направляя верификацию через внешние инструменты. В 2026 году этот паттерн поставляется во всех фреймворках как «evaluator-optimizer» (Anthropic) или цикл ограничителей (guardrail loop, OpenAI Agents SDK).

**Тип:** сборка (Build)
**Языки:** Python (stdlib)
**Предварительные знания:** Фаза 14 · 01 (цикл агента), Фаза 14 · 03 (Reflexion)
**Время:** ~60 минут

## Цели обучения

- Назвать три промпта Self-Refine (генерация, обратная связь, доработка) и объяснить, почему история важна для промпта доработки.
- Объяснить ключевое открытие CRITIC: LLM ненадёжны при самопроверке без внешней опоры (grounding).
- Реализовать цикл Self-Refine на stdlib с историей и опциональным внешним верификатором.
- Сопоставить этот паттерн с рабочим процессом (workflow) «evaluator-optimizer» от Anthropic и ограничителями выходных данных (output guardrails) OpenAI Agents SDK.

## Проблема

Агент (agent) выдаёт ответ, который почти правильный. Возможно, в строке кода есть синтаксическая ошибка. Возможно, саммари слишком длинное. Возможно, план пропускает граничный случай. Вы хотите, чтобы агент критиковал собственный выходной результат, а затем исправлял его.

Self-Refine показывает, что это работает с одной моделью (model), без обучающих данных, без обучения с подкреплением (RL). Но есть нюанс: LLM плохо справляются с самопроверкой на фактических данных. CRITIC называет решение — направить шаг проверки через внешние инструменты (поиск, интерпретатор кода, калькулятор, тестовый раннер).

Вместе эти две работы определяют паттерн по умолчанию на 2026 год для итеративного (iteration) улучшения: генерация, верификация (внешняя, когда это возможно), доработка, остановка при успешной проверке.

## Концепция

### Self-Refine (Madaan et al., NeurIPS 2023)

Одна LLM, три роли:

```
generate(task)            -> output_0
feedback(task, output_0)  -> critique_0
refine(task, output_0, critique_0, history) -> output_1
feedback(task, output_1)  -> critique_1
refine(task, output_1, critique_1, history) -> output_2
...
stop when feedback says "no issues" or budget exhausted.
```

Ключевая деталь: `refine` видит всю историю — все предыдущие выходы и критики — поэтому не повторяет ошибок. В работе проводится абляция: убираем историю — качество резко падает.

Заголовок: +20 абсолютных пунктов улучшения в среднем по 7 задачам (математика, код, акронимы, диалоги), включая GPT-4. Без обучения, без внешних инструментов, одна модель.

### CRITIC (Gou et al., arXiv:2305.11738, v4 Feb 2024)

Слабость Self-Refine: шаг обратной связи — это LLM, оценивающий сам себя. Для фактических утверждений это ненадёжно (галлюцинация (hallucination) часто выглядит убедительно для модели, её породившей). CRITIC заменяет `feedback(task, output)` на `verify(task, output, tools)`, где `tools` включает:

- Поисковую систему для фактических утверждений.
- Интерпретатор кода для проверки корректности кода.
- Калькулятор для арифметических вычислений.
- Специфические для домена верификаторы (unit-тесты, проверки типов, линтеры).

Верификатор выдаёт структурированную рецензию (critique), опирающуюся на результаты инструментов. Модуль доработки (refiner) затем использует эту рецензию в качестве условия.

Заголовок: CRITIC превосходит Self-Refine на фактических задачах, потому что рецензия имеет опору (grounding). На задачах без внешних верификаторов (творческое письмо, форматирование) CRITIC сводится к Self-Refine.

### Условие остановки (stop condition)

Две распространённые формы:

1. **Верификатор проходит проверку.** Внешний тест возвращает успех. Предпочтительно при наличии (unit-тесты, проверка типов, утверждение ограничителя).
2. **Обратная связь не выдана.** Модель утверждает, что «выходные данные в порядке». Дешевле, но ненадёжно; используйте с ограничением максимального числа итераций.

Паттерн по умолчанию на 2026 год: комбинация. «Остановиться, если верификатор прошёл проверку ИЛИ модель говорит «всё в порядке» И количество итераций >= 2 ИЛИ количество итераций >= max_iterations.»

### Evaluator-Optimizer (Anthropic, 2024)

Публикация Anthropic от декабря 2024 года называет это одним из пяти паттернов рабочего процесса (workflow). Две роли:

- Эвалюатор (Evaluator): оценивает выходные данные и выдаёт рецензию.
- Оптимизатор (Optimizer): пересматривает выходные данные на основании рецензии.

Цикл продолжается до тех пор, пока эвалюатор не пропустит результат. Это Self-Refine/CRITIC в формулировке Anthropic. Ключевая инженерная деталь, которую добавляет Anthropic: промпты эвалюатора и оптимизатора должны быть существенно различными, чтобы модель не просто ставила штамп одобрения.

### Ограничители выходных данных (output guardrails) OpenAI Agents SDK

OpenAI Agents SDK поставляет этот паттерн как «ограничители выходных данных» (output guardrails). Ограничитель (guardrail) — это валидатор, который запускается на финальном выходном агента. Если сработает ограничитель (выбрасывается `OutputGuardrailTripwireTriggered`), выходные данные отвергаются, и агент может повторить попытку. Ограничители могут вызывать инструменты (в стиле CRITIC) или быть чистыми функциями (в стиле Self-Refine).

### Типичные ошибки 2026 года

- **Циклы штамповки (rubber-stamp loops).** Одна и та же модель выполняет генерацию и критику с одинаковым стилем промптов — сходится к «выглядит хорошо». Используйте структурно различные промпты или более дешёвую маленькую модель для критики.
- **Чрезмерная доработка (over-refinement).** Каждый проход доработки добавляет задержку и токены (token). Заложите 1–3 прохода; после этого эскалируйте на ручную проверку.
- **CRITIC на тривиальных задачах.** Если нет внешнего верификатора, CRITIC деградирует до Self-Refine; не переплачивайте за задержку ради заглушки-верификатора.

## Собираем

`code/main.py` реализует Self-Refine и CRITIC на тривиальной задаче: составить короткий маркированный список по заданной теме. Верификатор проверяет формат (3 пункта, каждый не более 60 символов). CRITIC добавляет внешний «проверяющий факты», который штрафует известные галлюцинации (hallucinations).

Компоненты:

- `generate` — скриптовый генератор.
- `feedback` — самокритика в стиле LLM.
- `verify_external` — верификатор с опорой на инструменты в стиле CRITIC.
- `refine` — переписывает выходные данные с учётом истории.
- Условие остановки — верификатор прошёл проверку или максимум 4 итерации.

Запуск:

```
python3 code/main.py
```

Сравните результаты Self-Refine и CRITIC. CRITIC улавливает фактическую ошибку, которую Self-Refine пропустил, потому что внешний верификатор обладает опорой (grounding), которой нет у самокритика.

## Применение

Evaluator-optimizer от Anthropic — это тот же паттерн на языке, понятном Claude. Ограничители выходных данных (output guardrails) OpenAI Agents SDK имеют форму CRITIC (ограничители могут вызывать инструменты). LangGraph поставляет узел рефлексии (reflection), читающийся как Self-Refine. Google Gemini 2.5 Computer Use добавляет оценщик безопасности (safety evaluator) на каждом шаге — это вариант CRITIC: каждое действие проверяется перед фиксацией (commit).

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|--------|-------------|----------------------|
| Self-Refine | «LLM, которая исправляет сама себя» | Генерация → обратная связь → доработка в одной модели, с историей |
| CRITIC | «Верификация с опорой на инструменты» | Замена обратной связи внешним верификатором (поиск, код, калькулятор, тесты) |
| Evaluator-Optimizer | «Паттерн рабочего процесса Anthropic» | Две роли — эвалюатор оценивает, оптимизатор пересматривает — цикл до сходимости |
| Ограничитель выходных данных (output guardrail) | «Постфактум проверка» | Валидатор OpenAI Agents SDK, запускаемый после того, как агент выдал результат |
| Шаг проверки (verify step) | «Фаза критики» | Ключевое решение: с опорой на инструменты или с самооценкой |
| История доработки (refine history) | «То, что модель уже пробовала» | Предыдущие выходы + рецензии, добавляемые в начало промпта доработки; убираем — качество рушится |
| Цикл штамповки (rubber-stamp loop) | «Ошибка самосогласия» | Критика в том же стиле возвращает «выглядит хорошо»; исправляется структурно различными промптами |
| Условие остановки (stop condition) | «Критерий сходимости» | Верификатор прошёл проверку ИЛИ нет обратной связи И ограничение итераций; никогда не одно условие |
