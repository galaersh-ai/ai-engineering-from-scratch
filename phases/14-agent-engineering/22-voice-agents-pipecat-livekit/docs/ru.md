# Голосовые агенты (Voice Agents): Pipecat и LiveKit

> Голосовые агенты (voice agents) — полноценная продакшн-категория в 2026 году. Pipecat предоставляет фреймворк (framework) на Python для построения конвейеров (pipeline) на основе фреймов (frames): VAD → STT → LLM → TTS → транспорт. LiveKit Agents связывает модели ИИ с пользователями через WebRTC. Целевые показатели задержки (latency) в продакшне составляют 450–600 мс от начала до конца для оптимизированных стеков.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Фаза 14 · 01 (Цикл агента / Agent Loop), Фаза 14 · 12 (Шаблоны воркфлоу / Workflow Patterns)
**Время:** ~60 минут

## Цели обучения

- Описать фреймовый конвейер Pipecat: DOWNSTREAM (от источника к получателю) и UPSTREAM (управление).
- Перечислить стандартные этапы голосового конвейера и поддерживаемые Pipecat транспорты.
- Объяснить два класса голосовых агентов в LiveKit Agents (MultimodalAgent, VoicePipelineAgent) и случаи их применения.
- Обобщить ожидаемые показатели задержки в продакшне в 2026 году и их влияние на выбор архитектуры.

## Проблема

Голосовые агенты — это не текстовый цикл с подключённым синтезом речи (TTS). Бюджеты задержки очень жёсткие (~600 мс), частичное аудио — стандарт по умолчанию, определение конца хода речи (turn detection) реализуется моделью, а транспорты варьируются от телефонии SIP до WebRTC. Либо вы строите фреймовый конвейер (Pipecat), либо опираетесь на платформу (LiveKit).

## Концепция

### Pipecat (pipecat-ai/pipecat)

- Фреймворк для построения конвейеров на основе фреймов на Python.
- Цепочка `Frame` → `FrameProcessor`.
- Два направления потока данных:
  - **DOWNSTREAM** — от источника к получателю (source → sink): аудио на входе, синтез речи на выходе.
  - **UPSTREAM** — обратная связь и управление: отмена, метрики, перебивание (barge-in).
- `PipelineTask` управляет жизненным циклом с помощью событий (`on_pipeline_started`, `on_pipeline_finished`, `on_idle_timeout`) и наблюдателей (observers) для метрик/трассировки/RTVI.

Типичный конвейер:

```
VAD (Silero) → STT → LLM (контекст чередуется user/assistant) → TTS → транспорт
```

Транспорты: Daily, LiveKit, SmallWebRTCTransport, FastAPI WebSocket, WhatsApp.

Pipecat Flows добавляет структурированные диалоги (конечные автоматы). Pipecat Cloud — управляемая среда выполнения.

### LiveKit Agents (livekit/agents)

- Связывает модели ИИ с пользователями через WebRTC.
- Ключевые концепции: `Agent`, `AgentSession`, `entrypoint`, `AgentServer`.
- Два класса голосовых агентов:
  - **MultimodalAgent** — прямая передача аудио через OpenAI Realtime или аналог.
  - **VoicePipelineAgent** — каскад STT → LLM → TTS; обеспечивает контроль на уровне текста.
- Семантическое определение конца хода речи (semantic turn detection) с использованием модели трансформера (transformer).
- Нативная интеграция с MCP.
- Телефония через SIP.
- Более 50 моделей без ключей API (API keys) через LiveKit Inference; более 200 дополнительных — через плагины.

### Коммерческие платформы

Vapi (~450–600 мс на оптимизированном премиальном стеке) и Retell (~600 мс от начала до конца на 180 тестовых вызовах) построены поверх этих решений. Выбирайте платформу, когда вам нужен управляемый голосовой стек без команды WebRTC-специалистов.

### Типичные ошибки в данном подходе

- **Нет обработки перебивания (barge-in).** Пользователь перебивает; агент продолжает говорить. В Pipecat требуются фреймы отмены UPSTREAM, в LiveKit — аналогичный механизм.
- **Игнорирование достоверности STT.** Транскрипции с низкой достоверностью передаются в LLM как абсолютно надёжные. Необходим фильтр по порогу достоверности или запрос подтверждения.
- **Обрезка синтеза речи (TTS) на полуслове.** Когда конвейер отменяет текущее высказывание, TTS должен об этом знать или обрезать аудио.
- **Игнорирование бюджета задержки.** Каждый компонент добавляет 50–200 мс. Просуммируйте все этапы перед запуском.

### Типичные показатели задержки в 2026 году

- VAD: 20–60 мс
- STT (частичный результат): 100–250 мс
- LLM (первый токен / token): 150–400 мс
- TTS (первый фрагмент аудио): 100–200 мс
- Транспорт RTT: 30–80 мс

От начала до конца 450–600 мс — премиальный уровень. 800–1200 мс — типичный. Всё, что превышает 1500 мс, ощущается как неработающее.

## Реализация

`code/main.py` — это фреймовый тестовый конвейер (toy pipeline) с:

- Типами фреймов: аудио, транскрипция, текст, tts_audio, управление.
- Интерфейсом `Processor` с методом `process(frame)`.
- Пятиэтапным конвейером (VAD → STT → LLM → TTS → транспорт) в виде запрограммированных процессоров.
- Фреймом отмены UPSTREAM для демонстрации перебивания (barge-in).

Запуск:

```
python3 code/main.py
```

Трассировка показывает нормальный поток и перебивание (barge-in), которое останавливает синтез речи посреди высказывания.

## Применение

- **Pipecat** — для полного контроля: пользовательские процессоры, Python-ориентированный подход, подключаемые провайдеры.
- **LiveKit Agents** — для развёртывания (deployment) с приоритетом WebRTC и телефонии.
- **Vapi / Retell** — для хостинга голосовых агентов без команды WebRTC-специалистов.
- **OpenAI Realtime / Gemini Live** — для прямой передачи аудио на вход и выход (MultimodalAgent).

## Ключевые термины

| Термин | Что обычно говорят | Что это на самом деле |
|--------|-------------------|----------------------|
| Frame | «Событие» | Типизированная единица данных в конвейере (аудио, транскрипция, текст, управление) |
| Processor | «Этап конвейера» | Обработчик с методом process(frame) |
| DOWNSTREAM | «Прямой поток» | От источника к получателю: аудио на входе, речь на выходе |
| UPSTREAM | «Обратный поток» | Управление: отмена, метрики, перебивание (barge-in) |
| VAD | «Детекция голосовой активности» | Определяет, говорит ли пользователь |
| Semantic turn detection | «Умное окончание хода речи» | Решение модели о том, что пользователь закончил говорить |
| MultimodalAgent | «Прямой аудио-агент» | Аудио на входе, аудио на выходе; без промежуточного текста |
| VoicePipelineAgent | «Каскадный агент» | STT + LLM + TTS; контроль на уровне текста |
