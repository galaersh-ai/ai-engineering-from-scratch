# Паттерны оркестрации: супервайзер, рой, иерархия

> Четыре паттерна оркестрации повторяются в фреймворках 2026 года: супервайзер-работник, рой / пиринг, иерархия, дебаты. Руководство Anthropic: «Речь о построении правильной системы для ваших нужд». Начинайте просто; добавляйте топологию только когда один агент плюс пять паттернов воркфлоу недостаточны.

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 12 (Паттерны воркфлоу), Фаза 14 · 25 (Мультиагентные дебаты)
**Время:** ~60 минут

## Цели обучения

- Назвать четыре повторяющихся паттерна оркестрации и когда каждый подходит.
- Описать рекомендацию LangChain 2026: супервайзинг через вызовы инструментов vs библиотеки супервайзеров.
- Объяснить правило Anthropic «постройте правильную систему» и как оно ограничивает выбор топологии.
- Реализовать все четыре на stdlib против общего сценарного LLM.

## Проблема

Команды хватаются за «мультиагентный подход» до того, как он нужен. Четыре паттерна повторяются в фреймворках; когда вы можете их назвать, вы можете выбрать правильный — или отказаться от топологии вообще.

## Концепция

### Супервайзер-работник

- Центральный маршрутизирующий LLM направляет к специализированным агентам.
- Решает: вернуться к себе, передать специалисту, завершить.
- Специалисты не общаются друг с другом; вся маршрутизация проходит через супервайзера.

Фреймворки: LangGraph `create_supervisor`, Anthropic orchestrator-workers, CrewAI Hierarchical Process.

**Рекомендация LangChain 2026:** реализуйте супервайзинг через прямые вызовы инструментов, а не через `create_supervisor`. Даёт более точный контроль инженерии контекста — вы решаете, что именно видит каждый специалист.

### Рой / пиринг

- Агенты передают друг другу напрямую через разделяемую поверхность инструментов.
- Нет центрального маршрутизатора.
- Меньше латентность, чем у супервайзера (меньше хопов).
- Сложнее рассуждать (нет единой точки контроля).

Фреймворки: топология swarm в LangGraph, передачи OpenAI Agents SDK (когда все агенты могут передавать друг другу).

### Иерархия

- Супервайзеры управляют подсупервайзерами, которые управляют работниками.
- Реализуется как вложенные подграфы в LangGraph; вложенные экипажи в CrewAI.
- Масштабируется до больших популяций агентов ценой операционной сложности.

Когда нужно: когда контекстный бюджет одного супервайзера не вмещает описания всех специалистов.

### Дебаты

- Параллельные提案щики + итеративная перекрёстная критика (Урок 25).
- Не совсем оркестрация — скорее верификация — но появляется как выбор топологии во фреймворках.

### CrewAI Crew vs Flow

CrewAI формализует два режима развёртывания:

- **Flow** для детерминистического event-driven автоматизации (рекомендуемая отправная точка для продакшена).
- **Crew** для автономного ролевого сотрудничества.

Это ортогонально четырём паттернам выше, но отображается на топологию: Flow обычно супервайзер или иерархия; Crew обычно супервайзер с LLM-маршрутизатором.

### Руководство Anthropic

«Успех в пространстве LLM — не в построении самой сложной системы. Речь о построении правильной системы для ваших нужд.»

Порядок решений:

1. Один агент + паттерны воркфлоу (Урок 12) — начните здесь.
2. Супервайзер-работник — когда у вас 2-4 специалиста.
3. Рой — когда латентность важнее ясности рассуждений.
4. Иерархия — только когда контекстный бюджет супервайзера не справляется.
5. Дебаты — когда точность важнее стоимости.

### Где этот паттерн ломается

- **Топология-сначала мышление.** «Нам нужен мультиагентный подход» до определения какую задачу он решает.
- **Прыгающие передачи в рое.** A -> B -> A -> B. Используйте счётчики хопов.
- **Фиктивная иерархия.** Три слоя потому что «enterprise»; два реальных команды. Схлопните.

## Реализуем

`code/main.py` реализует все четыре паттерна на stdlib против сценарного LLM:

- `Supervisor` — центральный маршрутизатор.
- `Swarm` — пиринг с прямыми передачами.
- `Hierarchical` — супервайзеры супервайзеров.
- `Debate` — параллельные提案щики + критика.

Каждый паттерн обрабатывает одну и ту же задачу с тремя интентами (возврат / баг / продажи). Формы трассировки различаются.

Запуск:

```
python3 code/main.py
```

Вывод: трассировка на паттерн + количество операций. Супервайзер самый чистый; рой самый короткий; иерархия самая глубокая; дебаты самые дорогие.

## Применение

- **LangGraph** для супервайзера и иерархии (вложенные подграфы).
- **OpenAI Agents SDK** для передач-как-инструментов (в стиле супервайзера).
- **CrewAI Flow** для продакшен детерминистического.
- **Кастомное** для дебатов или когда нужен точный контроль.

## Упражнения

1. Конвертируйте супервайзера-работника в рой, удалив маршрутизатор. Что ломается? Что улучшается?
2. Добавьте счётчик хопов в рой: отказ после 3 передач. Ловит ли он прыжки A->B->A?
3. Постройте двухуровневую иерархическую систему для домена из 12 специалистов. Где ломается контекстный бюджет без вложенности?
4. Профилируйте четыре паттерна на продакшен workload. Какой побеждает по какой метрике (латентность, стоимость, точность, отладка)?
5. Прочитайте пост Anthropic «Building Effective Agents». Отобразите каждый из ваших продакшен-воркфлоу на один из четырёх. Есть ли те, которые не отображаются чисто?

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|----------------|------------------------|
| Супервайзер-работник | «Маршрутизатор + специалисты» | Центральный LLM направляет к специалистам; они не общаются друг с другом |
| Рой | «Пиринг» | Прямые передачи через разделяемые инструменты; нет центрального маршрутизатора |
| Иерархия | «Супервайзеры супервайзеров» | Вложенные подграфы для больших популяций |
| Дебаты | «Предлагающий + критик» | Параллельные提案щики, перекрёстная критика (Урок 25) |
| Супервайзинг через вызов инструментов | «Супервайзер без библиотеки» | Реализация супервайзера как прямых вызовов инструментов для контроля контекста |
| Экипаж | «Автономная команда» | Ролевое сотрудничество CrewAI |
| Флоу | «Детерминистический воркфлоу» | Event-driven продакшен-режим CrewAI |
