# Скрипты инициализации для агентов (Initialization Scripts for Agents)

> Каждая сессия, начинающаяся «с нуля», платит налог (setup tax). Агент читает одни и те же файлы, повторяет одни и те же проверки и заново открывает одни и те же пути. Скрипт инициализации платит этот налог один раз и записывает ответы в состояние (state).

**Тип:** Сборка (Build)
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 32 (Минимальная рабочая среда), Фаза 14 · 34 (Память репозитория)
**Время:** ~45 минут

## Цели обучения

- Определить работу, которую агент никогда не должен выполнять повторно в каждой сессии.
- Построить детерминированный скрипт инициализации (init script), проверяющий среду выполнения (runtime), зависимости (dependencies) и состояние репозитория.
- Сохранять результат проверки (probe result) так, чтобы агент читал его вместо повторного запуска проверок.
- Громко сообщать об ошибках, быстро останавливаться и указывать одно место для поиска при сбое инициализации.

## Проблема

Откройте сессию. Агент угадывает версию Python. Угадывает команду тестирования. Пять раз перечисляет корень репозитория, чтобы найти точку входа. Пытается импортировать пакет, который не установлен. Спрашивает пользователя, где находится файл конфигурации. К моменту, когда он вносит реальное изменение, десятки тысяч токенов (tokens) ушли на работу по настройке, которая должна была быть одним скриптом.

Решение — один скрипт инициализации, который запускается до того, как агент делает что-либо ещё, и записывает `init_report.json`, который агент читает при запуске.

## Концепция

```mermaid
flowchart TD
  Start[Начало сессии] --> Init[init_agent.py]
  Init --> Probes[Проверка среды / зависимостей / путей / окружения / тестов]
  Probes --> Report[init_report.json]
  Report --> Decision{Здоров?}
  Decision -- да --> Agent[Цикл агента (Agent Loop)]
  Decision -- нет --> Halt[Громкий отказ, остановка, уведомление человека]
```

### Что проверяет скрипт инициализации

| Проверка (Probe) | Почему это важно |
|-------|----------------|
| Версии среды выполнения | Неправильная версия Python или Node приводит к скрытым ошибкам из-за несовместимости версий |
| Доступность зависимостей | Недостающий пакет обойдётся потом в десять раз дороже, чем обнаружить его сейчас |
| Команда тестирования | Агент должен знать, как проверять; если команда отсутствует, рабочая среда (workbench) сломана |
| Пути репозитория | Захардкоженные пути устаревают; определите их один раз и зафиксируйте |
| Переменные окружения | Отсутствующий `OPENAI_API_KEY` — это проблема на поверхности, а не загадка времени выполнения |
| Свежесть состояния и доски | Устаревшее состояние (state) из упавшей сессии — ловушка |
| Последний известный хороший коммит | Якорь для дифа передачи (handoff) в конце сессии |

### Громкий отказ, быстрый отказ, отказ в одном месте

Сбой проверки (probe failure) означает остановку и уведомление человека. Никаких «агент разберётся сам». Суть инициализации в том, чтобы отказываться запускаться, когда рабочая среда сломана.

### Идемпотентность (Idempotent)

Запустите его дважды подряд. Второй запуск должен быть ничего не делающим, за исключением обновления временной метки. Идемпотентность (idempotency) — это то, что позволяет встроить скрипт в CI, хуки (hooks) или предварительную slash-команду задачи.

### Инициализация и правила запуска

Правила (Phase 14 · 33) описывают условия, которые должны быть выполнены для действия. Инициализация — это скрипт, устанавливающий, что эти правила можно проверить. Правила без инициализации превращаются в «будьте осторожны». Инициализация без правил превращается в отполированный отказ.

## Сборка

`code/main.py` реализует `init_agent.py`:

- Пять проверок: версия Python, перечисленные зависимости через `importlib.util.find_spec`, доступность команды тестирования, требуемые переменные окружения, свежесть файла состояния.
- Каждая проверка возвращает `(name, status, detail)`.
- Скрипт записывает `init_report.json` с полным набором проверок и завершается с кодом отличным от нуля, если любая проверка критической важности завершилась неудачей.

Запуск:

```
python3 code/main.py
```

Скрипт выводит таблицу проверок, записывает `init_report.json` и завершается с кодом нуля при успешном сценарии или с кодом отличным от нуля и списком неудачных проверок.

## Продукционные паттерны в реальном мире

Три паттерна отличают полезный скрипт инициализации от церемонии.

**Якорение по последнему известному хорошему коммиту (last-known-good commit anchoring).** Проверяйте текущий коммит относительно файла `LKG`, записанного при последнем успешном слиянии. Если диф превышает допустимый бюджет (по умолчанию 50 файлов), отказывайтесь запускаться и требуйте от человека подтверждения новой базовой линии. Именно так работает Cloudflare AI Code Review для ограничения области агентов-ревьюеров: каждая сессия ревью привязывается к одному и тому же последнему известному хорошему коммиту и никогда не накапливает дрейф (drift) между сессиями.

**Файлы блокировки с TTL.** Записывайте `prereqs.lock` после первого успешного прохождения проверок. Последующие запуски доверяют файлу блокировки в течение N часов (по умолчанию 24 часа) и пропускают дорогие проверки. Скрипт инициализации сначала читает файл блокировки; если он свежий и хеш манифеста зависимостей совпадает, проверка пропускается. Это тот же паттерн, который Docker использует для кэширования слоёв: идемпотентная проверка + хеш содержимого = пропуск.

**Нет сети, нет LLM, нет сюрпризов в горячем пути.** Проверки инициализации — это детерминированный plumbing. Проверка, которая вызывает LLM для классификации ошибки или обращается к внешнему сервису для проверки лицензии, — это не проверка; это воркфлоу (workflow). Если проверка длится более трёх секунд при пробном запуске, воспринимайте это как признак проблемы рабочей среды и либо выносите проверку за пределы инициализации, либо кэшируйте её результат.

## Применение

В продакшене:

- **Хуки Claude Code (Claude Code hooks).** Хук `pre-task` вызывает скрипт инициализации и отказывается запускать агента при его сбое.
- **GitHub Actions.** Задача `setup-agent` запускает скрипт инициализации; задача агента зависит от неё.
- **Точка входа Docker (Docker entrypoint).** Контейнер агента запускает скрипт инициализации перед exec рантайма агента; при сбое логи выводятся наружу.

Скрипт инициализации переносим, потому что не делает вызовов к конкретному фреймворку (framework). Bash, Make или файл задач (tasks file) могут его обернуть.

## Отправка

`outputs/skill-init-script.md` опрашивает проект, классифицирует его работу по настройке в проверки и генерирует проект-специфичный `init_agent.py` плюс CI-воркфлоу, запускающий его перед любым шагом агента.

## Упражнения

1. Добавьте проверку (probe), которая сравнивает текущий коммит с последним известным хорошим коммитом и отказывается запускаться, если изменено более 50 файлов.
2. Настройте скрипт для записи файла `prereqs.lock` и отказа в запуске, если файл блокировки старше семи дней.
3. Добавьте флаг `--fix`, который автоматически устанавливает недостающие зависимости для разработки, но никогда не изменяет зависимости среды выполнения без согласования.
4. Перенесите проверки из захардкоженных функций в YAML-реестр. Обоснуйте компромисс.
5. Добавьте временной бюджет на каждую проверку. Проверка, которая выполняется дольше трёх секунд, — признак проблемы рабочей среды.

## Ключевые термины

| Термин | Что говорят люди | Что это на самом деле |
|------|----------------|------------------------|
| Проверка (Probe) | «Проверка» | Детерминированная функция, возвращающая `(name, status, detail)` |
| Отчёт инициализации (Init report) | «Результат настройки» | JSON, записанный рядом с состоянием, содержащий результаты проверок |
| Идемпотентность (Idempotent) | «Безопасно перезапускать» | Два запуска подряд дают идентичные отчёты за вычетом временной метки |
| Громкий отказ (Fail loud) | «Не проглатывай» | Остановка и уведомление человека; никаких тихих обходных путей |
| Налог на настройку (Setup tax) | «Стоимость загрузки» | Токены (tokens), которые агент тратит в каждой сессии на повторное открытие очевидного |
