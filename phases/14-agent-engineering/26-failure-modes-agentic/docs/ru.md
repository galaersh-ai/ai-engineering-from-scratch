# Режимы отказов (Failure Modes): почему агенты ломаются

> MASFT (Беркли, 2025) описывает 14 режимов отказов мультиагентных систем (multi-agent systems) в 3 категориях. Таксономия Microsoft документирует то, как существующие сбои ИИ усиливаются в агентных средах. Отраслевые данные сходятся на пяти повторяющихся режимах: галлюцинации действий (hallucinated actions), расширение области задачи (scope creep), каскадные ошибки (cascading errors), потеря контекста (context loss), неправильное использование инструментов (tool misuse).

**Тип:** Теория + Практика
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 05 (Self-Refine и CRITIC), Фаза 14 · 24 (Наблюдаемость)
**Время:** ~60 минут

## Цели обучения

- Назвать три категории отказов MASFT и не менее четырёх конкретных режимов в каждой.
- Объяснить, почему агентные режимы отказов усиливают существующие сбои ИИ (смещение, галлюцинация).
- Описать пять повторяющихся отраслевых режимов и способы их смягчения.
- Реализовать детектор на stdlib, который размечает следы агентов (agent traces) метками режимов отказов.

## Проблема

Команды выпускают агентов, которые работают на 90% следов. Оставшиеся 10% отказов — не случайный шум: они подпадают под небольшое число повторяющихся категорий. Как только их можно назвать, за ними можно следить и исправлять их.

## Концепция

### MASFT (Беркли, arXiv:2503.13657)

Таксономия отказов мультиагентных систем (Multi-Agent System Failure Taxonomy). 14 режимов отказов, объединённых в 3 категории. Согласованность аннотаторов (Cohen's Kappa) — 0.88: категории надёжно различимы.

Ключевой тезис: отказы — это фундаментальные недостатки проектирования мультиагентных систем, а не ограничения LLM (LLM), которые можно исправить более совершенной базовой моделью.

### Таксономия режимов отказов в агентных системах ИИ от Microsoft

- Существующие сбои ИИ (смещение, галлюцинация, утечка данных) усиливаются в агентных средах.
- Новые сбои порождаются автономностью (autonomy): непреднамеренные действия в массовом масштабе, неправильное использование инструментов, отклонение от миссии (mission drift).
- Белая книга является реестром рисков для агентных продуктов.

### Характеристика сбоев в агентном ИИ (arXiv:2603.06847)

- Сбои возникают из-за оркестрации (orchestration), эволюции внутреннего состояния и взаимодействия с окружением.
- Это не просто «плохой код» или «плохой вывод модели».

### Обзор галлюцинаций LLM-агентов (arXiv:2509.18970)

Два основных проявления:

1. **Отклонение от следования инструкциям (Instruction-following Deviation)** — агент не следует системному промпту.
2. **Долгосрочное неправильное использование контекста (Long-range Contextual Misuse)** — агент забывает или неверно применяет контекст из предыдущих ходов.

Ошибки поднамерения: пропуск (omission) — пропущенный шаг; избыточность (redundancy) — повторный шаг; беспорядок (disorder) — шаги не в том порядке.

### Пять повторяющихся отраслевых режимов

Анализы отраслевых данных Arize, Galileo, NimbleBrain за 2024–2026 годы сходятся на следующем:

1. **Галлюцинации действий.** Агент вызывает инструмент, которого не существует, или выдумывает аргументы.
2. **Расширение области задачи (Scope creep).** Агент расширяет задачу за пределы запроса пользователя (создаёт дополнительные PR, отправляет лишние письма).
3. **Каскадные ошибки (Cascading errors).** Одна неверная операция запускает цепочку последствий. Галлюцинация несуществующего SKU порождает четыре вызова API — инцидент, затрагивающий несколько систем.
4. **Потеря контекста (Context loss).** При длительных задачах агент забывает ограничения начальных ходов.
5. **Неправильное использование инструментов (Tool misuse).** Вызов правильного инструмента с неверными аргументами или совсем неверного инструмента.

Каскадность — главная опасность. Агенты не могут различить «я не смог» и «задача невыполнима» и зачастую галлюцинируют сообщение об успехе при ошибках 400, чтобы замкнуть цикл.

### Смягчение: контрольные точки на каждом шаге

Автоматизированные контрольные ворота на каждом шаге цепочки рассуждений (chain-of-thought), проверяющие фактологическое обоснование относительно состояния окружения. Конкретно:

- Классификатор безопасности для каждого шага (Урок 21).
- Валидация аргументов вызова инструмента (Урок 06).
- Перекрёстная проверка извлечённого содержимого по сравнению с известными фактами (Урок 05, CRITIC).
- Обнаружение галлюцинации успеха путём повторного опроса состояния (был ли файл действительно создан?).

### Где мониторинг отказов даёт сбой

- **Разметка только аварий.** Большинство отказов агентов выглядят как корректный вывод. Необходима проверка на уровне содержимого.
- **Отсутствие базовой линии.** Обнаружение дрейфа (drift) требует последнего известного работоспособного состояния; без него невозможно утверждать, что ситуация ухудшается.
- **Избыточные оповещения.** Каждый отказ порождает уведомление. Необходима группировка и ограничение частоты оповещений.

## Реализация

`code/main.py` реализует теггер (tagger) режимов отказов на stdlib:

- Синтетический набор данных следов, охватывающий пять режимов.
- Функции-детекторы для каждого режима (сигнатурные паттерны на вызовах инструментов, выводах, повторяющихся действиях).
- Теггер, который размечает каждый след и выводит распределение режимов.

Запуск:

```
python3 code/main.py
```

Вывод: метки по следам + агрегированное распределение — недорогая реплика того, что кластеризация следов в Phoenix показывает визуально.

## Применение

- **Phoenix** для кластеризации дрейфа в продакшене (Урок 24).
- **Langfuse** для воспроизведения сессий (session replay) и аннотирования.
- **Кастомные решения** для доменно-специфических паттернов, которые ваша платформа наблюдаемости (observability) не может обнаружить.

## Выпуск

`outputs/skill-failure-detector.md` генерирует детекторы режимов отказов, адаптированные к вашей предметной области, подключённые к хранилищу следов.

## Упражнения

1. Добавить детектор «галлюцинации успеха»: агент возвращает успех, но целевое состояние не изменилось.
2. Разметить 100 реальных следов из продукта, который вы разрабатываете. Какой режим доминирует? Какова стоимость его исправления?
3. Реализовать метрику «радиус каскада»: при сбое на шаге N — сколько последующих шагов было затронуто?
4. Прочитать 14 режимов отказов MASFT. Выбрать три, актуальных для вашего продукта. Написать детекторы.
5. Подключить один детектор в CI-задачу: завершить сборку неуспешно, если ≥5% следов помечены как определённый режим.

## Ключевые термины

| Термин | Как обычно говорят | Что это на самом деле |
|--------|-------------------|----------------------|
| MASFT | «Таксономия отказов мультиагентных систем» | Категоризация Беркли с 14 режимами |
| Каскадная ошибка | «Провал с каскадным эффектом» | Одна ранняя ошибка проходит через N шагов |
| Потеря контекста | «Забыл ограничение» | Длительный ход теряет факты из начальных ходов |
| Неправильное использование инструментов | «Неверный инструмент / неверные аргументы» | Корректный вызов, неверная инвокация |
| Галлюцинация успеха | «Фиктивное завершение» | Агент заявляет об успехе при ошибке 400; состояние не изменено |
| Расширение области задачи | «Выход за рамки» | Агент делает больше, чем было запрошено |
| Отклонение от следования инструкциям | «Непослушание» | Игнорирует системный промпт или ограничение пользователя |
| Ошибки поднамерения | «Ошибки плана» | Пропуск, избыточность, беспорядок в исполнении плана |
