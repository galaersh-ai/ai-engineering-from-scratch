# AI-шлюзы (AI Gateways) — LiteLLM, Portkey, Kong AI Gateway, Bifrost

> Шлюз (gateway) находится между вашими приложениями и провайдерами моделей. Основные функции: маршрутизация провайдеров (provider routing), переключение при ошибках (fallback), повторные попытки (retries), ограничение частоты запросов (rate limiting), управление секретами, наблюдаемость (observability) и контрольные ограничения (guardrails). Распределение рынка в 2026 году: **LiteLLM** — проект с лицензией MIT, поддерживающий более 100 провайдеров, совместимый с OpenAI, но теряющий работоспособность при примерно 2000 RPS (потребление памяти 8 ГБ, каскадные сбои в опубликованных бенчмарках); лучше всего подходит для Python, менее 500 RPS, разработки и прототипирования. **Portkey** позиционируется как панель управления (guardrails, маскирование PII, обнаружение джейбрейков, журналы аудита), с марта 2026 года стал проектом с лицензией Apache 2.0, накладные задержки 20–40 мс, тариф Production за 49 долларов в месяц. **Kong AI Gateway** построен на базе Kong Gateway — собственный бенчмарк Kong на тех же 12 CPU: на 228% быстрее Portkey, на 859% быстрее LiteLLM; тарифная модель 100 долларов за модель в месяц (максимум 5 моделей на тарифе Plus); подходит для предприятий, уже использующих Kong. **Bifrost** (Maxim AI) — автоматические повторные попытки с настраиваемым откатом (backoff), переключение на Anthropic при ошибке 429 со стороны OpenAI. **Cloudflare / Vercel AI Gateways** — управляемые решения, нулевое администрирование, базовые повторные попытки. Выбор между самостоятельным размещением и облачным решением определяется требованиями к местоположению данных; Portkey и Kong занимают среднюю позицию с комбинацией открытого исходного кода и управляемой опции.

**Тип:** Обучающий модуль
**Языки:** Python (стандартная библиотека, имитатор маршрутизации шлюза)
**Предварительные знания:** Фаза 17 · 01 (Управляемые платформы LLM), Фаза 17 · 16 (Маршрутизация моделей)
**Время:** ~60 минут

## Цели обучения

- Перечислить шесть основных функций шлюза (маршрутизация, переключение при ошибках, повторные попытки, ограничения частоты, секреты, наблюдаемость, контрольные ограничения).
- Сопоставить четыре шлюза 2026 года (LiteLLM, Portkey, Kong AI, Bifrost) с предельными показателями масштаба и вариантами использования.
- Привести данные бенчмарка Kong (на 228% быстрее Portkey, на 859% быстрее LiteLLM) и объяснить их значение при нагрузке более 500 RPS.
- Выбрать между самостоятельным размещением и управляемым решением с учётом требований к местоположению данных и бюджета на эксплуатацию.

## Проблема

Ваш продукт использует OpenAI, Anthropic и самостоятельно развёрнутую модель Llama. У каждого провайдера собственный SDK, модель ошибок, ограничение частоты запросов и схема аутентификации. Вам необходим переключатель при ошибках (если OpenAI возвращает 429, попробовать Anthropic), единое хранилище учётных данных, унифицированная наблюдаемость и ограничения частоты запросов для каждого клиента.

Повторная реализация этой логики на уровне приложений приводит к сильному связанному коду между каждым сервисом и каждым провайдером. Уровень шлюза консолидирует всё в один процесс с единым API (обычно совместимым с OpenAI), который распределяет запросы по провайдерам.

## Концепция

### Шесть основных функций

1. **Маршрутизация провайдеров (Provider routing)** — OpenAI, Anthropic, Gemini, самостоятельно развёрнутые модели и другие за единым API.
2. **Переключение при ошибках (Fallback)** — при ошибке 429, 5xx или ошибке качества — повторная попытка у другого провайдера.
3. **Повторные попытки (Retries)** — экспоненциальный откат (exponential backoff), ограниченное количество попыток.
4. **Ограничения частоты (Rate limits)** — по клиенту, по ключу, по модели.
5. **Ссылки на секреты (Secret references)** — получение учётных данных из хранилища секретов (vault) во время выполнения (никогда не в приложении).
6. **Наблюдаемость (Observability)** — OpenTelemetry + атрибуты GenAI (Фаза 17 · 13) + отнесение затрат.
7. **Контрольные ограничения (Guardrails)** — маскирование PII, обнаружение джейбрейков (jailbreak detection), фильтры допустимых тем.

### LiteLLM — проект с лицензией MIT, Python

- Более 100 провайдеров, совместимость с OpenAI, конфигурация маршрутизатора, переключение при ошибках, базовая наблюдаемость.
- Теряет работоспособность при примерно 2000 RPS в бенчмарке Kong; потребление памяти 8 ГБ, каскадные сбои при постоянной нагрузке.
- Лучше всего подходит для: Python-приложений, менее 500 RPS, шлюзов для разработки и тестирования, экспериментальной маршрутизации.
- Стоимость: $0 для версии с открытым исходным кодом; доступен бесплатный облачный тариф.

### Portkey — позиционирование как панель управления

- Проект с лицензией Apache 2.0 с марта 2026 года. Контрольные ограничения, маскирование PII, обнаружение джейбрейков, журналы аудита.
- Накладные задержки 20–40 мс на каждый запрос.
- Тариф Production за 49 долларов в месяц с хранением данных и SLA.
- Лучше всего подходит для: регулируемых отраслей, которым необходимы контрольные ограничения и наблюдаемость в одном решении.

### Kong AI Gateway — решение для масштаба

- Построен на базе Kong Gateway (зрелый продукт API-шлюза, Lua + OpenResty).
- Собственный бенчмарк Kong на эквиваленте 12 CPU: на 228% быстрее Portkey, на 859% быстрее LiteLLM.
- Тарифная модель: 100 долларов за модель в месяц, максимум 5 моделей на тарифе Plus.
- Лучше всего подходит для: проектов уже использующих Kong; более 1000 RPS; готовность к лицензированию.

### Bifrost (Maxim AI)

- Автоматические повторные попытки с настраиваемым откатом (backoff).
- Переключение на Anthropic при ошибке 429 со стороны OpenAI — стандартный шаблон использования.
- Относительно новый участник рынка; коммерческий продукт.

### Cloudflare AI Gateway / Vercel AI Gateway

- Управляемое решение, нулевое администрирование. Базовые повторные попытки и наблюдаемость.
- Лучше всего подходит для: JavaScript-приложений, работающих на краю сети (edge) через Cloudflare/Vercel.
- Ограниченные возможности по сравнению с Kong/Portkey в отношении контрольных ограничений и управления частотой запросов.

### Самостоятельное развёртывание vs управляемое решение

Требования к местоположению данных (data residency) являются решающим фактором. Здравоохранение и финансы по умолчанию выбирают самостоятельное развёртывание (LiteLLM или Portkey с открытым исходным кодом или Kong). Потребительские продукты по умолчанию выбирают управляемое решение (Cloudflare AI Gateway) или средний вариант (Portkey managed). Гибридный подход: самостоятельное развёртывание для регулируемых клиентов, управляемое решение для остальных.

### Бюджет задержек

- LiteLLM: типичные накладные задержки 5–15 мс.
- Portkey: накладные задержки 20–40 мс.
- Kong: накладные задержки 3–8 мс.
- Cloudflare/Vercel: накладные задержки 1–3 мс (преимущество edge-расположения).

Задержки шлюза напрямую прибавляются ко времени до первого токена (TTFT). Для TTFT P99 < 100 мс в рамках SLA — Kong или Cloudflare. Для P99 < 500 мс — любое решение.

### Семантика ограничения частоты имеет значение

Простая модель с токен-бакетом (token-bucket) работает при умеренных нагрузках. Многоклиентские решения требуют скользящего окна (sliding window) с допуском всплесков (burst allowance) и дифференциацией по клиентам. LiteLLM использует токен-бакет; Kong использует скользящее окно; Portkey использует дифференциацию.

### Шлюз + наблюдаемость + маршрутизация: комплексная архитектура

Фаза 17 · 13 (наблюдаемость) + 16 (маршрутизация моделей) + 19 (шлюзы) в производстве представляют один слой. Выберите один инструмент, покрывающий все три функции, или тщательно объедините их: большинство развертываний в 2026 году комбинируют Helicone (наблюдаемость) или Portkey (контрольные ограничения) с Kong (масштаб) для разделения ролей.

### Цифры, которые стоит запомнить

- LiteLLM: теряет работоспособность при ~2000 RPS, 8 ГБ памяти.
- Portkey: накладные задержки 20–40 мс; Apache 2.0 с марта 2026 года.
- Kong: на 228% быстрее Portkey, на 859% быстрее LiteLLM.
- Тариф Kong: 100 долларов за модель в месяц, максимум 5 моделей на тарифе Plus.
- Cloudflare/Vercel: накладные задержки 1–3 мс на краю сети.

## Практическое применение

Файл `code/main.py` имитирует маршрутизацию шлюза с переключением при ошибках между тремя провайдерами при инъекции ошибок 429/5xx. Выводит статистику задержек, частоты повторных попыток и частоты срабатывания переключений.

## Готово к запуску

В результате выполнения этого урока формируется файл `outputs/skill-gateway-picker.md`. На основе показателей масштаба, эксплуатационных требований, соответствия нормативам и бюджета задерьки выбирает оптимальный шлюз.

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|--------|-------------|----------------------|
| Шлюз (Gateway) | «API-брокер» | Процесс, находящийся между приложениями и провайдерами |
| LiteLLM | «тот, что на MIT» | Проект на Python с открытым исходным кодом, более 100 провайдеров, теряет работоспособность при 2000 RPS |
| Portkey | «шлюз с контрольными ограничениями» | Панель управления + наблюдаемость, Apache 2.0 |
| Kong AI Gateway | «то, что масштабируется» | Построен на Kong Gateway, лидер бенчмарков |
| Bifrost | «шлюз от Maxim» | Повторные попытки + шаблон переключения на Anthropic |
| Cloudflare AI Gateway | «управляемый на краю сети» | Управляемый шлюз на краю сети, нулевое администрирование |
| Маскирование PII | «очистка данных» | Маскирование с помощью регулярных выражений и NER перед отправкой в модель |
| Обнаружение джейбрейков | «защита от инъекции промптов» | Классификатор для проверки пользовательского ввода |
| Журнал аудита | «нормативный журнал» | Неизменяемая запись каждого вызова LLM |
| Токен-бакет (Token-bucket) | «простое ограничение частоты» | Ограничитель частоты на основе пополнения |
| Скользящее окно (Sliding-window) | «точное ограничение частоты» | Ограничитель частоты на основе временного окна; более справедливый |
