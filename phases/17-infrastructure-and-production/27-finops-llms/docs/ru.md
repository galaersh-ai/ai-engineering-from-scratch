# FinOps для LLM — Единичная экономика и атрибуция для мультитенантных систем

> Классический FinOps ломается на расходах на LLM. Затраты — это токенные транзакции, а время работы ресурсов. Теги не сопоставляются — вызов API является транзакцией, а не ресурсом. Инженерные решения (дизайн промпта, контекстное окно, длина выхода) становятся финансовыми решениями. Практическое руководство 2026 года включает три измерения атрибуции (attribution), которые необходимо инструментировать с первого дня: по пользователю (`user_id`) для поseatовой (seat) ценообразования и расширения, по задаче (`task_id` + `route`) для стоимости поверхности продукта и приоритизации, по тенанту (`tenant_id`) для единичной экономики и обновления подписки. Четыре слоя токенов — промпт, инструменты, память, ответ — в одном контейнере скрывают расходы. Лесенка ограничений (enforcement ladder) для мультитенантных продуктов: лимит скорости на тенанта (2-3x от ожидаемого пика, возврат 429 + retry-after); дневной лимит расходов (1,5-3x от контрактного лимита; срабатывание: ужесточение лимита скорости + оповещение); аварийный выключатель (kill switch) при z-скоре расходов > 4 (автоприостановка + уведомление дежурного). Паттерны атрибуции: тегирование и агрегация, телеметрический соединитель (trace-ID → биллинг; максимальная точность), выборка и экстраполяция, модельная аллокация, событийная ориентированность (event-sourced), потоковая обработка в реальном времени. Единичная метрика: стоимость разрешённого запроса, стоимость сгенерированного артефакта — а не $/M токенов. Ретроспективное тегирование всегда упускает данные; инструментируйте при создании запроса.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, учебный симулятор атрибуции стоимости с аварийным выключателем)
**Предварительные требования:** Фаза 17 · 13 (Наблюдаемость), Фаза 17 · 14 (Кэширование)
**Время:** ~60 минут

## Цели обучения

- Объяснить, почему классический FinOps (теги + уровни) ломается на расходах на LLM, и назвать три новых измерения атрибуции.
- Перечислить четыре слоя токенов (промпт, инструменты, память, ответ) и объяснить, почему единый контейнер для биллинга скрывает стоимость.
- Спроектировать лесенку ограничений (скорость → лимит расходов → аварийный выключатель) для мультитенантного продукта.
- Выбрать единичную метрику (стоимость разрешённого запроса / артефакта) вместо $/M токенов.

## Проблема

Ваш счёт составляет $40 000. Вы не знаете:
- Какой тенант потратил эти средства.
- Какая функциональность продукта вызвала эти расходы.
- Был ли какой-либо отдельный пользователь накрутил расходы.
- Явились ли причиной раздутые промпты, вызовы инструментов или усиление памяти.

Тегирование и агрегация на стороне провайдера работает для облачных ресурсов (EC2, S3), где теги распространяются на строки детализации. Вызовы API LLM не тегируются автоматически — вам необходимо ставить пользовательскую/задачную/тенантную метку в месте вызова и передавать её далее. Ретроспективная атрибуция всегда упускает пограничные случаи.

## Концепция

### Три измерения атрибуции

**По пользователю** (`user_id`): кто и сколько стоит. Определяет поseatовое ценообразование, расширение (expansion), выявление активных пользователей.

**По задаче** (`task_id` + `route`): какая поверхность продукта стоит столько. Определяет приоритизацию функциональности, решения об отключении дорогих функций.

**По тенанту** (`tenant_id`): какой клиент является прибыльным. Определяет единичную экономику, ценообразование при обновлении подписки, пороги уровней.

Инструментируйте все три измерения в месте вызова с первого дня. Ретроспективный подход всегда хуже.

### Четыре слоя токенов

| Слой | Пример | Типичная доля от общего |
|-------|---------|---------------------|
| Промпт | системный + пользовательский ввод | 40-60% |
| Инструменты | результаты вызовов инструментов | 20-40% (агентные нагрузки) |
| Память | предыдущий диалог / извлечённые документы | 10-30% |
| Ответ | выход модели | 10-30% |

Объединение всех четырёх слоёв в один контейнер делает оптимизацию вслепую. Разделяйте их в вашей схеме атрибуции.

### Лесенка ограничений

1. **Лимит скорости** на тенанта. 2-3x от ожидаемого пика. Возврат 429 с `Retry-After`. Тенант видит ограничение; нет неожиданного счёта.

2. **Дневной лимит расходов** на тенанта. 1,5-3x от контрактного лимита. При срабатывании: ужесточение лимита скорости + оповещение клиентского сервиса.

3. **Аварийный выключатель** при z-скоре расходов > 4 относительно базовой линии тенанта. Автоприостановка тенанта; уведомление дежурного; эскалация в отдел эксплуатации и клиентский сервис.

### Паттерны атрибуции

- **Тегирование и агрегация**: нанесение метаданных в заголовки; агрегация позже. Простой; неточный.
- **Телеметрический соединитель**: соединение трассировок с биллингом через trace-ID. Максимальная точность. то, что делают зрелые команды.
- **Выборка + экстраполяция**: выборка 5-10%, умножение. Экономичная для грубых оценок расходов; пропускает хвосты.
- **Модельная аллокация**: регрессия для вывода источника затрат. Для унаследованных данных без тегов.
- **Событийная ориентированность**: стоимость как события в потоке (Kafka / Kinesis). В реальном времени.
- **Потоковая обработка в реальном времени**: обновление панели субсекундно.

### Стоимость за X — единичная метрика

$/M токенов — это язык поставщика. Метрики продукта:

- Стоимость разрешённого запроса в службу поддержки.
- Стоимость сгенерированной статьи.
- Стоимость успешной задачи агента.
- Стоимость пользовательской сессии в минуту.

Привяжите стоимость к результату продукта. В противном случае оптимизация будет не привязана к реальности.

### Форма трассировки атрибуции стоимости

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

Выдавайте трассировку при каждом вызове. Сохраняйте в хранилище данных (data lake). Агрегируйте по измерениям. Стек наблюдаемости Фазы 17 · 13 — это то место, где это живёт.

### Стек накопленной экономии

Стек: кэш + пакетная обработка + маршрутизация + шлюз. При использовании всех четырёх:
- Кэш L2 (Фаза 17 · 14): ~10x дешёвый ввод.
- Пакетная обработка (Фаза 17 · 15): скидка 50%.
- Маршрутизация на дешёвую модель (Фаза 17 · 16): снижение стоимости на 60%.
- Эффективность шлюза (Фаза 17 · 19): избыточность + повторные попытки.

Лучший сценарий с накоплением: ~5-10% от наивной базовой линии. Большинство команд используют 2-3 рычага; немногие используют все четыре.

### Цифры, которые стоит запомнить

- Измерения атрибуции: по пользователю, по задаче, по тенанту.
- Четыре слоя токенов: промпт, инструменты, память, ответ.
- Аварийный выключатель: z-скор расходов > 4.
- Единичная метрика: стоимость разрешённого запроса, а не $/M токенов.
- Накопленная оптимизация: возможно ~5-10% от базовой линии.

## Ключевые термины

| Термин | Что говорят | Что на самом деле означает |

| Атрибуция по тенанту | "стоимость клиента" | `tenant_id`; определяет единичную экономику |

| Лимит скорости | "защита 429" | Лимит на тенанта, обеспечиваемый шлюзом |
| Дневной лимит расходов | "дневной лимит" | Бюджет в масштабе тенанта с оповещением |
| Аварийный выключатель | "автоприостановка" | z-скор расходов > 4 запускает автоприостановку |
