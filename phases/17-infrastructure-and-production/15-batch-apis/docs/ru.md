# Пакетный API (Batch API) — скидка 50% как отраслевой стандарт

> Каждый крупный провайдер предоставляет асинхронный пакетный API со скидкой 50% и временем обработки ~24 часа. OpenAI, Anthropic, Google и большинство платформ инференса (Fireworks batch tier, Together batch) реализуют один и тот же подход. Стек пакетного API с кэшированием промптов и ночными конвейерами снижает стоимость до ~10% от синхронной некэшированной цены. Правило предельно простое: если запрос не является интерактивным, он должен обрабатываться пакетно. Конвейеры генерации контента, классификация документов, извлечение данных, генерация отчётов, массовая разметка, тегирование каталогов — всё, что допускает задержку в 24 часа, представляет собой нереализованную экономию, пока не будет переведено на пакетную обработку. Текущий production-паттерн 2026 года заключается в классификации каждой новой LLM-нагрузки на три потока: интерактивный (синхронный с кэшированием), полуинтерактивный (очередь асинхронных запросов с резервным кэшированием) и пакетный (ночной, с кэшированием входных данных). Нагрузки, которые маскируются под интерактивные, но допускают задержку в несколько минут, тратят большую часть бюджета.

**Тип:** Изучение
**Языки:** Python (stdlib, учебный симулятор стоимости пакетной и синхронной обработки)
**Предварительные требования:** Фаза 17 · 14 (Кэширование промптов и семантическое кэширование)
**Время:** ~45 минут

## Цели обучения

- Назвать три пакетных API провайдеров (OpenAI, Anthropic, Google) и общие параметры скидки 50% + гарантию обработки за 24 часа.
- Рассчитать стоимость комбинации пакетной обработки + кэшированного входа для ночной нагрузки по классификации и сравнить с синхронным некэшированным базовым сценарием.
- Классифицировать нагрузку как интерактивную / полуинтерактивную / пакетную и обосновать выбор потока.
- Назвать две ловушки: частичная интерактивность (пользователь ожидает результат быстрее 24 часов) и несоответствие выходной схемы (формат пакетного файла различается у разных провайдеров).

## Проблема

Ваша команда запускает ночной конвейер генерации отчётов. 50 000 документов, суммаризация каждого, кластеризация сводок, составление executives-брифа. При синхронной обработке это занимает 4 часа и стоит $2 000 за ночь. Вы узнаёте о пакетных API.

Пакетная обработка даёт скидку 50%. Вы также включаете кэширование промптов для системного промпта (общего для всех 50 000 вызовов). В стеке стоимость снижается до $180 за ночь — ~9% от базового уровня. Тот же конвейер, три изменения конфигурации.

Пакетная обработки — самый дешёвый рычаг снижения стоимости LLM, который никто не использует. Причина в основном организационная: команды думают «в реальном времени», хотя SLA фактически требует «к утру». Этот урок посвящён тому, как не терять 90% бюджета.

## Концепция

### Три пакетных API

**Пакетный API OpenAI**: загрузка JSONL-файла со списком запросов. Обещанное время обработки — 24 часа (обычно на практике ~2–8 часов). Скидка 50% на входные и выходные токены. Эндпоинт `/v1/batches`. Входные данные, подходящие для кэширования, также получают тариф кэшированного входа.

**Anthropic Message Batches**: загрузка JSONL. Время обработки — 24 часа. Скидка 50%. Поддержка `cache_control` — запись в кэш выполняется явно, чтение происходит автоматически в рамках пакета.

**Google Vertex AI Batch Prediction**: входные данные из BigQuery или GCS. Аналогичная скидка 50% для Gemini. Интеграция с Vertex Pipelines.

### Семантика: асинхронно, а не медленно

Пакетная обработка означает «я обещаю вернуть результат в течение 24 часов» — а не «это займёт 24 часа». Типичное P50 — 2–6 часов. Провайдер планирует выполнение вашего пакета в часы низкой нагрузки, когда GPU-ресурсы недозагружены.

### Комбинация с кэшированием

Суммаризация 50 000 документов с одинаковым системным промптом в 4 000 токенов:

- Синхронная некэшированная: 50 000 × ($вход × 4 000 + $выход × 200) по полным тарифам.
- Синхронная кэшированная: системный промпт кэшируется после первого вызова; оставшиеся 49 999 получают вход с 10-кратной скидкой.
- Пакетная кэшированная: всё вышеперечисленное плюс скидка 50% на чтение и запись.

Стек: пакетная обработки + кэширование = ~10% от синхронной некэшированной стоимости. Любая нагрузка, которая выполняется ночью и использует общий системный промпт, должна применять этот подход.

### Классификация нагрузки

**Интерактивная** — пользователь ожидает ответа. Важно время первого токена (TTFT). Синхронный вызов с кэшированием промптов. Пакетная обработка невозможна.

**Полуинтерактивная** — пользователь отправляет задачу и проверяет результат через несколько минут. Очередь асинхронных запросов с переключением на синхронный режим, если пакетная обработка недоступна. Пример: индексация RAG среднего объёма.

**Пакетная** — пользователь ожидает результат «к утру» или «в следующем часе». Конвейеры контента, массовая классификация, офлайн-анализ. Всегда пакетная обработка, всегда в стеке с кэшированием.

Типичная ошибка: классифицировать всё как интерактивное, потому что конвейер является production. Production — это не спецификация по задержке, SLA — вот что определяет требование.

### Ловушка частичной интерактивности

Некоторые функции выглядят интерактивными, но допускают задержку 5–10 минут. Пример: ночной отчёт о состоянии клиентов с кнопкой «обновить». Пользователь нажимает «обновить»; ожидание 10 минут допустимо. Команда реализует это как синхронный запрос. 50 одновременных обновлений стоят в 10 раз дороже, чем пакетная обработка с доставкой по электронной почте.

Вопрос для задания: «Что означает 24-часовая задержка для этого пользователя?» Если ответ — «он бы не заметил», используйте пакетную обработку.

### Ловушка выходной схемы

Форматы пакетных файлов различаются у разных провайдеров:

- OpenAI: JSONL, один запрос на строку.
- Anthropic: JSONL, одно сообщение на строку; формат ответа встроен.
- Vertex: таблица BigQuery или GCS-префикс с TFRecord.

Написание «единого пакетного клиента» для всех провайдеров требует написания адаптера для каждого провайдера. Шлюзы, предлагающие мультипровайдерную пакетную обработку (Portkey, LiteLLM в некоторых тарифах), по-прежнему выполняют простую обёртку над исходным форматом.

### Цифры, которые стоит запомнить

- Скидка на пакетную обработку у разных провайдеров: фиксированные 50% на вход и выход.
- Гарантия обработки: 24 часа гарантировано, 2–6 часов — типичное P50.
- Стек пакетной обработки + кэшированный вход: ~10% от синхронной некэшированной стоимости.
- Правило классификации нагрузки: если задержка 24 часа допустима, всегда используйте пакетную обработку.

## Использование

`code/main.py` рассчитывает стоимость для сценариев синхронной обработки, синхронной + кэширование, пакетной и пакетной + кэширование для нагрузки в 50 000 документов. Выводит экономию в долларах и процентах.

## Внедрение

В результате выполнения урока создаётся файл `outputs/skill-batch-triager.md`. На основе характеристик нагрузки выполняется классификация на интерактивную / полуинтерактивную / пакетную и оценивается экономия.

## Упражнения

1. Запустите `code/main.py`. Для конвейера в 100 000 документов с системным промптом в 3 000 токенов и выходом в 500 токенов рассчитайте экономию полного стека (пакетная обработка + кэширование) по сравнению с синхронным базовым сценарием.
2. Выберите три функции в реальном продукте, который вам известен. Классифицируйте каждую как интерактивную / полуинтерактивную / пакетную.
3. Пользователь жалуется, что его отчёт формировался 3 часа. Это была ошибочная классификация как пакетной или легитимная интерактивная обработка? Запишите критерий принятия решения.
4. Ваш SLA возврата пакетного API — 24 часа, но P99 — 20 часов. Как вы сообщите об этом пользователю — как себя ведёт система на граничном случае?
5. Рассчитайте точку безубыточности: при какой длине общего префикса пакетная обработка + кэширование становится дешевле, чем ночное выполнение на вашем выделенном GPU?

## Ключевые термины

| Термин | Что говорят | Что это означает |
|--------|------------|------------------|
| Пакетный API (Batch API) | «асинхронная скидка» | Скидка 50% с обработкой за 24 часа |
| JSONL | «формат пакета» | Один JSON-запрос на строку; стандарт OpenAI/Anthropic |
| Message Batches | «пакет Anthropic» | Название продукта пакетного API Anthropic |
| Batch prediction | «пакет Vertex» | Пакетный API продукта Vertex AI |
| Turnaround SLA | «обещание 24ч» | Гарантия, а не типичное значение; типичное — 2–6ч |
| Классификация нагрузки | «решение об интерактивности» | Маршрутизация: интерактивная / полуинтерактивная / пакетная |
| Выходная схема | «формат ответа» | Разметка JSONL для каждого провайдера; не переносимая |
| Стековая скидка | «пакет + кэш» | ~10% от некэшированного синхронного тарифа при применении обоих |
