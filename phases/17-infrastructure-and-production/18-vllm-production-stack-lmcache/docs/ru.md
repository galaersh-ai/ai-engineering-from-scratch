# Production Stack vLLM с LMCache KV Offloading

> Production Stack vLLM — это эталонное развёртывание на Kubernetes: маршрутизатор, движки и наблюдаемость, связанные воедино. LMCache — это уровень сброса KV-кэша (KV cache), который извлекает KV-кэш из памяти GPU и переиспользует его между запросами и движками (DRAM процессора, затем диск/Ceph). Соединитель (Connector) сброса KV vLLM 0.11.0 (январь 2026) делает эту операцию асинхронной и модульной через Connector API (v0.9.0+). Задержка сброса не отражается на пользовательском опыте. LMCache ценен даже при отсутствии общих префиксов — когда у GPU заканчиваются KV-слоты, прерванные запросы могут быть восстановлены из процессора вместо повторного вычисления префилла. Опубликованные бенчмарки на 16× H100 (80 ГБ HBM) в кластере из 4 узлов a3-highgpu-4g: когда KV-кэш превышает HBM, как нативный сброс в процессор, так и LMCache существенно повышают пропускную способность; при малом объёме KV все конфигурации соответствуют базовому уровню с небольшим накладным расходом.

**Тип:** Обучение
**Языки:** Python (стандартная библиотека, простой симулятор сброса KV)
**Предварительные требования:** Фаза 17 · 04 (Внутреннее устройство обслуживания vLLM), Фаза 17 · 06 (SGLang/RadixAttention)
**Время:** ~60 минут

## Цели обучения

- Составить схему уровней Production Stack vLLM: маршрутизатор, движки, сброс KV, наблюдаемость.
- Объяснить Connector API сброса KV (v0.9.0+) и то, как асинхронный путь версии 0.11.0 скрывает задержку сброса.
- Определить, когда LMCache с CPU DRAM помогает (KV > HBM) versus добавляет накладные расходы (KV достаточно мал для размещения в HBM).
- Выбрать между нативным сбросом CPU vLLM и коннектором LMCache с учётом ограничений развёртывания.

## Проблема

При обслуживании vLLM память GPU заполняется HBM на 100%, и при росте параллельности возникают события прерывания (preemption). Запросы вытесняются, ставятся в очередь заново, и вы повторно вычисляете префилл одного и того же промпта из 2K токенов четыре раза в минуту. GPU-вычисления тратятся на избыточные префиллы; полезная пропускная способность значительно ниже теоретической.

Добавление дополнительных GPU обходится линейно. Увеличить объём HBM невозможно. Но DRAM процессора дешёв: один сокет обеспечивает 512 ГБ+ при задержках на порядки хуже, чем у HBM, но достаточных для «временно горячего» KV-кэша.

LMCache извлекает KV-кэш в DRAM процессора, чтобы прерванные запросы восстанавливались быстро, а повторяющиеся префиксы между движками разделяли кэш без повторного префилла в каждом движке.

## Теория

### Production Stack vLLM

`github.com/vllm-project/production-stack` — эталонное развёртывание на Kubernetes:

- **Маршрутизатор (Router)** — с учётом кэша (Фаза 17 · 11). Потребляет события KV.
- **Движки (Engines)** — воркеры vLLM. По одному на GPU или на группу TP/PP.
- **Сброс KV-кэша (KV cache offload)** — развёртывание LMCache или нативный коннектор.
- **Наблюдаемость (Observability)** — скрапинг Prometheus, дашборды Grafana, трейсы OTel.
- **План управления (Control plane)** — обнаружение служб, конфигурация, скользящие обновления.

Поставляется как Helm-чарт + оператор.

### Connector API сброса KV (v0.9.0+)

vLLM 0.9.0 внедрил Connector API для модульных бэкендов KV-кэша. Движок сбрасывает блоки на коннектор; коннектор хранит их (RAM, диск, объектное хранилище, LMCache). Когда запросу нужен блок, коннектор загружает его обратно.

vLLM 0.11.0 (январь 2026) добавляет асинхронный путь сброса — сброс может происходить в фоне, и движок не блокируется в обычном случае. Конечная задержка и пропускная способность по-прежнему зависят от профиля нагрузки, коэффициента попаданий в KV-кэш и системного давления; в собственных примечаниях vLLM отмечается, что сброс с пользовательским ядром может снижать пропускную способность при низком коэффициенте попаданий и что асинхронное планирование имеет известные проблемы взаимодействия со спекулятивным декодированием.

### Нативный сброс CPU versus LMCache

**Нативный сброс CPU vLLM**: в пределах движка. Хранит KV-блоки в оперативной памяти хоста. Прост в реализации, не требует сетевых переходов. Не работает между движками.

**Коннектор LMCache**: на уровне кластера. Хранит блоки в общем сервере LMCache (DRAM процессора + уровень Ceph/S3). Блоки доступны любому движку. Опубликованы бенчмарки на 16× H100.

Выбирайте нативный вариант, когда давление на HBM испытывает один движок. Выбирайте LMCache, когда несколько движков разделяют префиксы (RAG с общими системными промптами, мультитенантность с общими шаблонами).

### Поведение в бенчмарках

Тест на 16× H100 (80 ГБ HBM), распределённых по 4 узлам a3-highgpu-4g:

- Малый объём KV (короткие промпты, низкая параллельность): все конфигурации соответствуют базовому уровню, LMCache добавляет ~3–5% накладных расходов.
- Средний объём: LMCache начинает помогать за счёт переиспользования префиксов между движками.
- KV превышает HBM: нативный сброс CPU и LMCache существенно повышают пропускную способность; LMCache даёт больший прирост благодаря кросс-движковому разделяемому кэшу.

### Когда LMCache решающий фактор

- Мультитенантное обслуживание, где системные промпты разделяются между арендаторами.
- RAG, где фрагменты документов повторяются в разных запросах.
- Дообученные варианты (LoRA) на одной базе, где повторное использование KV базовой модели сокращает избыточную работу.
- Нагрузки с частыми прерываниями: восстановление из процессора дешевле повторного префилла.

### Когда НЕ следует включать

- Малое давление на HBM — вы платите накладные расходы без выгоды.
- Короткие контексты (<1K токенов) — время передачи > повторного префилла.
- Однотенантная однопромптовая нагрузка — нечего переиспользовать.

### Интеграция с раздельным обслуживанием

Фаза 17 · 17 раздельное обслуживание (disaggregated serving) + LMCache усиливают друг друга: передачи KV от пула префиллов к пулу декодирования попадают в LMCache, если не используются; последующие запросы извлекают данные из LMCache. Маршрутизатор с учётом кэша из Фазы 17 · 11 может направлять запрос к движку, чей локальный или разделяемый через LMCache кэш совпадает.

### Цифры, которые стоит запомнить

- vLLM 0.9.0: выпущен Connector API.
- vLLM 0.11.0 (январь 2026): асинхронный путь сброса; влияние на конечную задержку зависит от нагрузки, коэффициента попаданий в KV и системного давления (не абсолютная гарантия).
- Бенчмарк на 16× H100: LMCache помогает, когда объём KV превышает HBM.
- Малое давление на HBM: 3–5% накладных расходов без выгоды.

## Применение

`code/main.py` симулирует нагрузку с частыми прерываниями с LMCache и без него. Выводит количество предотвращённых повторных префиллов, прирост пропускной способности и точку равновесия загрузки HBM.

## Упражнения

1. Запустите `code/main.py`. При какой загрузке HBM LMCache начинает окупаться?
2. Арендатор разделяет системный промпт из 6K токенов между 200 запросами в час. Вычислите ожидаемую экономию LMCache на арендатора.
3. Сервер LMCache является единственной точкой отказа. Спроектируйте стратегию высокой доступности (реплики, резервный переход на нативный вариант).
4. LMCache хранит данные в Ceph на жёстких дисках. Для KV из 4K токенов при 70B FP8 (500 МБ) каково время чтения по сравнению с повторным префиллом?
5. Обоснуйте, является ли асинхронный путь vLLM 0.11.0 «бесплатным» — где скрывается накладной расход?

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|--------|------------|----------------------|
| Production Stack | «эталонное развёртывание» | Helm-чарт + оператор vLLM для Kubernetes |
| Connector API | «интерфейс KV-бэкенда» | Модульный интерфейс хранения KV vLLM 0.9.0+ |
| Нативный сброс CPU | «сброс в пределах движка» | Хранение KV в оперативной памяти того же движка |
| LMCache | «кластерный KV-кэш» | Кросс-движковый сервер KV-кэша на DRAM процессора + диск |
| Асинхронный путь 0.11.0 | «неблокирующий сброс» | Сброс скрыт за потоком движка |
| Прерывание (preemption) | «вытеснение для освобождения места» | Перемещение KV-кэша при заполнении HBM |
| Переиспользование префиксов | «один и тот же системный промпт» | Несколько запросов разделяют начало; попадание в кэш |
| Уровень Ceph | «дисковый уровень» | Долговременное хранилище ниже DRAM в иерархии кэша |
