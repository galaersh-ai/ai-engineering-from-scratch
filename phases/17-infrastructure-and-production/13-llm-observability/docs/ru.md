# Выбор стека наблюдаемости (observability) за LLM

> Рынок наблюдаемости за LLM в 2026 году делится на две категории. Платформы для разработки (LangSmith, Langfuse, Comet Opik) объединяют мониторинг с оценками, управлением промптами, воспроизведением сессий. Инструменты шлюзов/инструментирования (Helicone, SigNoz, OpenLLMetry, Phoenix) фокусируются на телеметрии. Langfuse распространяется по лицензии MIT с сильным балансом открытого кода (50K событий/мес бесплатное облако). Phoenix — нативный OpenTelemetry под Elastic License 2.0, отлично подходит для визуализации дрифта и RAG, но не является постоянным производственным бэкендом. Arize AX использует zero-copy интеграцию с Iceberg/Parquet, утверждая, что в 100 раз дешевле монолитических решений для наблюдаемости. LangSmith лидирует для LangChain/LangGraph, $39/пользователь/мес, самохостинг только в Enterprise. Helicone — прокси-решение с настройкой за 15–30 минут, 100K запросов/мес бесплатно, но меньшая глубина трассировки агентов. Типичный производственный паттерн: шлюз (Helicone/Portkey) + платформа оценок (Phoenix/TruLens), связанные через OpenTelemetry.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, учебный симулятор выборочной трассировки)
**Предварительные требования:** Фаза 17 · 08 (Метрики инференса), Фаза 14 (Инженерия агентов)
**Время:** ~60 минут

## Цели обучения

- Различать платформы для разработки (комплексные: оценки + промпты + сессии) и инструменты шлюзов/телеметрии (только трассировка + метрики).
- Сопоставить шесть основных инструментов (Langfuse, LangSmith, Phoenix, Arize AX, Helicone, Opik) с их лицензиями, ценами и оптимальными сценариями применения.
- Объяснить паттерн связки OpenTelemetry (glue pattern), позволяющий комбинировать инструмент шлюза с отдельной платформой оценок.
- Назвать главный ценовой фактор 2026 года (zero-copy подход Arize AX против монолитического сбора данных) и указать приблизительный коэффициент 100x.

## Проблема

Вы выпустили функцию на базе LLM. Она работает. У вас нет видимости сбоя промптов, зацикливания инструментов, регрессий латентности, всплесков стоимости или частоты попаданий в кэш промптов. Вы гуглите «наблюдаемость за LLM» и получаете восемь инструментов, каждый из которых утверждает, что решает одну и ту же проблему по трём разным ценам.

На деле они решают разные проблемы. LangSmith отвечает на вопрос «почему этот запуск LangGraph провалился?» Phoenix отвечает «дрiftует ли мой RAG-пайплайн?» Helicone отвечает «какое приложение сжигает токены?» Langfuse отвечает «могу ли я развернуть всё это самостоятельно?» Разные инструменты — разные аудитории.

Выбор определяется четырьмя осями: стек (LangChain? чистый SDK? многопровайдерный?), допустимость лицензии (только MIT? Elastic допустима? коммерческая устроит?), бюджет (бесплатный тариф? $100/мес? $1000/мес?) и самохостинг (обязателен? желателен? никогда?).

## Концепция

### Две категории

**Платформы для разработки** объединяют наблюдаемость с оценками, управлением промптами, версионированием датасетов и воспроизведением сессий. Вы проводите эксперименты, видите, какой промпт сработал, тестируете новый промпт на регрессии датасета против предыдущих победителей. LangSmith, Langfuse, Comet Opik.

**Инструменты шлюзов/телеметрии** инструментируют вызовы инференса — промпт, ответ, токены, латентность, модель, стоимость. Helicone, SigNoz, OpenLLMetry, Phoenix. Минималистичны. Могут быть скомбинированы с отдельным инструментом оценок через OpenTelemetry.

### Langfuse — баланс открытого кода

- Ядро под лицензией Apache / MIT; самохостинг через Docker.
- Бесплатный облачный тариф: 50K событий/мес. Платный: $29/мес для команды.
- Оценки, управление промптами, трассировки (tracing), датасеты. Разумный охват всех четырёх ключевых возможностей платформы для разработки.
- Оптимальный сценарий: вам нужен функционал уровня LangSmith, но необходимо самохоститься или оставаться на открытой лицензии.

### Phoenix (Arize) — телеметрия на первом месте, нативный OpenTelemetry

- Elastic License 2.0; самохостинг не требует усилий.
- Отличная визуализация RAG и дрифта. Диаграммы рассеяния в пространстве эмбеддингов предоставляются как основной функционал.
- Не предназначен как постоянный производственный бэкенд — в первую очередь наблюдаемость на этапе разработки.
- Оптимальный сценарий: разработка RAG-пайплайна, отладка дрифта, в паре с отдельным шлюзом для продакшена.

### Arize AX — масштабное решение

- Коммерческий продукт. Интеграция с хранилищем данных через Iceberg/Parquet без копирования данных (zero-copy).
- Утверждает, что в ~100 раз дешевле монолитических решений для наблюдаемости (класса Datadog) в масштабе. Математика проста: вы храните трассировки в собственном Parquet на S3; Arize читает напрямую.
- Оптимальный сценарий: >10M трассировок/день, существующее хранилище данных, нужны LLM-специфичные дашборды без тарифов Datadog.

### LangSmith — приоритет LangChain/LangGraph

- Коммерческий продукт, $39/пользователь/мес. Самохостинг только в Enterprise.
- Лучший в своём классе для стеков LangChain и LangGraph. Если вы не используете ни один из них, он менее привлекателен.
- Оптимальный сценарий: команда работает с LangChain и готова платить.

### Helicone — прокси-решение «минимально жизнеспособное»

- Настройка за 15–30 минут путём замены `OPENAI_API_BASE` на прокси Helicone.
- Лицензия MIT; 100K запросов/мес бесплатно, платный тариф от $20/мес.
- Включает failover, кэширование, лимиты запросов — также работает как шлюз.
- Меньшая глубина трассировки агентов и многошаговых процессов.
- Оптимальный сценарий: быстрый старт, одностековое приложение, нужен шлюз + наблюдаемость в одном решении.

### Opik (Comet) — открытая платформа для разработки

- Apache 2.0, полностью открытый код.
- Аналогичный набор функций по сравнению с Langfuse, наследие Comet.
- Оптимальный сценарий: ML-команды, уже работающие с Comet, хотят наблюдаемость за LLM в том же интерфейсе.

### SigNoz — полноценный APM на базе OpenTelemetry

- Apache 2.0. Обрабатывает общий APM плюс LLM через OpenTelemetry.
- Оптимальный сценарий: единая наблюдаемость по сервисам и LLM-вызовам.

### Связующее звено: OpenTelemetry и семантические соглашения для GenAI

OpenTelemetry опубликовал семантические соглашения для GenAI в конце 2025 года (`gen_ai.system`, `gen_ai.request.model`, `gen_ai.usage.input_tokens`). Инструменты, потребляющие OTel, могут работать совместно. Формирующийся производственный паттерн:

1. Отправка OTel с GenAI-соглашениями из каждого LLM-вызова.
2. Маршрутизация в шлюз (Helicone / Portkey) для повседневной работы.
3. Двойная отправка на платформу оценок (Phoenix / Langfuse) для отслеживания регрессий.
4. Архивация в хранилище данных (Iceberg) для долгосрочного анализа через Arize AX или DuckDB.

### Ловушка: инструментирование на неверном уровне

Инструментирование внутри фреймворка агента (например, добавление трассировок LangSmith) привязывает вас к этому фреймворку. Инструментирование на уровне HTTP/OpenAI-SDK (через OpenLLMetry или ваш шлюз) является портируемым.

### Выборочная трассировка — невозможно хранить всё

При >1M запросов/день полное хранение трассировок стоит дороже самих LLM-вызовов. Выборка по правилам: 100% ошибок, 100% дорогих запросов, 5% успешных. Агрегаты храните всегда; сырые данные — для длинного хвоста.

### Цифры, которые стоит запомнить

- Langfuse бесплатное облако: 50K событий/мес.
- LangSmith: $39/пользователь/мес.
- Helicone бесплатно: 100K запросов/мес.
- Заявление Arize AX: ~100x дешевле монолитических решений в масштабе.
- Семантические соглашения OpenTelemetry для GenAI: выпущены в 2025, широкое внедрение в 2026.

## Практическое применение

`code/main.py` моделирует день с 1M трассировок по различным стратегиям хранения (полный сбор данных, выборка, выборка + ошибки). Выводит стоимость хранения и объём потерь при каждой стратегии.

## Упражнения

1. Ваша команда использует LangChain и хочет открытую платформу для наблюдаемости с самохостингом. Выберите Langfuse или Opik и обоснуйте выбор.
2. При 5M трассировок/день и предложении Datadog в $150K/мес рассчитайте точку безубыточности для Arize AX.
3. Разработайте набор атрибутов OpenTelemetry для GenAI, который ваша корпоративная политика должна требовать для каждого LLM-вызова.
4. Аргументируйте, достаточен ли Phoenix самостоятельно для продакшена. В каких случаях его недостаточно?
5. Overhead прокси Helicone составляет 20 мс. При P99 TTFT 300 мс это допустимо? А если SLA — 100 мс?
