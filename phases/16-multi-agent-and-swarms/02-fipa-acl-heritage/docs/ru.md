# Наследие FIPA-ACL и Речевых Актов

> Перед MCP, перед A2A существовал FIPA-ACL. В 2000 году IEEE Фонд Интеллектуальных Физических Агентов одобрил язык агентской коммуникации с двадцатью перформативами, двумя контентными языками и набором протоколов взаимодействия — contract net, subscribe/notify, request-when. Он исчез из промышленности из-за тяжелой семантической нагрузки для веба, но возрождение LLM мультиагентных систем без формальных семантик: JSON контракты заменяют перформативы, естественный язык заменяет семантику. Этот урок тщательно изучает FIPA-ACL, чтобы вы могли определить, какие решения протоколов 2026 года являются повторением, а какие — нововведения, и где текущая волна будет сталкиваться с проблемами уже решенными в 2000-х.

**Тип:** Учебный
**Языки:** Python (стандартная библиотека)
**Предварительные знания:** Фаза 16 · 01 (Почему Мультиагентные Системы?)
**Время:** ~60 минут

## Задача

2026 агентский-протокольный ландшафт загружен: MCP для инструментов, A2A для агентов, ACP для корпоративного аудита, ANP для децентрализованной доверенности, NLIP для естественного языка контента, а также CA-MCP и два десятка исследовательских предложений. Каждая спецификация объявляет себя фундаментальной.

Честное прочтение свидетельствует о том, что большинство из них восстанавливают очень конкретное двадцатилетнее решение дерева решений. Теория речевых актов от Остина (1962) и Сёрля (1969) дала нам "утверждения — это действия". KQML (1993) превратил это в протокол передачи данных. FIPA-ACL (одобренный в 2000 году) выработал стандартное определение: двадцать перформативов, контентные языки SL0/SL1, протоколы взаимодействия для contract net и subscribe/notify. JADE и JACK были Java-референсными платформами. Эффорт упал около 2010 года из-за тяжелой семантической нагрузки и победы веба.

Когда вы смотрите на MCP `tools/call`, A2A `task lifecycle` или CA-MCP `shared context store`, вы видите мягкий, JSON-native переработанный вариант решений FIPA. Знание истории говорит вам две вещи: какие новые "инновации" на самом деле являются повторениями, и какие старые проблемы новые спецификации будут восстанавливать.

## Концепция

### Речевые акты в одном абзаце

Остин заметил, что некоторые предложения не описывают мир — они меняют его. "Я обещаю." "Я прошу." "Я объявляю." Он назвал эти утверждения словесными делами. Сёрль формализовал пять категорий: assertive (утверждающее), directive (указательное), commissive (обещающее), expressive (выражение) и declarative (утверждение). KQML (Finin et al., 1993) сделал это операционным для программных агентов: сообщение — это словесное действие плюс контент (что делает это действие). FIPA-ACL устранил пробелы в KQML и стандартизировал около двадцати перформативов.

### Двадцать FIPA перформативов (частичный список)

| Перформатив | Цель |
|---|---|
| `inform` | "Я говорю вам, что P истинно" |
| `request` | "Я прошу вас сделать X" |
| `query-if` | "Истина ли P?" |
| `query-ref` | "Какова значение X?" |
| `propose` | "Я предлагаю мы сделаем X" |
| `accept-proposal` | "Я принимаю предложение" |
| `reject-proposal` | "Я отклоняю предложение" |
| `agree` | "Я согласен сделать X" |
| `refuse` | "Я отказываюсь делать X" |
| `confirm` | "Я подтверждаю, что P истинно" |
| `disconfirm` | "Я опровергаю P" |
| `not-understood` | "Ваше сообщение не распознано" |
| `cfp` | "Вызов к предложениям по X" |
| `subscribe` | "Уведомите меня, когда X изменится" |
| `cancel` | "Отмените текущее X" |
| `failure` | "Я попытался сделать X и провалился" |

Полный список находится в `fipa00037.pdf` (FIPA ACL Сообщение Структура). Цель не в том, чтобы запомнить его — цель в том, что каждый из этих перформативов соответствует примитиву, который в конечном итоге добавляет LLM протокол.

### Каноническое сообщение FIPA-ACL

```
(inform
  :sender       agent1@platform
  :receiver     agent2@platform
  :content      "((price IBM 83))"
  :language     SL0
  :ontology     finance
  :protocol     fipa-request
  :conversation-id   conv-42
  :reply-with   msg-17
)
```

Семь полей несут протокольную оболочку; одно поле (`content`) несет нагрузку. Остальные поля точно такие же, как вы каждый раз воссоздаете при присоединении попыток повтора, потока и семантики к JSON-протоколу.

### Две наследственные платформы

**JADE** (Java Агент Разработка Фреймворк, 1999–2020-е) был наиболее используемым FIPA-совместимым рантаймом. Агенты расширяли базовый класс, обменивались ACL сообщениями, запускались внутри контейнеров и coординировали с помощью "поведений". Библиотека протоколов взаимодействия поставлялась с contract-net, subscribe-notify, request-when и propose-accept.

**JACK** (Агент Ориентированное ПО, коммерческое) акцентировался на BDI (Belief-Desire-Intention) рационализации на основе FIPA сообщений. Более формальный, менее принятый.

Оба упали после того, как стек веба съел мультиагентные сценарии случаев. MCP и A2A являются рантаймом "контейнерами" 2026 года.

### Почему FIPA исчез

- **Семантическая нагрузка.** FIPA требовал совместного определения семантики для парсинга `content`. Согласование семантик — это многолетний процесс стандартизации. Веб просто использовал HTTP + JSON.
- **Формальные семантики, которые никто не использует.** SL (Semantic Language) давал строгое определение истинности, но большинство производственных систем использовали свободно форматированный контент и игнорировали формулизм.
- **Замкнутость инструментов.** JADE был Java-only; JACK был коммерческий. Полиглотные команды обходили оба.
- **Интернет выиграл стек.** REST, затем JSON-RPC, затем gRPC заменили ACL транспорт.

### Возрождение LLM — это FIPA-lite

Сравните FIPA `request` с MCP `tools/call`:

```
(request                                {
  :sender  agent1                         "jsonrpc": "2.0",
  :receiver tool-server                   "method":  "tools/call",
  :content "(lookup stock IBM)"           "params":  {"name":"lookup_stock",
  :ontology finance                                   "arguments":{"symbol":"IBM"}},
  :conversation-id c42                    "id": 42
)                                        }
```

Та же оболочка, другая синтаксис. Оба несут: кто, кого, намерение, нагрузку, идентификатор корреляции. Ни один из них не является революцией по сравнению с другим — они разные компромиссы на одной и той же дизайнерской концепции.

2025 обзор Луи et al. ("Обзор протоколов взаимодействия агентов: MCP, ACP, A2A, ANP", arXiv:2505.02279) делает эту линию явной: MCP соответствует речевым актам использования инструментов, A2A — речевым актам агентов-партнеров, ACP — речевым актам аудита, ANP — распределенной идентификации расширений. Новые спецификации являются потомками ACL с JSON синтаксисом и более лаконичными семантиками.

### Компромисс, выраженный прямо

**Что FIPA дал вам и современные спецификации убрали:**

- Формальные семантики — вы можете доказать, что `inform` подразумевает, что отправитель верит в содержимое.
- Канонический каталог перформативов — вы не должны заново аргументировать "должен ли у нас быть `cancel`?".
- Десятилетия шаблонов протоколов взаимодействия — contract-net, subscribe/notify, propose/accept — с известными свойствами корректности.

**Что современные спецификации дают вам и FIPA не дал:**

- JSON-native нагрузка, совместимая со всеми современными инструментами.
- Естественный язык контент, который LLM может интерпретировать без ручной кодовой семантики.
- Транспорт стека веба (HTTP, SSE, WebSocket).
- Открытие способностей через самоописательные документы (MCP `listTools`, A2A Агентная Карта).

Лаконичные семантики намерений для легче реализации. Это точный компромисс.

### Значимые протоколы, которые стоит перенести

FIPA отправил около 15 взаимодействующих протоколов. Три из них стоят переноса в мультиагентные системы LLM:

1. **Протокол Contract Net (CNP).** Менеджер выпускает `cfp` (вызов к предложениям); участники отвечают с `propose`; менеджер принимает или отклоняет. Это канонический шаблон рынка задач (Фаза 16 · 16 Неготация).
2. **Subscribe/Notify.** Подписчик отправляет `subscribe`; публишер отправляет `inform`, когда тема изменяется. Это каждая событийная шина в 2026 году.
3. **Request-When.** "Сделай X, когда условие Y выполнено." Задержанные действия с предусловиями. Канонический аналог в 2026 году — отложенные задачи в устойчивых движках потоков (Фаза 16 · 22 Производственная Масштабизация).

Каждый из них легко переносится на современные очереди сообщений, HTTP + опрос или SSE трансляции.

### Что ломается при отсутствии семантики

Без совместного определения семантики агенты инференцируют смысл из естественного языка контента. Документированный 2026-й сценарий провала — **семантический сдвиг**: два агента используют одно и то же слово (`"customer"`), но для слегка разных концепций, агент получает неправильное интерпретацию, и нет схемы валидатора, который это поймал бы. Требование FIPA по семантике отвергало сообщение при парсинге.

Митигации без полного определения семантики:

- JSON Schema на `content` — отклоняет структурные ошибки на уровне сети.
- Типизированные артефакты (A2A) — отклоняет неправильную модальность.
- Явный перформатив в оболочке — делает намерение неоднозначным даже при естественном языковом контенте.

### Спецификации 2026 года, отображенные на наследии речевых актов

| Современная спецификация | FIPA аналог | Что оно сохраняет | Что оно убирает |
|---|---|---|---|
| MCP `tools/call` | `request` | явное намерение, идентификатор корреляции | формальные семантики, семантика |
| MCP `resources/read` | `query-ref` | явное намерение, идентификатор корреляции | формальные семантики |
| A2A Task lifecycle | contract-net + request-when | асинхронный жизненный цикл, переходы состояния | формальные гарантии полной корректности |
| A2A streaming events | subscribe/notify | асинхронное толкание | типизированные предикатные подписки |
| CA-MCP shared context | черная доска (Hayes-Roth 1985) | мультиагентная совместная память | логическая модель согласованности |
| NLIP | естественный язык контент | LLM-native | схема |

Прочитав таблицу сверху вниз, паттерн: сохраняйте структурный примитив, убирайте формализм, позволяйте LLMам заполнить неопределенности.

## Сборка

`code/main.py` реализует переводчик FIPA-ACL на основе стандартной библиотеки Python. Он кодирует и декодирует каноническую ACL оболочку и показывает, как каждая современная форма сообщения MCP / A2A сводится к семи полям. Демо:

- Кодирует пять сообщений в стиле MCP и A2A в FIPA-ACL.
- Декодирует FIPA-ACL обратно в современное эквивалентное.
- Запускает игрушечную неготацию Contract Net между одним менеджером и тремя участниками, используя `cfp`, `propose`, `accept-proposal`, `reject-proposal`.

Запустите:

```
python3 code/main.py
```

Выход показывает параллельный отслеживание, где каждый современный протокол представлен как в его 2026 JSON форме, так и в FIPA-ACL форме, а затем обратно. Те же примитивы протоколов выживают после обратного перехода; только синтаксис отличается.

## Использование

`outputs/skill-fipa-mapper.md` — это навык, который читает любую спецификацию агентского протокола и производит карту FIPA-ACL. Используйте его перед принятием нового протокола для ответа на вопрос: "Это действительно новое или это `inform` с JSON синтаксисом?"

## Отправка

Не возвращайте FIPA-ACL. Возвращайте его список:

- Какой примитив намерения (перформатив) каждого сообщения?
- Есть ли идентификатор корреляции для запроса/ответа и отмены?
- Есть ли явное контентное языковое определение (JSON-RPC, простой текст, типизированный артефакт)?
- Суть протоколов взаимодействия первостепенная, или вы заново реализуете contract-net с нуля?
- Что происходит, когда два агента не согласны по смыслу контента (семантический сдвиг)?

Документируйте эти пять вопросов для любого нового протокола перед отправкой его в производство.

5. Разработайте минимальную JSON-Schema для поля `content` в перформативе `request` в вашей собственной системе. Что это дает вам, что не дает чистый естественный язык, и что это стоит?

## Ключевые термины

| Термин | Как говорят люди | На самом деле означает |
|------|----------------|------------------------|
| Речевой акт | "Утверждение, которое делает что-то" | Остин/Сёрль: утверждения как действия. Теоретический родитель ACL. |
| FIPA | "Старое XML вещь" | IEEE Фонд Интеллектуальных Физических Агентов. Стандартизировал ACL в 2000 году. |
| ACL | "Язык агентской коммуникации" | FIPA оболочка формата: перформатив + контент + метаданные. |
| Перформатив | "Глагол" | Целевая категория сообщения: `inform`, `request`, `propose`, `cfp` и т.д. |
| KQML | "Прежде всего FIPA" | Knowledge Query and Manipulation Language (1993). Проще, узкое. |
| Семантика | "Общий словарь" | Формальное определение концепций, о которых говорит контентный язык. |
| SL0 / SL1 | "Контентные языки FIPA" | Semantic Language уровней 0 и 1 — семантический контентный язык семейство. |
| Contract Net | "Рынок задач" | Менеджер выпускает `cfp`; участники отвечают с `propose`; менеджер принимает или отклоняет. Канонический протокол взаимодействия. |
| Протокол взаимодействия | "Паттерн сообщений" | Последовательность перформативов с известными свойствами корректности: request-when, subscribe/notify и т.д. |
