# Паттерны общей памяти и чёрной доски

(Translation note: "Blackboard" in the context of design patterns is often translated as "чёрная доска" or "доска-чернильница," but "чёрная доска" is more commonly used in this context.)

Двумя подходами в 2026 годах являются многокомпонентные системы агентов: **пул сообщений** (все видят все сообщения всех, как в AutoGen GroupChat или MetaGPT) и **черная доска с подпиской** (агенты подSCRIBE к релевантным событиям, как в Context-Aware MCP или фреймворке Matrix). Оба являются единственной состоятельной частью многокомпонентной системы — что означает, что обе части содержат самые интересные баги. Справочная модель сбоев — это **загрязнение памяти**: один агент выдумывает "факт," другие агенты принимают его за проверенный, и точность постепенно ухудшается в той степени, что это гораздо сложнее отлаживать, чем немедленный сбой. Эта лекция строит обе структуры из stdlib, внедряет атаку загрязнения памяти и показывает три меры защиты, которые действительно работают в производстве.

**Тип:** Учиться + Строить
**Языки:** Python (стандартная библиотека, `threading`)
**Предварительные знания:** Фаза 16 · 04 (Примитивная модель), Фаза 16 · 09 (Параллельные сети спутников)
**Время:** ~75 минут

## Проблема

Многомодельные системы нуждаются в месте, где агенты могут обмениваться фактами. Литеральный вариант — это «передавать всё через сообщения» — но это переинжиниривает общее состояние с дополнительной копировалью. Другой вариант — «дать всем глобальный журнал» — но глобальные журналы растут неограниченно и легко загрязняются. Третий вариант — «проектировать вид для каждого агента» — масштабируемый, но затратный по схеме.

Когда один из агентов видит галлюцинацию и записывает её в общее состояние, каждый последующий агент, читающий это состояние, принимает галлюцинацию за факт. Когда человек это заметит, цепочка рассуждений будет уже на пятом уровне, а корень проблемы — в третьем сообщении, когда-либо отправленном. Отладка уменьшения точности многокорпусной системы сложнее, чем отладка сбоя.

Это заражение памяти. Это второй по частоте зарегистрированный тип сбоев в классификации MAST (Cemri et al., arXiv:2503.13657) и он структурный: любая архитектура обмена памятью без доказательной базы и недописываемого проверяющего рано или поздно его покажет.

## Концепция

Два основных топологии:

Полный пул сообщений. Каждый агент читает каждое сообщение. AutoGen GroupChat и MetaGPT используют этот метод. Простой, прозрачный, проверяемый, но не масштабируется более ~10 агентов, так как контекст каждого агента заполняется работой других агентов.

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

Черная доска с подпиской. Агенты объявляют о своем интересе к темам; подложка передает только релевантные сообщения. CA-MCP (arXiv:2601.11595) и декцентрализованная фреймворк Matrix (arXiv:2511.21686) используют это. Это масштабируется, но требует предварительного дизайна схемы для того, чтобы подписки имели смысл.

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

Когда каждый выигрывает.

- **Полная бассейнная池**: выигрывает при небольшом количестве агентов (<10), неоднородных и короткой перспективе разговора. При этом тривиально определить, кто сказал что, так как все видят всё.
- **Черная доска板**: выигрывает при большом количестве агентов, однородных по роли, но множественных в экземплярах (сwarms), и долгосрочном разговоре. Роутинг экономит затраты на токены и предотвращает загрязнение контекста.

Продуктовые системы часто смешивают: небольшую полную池鱼不能跳过龙门：这句话的意思是小鱼无法跳出龙门，比喻能力有限的人难以超越自己现有的局限或达到更高的境界。在不同的语境中，这句话可以有不同的解释和应用。

В одном сценарии память может быть накачана несоответствующими данными — это называется заражением памяти.

Три агента работают над исследовательской задачей. Агент А — это агент извлечения информации. Агент Б — это сокращающий агент. Агент С — это аналитик.

1. А получает страницу и записывает сообщение в общее состояние: "Исследование показало улучшение точности на 42%. "
2. На самом деле страница говорила о "улучшении на 4,2%". А ошибся с десятичной запятой.
3. Б читает общее состояние и записывает: "Великое улучшение точности на 42% (источник: А)."
4. С читает общее состояние и записывает: "Рекомендуется принятие — повышение на 42% является трансформирующим."
5. Финальный отчет цитирует число 42%, которое никогда не существовало.

Ни один агент не сбился. Никаких тестов не провалило. Система "работала".hallucination перешла из контекста одного агента в размышления всех последующих агентов через общий состояние.

Ни один агент не вышел из строя. Никакие тесты не провалились. Система "работала". Ложная информация перешла из контекста одного агента в размышления всех последующих агентов через общее состояние.

Почему это структурное:

Без общей состояния, иллюзия агента А остается в контексте А. Дальнейшие агенты должны будут повторно получить или повторно вывести информацию и могут обнаружить ошибку. С использованием простой общей информации, контекст А становится общим для всех, и иллюзия превращается в факт.

Проблема не в том, что есть общее состояние в таком виде — проблема в том, что общее состояние отсутствует доказуемая история и без независимого проверяющего. ТриMitigations решают эту проблему:

1. **Отметьте происхождение каждого изменения.** Каждая запись в общем состоянии фиксирует, кто её сделал, когда и при каком запросе, а также (если применимо), из какого источника взяла информацию агент. Агенты ниже по топологии читают эти записи с подозрением, ориентируясь на происхождение.
2. **Версионное записывание; относитесь к нему как к записи только в конец.** Корректировка — это новая запись, которая заменяет старую, а не изменение на месте. Аудиторская история сохраняется.
3. **Поддерживайте хотя бы одного агента, который не может писать в общее состояние.** Чисто читающий агент-верификатор проверяет записи, перезагружает источники и обнаруживает противоречия. Поскольку он не может записывать в систему, его нельзя заразить через неё.

Пример на чёрном доске (Хейз-Рот, 1985)

Черновиковая схема предшествует агентам LLM на четыре десятилетия. Хейз-Рот (1985, "Аrchitecture for Control на черновиковой схеме") описал специализированные Источники Знаний, которые наблюдают за глобальной черновиковой схемой, вносят частичные решения и триггеризируют другие источники. Черновиковая схема 2026 года (CA-MCP, Matrix) является той же схемой, но с использованием агентов LLM как Источников Знаний и JSON-объектов как частичных решений. Старая литература документировала решения для записи конфликтов, оппортунистического контроля и согласованности, которые современные системы пересматривают.

Преобrazование проекции vs полный обзор

Полная чистая доска позволяет каждому подписчику получить одинаковое проекционное представление (ограниченное темой). Более агрессивный подход — это **проекция по агенту**: каждый агент получает кастомизированное представление в зависимости от своей роли. Состояние редукторов LangGraph является каноническим реализациям 2026 года — функция редуктора сворачивает глобальное состояние в срез, специфичный для роли.

Персонализированные проекции масштабируются лучше, но требуют схемы. Без неё приходится вручную воссоздавать проекцию в каждом запросе агента.

Многие агенты, пишущие одновременно, является проблемой параллелизма, а не только проблемой LLM. Три шаблона работают:

- **Последовательный писатель (один производитель).** Все записи проходят через одного координатора-агент, который серилизует их. Просто, но это бottleneck.
- **Оптимистическая конкуренция с версионированием.** Каждый элемент имеет версию; писатели терпят неудачу при несоответствии версий и повторяют попытку. Классический метод баз данных.
- **Разделение по темам.** Разные агенты владеют разными темами. Нет пересечения тем в контесте конкуренции. Требует заранее спроектированных границ разделения тем.

В большинстве фреймворков 2026 по умолчанию используется последовательный писатель, потому что вызовы LLM медленные настолько, что конкуренция редко возникает и бottleneck не оказывает влияния.

Недописываемый верификатор

Наиболее эффективная мера смягчения нагрузки — это только для чтения верификатор. Правила реализации:

- Проверяющий обменивается информацией с командой (читает доску или бассейн).
- У проверяющего нет права записи в общее состояние — только в отдельный канал проверки.
- Проверяющий самостоятельно извлекает исходные данные, упомянутые в записях. Устанавливает флаг несогласия.
- Выходные данные проверяющего направляются к человеку или отдельному агенту принятия решений, никогда не возвращаются обратно в бассейн.

Без этого разделения выводы верификатора становятся новыми записями в пуле, что означает, что заражённый пул заражает верификатора, который затем заражает его проверки.

`code/main.py` реализует оба топологии из стандартной библиотеки Python, а также игрушечную атаку подделки и три митигации.

- `MessagePool` — потокобезопасный только для записи журнал с полным чтением.
- `Blackboard` — публикация/подписка по темам с персональными подписками агентов.
- `ProvenanceEntry` — каждый запись отмечает (автор, время создания, хеш запроса, исходный URI).
- `PoisoningScenario` — запускает исследовательскую задачу для трех агентов, где агент A генерирует случайное число. Выводит最终报告。
- `Verifier` — только для чтения агент, который перезагружает источники и обнаруживает несоответствия. Запускает ту же сценарий с присутствием валидатора.

## Отправляем!

Для любого дизайна с общей памятью:

- Записывайте происхождение при каждой записи: `(автор, метка времени, хеш запроса, использованные инструменты, источник)`.
- Сделайте журнал только для добавления. Исправления — это новые записи, ссылающиеся на заменяемую.
- Разверните хотя бы одного агента-верификатора только для чтения с независимым доступом к источникам.
- Направляйте вывод верификатора в отдельный канал, а не обратно в общий пул.
- Отслеживайте долю записей, которые являются заменой предыдущих — растущая доля является ранним признаком паттернов галлюцинаций.

## Ключевые термины

| Термин | Что говорят люди | На самом деле означает |
|--------|------------------|----------------------|
| Пул сообщений | "Общий чат-хроник" | Только для чтения журнал, который каждый агент просматривает. Полная прозрачность, плохая масштабируемость. |
| Доска объявлений | "Общее рабочее пространство" | Тема-ориентированное публикационно-подписное сообщество. Агенты подписываются на релевантные темы. Масштабируется лучше. |
| Происхождение | "Кто написал что" | Метаданные для каждого записи: автор, время создания, запрос, источники. |
| Пoisoned память | "Распространение иллюзий" | Ошибка одного агента попадает в общее состояние, и последующие агенты принимают её за факту. |
| Только для добавления | "Нет местных обновлений" | Корректировки являются новыми записями, которые заменяют старые. Сохраняет историю изменений. |
| Неписательный верификатор | "Независимый аудитор" | Только для чтения агент, который пересматривает источники и обнаруживает несоответствия. |
| Проекция | "Ограниченное представление" | Персональное представление каждого агента, вычисленное из глобального состояния. Редукторы LangGraph являются каноническими примерами. |
| Источник знаний | "Экспертный агент" | Термин Хейз-Рота 1985 года для участника доски объявлений. |
