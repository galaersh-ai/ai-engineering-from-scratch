1|# Почему мультиагентная система?
2|
3|> Одному агенту попался на пути стенка. Умный ход не в том, чтобы сделать его больше — а в том, чтобы использовать несколько агентов.
4|
5|**Тип:** Изучение
6|**Языки:** TypeScript
7|**Предварительные требования:** Фаза 14 (Инженерия агента)
8|**Время:** ~60 минут
9|
10|## Цели обучения
11|
12|- Выявить потолок одиночного агента (переполнение контекста, смешанная экспертиза, последовательный узкое место) и объяснить когда разбиение на несколько агентов является правильным шагом
13|- Сравнить шаблоны оркестрации (пайплайн, параллельное распределение, супервайзер, иерархия) и выбрать подходящий для заданной структуры задачи
14|- Проектировать мультиагентную систему с четкими границами ролей, общей состояния и контрактом коммуникации
15|- Анализировать tradeoffs мультиагентной сложности (задержка, стоимость, сложность отладки) по сравнению с простотой одиночного агента
16|
17|## Проблема
18|
19|Вы создали одиночного агента в фазе 14. Он работает. Он может читать файлы, выполнять команды, вызывать API и размышлять о результатах. Затем вы направили его на реальный код: 200 файлов, три языка, тесты, зависящие от инфраструктуры, и требование исследовать внешние API перед написанием кода.
20|
21|Агент застрял. Не потому что LLM глуп, а потому что задача превышает то, что может справиться один цикл агента. Контекстное окно заполняется содержимым файлов. Агент забывает, что прочитал 40 вызовов инструментов назад. Он пытается быть исследователем, программистом и ревьюером одновременно, и делает все это плохо.
22|
23|Это потолок одиночного агента. Вы его достигаете каждый раз, когда задача требует:
24|
25|- **Больше контекста, чем помещается в одно окно** - чтение 50 файлов превышает 200k токенов
26|- **Разные экспертизы на разных этапах** - исследование требует другого запроса, чем генерация кода
27|- **Работа, которая может выполняться параллельно** - почему читать три файла последовательно, когда можно это сделать одновременно?
28|
29|## Концепция
30|
31|### Потолок одиночного агента
32|
33|Одиночный агент — один цикл, одно контекстное окно, одна система запроса. Представьте себе:
34|
35|```
36|┌─────────────────────────────────────────┐
37|│            SINGLE AGENT                 │
38|│                                         │
39|│  ┌───────────────────────────────────┐  │
40|│  │         Context Window            │  │
41|│  │                                   │  │
42|│  │  research notes                   │  │
43|│  │  + code files                     │  │
44|│  │  + test output                    │  │
45|│  │  + review feedback                │  │
46|│  │  + API docs                       │  │
47|│  │  + ...                            │  │
48|│  │                                   │  │
49|│  │  ██████████████████████ FULL ███  │  │
50|│  └───────────────────────────────────┘  │
51|│                                         │
52|│  One system prompt tries to cover       │
53|│  research + coding + review + testing   │
54|│                                         │
55|│  Result: mediocre at everything         │
56|└─────────────────────────────────────────┘
57|```
58|
59|Три вещи сломались:
60|
61|1. **Переполнение контекста** - результаты инструментов накапливаются. На третьей итерации агент уже съел 150k токенов содержимого файлов, выводов команд и предыдущих размышлений. Критические детали из пятой итерации потерялись.
62|
63|2. **Смешение ролей** - система запроса "ты исследователь, программист, ревьюер и тестировщик" выдает агента, который лишь наполовину исследует, лишь наполовину пишет код и никогда не завершает ревью.
64|
65|3. **Последовательный узкое место** - агент читает файл A, затем файл B, затем файл C. Три последовательных вызова LLM. Три последовательных выполнения инструментов. Нет параллелизма.
66|
67|### Многокорпусное решение
68|
69|Разделите работу. Дайте каждому агенту одну задачу, одно контекстное окно и системный запрос, настроенный под эту задачу:
70|
71|```
72|┌──────────────────────────────────────────────────────────┐
73|│                    ORCHESTRATOR                          │
74|│                                                          │
75|│  "Build a REST API for user management"                  │
76|│                                                          │
77|│         ┌──────────┬──────────┬──────────┐               │
78|│         │          │          │          │               │
79|│         ▼          ▼          ▼          ▼               │
80|│   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
81|│   │RESEARCHER│ │  CODER   │ │ REVIEWER │ │  TESTER  │  │
82|│   │          │ │          │ │          │ │          │  │
83|│   │ Reads    │ │ Writes   │ │ Checks   │ │ Runs     │  │
84|│   │ docs,    │ │ code     │ │ code     │ │ tests,   │  │
85|│   │ finds    │ │ based on │ │ quality, │ │ reports  │  │
86|│   │ patterns │ │ research │ │ finds    │ │ results  │  │
87|│   │          │ │ + spec   │ │ bugs     │ │          │  │
88|│   └─────┬────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  │
89|│         │           │            │             │         │
90|│         └───────────┴────────────┴─────────────┘         │
91|│                          │                               │
92|│                     Merge results                        │
93|└──────────────────────────────────────────────────────────┘
94|```
95|
96|Каждый агент имеет:
97|- Фокусированный системный запрос ("Вы кодовый ревьюер. Ваша единственная задача — находить баги.")
98|- Свое собственное контекстное окно (не загрязненное работой других агентов)
99|- Ясную схему ввода/вывода (получает заметки исследований, выдает код)
100|
101|### Реальные системы, которые это делают
102|
103|**Claude Code субагенты** - когда Claude Code создает субагент с инструментом `Task`, он создает дочерний агент с ограниченной задачей. Родитель сохраняет свой контекст чистым. Дочерний выполняет фокусированную работу и возвращает сводку.
104|
105|**Devin** - запускает планировщик-агент, программист-агент и браузер-агент. Планировщик разбивает работу на шаги. Программист пишет код. Браузер исследует документацию. Каждый имеет отдельный контекст.
106|
107|**Мультиагентные команды программистов (SWE-bench)** - лучшие системы на SWE-bench используют исследователя, читающего код, планировщика, который проектирует исправление, и программиста, который реализует его. Одноагентные системы получают меньший балл.
108|
109|**ChatGPT Глубокое Исследование** - запускает несколько поисковых агентов параллельно, каждый исследуя разные направления, затем синтезирует результаты.

111|### Спектр
112|
113|Мультиагентность не бинарна. Это спектр:
114|
115|```
116|SIMPLE ──────────────────────────────────────────── COMPLEX
117|
118| Single        Sub-         Pipeline      Team         Swarm
119| Agent         agents
120|
121| ┌───┐       ┌───┐        ┌───┐───┐    ┌───┐───┐    ┌─┐┌─┐┌─┐
122| │ A │       │ A │        │ A │ B │    │ A │ B │    │ ││ ││ │
123| └───┘       └─┬─┘        └───┘─┬─┘    └─┬─┘─┬─┘    └┬┘└┬┘└┬┘
124|               │                │        │   │       ┌┴──┴──┴┐
125|             ┌─┴─┐          ┌───┘───┐    │   │       │shared │
126|             │ a │          │ C │ D │  ┌─┴───┴─┐    │ state │
127|             └───┘          └───┘───┘  │  msg   │    └───────┘
128|                                       │  bus   │
129| 1 loop      Parent +      Stage by    │       │    N peers,
130| 1 context   child tasks   stage       └───────┘    emergent
131|                                       Explicit      behavior
132|                                       roles
133|```
134|
135|**Одиночный агент** - один цикл, один запрос. Хорош для простых задач.
136|
137|**Субагенты** - родитель создает дочерние агенты для фокусированных подзадач. Родитель сохраняет план. Дочерние отчитываются о результатах. Это то, что делает Claude Code.
138|
139|**Пайплайн** - агенты выполняют последовательно. Выход одного агента становится входом другого. Хорош для стадийных процессов: исследование -> код -> ревью -> тестирование.
140|
141|**Команда** - агенты выполняют параллельно с общим шиной сообщений. Каждый имеет роль. Оркестратор coординирует. Хорошо, когда нужны разные навыки одновременно.
142|
143|**Рой (Swarm)** — множество идентичных или очень похожих агентов с общим состоянием. Нет фиксированного оркестратора. Агенты берут работу из очереди. Хорош для параллельных задач высокой пропускной способности.

145|### Четыре шаблона мультиагентной системы
146|
147|#### Шаблон 1: Пайплайн
148|
149|```
150|Input ──▶ Agent A ──▶ Agent B ──▶ Agent C ──▶ Output
151|          (research)  (code)      (review)
152|```
153|
154|Каждый агент преобразует данные и передает их дальше. Просто в терминах рассуждений. Ошибка на одном этапе блокирует остальные.
155|
156|#### Шаблон 2: Распределение / Сбор
157|
158|```
159|                ┌──▶ Agent A ──┐
160|                │              │
161|Input ──▶ Split ├──▶ Agent B ──├──▶ Merge ──▶ Output
162|                │              │
163|                └──▶ Agent C ──┘
164|```
165|
166|Распределяйте работу между параллельными агентами, затем объединяйте результаты. Хорошо для задач, которые разлагаются на независимые подзадачи.
167|
168|#### Шаблон 3: Оркестратор-работник
169|
170|```
171|                    ┌──────────┐
172|                    │  Orch.   │
173|                    └──┬───┬───┘
174|                  task │   │ task
175|                 ┌─────┘   └─────┐
176|                 ▼               ▼
177|           ┌──────────┐   ┌──────────┐
178|           │ Worker A │   │ Worker B │
179|           └──────────┘   └──────────┘
180|```
181|
182|Умный оркестратор решает, что делать, делегирует работу работникам и синтезирует результаты. Оркестратор сам по себе агент с инструментами для запуска работников.
183|
184|#### Шаблон 4: Плоский рой
185|
186|```
187|         ┌───┐ ◄──── msg ────▶ ┌───┐
188|         │ A │                  │ B │
189|         └─┬─┘                  └─┬─┘
190|           │                      │
191|      msg  │    ┌───────────┐     │ msg
192|           └───▶│  Shared   │◄────┘
193|                │  State    │
194|           ┌───▶│  / Queue  │◄────┐
195|           │    └───────────┘     │
196|      msg  │                      │ msg
197|         ┌─┴─┐                  ┌─┴─┐
198|         │ C │ ◄──── msg ────▶ │ D │
199|         └───┘                  └───┘
200|```
201|
202|Нет фиксированного оркестратора. Агенты общаются напрямую друг с другом. Поведение возникает из взаимодействия. Труднее отлаживать, но масштабируется до многих агентов.
203|
204|### Когда НЕ использовать мультиагентную систему
205|
206|Мультиагентность добавляет сложность. Каждое сообщение между агентами — потенциальная точка сбоя. Отладка переходит от "прочтите одну беседу" к "следуйте сообщениям через пять агентов".
207|
208|**Оставайтесь одиночным агентом, когда:**
209|- Задача помещается в одно контекстное окно (менее ~100k токенов рабочих данных)
210|- Не требуется разные системные запросы для разных этапов
211|- Последовательное выполнение достаточно быстро
212|- Задача простая настолько, что ее разделение добавляет больше издержек, чем пользы
213|
214|**Стоимость сложности:**
215|- Каждый граница агента — это убыточный шаг сжатия: контекст агента A суммируется в сообщении для агента B
216|- Логика координации (кто что делает, когда и в каком порядке) является источником ошибок
217|- Задержка увеличивается: N агентов означают N последовательных вызовов LLM минимум, больше если они должны обмениваться сообщениями друг с другом
218|- Стоимость умножается: каждый агент сжигает токены независимо
219|
220|Правило пальца: если задача требует менее 20 вызовов инструментов и помещается в 100k токенов, оставьте ее одиночным агентом.

223|
224|### Шаг 1: Перегруженный одиночный агент
225|
226|Вот одиночный агент, который пытается делать все. У него один большой системный запрос и одно контекстное окно с исследованием, кодом и ревью:
227|
228|```typescript
229|type AgentResult = {
230|  content: string;
231|  tokensUsed: number;
232|  toolCalls: number;
233|};
234|
235|async function singleAgentApproach(task: string): Promise<AgentResult> {
236|  const systemPrompt = `You are a full-stack developer. You must:
237|1. Research the requirements
238|2. Write the code
239|3. Review the code for bugs
240|4. Write tests
241|Do ALL of these in a single conversation.`;
242|
243|  const contextWindow: string[] = [];
244|  let totalTokens = 0;
245|  let totalToolCalls = 0;
246|
247|  const research = await fakeLLMCall(systemPrompt, `Research: ${task}`);
248|  contextWindow.push(research.output);
249|  totalTokens += research.tokens;
250|  totalToolCalls += research.calls;
251|
252|  const code = await fakeLLMCall(
253|    systemPrompt,
254|    `Given this research:\n${contextWindow.join("\n")}\n\nNow write code for: ${task}`
255|  );
256|  contextWindow.push(code.output);
257|  totalTokens += code.tokens;
258|  totalToolCalls += code.calls;
259|
260|  const review = await fakeLLMCall(
261|    systemPrompt,
262|    `Given all previous context:\n${contextWindow.join("\n")}\n\nReview the code.`
263|  );
264|  contextWindow.push(review.output);
265|  totalTokens += review.tokens;
266|  totalToolCalls += review.calls;
267|
268|  return {
269|    content: contextWindow.join("\n---\n"),
270|    tokensUsed: totalTokens,
271|    toolCalls: totalToolCalls,
272|  };
273|}
274|```
275|
276|Проблемы с этим подходом:
277|- Контекстное окно растет на каждом этапе. На этапе ревью оно содержит заметки исследований, код и предыдущие размышления.
278|- Системный запрос универсален. Он не может быть настроен для каждого этапа.
279|- Ничто не выполняется параллельно.

281|### Шаг 2: Специализированные агенты
282|
283|Теперь разделите его. Каждый агент получает одну задачу:
284|
285|```typescript
286|type SpecialistAgent = {
287|  name: string;
288|  systemPrompt: string;
289|  run: (input: string) => Promise<AgentResult>;
290|};
291|
292|function createSpecialist(name: string, systemPrompt: string): SpecialistAgent {
293|  return {
294|    name,
295|    systemPrompt,
296|    run: async (input: string) => {
297|      const result = await fakeLLMCall(systemPrompt, input);
298|      return {
299|        content: result.output,
300|        tokensUsed: result.tokens,
301|        toolCalls: result.calls,
302|      };
303|    },
304|  };
305|}
306|
307|const researcher = createSpecialist(
308|  "researcher",
309|  "You are a technical researcher. Read documentation, find patterns, and summarize findings. Output only the facts needed for implementation."
310|);
311|
312|const coder = createSpecialist(
313|  "coder",
314|  "You are a senior TypeScript developer. Given requirements and research notes, write clean, tested code. Nothing else."
315|);
316|
317|const reviewer = createSpecialist(
318|  "reviewer",
319|  "You are a code reviewer. Find bugs, security issues, and logic errors. Be specific. Cite line numbers."
320|);
321|```
322|
323|Каждый специалист имеет фокусированный запрос. Каждый получает чистое контекстное окно с только необходимыми данными.
324|
325|### Шаг 3: Координация через сообщения
326|
327|Соедините специалистов с явным обменом сообщениями:
328|
329|```typescript
330|type AgentMessage = {
331|  from: string;
332|  to: string;
333|  content: string;
334|  timestamp: number;
335|};
336|
337|async function multiAgentApproach(task: string): Promise<AgentResult> {
338|  const messages: AgentMessage[] = [];
339|  let totalTokens = 0;
340|  let totalToolCalls = 0;
341|
342|  const researchResult = await researcher.run(task);
343|  messages.push({
344|    from: "researcher",
345|    to: "coder",
346|    content: researchResult.content,
347|    timestamp: Date.now(),
348|  });
349|  totalTokens += researchResult.tokensUsed;
350|  totalToolCalls += researchResult.toolCalls;
351|
352|  const coderInput = messages
353|    .filter((m) => m.to === "coder")
354|    .map((m) => `[From ${m.from}]: ${m.content}`)
355|    .join("\n");
356|
357|  const codeResult = await coder.run(coderInput);
358|  messages.push({
359|    from: "coder",
360|    to: "reviewer",
361|    content: codeResult.content,
362|    timestamp: Date.now(),
363|  });
364|  totalTokens += codeResult.tokensUsed;
365|  totalToolCalls += codeResult.toolCalls;
366|
367|  const reviewerInput = messages
368|    .filter((m) => m.to === "reviewer")
369|    .map((m) => `[From ${m.from}]: ${m.content}`)
370|    .join("\n");
371|
372|  const reviewResult = await reviewer.run(reviewerInput);
373|  messages.push({
374|    from: "reviewer",
375|    to: "orchestrator",
376|    content: reviewResult.content,
377|    timestamp: Date.now(),
378|  });
379|  totalTokens += reviewResult.tokensUsed;
380|  totalToolCalls += reviewResult.toolCalls;
381|
382|  return {
383|    content: messages.map((m) => `[${m.from} -> ${m.to}]: ${m.content}`).join("\n\n"),
384|    tokensUsed: totalTokens,
385|    toolCalls: totalToolCalls,
386|  };
387|}
388|```
389|
390|Каждый агент получает только сообщения, адресованные ему. Нет загрязнения контекста. Заметки исследователя с 50k токенами документации никогда не попадают в контекст ревьюера.

392|### Шаг 4: Сравнение
393|
394|```typescript
395|async function compare() {
396|  const task = "Build a rate limiter middleware for an Express.js API";
397|
398|  console.log("=== Single Agent ===");
399|  const single = await singleAgentApproach(task);
400|  console.log(`Tokens: ${single.tokensUsed}`);
401|  console.log(`Tool calls: ${single.toolCalls}`);
402|
403|  console.log("\n=== Multi-Agent ===");

405|  console.log(`Tokens: ${multi.tokensUsed}`);
406|  console.log(`Tool calls: ${multi.toolCalls}`);
407|}

422|## Ключевые термины
423|
424|| Термин | Что говорят люди | Что на самом деле означает |
425||------|----------------|----------------------|
426|| Рой (Swarm) | "Многоагентная система" | Сеть равноправных агентов с общим состоянием и без фиксированного лидера. Поведение возникает из локального взаимодействия. |
427|| Оркестратор | "Босс-агент" | Агент, у которого есть инструменты для запуска и управления другими агентами. Он планирует и делегирует работу, но может не выполнять ее сам. |
428|| Координатор | "Трафикопроводник" | Нераспределенная компонента (часто просто код, а не LLM), которая направляет сообщения между агентами на основе правил. |
429|| Консенсус | "Агенты согласились" | Протокол, при котором несколько агентов должны достичь согласия перед продолжением. Используется, когда конфликтующие результаты требуют решения. |
430|| Эмерджентное поведение | "Агенты разобрались сами" | Системные паттерны, возникающие из взаимодействий агентов, но не были явно программированы. Может быть полезным или вредным. |
431|| Распределение / сбор | "Map-reduce для агентов" | Разделение задачи между параллельными агентами (распределение), затем объединение их результатов (сбор). |
432|| Обмен сообщениями | "Агенты общаются друг с другом" | Механизм коммуникации между агентами: структурированные данные, отправляемые одним агентом другому, заменяющие общее контекстное окно. |
