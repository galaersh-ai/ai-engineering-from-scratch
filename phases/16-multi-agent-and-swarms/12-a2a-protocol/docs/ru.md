# A2A — Протокол агента-агенту

> В апреле 2025 года Google анонсировала A2A; к апрелю 2026 года спецификация доступна по адресу https://a2a-protocol.org/latest/specification/ и поддерживает её более 150 организаций. A2A является горизонтальным дополнением к MCP (урок 13): где MCP вертикальный (агент ↔ инструменты), A2A — это взаимодействие на равных (агент ↔ агент). Она определяет Карты Агентов (поиск), задачи с артефактами (текст, структурированные данные, видео), непрозрачные циклы жизни задач и авторизацию. Производственные системы всё чаще используют MCP и A2A вместе. В течение 2025-2026 годов Google Cloud интегрировала поддержку A2A в Vertex AI Agent Builder.

**Тип:** Учиться + Строить
**Языки программирования:** Python (стандартная библиотека, `http.server`, `json`)
**Предварительные знания:** Фаза 16 · 04 (Примитивная модель)
**Время:** ~75 минут

## Проблема

Ваш агент должен позвонить другому агенту на другой системе. Как это сделать? Вы можете экспонировать точку входа HTTP, определить собственный схему JSON и надеяться, что сторона другая разбирается в ней. Каждая пара агентов становится кастомной интеграцией.

A2A является универсальным проводным протоколом для этого вызова. Стандартная发现，标准任务模型，标准传输，标准制品。就像HTTP+REST，但将代理视为一等公民。

A2A является универсальным проводным протоколом для этого вызова. Стандартная система обнаружения, стандартная модель задач, стандартный транспорт, стандартные артефакты. Как HTTP+REST, но с代理视为一等公民。

### Авторизация

A2A поддерживает три распространенных шаблона:

- **Токен bearer** — OAuth2 или непрозрачный.
- **mTLS** — взаимная проверка подлинности по TLS; организации доказывают свою идентичность друг другу.
- **Подписанные запросы** — HMAC над нагрузкой.

Авторизация объявляется на карточке Агента; клиенты обнаруживают ее и соблюдают.

К апрелю 2026 года более 150 организаций

Адаптация предприятий привела к масштабированию A2A. Заголовок: A2A стал способом преодоления границ доверия для систем агентов в предприятиях. Google Cloud выпустила поддержку Vertex AI Agent Builder A2A; Microsoft Framework для агентов поддерживает его; большинство основных фреймворков (LangGraph, CrewAI, AutoGen) выпускают адаптеры A2A.

Где A2A выигрывает

- **Перекрестные звонки между организациями.** Агент из компании A звонит агенту из компании B. Без A2A каждая пара агентов заключает индивидуальное соглашение.
- **Разнообразные фреймворки.** Агент LangGraph вызывает агента CrewAI, который в свою очередь вызывает пользовательский агент на Python. A2A нормализует процесс.
- **Типизированные артефакты.** Видеорезультаты, структурированный JSON и аудио — все являются первостепенными элементами.
- **Долгосрочные задачи.** Неопределенная жизненный цикл плюс периодическая проверка делают выполнение длительных задач простым.

Где A2A сталкивается с трудностями

- **Чувствительные к задержке микрокалибы.** Жизненный цикл A2A асинхронный. Агенты之间的亚毫秒级通信不符合要求；使用直接的RPC。
- **Тightly coupled впроцессных агентах.** Если оба агента выполняются в одной Python процессе, HTTP-обмен между ними является перебором.
- **Маленькие команды.** Надмерная спецификация действительно существует; внутренние только для использования агенты могут не нуждаться в формальности.

### A2A против ACP, ANP, NLIP

Несколько связанных спецификаций появились в 2024-2026 годах:

- **ACP** (IBM/Linux Фондация) — предшественник A2A, более узкий диапазон.
- **ANP** (Протокол Сетевого Агента) — сильный акцент на обнаружении равных, децентрализованный подход первостепенный.
- **NLIP** (Экма Протокол Природного Языка взаимодействия, стандартизирован в декабре 2025 года) — тип контента на природном языке.

A2A к апрелю 2026 года является наиболее распространенным протоколом сверки сводных данных. Подробнее о сравнении см. в статье arXiv:2505.02279 (Liu et al., "Обзор протоколов интероперабельности агентов").

## Собери его

`code/main.py` реализует минимальный сервер и клиент для A2A (Assurance 2 Authorities) с использованием модулей `http.server` и JSON. Сервер:

- экспонирует `/\.well-known\/agent\.json`,
- принимает `POST /tasks`,
- управляет состоянием задач,
- возвращает артефакты по `GET /tasks/{id}`.

Клиент:

- получает Агентскую карту,
- отправляет задачу,
- опрашивает до завершения,
- читает артефакт.

Запустить:

```
python3 code/main.py
```

Скрипт запускает сервер в фоновом потоке, а затем запускает клиентское приложение для взаимодействия с ним. Вы видите полную последовательность действий: обнаружение, отправка, проверка, артефакт.

### Список проверки:

- **Закрепить версию спецификации.** A2A всё ещё эволюционирует; карточка агента должна объявлять версию протокола.
- **Идемпотентное создание задач.** Дублирующиеся отправки (сетевые повторы) должны генерировать одну задачу.
- **Схемы артефактов.** Объявите, какие формы возвращаются агентом; потребители должны проверять их.
- **Ограничения скорости + аутентификация.** A2A является публичным интерфейсом; применяйте стандартную веб-безопасность.
- **Канал для неудачных задач.** Инспектируйте шаблоны по мере времени для типов повторяющихся ошибок.

## Упражнения

1. Запустите `code/main.py`. Подтвердите, что клиент обнаруживает сервер и получает правильный артефакт.
2. Добавьте вторую навык на сервер (например, "summarize"). Обновите карточку Агента. Напишите клиента, который выбирает навык в зависимости от типа задачи.
3. Реализуйте эндпоинт потоковой передачи SSE: `/tasks/{id}/events`, который излучает изменения состояния. Что клиенту нужно делать по-другому?
4. Прочитайте спецификацию A2A (https://a2a-protocol.org/latest/specification/). Выделите три вещи, которые спецификация требует, но этот демонstration не реализовал.
5. Сравните A2A (обнаружение карточки агента) с MCP (серверное перечисление возможностей через `listTools`). Какая есть tradeoff между самопроизвольно описываемыми агентами и поиском возможностей?
