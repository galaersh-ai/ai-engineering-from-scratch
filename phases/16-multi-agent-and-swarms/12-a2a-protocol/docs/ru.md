# A2A — протокол агент-агент

> Google объявила A2A в апреле 2025; к апрелю 2026 спецификация доступна по адресу https://a2a-protocol.org/latest/specification/ и её поддерживают 150+ организаций. A2A — горизонтальный дополнитель к MCP (Урок 13): MCP — вертикальный (агент ↔ инструменты), A2A — пиринговый (агент ↔ агент). Он определяет Карты агентов (обнаружение), задачи с артефактами (текст, структурированные данные, видео), непрозрачные жизненные циклы задач и аутентификацию. Продакшен-системы всё больше сочетают MCP с A2A. Google Cloud внедрила поддержку A2A в Vertex AI Agent Builder в 2025-2026.

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib, `http.server`, `json`)
**Предварительные требования:** Фаза 16 · 04 (Примитивная модель)
**Время:** ~75 минут

## Проблема

Вашему агенту нужно вызвать другого агента на другой системе. Как? Вы можете открыть HTTP-эндпоинт, определить кастомную JSON-схему и надеяться, что другая сторона говорит на ней. Каждая пара агентов становится кастомной интеграцией.

A2A — универсальный wire-протокол для этого вызова. Стандартное обнаружение, стандартная модель задач, стандартный транспорт, стандартные артефакты. Как HTTP+REST, но для агентов как первоклассных граждан.

## Концепция

### Четыре элемента

**Карта агента.** JSON-документ по адресу `/.well-known/agent.json`, описывающий агента: имя, навыки, эндпоинты, поддерживаемые модальности, требования аутентификации. Обнаружение происходит чтением карты.

```
GET https://agent.example.com/.well-known/agent.json
→ {
    "name": "code-review-agent",
    "skills": ["review-python", "review-typescript"],
    "endpoints": {
      "tasks": "https://agent.example.com/tasks"
    },
    "auth": {"type": "bearer"},
    "modalities": ["text", "structured"]
  }
```

**Задача.** Единица работы. Асинхронный состоящий объект с жизненным циклом: `submitted → working → completed / failed / canceled`. Клиент отправляет задачу, опрашивает или подписывается на обновления.

**Артефакт.** Тип результата, производимого задачей. Текст, структурированный JSON, изображение, видео, аудио. Артефакты типизированы, так что разные модальности — первоклассные.

**Непрозрачный жизненный цикл.** A2A не предписывает, *как* удалённый агент решает задачу. Клиент видит переходы состояний и артефакты; реализация свободна использовать любой фреймворк.

### Разделение MCP/A2A

- **MCP** (Урок 13): агент ↔ инструмент. Агент читает/пишет через JSON-RPC на сервер инструментов. Без состояния по умолчанию.
- **A2A**: агент ↔ агент. Пиринговый протокол; обе стороны — агенты с собственными рассуждениями.

Продакшен мультиагентные системы используют оба. A2A-пир вызывает MCP-инструменты на своей стороне. Разделение giữает два интереса чистыми.

### Поток обнаружения

```
Клиент                     Сервер агента
  ├──GET /.well-known/agent.json──>
  <──JSON Карты агента─────────────
  ├──POST /tasks {skill, input}──>
  <──201 task_id, state=submitted
  ├──GET /tasks/{id}──────────────>
  <──state=working, 42% done──────
  ├──GET /tasks/{id}──────────────>
  <──state=completed, artifacts──
```

Или со стримингом: SSE-подписка на `/tasks/{id}/events` для push-обновлений.

### Аутентификация

A2A поддерживает три типичных паттерна:

- **Bearer-токен** — OAuth2 или непрозрачный.
- **mTLS** — взаимный TLS; организации доказывают личность друг другу.
- **Подписанные запросы** — HMAC над полезной нагрузкой.

Аутентификация объявлена в Карте агента; клиенты обнаруживают и соблюдают.

### 150+ организаций к апрелю 2026

Корпоративное внедрение обеспечило масштаб A2A. Ключевое: A2A стал способом, которым корпоративные агентные системы пересекают границы доверия. Google Cloud выпустила поддержку A2A в Vertex AI Agent Builder; Microsoft Agent Framework поддерживает её; большинство крупных фреймворков (LangGraph, CrewAI, AutoGen) поставляют A2A-адаптеры.

### Где A2A выигрывает

- **Межорганизационные вызовы.** Агент компании A вызывает агента компании B. Без A2A каждая пара — кастомный контракт.
- **Гетерогенные фреймворки.** Агент LangGraph вызывает агента CrewAI вызывает кастомного Python-агента. A2A нормализует.
- **Типизированные артефакты.** Видео-результат, структурированный JSON, аудио — всё первоклассное.
- **Долгосрочные задачи.** Непрозрачный жизненный цикл + опрос делает задачи продолжительностью часы простыми.

### Где A2A затрудняется

- **Латентностно-чувствительные микро-вызовы.** Жизненный цикл A2A асинхронный. Субмиллисекундные агент-к-агенту не подходят; используйте прямой RPC.
- **Плотно связанные внутрипроцессные агенты.** Если оба агента работают в одном Python-процессе, HTTP-круг A2A избыточен.
- **Маленькие команды.** Накладные расходы спецификации реальны; агенты только для внутреннего использования могут не нуждаться в формальности.

### A2A vs ACP, ANP, NLIP

Несколько связанных спецификаций возникли в 2024-2026:

- **ACP** (IBM/Linux Foundation) — предшественник A2A, более узкий охват.
- **ANP** (Agent Network Protocol) — с упором на пиринговое обнаружение, децентрализованный-first.
- **NLIP** (Ecma Natural Language Interaction Protocol, стандартизован декабрь 2025) — тип контента на естественном языке.

A2A — самый внедрённый пиринговый протокол на апрель 2026. См. arXiv:2505.02279 (Liu et al., «A Survey of Agent Interoperability Protocols») для сравнения.

## Реализуем

`code/main.py` реализует A2A-минимальный сервер и клиент с использованием `http.server` и JSON. Сервер:

- открывает `/.well-known/agent.json`,
- принимает `POST /tasks`,
- управляет состоянием задач,
- возвращает артефакты на `GET /tasks/{id}`.

Клиент:

- получает Карту агента,
- отправляет задачу,
- опрашивает до завершения,
- читает артефакт.

Запуск:

```
python3 code/main.py
```

Скрипт запускает сервер в фоновом потоке, затем запускает клиента против него. Вы видите полный поток: обнаружение, отправка, опрос, артефакт.

## Результат

Чеклист:

- **Зафиксируйте версию спецификации.** A2A всё ещё развивается; Карта агента должна объявлять версию протокола.
- **Идемпотентное создание задач.** Дублированные отправки (сетевые повторы) должны производить одну задачу.
- **Схемы артефактов.** Объявите, какие формы возвращает агент; потребители должны валидировать.
- **Лимиты скорости + аутентификация.** A2A публичный; применяйте стандартную веб-безопасность.
- **Dead-letter для проваленных задач.** Анализируйте паттерны со временем для повторяющихся типов сбоев.

## Упражнения

1. Запустите `code/main.py`. Подтвердите, что клиент обнаруживает сервер и получает правильный артефакт.
2. Добавьте второй навык на сервер (например, «суммаризировать»). Обновите Карту агента. Напишите клиента, который выбирает навык по типу задачи.
3. Реализуйте SSE-стриминговый эндпоинт: `/tasks/{id}/events`, генерирующий изменения состояний. Что клиент должен делать по-другому?
4. Прочитайте спецификацию A2A (https://a2a-protocol.org/latest/specification/). Определите три вещи, которые спецификация требует, а это демо не реализует.
5. Сравните A2A (обнаружение через Карту агента) с MCP (вывод возможностей на стороне сервера через `listTools`). В чём компромисс между сам описываемыми агентами и зондированием возможностей?
