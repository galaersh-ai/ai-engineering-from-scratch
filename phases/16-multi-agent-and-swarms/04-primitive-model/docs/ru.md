Каждые шесть месяцев поступает в продажу новая многоагентная платформа. AutoGen в 2023 году. CrewAI в 2024 году. LangGraph и OpenAI Swarm в 2024 году. Google ADK в апреле 2025 года. ## Концепция

### Четыре примитива

1. **Агент** — это системное сообщение плюс список инструментов. Без состояния; каждое выполнение начинается с его системного сообщения и текущей истории сообщений.
2. **Передача управления** — структурированная передача контроля от одного агента к другому. Механически, это вызов инструмента, который возвращает новый агент или ребро графа, которое следует за условием.
3. **Общее состояние** — любая структура данных, которую могут читать (иногда и записывать) более одного агента. Пул сообщений, черная доска, хранилище ключ-значение, векторное память.
4. **Оркестратор** — тот, кто решает, кто говорит следующим. Варианты: явный граф (детерминированный), LLM выборщик говорящего (мягкий), последний говорящий передает вызов передачи управления (OpenAI Swarm) или планировщик над очередью (архитектура стаи).

Это и есть все пространство дизайна. Каждая платформа выбирает значения по умолчанию для каждой оси; остальное — поверхность синтаксиса.

### Как каждая платформа 2026 года отображается на этом

| Платформа | Агент | Передача управления | Общее состояние | Оркестратор |
|-----------|-------|---------|--------------|--------------|
| OpenAI Swarm / Agents SDK | `Agent(instructions, tools)` | инструмент возвращает Agent | проблема вызывающего | следующий вызов передачи управления LLM |
| AutoGen v0.4 / AG2 | `ConversableAgent` | выборщик говорящего на GroupChat | пул сообщений | функция-выборщик (LLM или round-robin) |
| CrewAI | `Agent(role, goal, backstory)` | `Process.Sequential / Hierarchical` | выводы задачи связаны цепочкой | LLM менеджера или статический порядок |
| LangGraph | узловая функция | ребро графа + условие | `StateGraph` редуктор | граф, детерминированный |
| Microsoft Agent Framework | агент + паттерны оркестрации | паттерн-специфический | нить / контекст | паттерн-специфический |
| Google ADK | агент + карта A2A | задача A2A | артефакты A2A | хост решает |

Поверхностные различия кажутся огромными. Под поверхностью: те же четыре регулятора.

### Почему это важно

Как только Вы увидите примитивы, сравнение платформ превращается в короткий список:

- Оркестратор доверяет ли LLM маршрутизации (Swarm) или закрепляет маршрут в коде (LangGraph)?
- Является ли общее состояние полной историей (GroupChat) или проекцией (редуктор StateGraph)?
- Могут ли агенты изменять сообщения друг друга (менеджер CrewAI) или только передавать управление (Swarm)?

Эти три вопроса ответят на 80% того, какая платформа подходит для данной задачи. Вы перестаете искать "лучшую многоагентную платформу" и начинаете проектировать для осей, которые Вам действительно важны.

### Безостоятый взгляд

Каждый примитив, кроме общего состояния, безостоятелен. Агент — это функция (сообщение, инструменты). Передача управления — вызов функции. Оркестратор — планировщик. **Единственная статическая часть системы — общее состояние.** Именно там живут все интересные ошибки: отравление памяти (Урок 15), порядок сообщений, версионирование, конфликт записи.

Платформы, которые скрывают общее состояние (Swarm), переносят проблему на вызывающего. Платформы, централизующие его (LangGraph checkpoint, AutoGen pool) делают его проверяемым, но смещают стоимость координации на реализацию общего состояния.

### Анатомия одного примитива

#### Агент

```
Agent = (system_prompt, tools, model, optional_name)
```

Нет памяти. Нет состояния. Два агента с одинаковым системным сообщением и инструментами взаимозаменяемы. Все, что выглядит как состояние агента, на самом деле находится в общем состоянии или протоколе передачи управления.

#### Передача управления

```
Handoff = (from_agent, to_agent, reason, payload)
```

Три реализации доминируют:

- **Возврат функции** — инструмент возвращает следующего агента. Это паттерн OpenAI Swarm. Агенты несут маршрутизацию в своих схемах инструментов.
- **Ребро графа** — LangGraph. Ребра декларативны. LLM производит значение; условие выбирает следующий узел.
- **Выбор говорящего** — AutoGen GroupChat. Функция выборщика (иногда сам вызов LLM) читает пул и выбирает, кто говорит дальше.

#### Общее состояние

```
SharedState = { messages: [], artifacts: {}, context: {} }
```

Минимумом является список сообщений. Часто больше: структурированные артефакты (выходы задач CrewAI), типизированный контекст (редукторы LangGraph), внешняя память (MCP, векторное хранилище).

Две топологии: **полный пула** (каждый агент видит все сообщения) и **проекция** (агенты видят роль-специфическую версию). Полные пулы просты, но плохо масштабируются. Проекции масштабируются, но требуют предварительного проектирования схемы.

#### Оркестратор

```
Orchestrator = ({state, last_speaker}) -> next_agent
```

Четыре вкупе:

- **Статический** — граф фиксирован на этапе сборки (детерминированный LangGraph, последовательный CrewAI).
- **Выбор LLM** — LLM читает пул и выбирает следующего говорящего (AutoGen, иерархический CrewAI).
- **Передача управления** — текущий агент решает, вызывая инструмент передачи управления (Swarm).
- **Очередь** — работники извлекают работу из общей очереди; нет явного следующего говорящего (архитектура стаи, Matrix).

### Что меняется между платформами

Как только примитивы фиксируются, оставшиеся дизайнерские решения:

- **Стратегия памяти** — эфемерная или устойчивая точка контроля (LangGraph checkpointer).
- **Граница безопасности** — кто может одобрить передачу управления (человек в цикле).
- **Учет затрат** — токенные бюджеты на агента.
- **Наблюдаемость** — отслеживание передачи управления, сохранение состояния для воспроизведения.

Все реализуемо поверх примитивов. Ничего из этого не является новыми примитивами.

## Постройте это

`code/main.py` реализует четыре примитива в ~150 строках стандартной библиотеки Python. Без настоящего LLM — каждый агент представляет собой скриптовую политику, чтобы фокус оставался на структуре координации.

Файл экспортирует:

- `Agent` — класс данных с именем, системным сообщением, инструментами и функцией политики.
- `Handoff` — функция, которая возвращает новый агент.
- `SharedState` — потокобезопасный пул сообщений.
- `Orchestrator` — три варианта: `StaticOrchestrator`, `HandoffOrchestrator`, `LLMSelectorOrchestrator` (симулированный).

Демо запускает ту же трехагентную трубу (исследование → написание → проверка) через все три типа оркестратора и выводит пул сообщений в конце. Вы можете увидеть, что выходы различаются только тем, кто выбирает следующего; агенты и общее состояние одинаковы во всех запусках.

Запустите его:

```
python3 code/main.py
```

Ожидаемый вывод: три запуска оркестратора, по одному на паттерн. Каждый выводит финальный пул сообщений. Запуск управления передачей достигает меньшего числа агентов, если исследователь решает закончить работу заранее — это миниатюрное представление торговой сделки маршрутизации LLM.

## Основные термины

| Термин | Что люди говорят | На самом деле означает |
|------|----------------|------------------------|
| Агент | "LLM с инструментами" | Тройка `(system_prompt, tools, model)`. Безостоятелен. |
| Передача управления | "Передача контроля" | Структурированный вызов, который указывает следующего агента и необязательный пакет. Три реализации: возврат функции, ребро графа, выбор говорящего. |
| Общее состояние | "Память" / "контекст" | Единственная статическая часть многоагентной системы. Пул сообщений или черная доска. |
| Оркестратор | "Координатор" | Кто решает, кто следующий. Статический граф, выбор LLM, управление передачей, или очередь-драйвенный. |

| Проекция состояния | "Специфическая роль" | Роль-специфический вид на общее состояние. Масштабируется, требует проектирования схемы. |
| Выбор говорящего | "Кто говорит следующим" | Паттерн оркестратора, где функция (часто LLM) выбирает следующего агента из группы. |
