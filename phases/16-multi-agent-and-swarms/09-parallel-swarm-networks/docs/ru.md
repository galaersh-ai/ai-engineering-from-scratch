# Параллельные / Сwarm- / Сетевые Архитектуры

> В对照与主管：没有中央决策者。代理读取共享事件总线，异步接收任务并写回结果。LangGraph явно поддерживает "Сwarm-архитектуру" для децентрализованных, динамических сред. Матрица (arXiv:2511.21686) представляет как контрольные, так и данные потоки сериализованными сообщениями, передаваемыми через распределенные очереди для устранения бottleneck-а оркестратора. Торговля здесь явна: детерминированность и отслеживаемость за счет масштабируемости. Swarm подходит для задач с множеством независимых подзадач; он не подходит для задач, требующих единого плана.

Руководитель масштабируется до нескольких работников. Что делать, если их несколько сотен? Сам руководитель становится бottleneck'ом: каждое решение о том, кто doing什么工作分配都通过一个代理来做出。一项缓慢的计划步骤会阻碍整个系统。

Архитектуры стаффов переворачивают дизайн. Вместо центрального планировщика, распределяющего работу, работники берут задачи с общего очереди. "Координация" встроена в семантику шины событий. Нет оркестратора; система масштабируется до тех пор, пока не достигнет предела очереди.

## Концепция

Форма

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

Без оркестратора. Каждый рабочий повторяет следующие действия: забирает задачу, обрабатывает ее, записывает результат (и при необходимости.enqueue последующие задачи).

Когда стадо подходит (по смыслу переведено, так как "swarm" может иметь разные значения в зависимости от контекста)

- **Множество независимых задач.** Сcraping, преобразование, классификация. Задачи не зависят друг от друга.
- **При变量持续时间的工作。** Если некоторые задачи выполняются за 100 мс, а другие — за 10 секунд, стая автоматически распределяет нагрузку — быстрые работники берут следующие задания. Супервайзер должен учитывать продолжительность.
- **Потоковость над детерминизмом.** Вам важен общий время завершения, а не строгое порядок выполнения задач.

Матрица (арXив:2511.21686)

Матрица — это статья 2025 года, которая доводит концепцию спорма до логического завершения: управление потоком данных и управления потоком данных реализованы как сериализованные сообщения на распределенных очередях. Нет центрального координатора. Устойчивость к отказам обеспечивается благодаря устойчивости сообщений. Масштабируемость — это проблема брокера сообщений, а не системы.

Вклад: модель программирования, где координация многоагентной системы определяется как "какая тема сообщений подписан на этой агенте?", вместо того чтобы задавать это как "который агент выберет надзорный агент следующим?". Это делает систему выглядеть как публикационно-подписческую сеть событий.

LangGraph 2025 документы явно описывают "Сwarм-архитектуру" как один из многоагентных шаблонов: агенты являются узлами, но ребра формируют ориентированный граф с циклами, и любой узел может быть активирован из пула. Работник выбирает доступную работу по условию, а не по назначению руководителя.

Режим сбоев: голодание и ск

Если все работники берут самый быстро доступную задачу, длинные задачи никогда не будут выбраны до тех пор, пока они не останутся единственными. Классическая ситуация голодания очереди.

Меры по снижению рисков:
- Приоритетные очереди с явным старением (увеличение приоритета с увеличением времени ожидания).
- Специализация рабочих: некоторые рабочие обрабатывают только "длинные" задачи.
- Нагрузочное ограничение: ограничивать количество быстрых задач, входящих в очередь.

Контентно-ориентировочный маршрутинг-связь

Сwarm естественно сочетается с содержательно-ориентированным роутингом (урок 22). Вместо общего очереди используйте одну очередь для каждого типа сообщения. Специализированные рабочие подписываются только на свой тип сообщений. Это основа для архитектур сообщений-бусов, которые масштабируются до тысяч агентов.

## Собери его

`code/main.py` реализует сворм из 4 рабочих потоков, которые берут задачи из общего `queue.Queue`. Задачи имеют переменную продолжительность (некоторые быстро выполняются, а некоторые медленно). Демонstration сравнивает:

- **Последовательная базовая линия:** один работник обрабатывает все задачи последовательно.
- **Фиксированное назначение:** каждая задача предварительно назначена конкретному работнику (по стилю супервайзера).
- **Сwarм:** работники берут задачи из общего очереди.

Сwarm автоматически распределяет нагрузку; фиксированное назначение оставляет быстрых работников без дела, когда их присвоенная задача медленная.

Запустить:

```
python3 code/main.py
```

Передано показания по количеству задач на каждого работника (сwarm распределяет неравномерно, но оптимально) и времени по часам.

## Используйте Его

`outputs/skill-swarm-fit.md` оценивает, следует ли использовать сворм вместо супервизора для задачи. Входные данные: независимость задачи, разброс продолжительности, требования к порядку выполнения и потребность в отладке.

Отправить его

### Список проверки:

- **Приоритетная очередь с устареванием.** Предотвращает голодание задач с длительным выполнением.
- **Идемпотентность работников.** Задача может быть вытащена несколько раз, если работник внезапно остановится во время выполнения. Работники должны быть идемпотентными.
- **Долговечная очередь.** Используйте Kafka, Redis Streams или очередь с поддержкой базы данных для продакшна. `queue.Queue` является только в памяти.
- **Мониторинг на уровне задачи.** Каждая задача имеет идентификатор трассировки; каждый работник логирует начало/окончание с этим идентификатором.
- **Обратная нагрузка (back-pressure).** Если очередь растет быстрее, чем работники её опустошают, замедлите производителя.
