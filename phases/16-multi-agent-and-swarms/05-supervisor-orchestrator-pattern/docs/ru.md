# Паттерн супервайзера / оркестратор-работник

> Один ведущий агент планирует и делегирует; специализированные работники выполняют в параллельных контекстах и отчитываются. Это паттерн, лежащий в основе системы Research от Anthropic (Claude Opus 4 как ведущий, Sonnet 4 как субагенты), показавшей +90.2% по сравнению с одиночным Opus 4 на внутренних исследованиях. Инженерный пост Anthropic сообщает, что 80% дисперсии на BrowseComp объясняется использованием токенов — мультиагентный подход выигрывает главным образом потому, что каждый субагент получает свежее контекстное окно. Этот урок построит паттерн супервайзера из примитивов и рассмотрит инженерные уроки 2026 года из продакшен-развёртываний.

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib, `threading`)
**Предварительные требования:** Фаза 16 · 04 (Примитивная модель)
**Время:** ~75 минут

## Проблема

Исследование — это типичная задача, с которой не справляются системы с одним агентом. Вы спрашиваете: «что изменилось в мультиагентных системах между 2023 и 2026 годами?» Один агент читает пять статей последовательно, заполняет половину контекста их текстом, а затем должен рассуждать о них всех вместе. Он забывает первую статью к тому моменту, когда добирается до пятой. Он не может параллелизовать.

Паттерн супервайзера решает это: один ведущий агент планирует поиск, делегирует каждый подвопрос работнику, а затем синтезирует результаты. Каждый работник получает собственное окно на 200k токенов для узкого вопроса. Ведущий никогда не видит сырые статьи — только резюме работников.

Продакшен-система Research от Anthropic сообщает о +90.2% на внутренних исследованиях по сравнению с одиночным Opus 4. Тот же пост отмечает, что 80% дисперсии BrowseComp объясняется *использованием токенов*. Свежий контекст для каждого субагента — основной механизм.

## Концепция

### Паттерн

```
                 ┌──────────────┐
                 │   Lead       │  планирует, декомпозирует,
                 │  (Opus 4)    │  синтезирует
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         свежий       свежий        свежий
         контекст    контекст      контекст
```

Ведущий никогда не читает сырые материалы. Работники не видят работу друг друга до тех пор, пока ведущий не синтезирует. Каждая стрелка — это передача (handoff) с узким артефактом.

### Почему это работает

Три механизма:

1. **Свежий контекст для каждого субагента.** Работник, изучающий «наследие FIPA-ACL», не несёт 40k токенов, которые ведущий потратил на планирование. Он получает окно 200k для одного вопроса.
2. **Специализация через промпт.** Промпт ведущего — «декомпозируй и синтезируй», а не «исследуй». Промпт каждого работника узкий: «найди, что изменилось в X». Фокусированные промпты дают фокусированные результаты.
3. **Параллелизм.** Работники выполняются параллельно. Реальное время примерно `max(времена_работников) + планирование + синтез`, а не `сумма(времена_работников)`.

### Инженерные уроки (Anthropic 2025)

Пост Anthropic перечисляет несколько продакшен-уроков, актуальных и в 2026 году:

- **Масштабируйте усилия под сложность запроса.** Простые запросы: один агент, 3-10 вызовов инструментов. Сложные: 10+ агентов. Ведущий должен оценивать это, а не вызывающий.
- **Сначала широко, потом узко.** Декомпозируйте на широкие подвопросы, затем запускайте больше работников на каждый подвопрос, если ответ требует глубины.
- **Радужные развёртывания.** Агенты — долгоживущие и состоящие. Традиционный blue-green не работает. Anthropic использует rainbow: постепенный rollout новых версий при вытеснении старых.
- **Использование токенов доминирует.** Мультиагентный подход потребляет ~15× токенов по сравнению с одиночным. Запускайте его только когда стоимость задачи оправдывает расходы.

### Поворот LangGraph

LangGraph изначально поставлял библиотеку `langgraph-supervisor` с высокоуровневым хелпером `create_supervisor`. В 2025 году LangChain рекомендовал реализовать паттерн супервайзера через tool-calling напрямую, потому что вызовы инструментов дают больше контроля над *тем, что видит супервайзер* (инженерия контекста). Библиотека всё ещё работает; документация теперь рекомендует форму с tool-calling.

### Режимы отказа

- **Ведущий галлюцинирует план.** Если ведущий генерирует подвопросы, которые не декомпозируют реальный вопрос, работники проводят точное исследование по неправильной цели.
- **Работники чрезмерно исследуют.** Без явных границ объёма работники выходят за пределы назначенного подвопроса и загрязняют шаг синтеза.
- **Конфликты синтеза.** Два работника возвращают противоречащие факты. Ведущий должен либо переспросить (добавить раунд), либо явно зафиксировать разногласие. Молчаливый выбор одной стороны — худший провал: пользователь никогда не узнает, что было разногласие.

### Когда супервайзер неправилен

- **Последовательные задачи.** Если шаг 2 буквально требует выходные данные шага 1, параллелизм ничего не даёт. Используйте пайплайн (CrewAI Sequential, LangGraph linear graph).
- **Простые запросы.** Одиночный агент справляется быстрее и дешевле. Используйте проверку ведущего на «масштабирование усилий» перед запуском работников.
- **Строгая детерминированность.** Супервайзер использует делегирование, выбираемое LLM. Статические графы лучше, когда аудит/перезапуск важнее адаптивности.

## Реализуем

`code/main.py` реализует супервайзера из трёх параллельных работников с использованием `threading`. Ведущий декомпозирует запрос на подвопросы, работники выполняются параллельно для каждого подвопроса, затем ведущий синтезирует. Без реальных LLM — работники смоделированы для имитации загрузки и резюмирования.

Ключевая структура:

- `Lead.plan(query)` разбивает запрос на 3 подвопроса.
- `Worker.run(sub_q)` возвращает фиктивное резюме (в продакшене это может быть любой агент, использующий инструменты).
- `Lead.run(query)` запускает работников в потоках, ждёт завершения и синтезирует.

Запуск:

```
python3 code/main.py
```

Вывод показывает план, параллельные трейсы работников с временными метками начала/конца и финальный синтез. Видно выигрыш в реальном времени: три работника по 0.3 секунды работают ~0.35 секунды, а не 0.9.

## Применение

`outputs/skill-supervisor-designer.md` принимает запрос пользователя и генерирует дизайн паттерна супервайзера: системный промпт ведущего, роли работников, правила декомпозиции подвопросов и шаблон синтеза. Используйте это перед построением новой исследовательской системы.

## Результат

Чеклист перед развёртыванием паттерна супервайзера:

- **Парирование моделей.** Ведущий на модели уровня рассуждений (класс Opus, `o3`). Работники на более быстрой и дешёвой модели (Sonnet, `o4-mini`).
- **Тайм-аут работника.** Любой работник, превышающий 2× медианного времени выполнения, убивается; ведущий либо перезапускает с узким объёмом, либо продолжает без него.
- **Лимит токенов на работника.** Жёсткий лимит (например, 10× ожидаемого входа синтеза) предотвращает утечку токенов из-за сбежавшего работника.
- **Наблюдаемость.** Трассируйте план ведущего, вызовы инструментов каждого работника и синтез. Это основа для любого постмортема.
- **Радужный rollout.** Состоящие долгоживущие агенты требуют постепенного перехода версий, а не горячей замены.

## Упражнения

1. Запустите `code/main.py`, затем измените ведущего так, чтобы он запускал 5 работников вместо 3. Наблюдайте эффект реального времени. При каком количестве работников накладные расходы на запуск превышают выигрыш от параллелизма в этой демонстрации?
2. Реализуйте тайм-аут работника: убивайте любого работника, выполняющегося дольше 0.5 секунды, и заставляйте ведущего синтезировать оставшиеся результаты. Какая наблюдаемость нужна, чтобы знать, что работник был обрезан?
3. Добавьте шаг обнаружения конфликтов в синтез ведущего: если два работника возвращают противоречащие ответы, ведущий фиксирует разногласие, а не выбирает одну сторону. Как обнаружить противоречие без вызова LLM?
4. Прочитайте инженерный пост Anthropic о системе Research. Перечислите три практики, которые этой игрушечной демонстрации нужно Adoption для запуска в продакшене.
5. Сравните `create_supervisor` от LangGraph (устаревший) с новой рекомендацией tool-calling. Что даёт лучший контроль над тем, что видит супервайзер? Почему Anthropic явно передаёт только подответы, а не полный контекст работников, в синтез?

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|----------------|------------------------|
| Супервайзер | «Ведущий агент» | Оркестратор, который планирует, делегирует и синтезирует. Сам не выполняет работу. |
| Работник | «Субагент» | Фокусированный агент, вызываемый супервайзером с узким объёмом и собственным контекстным окном. |
| Оркестратор-работник | «Паттерн супервайзера» | То же самое, другое название. Литература 2026 года использует оба. |
| Свежий контекст | «Чистое окно» | Контекст работника начинается с его системного промпта и назначенного вопроса, а не из истории ведущего. |
| Радужное развёртывание | «Постепенный rollout» | Долгоживущие состоящие агенты требуют постепенной замены версий, а не blue-green. |
| Доминирование токенов | «Контекст — главная переменная» | 80% дисперсии исследовательских оценок объясняется общим числом использованных токенов, а не выбором модели, по данным Anthropic. |
| Масштабирование усилий | «Соизмеряйте число агентов со сложностью» | Ведущий оценивает сложность запроса и запускает 1 vs 10+ работников соответственно. |
| Конфликт синтеза | «Работники не согласны» | Два работника возвращают противоречащие факты; ведущий должен выявить разногласие, а не молча выбрать одну сторону. |
