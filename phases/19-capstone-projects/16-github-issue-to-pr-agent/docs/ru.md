# Выпускной проект 16 — Автономный агент (Agent) от задачи (Issue) к PR на GitHub

> AWS Remote SWE Agents, Cursor Background Agents, OpenAI Codex cloud и Google Jules — все они реализуют один и тот же продуктовый шаблон 2026 года: пометьте задачу — получите PR. Агент запускается в облачном песочнице, проверяет прохождение тестов и публикует PR, готовый к обзору, с обоснованием изменений. Сложные задачи — автоматическое воспроизведение среды сборки репозитория, предотвращение утечки учётных данных, соблюдение бюджетов для каждого репозитория и защита от принудительной отправки (force-push). В рамках данного выпускного проекта (capstone) создаётся самостоятельная версия и сравнивается по стоимости и проценту успешности с hosted-альтернативами.

**Тип:** Выпускной проект (Capstone)
**Языки:** Python (агент), TypeScript (GitHub App), YAML (Actions)
**Предварительные требования:** Фаза 11 (инженерия LLM), Фаза 13 (инструменты), Фаза 14 (агенты), Фаза 15 (автономные агенты), Фаза 17 (инфраструктура)
**Задействованные фазы:** P11 · P13 · P14 · P15 · P17
**Время:** 30 часов

## Проблема

Асинхронный облачный кодирующий агент (async cloud coding agent) — это отдельная продуктовая категория в отличие от интерактивных кодирующих агентов (выпускной проект 01). Пользовательский интерфейс — это метка GitHub. Вы помечаете задачу `@agent fix this`, в облачном песочнице запускается воркер, клонирует репозиторий, запускает тесты, редактирует файлы, проверяет результат и открывает PR с обоснованием агента в описании. Нет интерактивного цикла, нет терминала. AWS Remote SWE Agents, Cursor Background Agents, OpenAI Codex cloud, Google Jules и Factory Droids — все идут по этому пути.

Инженерные задачи конкретны: воспроизведение среды (агент должен собрать репозиторий с нуля без кешированного образа разработки), ненадёжные тесты (необходимо перезапускать или изолировать), ограничение области действия учётных данных (GitHub App с минимальными детализированными разрешениями), соблюдение бюджетов для каждого репозитория в день и политика запрета force-push. Выпускной проект измеряет процент успешности, стоимость и безопасность в сравнении с hosted-альтернативами.

## Концепция

Триггером является вебхук GitHub (метка задачи или комментарий к PR). Диспетчер ставит задачу в очередь на ECS Fargate или Lambda. Воркер скачивает репозиторий в песочницу Daytona или E2B с универсальным Dockerfile, выводимым из репозитория (язык, фреймворк). Агент запускает цикл mini-swe-agent или SWE-agent v2 с Claude Opus 4.7 или GPT-5.4-Codex. Он итеративно: читает код, предлагает исправление, применяет патч, запускает тесты.

Верификация (verification) — это контрольный этап. Полный CI должен пройти в песочнице перед открытием PR. Вычисляется дельта покрытия; если она отрицательна сверх допустимого порога, PR открывается, но получает метку `needs-review`. Агент публикует обоснование в виде описания PR, а также создаёт ветку обсуждения `@agent`, к которой рецензент может обратиться за уточнениями.

Безопасность обеспечивается через два различных интерфейса GitHub: приложение предоставляет краткосрочный токен установки с разрешениями `workflows: read` и узкими областями доступа к содержимому репозитория и PR; защита веток (а не разрешения приложения) обеспечивает «запрет прямой записи в `main`» и «запрет force-push» — приложение никогда не добавляется в список обхода (bypass list). Доступ с ограничением по пути к `.github/workflows` для чтения не является стандартным примитивом GitHub App, поэтому список допустимых файлов для редактирования должен обеспечивать это ограничение на стороне воркера. Дневные лимиты бюджетов для каждого репозитория соблюдаются на уровне диспетчера (например, максимум 5 PR на репозиторий в день, $20 за PR).

## Архитектура

```
Метка задачи GitHub @agent fix или комментарий к PR
            |
            v
    GitHub App webhook -> диспетчер AWS Lambda
            |
            v
    Задача ECS Fargate (или self-hosted runner GitHub Actions)
       - скачать репозиторий
       - вывести Dockerfile (язык, менеджер пакетов)
       - песочница Daytona / E2B с целевой средой выполнения
       - клонирование -> git worktree -> ветка агента
            |
            v
    Цикл mini-swe-agent / SWE-agent v2
       Claude Opus 4.7 или GPT-5.4-Codex
       инструменты: ripgrep, tree-sitter, чтение/редактирование, run_tests, git
            |
            v
    Проверка прохождения CI в песочнице + проверка дельты покрытия
            |
            v (проверено)
    git push + открытие PR через GitHub App
       Описание PR = обоснование + сводка диффов + URL трассировки
       метка: needs-review
            |
            v
    Оператор проверяет; может упомянуть агента (@-mention) для уточнений
```

## Стек технологий

- Триггер: GitHub App с детализированным токеном; приём вебхуков через Lambda или Fly.io
- Воркер: задача ECS Fargate (или self-hosted runner GitHub Actions)
- Песочница: devcontainer Daytona или песочница E2B для каждой задачи
- Цикл агента: mini-swe-agent (базовый вариант) или SWE-agent v2 с Claude Opus 4.7 / GPT-5.4-Codex
- Поиск: repo-map на tree-sitter + ripgrep
- Верификация: полный CI в песочнице + порог дельты покрытия
- Наблюдаемость: Langfuse с архивом трассировок для каждого PR, ссылка на который размещается в описании PR
- Бюджет: дневной лимит в долларах для каждого репозитория; максимальное количество PR на репозиторий в день

## Сборка

1. **GitHub App.** Детализированный токен установки (installation token): issues read+write, pull_requests write, contents read+write, workflows read. Защита веток (единственный интерфейс, способный обеспечить это) гарантирует «запрет прямой отправки в `main`» и «запрет force-push»; приложение не включено в список обхода. Воркер обеспечивает «запрет записи в `.github/workflows`» посредством проверки списка допустимых файлов на основе предлагаемого диффа, поскольку разрешения GitHub App не поддерживают ограничение по пути.

2. **Приём вебхуков.** Функция Lambda принимает вебхуки меток задач и комментариев к PR. Фильтрует по метке `@agent fix this`. Ставит задачи в очередь SQS.

3. **Диспетчер.** Извлекает задачи из SQS. Обеспечивает соблюдение дневного бюджета для каждого репозитория. Запускает задачу ECS Fargate с URL репозитория, телом задачи и новой песочницей Daytona.

4. **Вывод среды.** Определение языка (Python, Node, Go, Rust) и менеджера пакетов (uv, pnpm, go mod, cargo). Динамическая генерация Dockerfile, если он отсутствует.

5. **Цикл агента.** mini-swe-agent или SWE-agent v2 с Claude Opus 4.7. Инструменты: ripgrep, repo-map на tree-sitter, read_file, edit_file, run_tests, git. Жёсткие ограничения: $20 стоимости, 30 минут реального времени, 30 ходов агента.

6. **Верификация.** После завершения цикла запуск полного набора тестов в песочнице. Вычисление дельты покрытия через jacoco / coverage.py. Если CI красный: остановка, PR не открывается. Если покрытие снижается более чем на 2%: PR открывается с меткой `needs-review`.

7. **Публикация PR.** Отправка ветки агента. Открытие PR через GitHub API с: заголовком, обоснованием, сводкой диффов, URL трассировки, стоимостью, количеством ходов.

8. **Гигиена учётных данных.** Воркер работает с краткосрочным токеном установки GitHub App. Логи очищаются от секретов перед архивацией.

9. **Оценка.** 30 засеянных внутренних задач различной сложности. Измерение процента успешности, качества PR (размер диффа, стиль, покрытие), стоимости, задержки. Сравнение с Cursor Background Agents и AWS Remote SWE Agents на тех же задачах.

## Результат

`outputs/skill-issue-to-pr.md` — это результат проекта. GitHub App + асинхронный облачный воркер, который превращает помеченные задачи в PR, готовые к обзору, с ограниченной стоимостью и ограниченными областями действия учётных данных.

## Упражнения

1. Добавьте режим «исправление ненадёжного теста»: метка `@agent stabilize-flake TestX` запускает тест 50 раз в песочнице и предлагает минимальное изменение для стабилизации.

2. Сравните стоимость с Cursor Background Agents на трёх общих задачах. Отчитайтесь, какой инструмент где выигрывает.

3. Реализуйте панель бюджета: стоимость по репозиториям в день, стоимость по пользователям. Оповещение об аномалиях.

4. Создайте режим «пробного запуска» (dry-run), который открывает черновик PR без запуска CI, чтобы рецензенты могли дёшево изучить план.

5. Добавьте политику хранения: ветки PR старше 7 дней без слияния удаляются автоматически.

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|------|-----------------|------------------------|
| GitHub App (приложение GitHub) | «Ограниченная учётная запись бота» | Приложение с детализированными разрешениями + краткосрочный токен установки |
| Асинхронный облачный агент | «Фоновый агент» | Неинтерактивный воркер, работающий в облачном песочнице, а не в терминале |
| Вывод среды | «Синтез Dockerfile» | Определение языка + менеджера пакетов, генерация Dockerfile при его отсутствии |
| Верификация | «CI в песочнице» | Запуск полного набора тестов внутри воркера перед открытием PR |
| Дельта покрытия | «Сохранение покрытия» | Изменение процента покрытия тестами от базовой ветки до ветки агента |
| Бюджет на репозиторий | «Дневной лимит» | Ограничение по долларам и количеству PR, обеспечиваемое на уровне диспетчера |
| Обоснование | «Пояснение в описании PR» | Резюме агента о том, что изменилось и почему; обязательно размещается в описании PR |
