# Выпускной проект 02 — RAG над базой кода (семантический поиск по репозиториям)

> Каждая серьёзная инженерская организация в 2026 году использует внутренний поиск по коду, который понимает смысл, а не только строки. Sourcegraph Amp, ответы Cursor по базе кода, граф Augment для предприятий, repomap от Aider, внутренний MCP Pinterest — один и тот же подход. Сбор данных из множества репозиториев, парсинг через tree-sitter, эмбеддинг (embedding) чанков на уровне функций и классов, гибридный поиск, переранжирование, ответ с цитированием. Этот выпускной проект (capstone) предлагает вам построить систему, способную обработать 2 миллиона строк кода в 10 репозиториях и выдерживать инкрементальную переиндексацию при каждом git push.

**Тип:** Выпускной проект
**Языки:** Python (сбор данных), TypeScript (API + UI)
**Предварительные требования:** Фаза 5 (основы NLP), Фаза 7 (трансформеры), Фаза 11 (инженерия LLM), Фаза 13 (инструменты), Фаза 17 (инфраструктура)
**Задействованные фазы:** P5 · P7 · P11 · P13 · P17
**Время:** 30 часов

## Проблема

К 2026 году каждый современный кодогенерирующий агент поставляется со слоем извлечения из базы кода, поскольку окна контекста alone не решают вопросы о коде в нескольких репозиториях. Контекст Claude на 1 миллион токенов помогает, но не устраняет потребность в ранжированном извлечении. Наивный поиск по косинусному сходству на необработанных чанках ухудшает результаты для сгенерированного кода, дублирования в монорепозиториях и редко импортируемых символов из длинного хвоста. Производственным решением является гибридный (плотный + BM25) поиск по чанкам с учётом AST (абстрактного синтаксического дерева) с переранжированием, подкреплённый графом ссылок на символы.

Вы изучите это, индексируя реальный пул репозиториев — а не один учебный репозиторий — и измеряя MRR@10, достоверность цитирования и инкрементальную актуальность. Режимы отказа являются инфраструктурными: монорепозиторий на 100 тысяч файлов, push, затрагивающий половину файлов, запрос, требующий пересечения четырёх репозиториев для корректного ответа.

## Концепция

Парсерный конвейер (pipeline), осведомлённый о AST, обрабатывает каждый файл с помощью tree-sitter, извлекает узлы функций и классов и разбивает код на чанки на границах узлов, а не фиксированных окон токенов. Каждый чанк получает три представления: плотный эмбеддинг (Voyage-code-3 или nomic-embed-code), разреженные термины BM25 и краткую аннотацию на естественном языке. Аннотация добавляет третью модальность для поиска — пользователи спрашивают «как выполняется авторизация X», и аннотация содержит слово «authz», даже если в коде встречается только `check_permission`.

Извлечение осуществляется гибридным образом. Запрос запускает одновременно плотный и BM25 поиск, объединяет top-k результатов и передаёт объединённую выборку переранжировщику на основе кросс-энкодера (Cohere rerank-3 или bge-reranker-v2-gemma-2b). Отранжированный список передаётся синтезатору с длинным контекстом (Claude Sonnet 4.7 с кэшированием промптов или Llama 3.3 70B с самостоятельным развертыванием) с указанием цитировать каждое утверждение с указанием файла и диапазона строк. Ответы без цитирований отклоняются постфильтром.

Инкрементальная актуальность — это инфраструктурная задача. Git push запускает вычисление дифференциала: какие файлы изменились, какие символы изменились. Переэббеддингиваются только затронутые чанки. Затронутые межфайловые рёбра символов (импорты, вызовы методов) пересчитываются. Индекс остаётся согласованным без необходимости повторной обработки 2 миллионов строк при каждом коммите.

## Архитектура

```
git push --> webhook --> воркер сбора данных (LlamaIndex Workflow)
                           |
                           v
             tree-sitter парсинг + AST-чанкинг
                           |
            +--------------+----------------+
            v              v                v
          плотный      BM25-индекс      аннотация (LLM)
        (Voyage / bge)  (Tantivy)      (Haiku 4.5)
            |              |                |
            +------> Qdrant / pgvector <----+
                            |
                            v
                      граф символов (Neo4j / kuzu)
                            |
  запрос --> агент LangGraph (извлечение -> переранжирование -> синтез)
                            |
                            v
                 Claude Sonnet 4.7, 1M контекст
                            |
                            v
                 ответ + цитирование файл:строка
```

## Технический стек

- Парсинг: tree-sitter с 17 грамматиками языков (Python, TS, Rust, Go, Java, C++ и др.)
- Плотные эмбеддинги: Voyage-code-3 (хостинг) или nomic-embed-code-v1.5 (самостоятельное развертывание), bge-code-v1 как запасной вариант
- Разреженный индекс: Tantivy (Rust) с BM25F, взвешивание по полям для имени символа относительно тела
- Векторная БД: Qdrant 1.12 с гибридным поиском или pgvector + pgvectorscale для проектов с объёмом до 50 миллионов векторов
- Модель аннотирования чанков: Claude Haiku 4.5 или Gemini 2.5 Flash с кэшированием промптов
- Переранжировщик: Cohere rerank-3 или bge-reranker-v2-gemma-2b с самостоятельным развертыванием
- Оркестрация: LlamaIndex Workflows для сбора данных, LangGraph для агента запросов
- Синтезатор: Claude Sonnet 4.7 (1M контекст) с кэшированием промптов
- Граф символов: Neo4j (управляемый) или kuzu (встраиваемый) для рёбер импортов и вызовов
- Наблюдаемость: трассировки Langfuse для каждого шага извлечения и синтеза

## Реализация

1. **Обходчик для сбора данных.** При каждом push-хуке проходите по git-истории. Собирайте изменённые файлы. Для каждого файла выполняйте парсинг через tree-sitter, извлекайте узлы функций и классов с их полным исходным диапазоном. Генерируйте записи чанков `{repo, path, start_line, end_line, symbol, body}`.

2. **Аннотатор чанков.** Группируйте чанки в пакетные вызовы Haiku 4.5 с кэшированием системного прелюда. Промпт: «Суммируйте эту функцию одним предложением, указав её публичный контракт и побочные эффекты». Сохраняйте аннотацию вместе с чанком.

3. **Пул эмбеддингов.** Две параллельные очереди: плотные эмбеддинги (Voyage-code-3, пакет по 128) и аннотации (та же модель, но по строке аннотации). Записывайте векторы в Qdrant с полезной нагрузкой `{repo, path, start_line, end_line, symbol, kind}`.

4. **BM25-индекс.** Индекс Tantivy с взвешиванием по полям: вес имени символа — 4, вес тела символа — 1, вес аннотации — 2. Это позволяет обрабатывать как запросы «найти функцию с именем X», так и «найти функцию, которая делает X».

5. **Граф символов.** Для каждого чанка записывайте рёбра: импорты (данный файл использует символ Y из репозитория Z), вызовы (данная функция вызывает метод M у класса C), наследование. Сохраняйте в kuzu. Используйте при запросах для расширения поиска за пределами репозитория.

6. **Агент запросов.** LangGraph с тремя узлами. `retrieve` запускает плотный и BM25 поиск параллельно, дедуплицирует по (repo, path, symbol). `rerank` выполняет кросс-энкодер для top-50 и оставляет top-10. `synth` вызывает Claude Sonnet 4.7 с отранжированными чанками в контексте, кэширует системный промпт и требует цитирование файл:строка.

7. **Обеспечение цитирования.** Анализируйте вывод модели; любое утверждение без якоря `(repo/path:start-end)` помечается для повторного запроса или отбрасывается. Возвращайте пользователю только ответ с цитированием.

8. **Инкрементальная переиндексация.** При каждом вебхуке вычисляйте дифференциал на уровне символов. Переэббеддингивайте только те чанки, чей текст изменился. Пересчитывайте рёбра символов для чанков, у которых изменились импорты. Измеряйте: push с 50 файлами переиндексируется менее чем за 60 секунд для пула в 2 миллиона строк кода.

9. **Оценка.** Разметьте 100 кросс-репозиторных вопросов с эталонными ответами файл:строка. Измеряйте MRR@10, nDCG@10, достоверность цитирования (долю утверждений с проверяемыми якорями) и латентность p50/p99.

## Отправка

Результатом является навык `outputs/skill-codebase-rag.md`. При наличии корпуса репозиториев он запускает конвейер сбора данных, гибридный индекс и агент запросов, возвращая ответ с цитированием для любого кросс-репозиторного вопроса. Критерии оценки:

## Упражнения

1. Замените Voyage-code-3 на nomic-embed-code с самостоятельным развертыванием. Измерьте разницу MRR@10. Определите, сокращается ли разрыв при включённом переранжировании.

2. Внедрите 20% сгенерированного кода (шаблонов, produced LLM) в корпус и повторите оценку. Наблюдайте за ухудшением результатов поиска. Добавьте флаг «generated» в полезную нагрузку и понизьте вес таких результатов.

3. Сравните гибридный поиск Qdrant и pgvector + pgvectorscale для размера вашего корпуса. Определите p99 при размере пакета 1.

4. Добавьте проверку дрифта на основе выборки: еженедельно повторяйте оценку из 100 вопросов. Отправляйте оповещение при падении MRR@10 более чем на 5%.

5. Расширьте систему для кросс-язычного разрешения символов: функция на Python, которая вызывает сервис на Go через gRPC. Используйте граф символов для связывания этих элементов.

## Ключевые термины

| Термин | Как говорят люди | Что это означает на самом деле |
|------|-----------------|------------------------|
| Чанкинг с учётом AST | «Разбиение на уровне функций» | Разделение кода на границах узлов tree-sitter вместо фиксированных окон токенов |
| Гибридный поиск | «Плотный + разреженный» | Параллельное выполнение BM25 и векторного поиска, объединение top-k, переранжирование |
| Переранжирование кросс-энкодером | «Второй этап ранжирования» | Модель, которая совместно оценивает каждую пару (запрос, кандидат), более точная, чем косинусное сходство |
| Кэширование промптов | «Кэшированный системный промпт» | Функция Claude/OpenAI в 2026 году, снижающая стоимость повторяющихся префиксных токенов до 90% |
| Граф символов | «Кодовый граф» | Рёбра для импортов, вызовов, наследования файлов и репозиториев |
| Достоверность цитирования | «Процент обоснованных ответов» | Доля утверждений, которые пользователь может проверить, щёлкнув на якорь и прочитав указанный фрагмент |
| Инкрементальная переиндексация | «Время от push до поиска» | Реальное время от git push до того, как изменённые символы становятся доступными для запросов |
