# Выпускной проект 01 — Кодирующий агент с нативной терминальной интеграцией

> К 2026 году облик кодирующего агента (coding agent) окончательно определился: терминальная обвязка (harness), консистентный план, изолированный инструментарий, цикл планирования, действия, наблюдения и восстановления. Claude Code, Cursor 3 и OpenCode выглядят одинаково на расстоянии пятидесяти футов. В данном выпускном проекте (capstone) предлагается собрать такой агент с нуля — от CLI до пул-реквеста — и сравнить его с mini-swe-agent и Live-SWE-agent на SWE-bench Pro. Вы убедитесь, что сложная часть заключается не в вызове модели, а в инструментальном цикле, песочнице (sandbox) и ценовом лимите при запуске на 50 ходов.

**Тип:** Выпускной проект
**Языки:** TypeScript / Bun (обвязка), Python (скрипты оценки)
**Предварительные требования:** Фаза 11 (инженерия LLM), Фаза 13 (инструменты и протоколы), Фаза 14 (агенты), Фаза 15 (автономные системы), Фаза 17 (инфраструктура)
**Задействованные фазы:** P0 · P5 · P7 · P10 · P11 · P13 · P14 · P15 · P17 · P18
**Время:** 35 часов

## Проблема

Кодирующие агенты стали доминирующей категорией ИИ-приложений в 2026 году. Claude Code (Anthropic), Cursor 3 с Composer 2 и Agent Tabs (Cursor), Amp (Sourcegraph), OpenCode (112 000 звёзд на GitHub), Factory Droids и Google Jules представляют вариации одной и той же архитектуры: терминальная обвязка, изолированный инструментарий с разграничением прав, песочница и цикл «план — действие — наблюдение», построенный на базе фронтовой модели. Фронт узкий — Live-SWE-agent достиг 79,2% на SWE-bench Verified с Opus 4.5 — но инженерная практика широка. Большинство режимов сбоя — это не ошибки модели. Это нестабильность инструментального цикла, отравление контекста, неуправляемый расход токенов и деструктивные операции с файловой системой.

Невозможно понять этих агентов со стороны. Нужно собрать одного из них, наблюдать, как цикл падает на 47-м ходе, когда ripgrep возвращает 8 МБ совпадений, и заново реализовать уровень усечения. В этом и заключается суть данного выпускного проекта.

## Концепция

У обвязки четыре поверхности. **План** поддерживает объект состояния в стиле TodoWrite, который модель перезаписывает на каждом ходе. **Действие** диспетчеризирует вызовы инструментов (чтение, редактирование, выполнение, поиск, git). **Наблюдение** фиксирует stdout / stderr / коды возврата, выполняет усечение и передаёт сводку обратно. **Восстановление** обрабатывает ошибки инструментов без истощения окна контекста и бесконечного зацикливания. В 2026 году добавился ещё один элемент — **хуки** (hooks). `PreToolUse`, `PostToolUse`, `SessionStart`, `SessionEnd`, `UserPromptSubmit`, `Notification`, `Stop` и `PreCompact` — настраиваемые точки расширения, через которые оператор внедряет политику, телеметрию и защитные механизмы.

Песочница — это E2B или Daytona. Каждая задача выполняется в свежем devcontainer с git worktree, смонтированным на чтение и запись. Обвязка никогда не обращается к файловой системе хоста. Worktree уничтожается по завершении — успешном или нет. Управление стоимостью осуществляется на трёх уровнях: лимит токенов на ход, денежный бюджет на сессию и жёсткий предел ходов (обычно 50). Слой наблюдаемости строится на OpenTelemetry спанах с семантическими соглашениями GenAI и доставляется в самостоятельно размещённый Langfuse.

## Архитектура

```
  CLI пользователя  ->  обвязка (Bun + Ink TUI)
                           |
                           v
               цикл план / действие / наблюдение  <--->  Claude Sonnet 4.7 / GPT-5.4-Codex / Gemini 3 Pro
                           |                              (через OpenRouter, модель-agnostic)
                           v
               диспетчер инструментов (MCP StreamableHTTP клиент)
                           |
          +----------------+----------------+----------+
          v                v                v          v
       read/edit        ripgrep        tree-sitter   git/run
          |                |                |          |
          +----------------+----------------+----------+
                           |
                           v
               E2B / Daytona песочница  (worktree изолирован)
                           |
                           v
               хуки: Pre/Post, Session, Prompt, Compact
                           |
                           v
               OpenTelemetry -> Langfuse (спаны, токены, $)
                           |
                           v
               PR через GitHub-приложение
```

## Технологический стек

- Среда выполнения обвязки: Bun 1.2 + Ink 5 (React-in-terminal)
- Доступ к моделям: единый API OpenRouter с Claude Sonnet 4.7, GPT-5.4-Codex, Gemini 3 Pro, Opus 4.5 (для наиболее сложных задач)
- Транспорт инструментов: Model Context Protocol StreamableHTTP (редакция MCP 2026)
- Песочница: E2B песочницы (JS SDK) или Daytona devcontainers
- Поиск по коду: подпроцесс ripgrep, парсеры tree-sitter для 17 языков (предкомпилированные)
- Изоляция: `git worktree add` для каждой задачи, очистка при успехе / неудаче
- Оценочный стенд: SWE-bench Pro (проверенный поднабор) + Terminal-Bench 2.0 + собственный отложенный набор из 30 задач
- Наблюдаемость: SDK OpenTelemetry с семантическими соглашениями `gen_ai.*` → самостоятельно размещённый Langfuse
- Публикация PR: GitHub App с детализированным токеном, ограниченным целевым репозиторием

## Сборка

1. **TUI и цикл команд.** Инициализируйте проект Bun с Ink. Принимайте `agent run <repo> "<task>"`. Выводите раздельный вид: панель плана (сверху), поток вызовов инструментов (посередине), бюджет токенов (снизу). Добавьте отмену по Ctrl-C, которая перед выходом вызывает хук `SessionEnd`.

2. **Состояние плана.** Определите типизированную схему TodoWrite (элементы pending / in_progress / done с примечаниями). Модель перезаписывает всё состояние целиком на каждом ходе в виде вызова инструмента — не допускайте инкрементальных мутаций. Сохраняйте план в `.agent/state.json`, чтобы после сбоя можно было возобновить выполнение.

3. **Инструментарий.** Определите шесть инструментов: `read_file`, `edit_file` (с предварительным просмотром diff), `ripgrep`, `tree_sitter_symbols`, `run_shell` (с таймаутом), `git` (status / diff / commit / push). Экспортируйте их через MCP StreamableHTTP, чтобы обвязка была транспортно-независимой. Каждый инструмент возвращает усечённый вывод (лимит 4000 токенов на вызов).

4. **Обёртка песочницы.** Каждая задача порождает E2B песочницу. `git worktree add -b agent/$TASK_ID` создаёт свежую ветку. Все вызовы инструментов выполняются внутри песочницы. Файловая система хоста недоступна.

5. **Хуки.** Реализуйте все восемь типов хуков 2026 года. Подключите как минимум четыре пользовательских хука: (а) `PreToolUse` — защита от деструктивных команд, блокирующая `rm -rf` за пределами worktree; (б) `PostToolUse` — учёт токенов; (в) `SessionStart` — инициализация бюджета; (г) `Stop` — запись финального трассировочного пакета.

6. **Оценочный цикл.** Клонируйте поднабор из 30 задач SWE-bench Pro на Python. Запустите обвязку против каждой. Сравните результаты с mini-swe-agent (минимальный базовый уровень) по pass@1, количеству ходов на задачу и стоимости задачи. Запишите результаты в `eval/results.jsonl`.

7. **Управление стоимостью.** Жёсткие ограничения: 50 ходов, 200 000 контекста, $5 за задачу. Хук `PreCompact` обобщает более ранние ходы в блок предыдущего состояния на отметке 150 000, освобождая место для новых наблюдений без потери плана.

8. **Публикация PR.** При успешном завершении последний шаг — `git push` + вызов GitHub API, открывающий PR с планом и сводкой diff в теле.

## Отправка

Результатом проекта является навык, размещённый в `outputs/skill-terminal-coding-agent.md`. Получив путь к репозиторию и описание задачи, он выполняет полный цикл план — действие — наблюдение в песочнице и возвращает URL пул-реквеста вместе с трассировочным пакетом. Критерии оценки данного выпускного проекта:

| Вес | Критерий | Способ измерения |
|:-:|---|---|
| 25 | SWE-bench Pro pass@1 относительно базового уровня | Ваша обвязка vs mini-swe-agent на 30 сопоставленных задачах на Python |
| 20 | Чёткость архитектуры | Разделение план/действие/наблюдение, поверхность хуков, схема инструментов — по сравнению со структурой Live-SWE-agent |
| 20 | Безопасность | Тесты на побег из песочницы, запросы разрешений, защита от деструктивных команд проходят.red-тест |
| 20 | Наблюдаемость | Полнота трассировки (100% вызовов инструментов в спанах), учёт токенов по ходам |
| 15 | UX разработчика | Холодный запуск < 2 с, восстановление после сбоя возобновляет план, Ctrl-C корректно отменяет выполняющийся инструмент |
| **100** | | |

## Упражнения

1. Замените базовую модель Claude Sonnet 4.7 на Qwen3-Coder-30B, развёрнутую на vLLM. Сравните pass@1 и стоимость задачи. Определите, в каких сценариях открытая модель уступает.

2. Добавьте субагента «рецензент», который читает diff перед публикацией PR и может запросить цикл доработки. Измерьте, снижают ли ложноположительные рецензии частоту прохождения SWE-bench ниже базового уровня одного агента (подсказка: обычно да).

3. Нагрузочный тест песочницы: напишите задачу, пытающуюся выполнить `curl` к внешнему URL, и задачу, записывающую за пределы worktree. Убедитесь, что обе заблокированы хуком PreToolUse. Зарегистрируйте попытки.

4. Реализуйте обобщение в `PreCompact` с использованием меньшей модели (Haiku 4.5). Измерьте, насколько снижается точность плана при 3-кратном сжатии.

5. Замените транспорт MCP StreamableHTTP на stdio. Измерьте время холодного запуска и задержку вызова. Выберите оптимальный вариант для использования только на локальной машине.

## Ключевые термины

| Термин | Что говорят | Что это значит на самом деле |
|------|-----------------|------------------------|
| Обвязка (harness) | «Агентный цикл» | Код вокруг модели, который диспетчеризирует инструменты, поддерживает состояние плана и соблюдает бюджеты |
| Хук (hook) | «Обработчик событий агента» | Пользовательский скрипт, выполняемый на одном из восьми событий жизненного цикла обвязкой |
| Worktree | «Git-песочница» | Связанная git-инсталляция по отдельному пути; уничтожаемая без затрагивания основного клонирования |
| TodoWrite | «Состояние плана» | Типизированный список элементов pending/in_progress/done, который модель перезаписывает на каждом ходе |
| StreamableHTTP | «Транспорт MCP» | Редакция MCP 2026: долгоживущее HTTP-соединение с двунаправленной потоковой передачей; заменяет SSE |
| Лимит токенов (token ceiling) | «Бюджет контекста» | Ограничение на ввод+вывод токенов за ход или за сессию; запускает компрессию или завершение |
| pass@1 | «Частота прохождения с первой попытки» | Доля задач SWE-bench, решённых при первом запуске без повторных попыток и подглядывания в тестовый набор |
