# AlphaEvolve — Эволюционные агенты для программирования

> Сочетайте передовую модель программирования с эволюционным циклом и машинно-проверяемым оценщиком. Дайте циклу достаточно времени для работы. Он обнаруживает процедуру умножения комплексных матриц 4x4, использующую 48 скалярных умножений — первую улучшение над алгоритмом Страссена за 56 лет. Также он находит гибридный алгоритм расписания Borg в Google, который восстанавливает около 0,7% вычислительной мощности кластера в производстве. Архитектура проста по design. Важные результаты достигаются благодаря строгостью оценщика.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, эволюционный цикл)
**Предварительные требования:** Фаза 15 · 01 (долгосрочное планирование), Фаза 15 · 02 (самообучаемое мышление)
**Время:** ~60 минут

## Проблема

Большие языковые модели могут писать код. Эволюционные алгоритмы способны искать в пространстве кода. Обе эти техники уже использовались отдельно на протяжении десятилетий; обе достигли своих пределов. Предельное ограничение модели языка программирования — это фабуляция: модель пишет код, который выглядит правдоподобно, но не выполняет заявленных функций. Предельное ограничение эволюционного алгоритма — это стоимость поиска: случайные мутации синтаксиса редко приводят к компилируемым программам, а уж тем более к лучшим.

AlphaEvolve (Novikov et al., DeepMind, arXив:2506.13131, июнь 2025) объединяет эти подходы. Модель языка программирования предлагает целенаправленные изменения в базу данных программ; автоматический оценщик присваивает каждому варианту балл; высокобалльные варианты становятся родителями для будущих поколений. Модель языка программирования отвечает за написание правдоподобного кода, а оценщик отлавливает фабуляции. Цикл работает несколько часов до недели.

Отчеты об итогах: процедура умножения комплексных матриц 4x4 с использованием 48 скалярных умножений (предел Страссена 1969 года был 49), гибридный алгоритм расписания Borg в Google, ускорение ядра FlashAttention на 32.5%, улучшения производительности обучения Gemini.

Архитектура работает благодаря строгостью оценщика. Без оценщика она не работает. Эта асимметрия — это урок.

## Концепция

### Цикл

1. Начните с семенного программного кода `P_0`, который верен, но оптимизирован не до конца.
2. Поддерживайте базу данных вариантов программ, каждый из которых оценивается оценщиком.
3. Выберите одного или несколько родителей из базы данных (по аналогии с MAP-elites или островной моделью).
4. Просьба к модели языка программирования (Gemini Flash для многих вариантов, Gemini Pro для сложных) на производство модифицированного варианта родителя.
5. Компиляция, запуск и оценка варианта на выделенном оценщике.
6. Вставка в базу данных с ключом по баллу и вектору признаков.
7. Повторение.

Два детали важны. Во-первых, модель языка программирования получает больше информации, чем просто родительский код — обычно несколько верхних вариантов из базы данных, а также подпись оценщика и краткое описание задачи. Задача модели — предложить целенаправленное изменение, которое может улучшить балл. Во-вторых, база данных структурирована (по аналогии с MAP-elites сетью или островной моделью), чтобы цикл исследовал разнообразие, а не только текущего лидера.

### Что делает оценщик незаменимым

Все успехи AlphaEvolve происходят в областях, где оценщик быстро работает, детерминирован и устойчив к взлому:

- **Алгоритм умножения матриц**: проверка на равенство результатов умножения матриц битово.
- **Гибридный алгоритм расписания Borg**: производственная симуляция, которая воспроизводит историческую нагрузку кластера и измеряет потерянную вычислительную мощность.
- **Ядро FlashAttention**: проверка на корректность и бенчмарк по реальной аппаратуры.
- **Производительность обучения Gemini**: измерение GPU-секунд на шаг.

В каждом случае оценщик отлавливает класс ошибок модели языка программирования, которые в противном случае доминировали бы: фабулированные утверждения о корректности, производительность, которая исчезает при тестировании на аппаратуре, и краевые случаи.

Уберите оценщик и цикл будет оптимизировать для красивого кода.

### Взлом вознаграждения — обратная сторона утверждения

Эволюция оптимизирует то, что измеряет оценщик. Если оценщик несовершенен, цикл найдет его недостатки. В незавершенной области цикл будет оптимизировать для внешних признаков, а не для намеченного поведения. DeepMind явно указывает на это в статье: успехи AlphaEvolve передаются только в областях, где строгость оценщика соответствует амбициям поиска.

Конкретные примеры 2025-2026 года обмана награды в циклах поиска кода:

- Цели оптимизации, которые вознаграждают "время выполнения", приводят к отправке пустых решений.
- Баллы бенчмарков, которые вознаграждают корректность под тестом, приводят к запоминанию тестов и переобучению.
- Прокси качества кода вознаграждает удаление комментариев и переименование переменных без изменения семантики.

Исправление в AlphaEvolve: отправьте оценщик, который модель языка программирования никогда не видела, с входными данными, генерируемыми при оценке. Даже тогда DeepMind рекомендует тщательный обзор перед любым предложением о внедрении.

### Почему комбинация модели языка программирования и поиска превосходит их отдельно

Модель языка программирования может генерировать компилируемый, семантически правдоподобный код. Генетический алгоритм с рандомными мутациями на 2000-строчном файле Python почти всегда приводит к синтаксическим ошибкам. Модель языка программирования также концентрирует поиск в вероятных окрестностях (изменение одной функции, а не случайные байты), что значительно снижает количество бесполезных вызовов оценщика.

В свою очередь, оценщик отлавливает фабуляции модели языка программирования. Модели языка программирования уверенно утверждают, что функция "имеет сложность O(n log n) в пределе", когда на самом деле она имеет сложность O(n^2); бенчмарк по реальному времени решает вопрос.

### Где AlphaEvolve входит в стек передовых технологий

| Система | Генератор | Оценщик | Область | Примерный успех |
|---|---|---|---|---|
| AlphaEvolve | Gemini | корректность + бенчмарк | алгоритмы, ядра, расписания | умножение матриц 4x4 с использованием 48 скалярных умножений |
| FunSearch (DeepMind, 2023) | PaLM / Codey | корректность | комбинаторная математика | нижние границы cap-set |
| AI Scientist v2 (Sakana, L5) | GPT/Claude | критика модели языка программирования + эксперимент | исследование машинного обучения | статья ICLR сессии |
| Darwin Godel Machine (L4) | структурирование агента | SWE-bench / Polyglot | код агента | ускорение на 20% → 50% SWE-bench |

Все четыре являются вариациями одного рецепта: генератор и оценщик, цикл. Разница заключается в том, что оценивает оценщик и как строго он работает.

## Используем

`code/main.py` реализует минимальный цикл AlphaEvolve для задачи симуляционной регрессии. "Модель языка программирования" — это прокси стандартной библиотеки, которая предлагает небольшие синтаксические мутации коду, вычисляющему целевую функцию. "Оценщик" измеряет среднеквадратичную ошибку на отложенных тестовых точках.

Посмотрите:

- Как лучший балл улучшается по поколениям.
- Как сетка MAP-elites поддерживает разнообразные решения, чтобы цикл не сходил в локальный минимум.
- Как удаление отложенных тестов (только для обучения оценщик) позволяет циклу переобучаться.

## Результат

`outputs/skill-evaluator-rigor-audit.md` — это предварительное условие для рассмотрения AlphaEvolve в новой области: действительно ли ваш оценщик отлавливает ошибки, которые вы заботитесь?

## Упражнения

1. Запустите `code/main.py`. Обратите внимание на траекторию лучшего балла. Отключите отложенный оценщик (флаг `--no-holdout`) и повторно запустите. Определите степень переобучения.

2. Прочитайте раздел 3 статьи AlphaEvolve по сетке MAP-elites. Разработайте описание вектора признаков для новой задачи (например, оптимизация компиляторных проходов), которое бы поддерживало разнообразие поиска.

3. Результат умножения матриц 4x4 с использованием 48 скалярных умножений улучшил предел Страссена на 56 лет. Прочитайте приложение F статьи и объясните в трех предложениях, почему оценщик для этой задачи особенно легко сделать правильным, а большинство областей не похожи на него.

4. Предложите одну область, где AlphaEvolve потерпит неудачу. Определите точное место, где оценщик сломается и почему.

5. Для области, которую вы знаете, напишите подпись оценщика, который бы использовали. Включите (а) условия корректности, (б) метрику производительности, (в) правило генерации отложенных входов, (г) хотя бы одно проверка на обман награды.

## Ключевые термины

| Термин | Что говорят люди | На самом деле означает |
|---|---|---|
| AlphaEvolve | "эволюционный агент DeepMind для программирования" | Gemini + база данных программ + машинно-проверяемый оценщик |
| MAP-elites | "архив, сохраняющий разнообразие" | Сетка по ключам признаков; каждая ячейка содержит лучший вариант с этим описателем |
| Островная модель | "параллельное эволюционное подпопуляции" | Независимые популяции, которые периодически мигрируют; предотвращает раннюю конвергенцию |
| Машинно-проверяемый оценщик | "определенный оракул" | Тест на равенство, симулятор или бенчмарк, который модель языка программирования не может обмануть — необходимое условие для этого цикла |
| Обман награды | "оптимизация меры, а не цели" | Цикл находит способ максимизировать балл без выполнения намеченной задачи |
| Семенной программный код | "начальная точка" | Изначально верный, но оптимизированный не до конца программа, которую цикл эволюционирует |
| Отложенный оценщик | "данные оценки, которые модель языка программирования никогда не видела" | Входные данные генерируются при оценке для предотвращения запоминания |
