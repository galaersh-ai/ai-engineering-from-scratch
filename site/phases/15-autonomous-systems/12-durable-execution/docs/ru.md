# Долгосрочные фоновые агенты: постоянное исполнение

> Продакшен-агенты с долгосрочным горизонтом не работают в `while True`. Каждый вызов LLM становится активностью с чекпоинтом, повтором и воспроизведением. Интеграция Temporal с OpenAI Agents SDK вышла в GA в марте 2026. Claude Code Routines (Anthropic) запускает запланированные вызовы Claude Code без постоянного локального процесса. Сессии приостанавливаются при вводе человека, переживают развёртывания и возобновляются с последнего чекпоинта, привязанного к `thread_id`. За новой эргономикой стоит старый паттерн — оркестрация воркфлоу — с одним новым входом: вызовы LLM как недетерминированные активности, которые должны быть детерминированно воспроизведены при восстановлении.

**Тип:** Учебный материал
**Языки:** Python (стандартная библиотека, минималистичный конечный автомат постоянного исполнения)
**Предварительные требования:** Фаза 15 · 10 (Режимы разрешений), Фаза 15 · 01 (Долгосрочные агенты)
**Время:** ~60 минут

## Проблема

Рассмотрим агента, работающего четыре часа. Он вызывает три инструмента, дважды запрашивает пользователя и делает сорок вызовов LLM. На середине хост, на котором он работает, перезагружается. Что происходит?

- В наивном цикле `while True`: всё потеряно. Запуск.restart с нуля. Три вызова инструментов (с реальными побочными эффектами) выполняются снова. Пользователь снова запрашивается для вещей, которые уже одобрил. Сорок вызовов LLM billed повторно.
- С постоянным исполнением: запуск возобновляется с последнего чекпоинта. Уже завершённые активности не выполняются заново; их результаты воспроизводятся из постоянного журнала. Пользователь не повторно одобряет то, что уже одобрил. Уже выполненные вызовы LLM не billed повторно.

Это тот же паттерн, который движки воркфлоу поставляют уже десятилетие (Temporal, Cadence, Cherami от Uber). Новизна в том, что вызовы LLM теперь являются видом активности — недетерминированными, дорогими, с побочными эффектами — и они вписываются в этот паттерн чисто.

Главная тема урока: надёжность долгосрочных агентов деградирует (METR наблюдает «деградацию на 35 минут» — частота успеха падает примерно квадратично с горизонтом). Постоянное исполнение позволяет запускам быть длиннее, чем профиль надёжности поддерживает, что является новым способом безопасно завершиться, если дизайн правильный, и небезопасно, если дизайн неправильный.

## Концепция

### Активности, воркфлоу и воспроизведение

- **Воркфлоу**: детерминированный код оркестрации. Определяет последовательность активностей, ветвления, ожидания. Должен быть детерминированным, чтобы его можно было воспроизвести из журнала событий без неожиданных расхождений.
- **Активность**: недетерминированная, потенциально завершающаяся неудачей единица работы. Вызов LLM, вызов инструмента, запись файла, HTTP-запрос. Каждая активность логируется со своими входами и (после завершения) выходами.
- **Журнал событий**: постоянное хранилище. Каждое начало, завершение, сбой, повтор активности и каждое решение воркфлоу записываются.
- **Воспроизведение**: при восстановлении код воркфлоу запускается заново с начала; каждая уже завершённая активность возвращает свой залогированный результат без повторного выполнения. Только активности, которые не были завершены, фактически запускаются.

Это та же структура, что и повторный рендер React через виртуальный DOM, или Git, восстанавливающий рабочее дерево из коммитов. Детерминизм в оркестраторе — это то, что делает постоянство дешёвым.

### Почему вызовы LLM вписываются в паттерн

Вызовы LLM:
- Недетерминированы (температура > 0; даже температура 0 дрейфует между версиями модели).
- Дороги (деньги и латентность).
- Потенциально завершаются неудачей (ограничения частоты, таймауты).
- Имеют побочные эффекты (если вызывают инструменты).

Это в точности профиль активности. Обёртывание каждого вызова LLM как активности даёт вам повтор с экспоненциальным отступом, чекпоинты через перезапуски и воспроизводимый трейс для отладки.

### Чекпоинты, привязанные к `thread_id`

LangGraph, Microsoft Agent Framework, Cloudflare Durable Objects и Claude Code Routines сошлись на одной форме API: `thread_id` (или эквивалент) идентифицирует сессию; каждый переход состояния сохраняется в бэкенд (PostgreSQL по умолчанию, SQLite для разработки, Redis для кэша); возобновление читает последний чекпоинт.

Выбор бэкенда важен:

- **PostgreSQL**: постоянный, запросимый, переживает развёртывания. По умолчанию для LangGraph.
- **SQLite**: только для локальной разработки; теряет данные через хосты.
- **Redis**: быстрый, но эфемерный, если не настроен AOF/снимок.
- **Cloudflare Durable Objects**: прозрачно распределённый; ограничен уникальным ключом; живёт часы-недели.

### Ввод человека как состояние первого класса

«Предложить, затем выполнить» (Урок 15) требует постоянного состояния «ожидание человека». Воркфлоу приостанавливается, внешняя очередь удерживает ожидающий запрос, а одобрение возобновляет именно с этой точки. Без постоянства это best-effort; с ним, ночное одобрение прибывает, и воркфлоу продолжается утром.

### Деградация на 35 минут

METR наблюдал, что каждый измеренный класс агентов показывает деградацию надёжности за пределами ~35 минут непрерывной работы. Удвоение длительности задачи примерно утраивает частоту отказов. Постоянное исполнение не исправляет это; оно позволяет запускаться дольше, чем профиль надёжности поддерживает. Безопасный паттерн — комбинировать постоянство с чекпоинтами, требующими свежего HITL при повторном входе, и с бюджетными аварийными выключателями (Урок 13), ограничивающими общее вычисление независимо от реального времени.

### Когда постоянное исполнение — неправильный ответ

- Запуски короче нескольких минут без ввода человека. Накладные расходы > выгоды.
- Строго только чтение информации.
- Задачи, где корректность требует end-to-end внутри одного контекстного окна (некоторые задачи рассуждений; некоторые одноразовые генерации).

## Применение

`code/main.py` реализует минималистичный движок постоянного исполнения на стандартной библиотеке Python. Он поддерживает:

- Декоратор `@activity`, логирующий входы и выходы в JSON-журнал событий.
- Функцию воркфлоу, последовательно выполняющую активности.
- Функцию `run_or_replay(workflow, event_log)`, воспроизводящую завершённые активности без повторного выполнения.

Драйвер моделирует трёхактивностный воркфлоу, падает на середине и показывает (а) наивный повтор, выполняющий всё, versus (b) воспроизведение, запускающее только недостающую активность.

## Доработка

`outputs/skill-durable-execution-review.md` проверяет предложенный запуск долгосрочного агента на корректную форму постоянного исполнения: активности, детерминизм, бэкенд чекпоинтов, состояние ввода человека и политика HITL при возобновлении.

## Упражнения

1. Запустите `code/main.py`. Наблюдайте разницу в количестве выполнений активностей между наивным повтором и воспроизведением. Измените точку падения и покажите, что количество воспроизведений меняется соответствующе.

2. Преобразуйте игрушечный движок для явного использования `thread_id`. Симулируйте две параллельные сессии, использующие один движок, и подтвердите, что их журналы событий не пересекаются.

3. Возьмите одну активность в игрушечном движке. Введите недетерминизм (метку реального времени внутри решения воркфлоу). Продемонстрируйте расхождение при воспроизведении. Объясните, как реальные движки решают это (регистрация побочных эффектов, API `Workflow.now()`).

4. Прочитайте пост LangChain «Runtime behind production deep agents». Перечислите каждое состояние, которое рантайм сохраняет, и назовите, какой режим отказа каждое покрывает.

5. Спроектируйте политику чекпоинтов для 6-часовой автономной кодирующей задачи. Где вы делаете чекпоинты? Как выглядит возобновление после падения? Что требует свежего HITL?

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|---|---|---|
| Воркфлоу | «Скрипт агента» | Детерминированный код оркестрации; воспроизводим из журнала событий |
| Активность | «Шаг» | Недетерминированная единица (вызов LLM, вызов инструмента); логируется до и после |
| Журнал событий | «Хранилище» | Постоянная запись каждого перехода состояния |
| Воспроизведение | «Возобновление» | Повторный запуск воркфлоу; завершённые активности возвращают залогированные результаты без повторного выполнения |
| Чекпоинт | «Точка сохранения» | Сохранённое состояние, привязанное к thread_id; при возобновлении побеждает последний |
| thread_id | «Ключ сессии» | Идентификатор, ограничивающий постоянное состояние |
| Деградация на 35 минут | «Деградация надёжности» | METR: частота успеха падает ~квадратично с горизонтом |
| Недетерминизм | «Дрейф при воспроизведении» | Реальное время, случайность, выход LLM; должен быть зарегистрирован как побочный эффект |
