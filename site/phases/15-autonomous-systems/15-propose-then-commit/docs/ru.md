# HITL: предложить, затем выполнить

> Консенсус 2026 года по HITL конкретен. Это не «агент спрашивает, пользователь нажимает Одобрить». Это propose-then-commit: предложенное действие сохраняется в постоянное хранилище с ключом идемпотентности; представляется рецензенту с намерением, происхождением данных, затронутыми разрешениями, радиусом поражения и планом отката; выполняется только после положительного подтверждения; проверяется после выполнения, чтобы убедиться, что побочный эффект действительно произошёл. `interrupt()` от LangGraph с чекпоинтингом в PostgreSQL, `RequestInfoEvent` от Microsoft Agent Framework и `waitForApproval()` от Cloudflare реализуют одну и ту же форму. Канонический режим отказа — одобрение штампом: «Одобрить?» нажимается без проверки. Задокументированное средство смягчения — чеклист с вызовом и ответом.

**Тип:** Учебный материал
**Языки:** Python (стандартная библиотека, конечный автомат propose-then-commit с идемпотентностью)
**Предварительные требования:** Фаза 15 · 12 (Постоянное исполнение), Фаза 15 · 14 (Срабатывания)
**Время:** ~60 минут

## Проблема

Агент выполняет действие. Пользователь должен решить: одобрить или нет. Если решение мгновенное, это, вероятно, не проверка. Если решение структурированное, оно медленное, но надёжное. Инженерный вопрос — как сделать структурированную проверку путём наименьшего сопротивления.

Паттерн HITL 2023 года был синхронным запросом: «Агент хочет отправить письмо на X с текстом Y — одобрить?» Пользователь нажимает Одобрить. Все чувствуют, что система безопасна. На практике эта поверхность сильно штампуется: пользователи одобряют быстро, одобрения мало предсказывают, и когда агент ошибается, аудит-трейл показывает длинную историю одобрений, которые пользователь не помнит.

Паттерн 2026 года — propose-then-commit — переносит HITL на постоянный субстрат, прикрепляет структурированные метаданные и требует явного подтверждения. Каждый управляемый агентский SDK поставляет версию: `interrupt()` от LangGraph, `RequestInfoEvent` от Microsoft Agent Framework, `waitForApproval()` от Cloudflare. Имена API различаются; форма — нет.

## Концепция

### Конечный автомат propose-then-commit

1. **Предложить.** Агент производит предложенное действие. Сохраняется в постоянное хранилище (PostgreSQL, Redis, Durable Object). Включает:
   - намерение (почему агент это делает)
   - происхождение данных (какой источник привёл к этому предложению)
   - затронутые разрешения (какие области / файлы / конечные точки)
   - радиус поражения (что в худшем случае)
   - план отката (если выполнено, как это отменить)
   - ключ идемпотентности (уникальный для каждого предложения; повторная отправка возвращает ту же запись)
2. **Представить.** Рецензент видит предложение со всеми метаданными. Рецензент — человек (не агент, проверяющий сам себя).
3. **Выполнить.** Положительное подтверждение. Действие выполняется.
4. **Проверить.** После выполнения побочный эффект считывается обратно и подтверждается. Если шаг проверки завершается неудачей, система находится в известно-плохом состоянии и включается оповещение.

### Ключ идемпотентности

Без ключа идемпотентности повтор при транзиентном сбое может выполнить одобренное действие дважды. Конкретный пример: пользователь одобряет «перевести $100 с A на B». Сеть мерцает. Воркфлоу повторяется. Пользователь одобрил один раз, но перевод выполняется дважды. Ключ идемпотентности привязывает одобрение к одному уникальному побочному эффекту; второе выполнение — no-op.

Это тот же паттерн идемпотентности, который используют API Stripe и AWS. Его повторное использование для одобрений агентов явно указано в документации Microsoft Agent Framework.

### Постоянство: почему одобрения переживают процессы

Комната ожидания одобрений — это кусок состояния, которым агент не владеет. Воркфлоу приостанавливается (Урок 12). Когда одобрение приходит, воркфлоу возобновляется именно с этой точки. Поэтому LangGraph парит `interrupt()` с чекпоинтингом в PostgreSQL, а не просто с состоянием в памяти — одобрение через два дня всё ещё находит воркфлоу целым.

### Штампованные одобрения и средство смягчения «вызов и ответ»

UI по умолчанию для HITL (кнопки «Одобрить» / «Отклонить») производит быстрые одобрения без реальной проверки. Задокументированное средство смягчения: чеклист с вызовом и ответом, требующий положительных ответов на конкретные вопросы перед включением кнопки Одобрить. Конкретная форма:

- «Вы понимаете, какой ресурс это затрагивает? [ ]»
- «Вы проверили, что радиус поражения приемлем? [ ]»
- «У вас есть план отката, если это завершится неудачей? [ ]»

Не бюрократия ради бюрократии — функция принуждения. Рецензент, не able поставить галочки, либо запрашивает уточнение (эскалация), либо отказывается (безопасный дефолт). Исследования безопасности агентов Anthropic явно указывают чеклист-driven HITL как средство смягчения паттернов штампования.

### Что считается значимым

Не каждое действие требует propose-then-commit. Рекомендации 2026 года:

- **Значимые действия** (всегда HITL): необратимые записи, финансовые транзакции, исходящая коммуникация, изменения продакшен-базы данных, деструктивные операции файловой системы.
- **Обратимые действия** (иногда HITL): правки локальных файлов, изменения staging-окружения, обратимые записи с чётким откатом.
- **Чтения и проверки** (никогда HITL): чтение файла, перечисление ресурсов, вызов read-only API.

### Пост-действие проверка

«Коммит выполнился» — это не то же, что «побочный эффект произошёл». Сетевые разделения и состояния гонки могут породить воркфлоу, который думает, что он успешен, тогда как бэкенд не сохранил состояние. Шаг проверки повторно считывает целевой ресурс после коммита для подтверждения. Это тот же паттерн, что и транзакции баз данных с `RETURNING` или AWS `GetObject` после `PutObject`.

### Статья 14 EU AI Act

Статья 14 требует эффективного надзора человека для систем высокого риска в ЕС. «Эффективный» — не украшение. Регуляторный язык явно исключает паттерны штампования. Propose-then-commit с вызовом и ответом — это форма, выдерживающая проверку Статьёй 14 в документации по соответствию Microsoft Agent Governance Toolkit.

`code/main.py` реализует конечный автомат propose-then-commit на стандартной библиотеке Python. Постоянное хранилище — JSON-файл. Ключ идемпотентности — хеш от (thread_id, action_signature). ## Упражнения

1. Запустите `code/main.py`. Подтвердите, что повтор одобренного предложения использует постоянную запись и не выполняется повторно. Теперь измените ключ идемпотентности, добавив временную метку, и покажите, что повтор выполняется дважды.

2. Расширьте запись предложения полем `rollback`. Симулируйте выполнение, чей шаг проверки завершается неудачей. Покажите автоматический запуск отката.

3. Прочитайте документацию Microsoft Agent Framework по `RequestInfoEvent`. Определите одно поле метаданных, которое API включает, а игрушечный движок не имеет. Добавьте его и объясните, от чего оно защищает.

4. Спроектируйте чеклист вызова и ответа для конкретного действия (например, «опубликовать в публичный аккаунт Twitter»). Какие три вопроса должен ответить рецензент? Почему именно эти три?

5. Выберите один случай, где синхронный запрос «Одобрить?» был бы достаточным (постоянное хранилище не нужно). Объясните почему и назовите принимаемый класс риска.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|---|---|---|
| Propose-then-commit | «Двухфазное одобрение» | Сохранённое предложение + явное выполнение + проверка |
| Ключ идемпотентности | «Повторно-безопасный токен» | Уникальный для каждого предложения; второе выполнение — no-op |
| Происхождение данных | «Откуда взялось» | Конкретный исходный контент, приведший к предложению |
| Радиус поражения | «Худший случай» | Область действия, если действие пойдёт не так |
| Штампование | «Быстрое одобрение» | «Одобрить» нажимается без реальной проверки |
| Вызов и ответ | «Чеклист принуждения» | Рецензент должен явно подтвердить конкретные вопросы |
| RequestInfoEvent | «Примитив MS Agent Framework» | Постоянный HITL-запрос со структурированными метаданными |
| `interrupt()` / `waitForApproval()` | «Примитивы фреймворка» | LangGraph / Cloudflare эквиваленты одной формы |
