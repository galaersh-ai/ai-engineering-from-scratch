# Нагрузочное тестирование LLM API — Почему k6 и Locust дают ложные результаты

> Традиционные инструменты нагрузочного тестирования (load testing) не предназначены для потоковых ответов (streaming), переменной длины вывода, метрик на уровне токенов или насыщения GPU. Существуют две ловушки, в которые попадает большинство команда. Ловушка GIL: измерение на уровне токенов в Locust выполняет токенизацию в рамках GIL Python, что конкурирует с генерацией запросов при высокой одновременности; задержка токенизации раздувает отчётуемую межтокенную латентность (latency) — узким местом оказывается клиент, а не сервер. Ловушка однородности промптов: одинаковые промпты в цикле тестируют одну точку распределения токенов; реальный трафик имеет переменную длину и разнообразные совпадения префиксов. LLMPerf решает эту проблему с помощью `--mean-input-tokens` + `--stddev-input-tokens`. Соответствие инструментов в 2026 году: специализированные для LLM (GenAI-Perf, LLMPerf, LLM-Locust, guidellm) — для точности на уровне токенов; **k6 v2026.1.0** + **k6 Operator 1.0 GA (сентябрь 2025)** — поддерживают streaming, распределённое тестирование в Kubernetes через CRD TestRun/PrivateLoadZone, оптимальны для ворот CI/CD; Vegeta — насыщение с постоянной частотой на Go; Locust 2.43.3 — только с расширением LLM-Locust для streaming. Паттерны нагрузки: стационарный (steady-state), нарастающий (ramp), пиковый (spike, тестирование автоскейлинга), длительный (soak, утечки памяти).

**Тип:** Практическая работа
**Языки:** Python (стандартная библиотека, простой генератор реалистичных промптов + сборщик латентности)
**Предварительные требования:** Фаза 17 · 08 (Метрики инференса), Фаза 17 · 03 (Автоскейлинг GPU)
**Время:** ~75 минут

## Цели обучения

- Объяснить два антипаттерна (ловушка GIL, ловушка однородности промптов), из-за которых универсальные инструменты нагрузочного тестирования дают ложные результаты для LLM API.
- Выбрать инструмент для конкретной задачи: LLMPerf (бенчмарк-запуск), k6 + расширение для streaming (ворота CI), guidellm (масштабное синтетическое тестирование), GenAI-Perf (эталон NVIDIA).
- Спроектировать четыре паттерна нагрузки (steady, ramp, spike, soak) и назвать вид сбоя, который каждый из них выявляет.
- Построить реалистичное распределение промптов, используя среднее и стандартное отклонение количества входных токенов вместо фиксированной длины.

## Проблема

Вы провели нагрузочное тестирование LLM-эндпоинта с помощью k6 на 500 одновременных пользователей. Он выдержал. Вы развернули в продакшне. При 200 реальных пользователях сервис упал — P99 TTFT (Time To First Token) взлетел, GPU залипли.

Прошло две вещи. Во-первых, k6 отправил 500 одинаковых промптов — ваш когалескинг запросов и кеширование префиксов создали впечатление, что вы обрабатываете 500 параллельных декодов, хотя на деле обрабатывали один. Во-вторых, k6 не отслеживает межтокенную латентность (latency) потоковых ответов так, как это воспринимает человек; он видит одно HTTP-соединение, а не 500 токенов, прибывающих с переменными интервалами.

Нагрузочное тестирование (load testing) для LLM — это отдельная дисциплина.

## Концепция

### Ловушка GIL (Locust)

Locust использует Python и выполняет токенизацию на стороне клиента в рамках GIL. При высокой одновременности токенизатор оказывается в очереди за генерацией запросов. Отчётная межтокенная латентность (latency) включает задержку токенизации на стороне клиента. Вы думаете, что сервер медленный; на самом деле проблема в тестовом стенде.

Решение: расширение LLM-Locust переносит токенизацию в отдельные процессы, либо используйте тестовый стенд на компилируемом языке (k6, LLMPerf с использованием tokenizers.rs).

### Ловушка однородности промптов

Все известные инструменты нагрузочного тестирования позволяют настроить один промпт. В цикле из 10 000 итераций каждый раз отправляется один и тот же промпт. Сервер видит один и тот же префикс — попадания в кеш префиксов приближаются к 100%, пропускная способность выглядит превосходно.

Решение: сэмплируйте из распределения промптов. LLMPerf использует `--mean-input-tokens 500 --stddev-input-tokens 150` — разнообразные длины, разнообразное содержание.

### Четыре паттерна нагрузки

1. **Стационарный (steady-state)** — постоянная частота запросов (RPS) в течение 30–60 минут. Выявляет: регрессию базовых показателей производительности.
2. **Нарастающий (ramp)** — линейное увеличение RPS от 0 до целевого значения за 15 минут. Выявляет: точку отказа по ёмкости, аномалии прогрева.
3. **Пиковый (spike)** — резкое увеличение RPS в 3–10 раз на 2 минуты с последующим возвратом. Выявляет: латентность автоскейлинга, переполнение очереди, влияние холодного старта.
4. **Длительный (soak)** — стационарный режим в течение 4–8 часов. Выявляет: утечки памяти, дрейф пула соединений, переполнение систем наблюдаемости.

### Соответствие инструментов в 2026 году

**LLMPerf** (Anyscale) — Python, но токенизация на базе Rust. Среднее/стандартное отклонение промптов. Поддержка streaming. Лучший вариант по умолчанию для замеров производительности.

**NVIDIA GenAI-Perf** — эталон NVIDIA. Использует клиент Triton; комплексное покрытие метрик. Обратите внимание: его ITL (Inter-Token Latency) не включает TTFT; LLMPerf его включает. Два инструмента выдают разный TPOT для одного сервера.

**LLM-Locust** (TrueFoundry) — расширение Locust, исправляющее ловушку GIL. Привычный DSL Locust + метрики streaming.

**guidellm** — масштабное синтетическое бенчмаркинг.

**k6 v2026.1.0** + **k6 Operator 1.0 GA (сентябрь 2025)**:
- k6 (Go, компилируемый, без GIL) добавил метрики с поддержкой streaming.
- k6 Operator использует CRD TestRun / PrivateLoadZone для распределённого тестирования с нативной поддержкой Kubernetes.
- Лучший вариант для ворот CI/CD и тестирования SLA.

**Vegeta** — Go, проще, чем k6. Насыщение HTTP с постоянной частотой. Не учитывает специфику LLM, но подходит для тестирования шлюзов и лимитов частоты запросов.

**Locust 2.43.3** — стандартный. Имеет ловушку GIL для LLM. Только с расширением LLM-Locust.

### Ворота SLA в CI

Запускайте k6 на PR с:

- 30–50 итерациями каждая на базовой частоте RPS.
- Воротами: P50/P95 TTFT, доля ошибок 5xx < 5%, TPOT ниже порога.
- Срывайте сборку при нарушении.

### Реалистичное распределение промптов
