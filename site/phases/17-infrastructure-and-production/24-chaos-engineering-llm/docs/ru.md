# Хаос-инженерия (chaos engineering) для LLM в продакшене

> Хаос-инженерия для LLM — это отдельная дисциплина в 2026 году. Предварительные условия перед запуском экспериментов в продакшене: определённые SLI/SLO, наблюдаемость (observability) через трейсы, метрики и логи, автоматический откат, рантбуки, дежурная служба. Архитектура включает четыре плоскости: управления (планировщик экспериментов), целевую (сервисы, инфраструктура, хранилища данных), безопасности (защитные механизмы + аварийная остановка + фильтры трафика), наблюдательности (метрики + трейсы + логи), обратной связи (корректировка SLO). Защитные ограничения обязательны: оповещения о скорости сжигания (burn-rate) приостанавливают эксперименты, если суточное потребление бюджета ошибок превышает ожидаемое в 2 раза; окна подавления + корреляция по trace-ID устраняют шум оповещений. Регулярность: еженедельные небольшие канареечные эксперименты + пересмотр SLO; ежемесячные игровые дни (game day) + разбор инцидентов; ежеквартальный кросс-командный аудит устойчивости (resilience) + карта зависимостей. Эксперименты, специфичные для LLM: перегрузка памяти, сетевые сбои, простои провайдеров, некорректные промпты, штормы вытеснения из KV-кеша. Инструменты: Harness Chaos Engineering (рекомендации на основе ИИ, масштабирование радиуса воздействия, интеграция MCP-инструментов); LitmusChaos (CNCF); Chaos Mesh (CNCF, нативный для Kubernetes).

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, учебный запускальщик экспериментов)
**Предварительные условия:** Фаза 17 · 23 (SRE для ИИ), Фаза 17 · 13 (Наблюдаемость)
**Время:** ~60 минут

## Цели обучения

- Назвать пять предварительных условий хаос-инженерии (SLI/SLO, наблюдаемость, откат, рантбуки, дежурная служба) и объяснить, почему пропуск любого из них нарушает практику.
- Схематично представить четыре плоскости (управления, целевую, безопасности, наблюдательности) и петлю обратной связи в SLO.
- Перечислить пять экспериментов, специфичных для LLM (перегрузка памяти, сетевой сбой, простои провайдера, некорректный промпт, шторм вытеснения KV-кеша).
- Выбрать инструмент — Harness, LitmusChaos или Chaos Mesh — в зависимости от используемого стека.

## Проблема

Хаос-тестирование в традиционных стеках является устоявшейся практикой. Стеки с LLM добавляют новые типы отказов. Промпт на 4 000 токенов со специальным символом-отравителем задерживает токенизатор на 12 секунд. Вышестоящий провайдер возвращает 429; ваш шлюз выполняет повторные попытки; ваш сервис падает с переполнением памяти из-за усиленной повторными попытками нагрузки на параллельную обработку. Шторм вытеснения из KV-кеша при пиковом трафике вызывает каскадное повторное заполнение, которое перегружает вычислительные ресурсы.

Ничего из этого не обнаруживается в модульных тестах. Хаос-инженерия — это способ обнаружить эти проблемы до того, как они обнаружат пользователи.

## Концепция

### Предварительные условия

Не запускайте хаос-инженерию в продакшене без:

1. **SLI/SLO** — определённых индикаторов и целей уровня обслуживания.
2. **Наблюдаемости** — трейсы, метрики, логи, подключённые к панелям мониторинга.
3. **Автоматического отката** — политика отката по флагу из Фазы 17 · 20.
4. **Рантбуков** — структурированных, из Фазы 17 · 23.
5. **Дежурной службы** — кто-то должен реагировать.

Отсутствие любого из них означает, что хаос-инженерия превращается в реальный инцидент.

### Четыре плоскости + обратная связь

**Плоскость управления** — планировщик экспериментов (workflow в Litmus, расписание Chaos Mesh, интерфейс Harness).

**Целевая плоскость** — сервисы, поды, узлы, балансировщики нагрузки, хранилища данных.

**Плоскость безопасности** — аварийный выключатель, окна подавления, ограничения радиуса воздействия, контрольные точки бюджета ошибок.

**Плоскость наблюдательности** — стандартные метрики + корреляция по trace-ID для различения хаос-индуцированных и естественных отказов.

**Петля обратной связи** — результаты используются для корректировки SLO, обновления рантбуков и исправления кода.

### Защитные ограничения обязательны

- **Оповещение о скорости сжигания (burn-rate)**: приостановка эксперимента, если суточное потребление бюджета ошибок превышает ожидаемое в 2 раза.
- **Окна подавления**: подавление оповещений, не связанных с экспериментом, в радиусе воздействия во время проведения эксперимента.
- **Корреляция по trace-ID**: все ошибки, вызванные экспериментом, маркируются тегом, чтобы дежурная служба могла устранять дубликаты.

### Пять экспериментов, специфичных для LLM

1. **Перегрузка памяти** — вызов шторма вытеснения из KV-кеша отправкой запросов с длинным контекстом при высокой параллельной нагрузке. Наблюдение: сервис корректно отбрасывает лишнее или падает?

2. **Сетевой сбой** — обрыв связи между шлюзом вывода и провайдером. Наблюдение: переключение на запасной вариант (fallback) происходит в пределах SLA? (Фаза 17 · 19)

3. **Имитация простоев провайдера** — 100% ответов 429 от OpenAI. Наблюдение: маршрутизация переключается на Anthropic? (Фаза 17 · 16, 19)

4. **Некорректный промпт** — внедрение нагрузки, блокирующей токенизатор (например, глубоко вложенный Unicode, кодовая точка огромного размера в UTF-8). Наблюдение: один запрос блокирует обработчик?

5. **Шторм вытеснения KV-кеша** — вытеснение путём переполнения лимита блоков vLLM. Наблюдение: LMCache восстанавливается или сервис деградирует?

### Регулярность

- **Еженедельно** — небольшие канареечные эксперименты в премодукшне, возможно 5% продакшена.
- **Ежемесячно** — запланированный игровой день по конкретному сценарию; участие кросс-командное; разбор инцидентов (postmortem).
- **Ежеквартально** — кросс-командный аудит устойчивости; обновление карты зависимостей.

### Инструменты

- **Harness Chaos Engineering** — коммерческий; рекомендации экспериментов на основе ИИ; масштабирование радиуса воздействия; интеграция MCP-инструментов.
- **LitmusChaos** — CNCF (выпускная стадия); на основе workflow Kubernetes.
- **Chaos Mesh** — CNCF (песочница); нативный для Kubernetes стиль CRD.
- **Gremlin** — коммерческий; широкая поддержка.
- **AWS FIS** / **Azure Chaos Studio** — управляемые облачные решения.

### Начинайте с малого

Первый эксперимент: удаление одного реплика-декодера (pod-kill) при стабильном трафике. Наблюдайте за перенаправлением и восстановлением. Если всё работает безопасно, переходите к сетевым экспериментам.

Первый эксперимент, специфичный для LLM: внедрение одного ответа 429 от провайдера на 5 минут. Наблюдайте за переключением на запасной вариант. Большинство команд обнаруживают, что их fallback-механизм не был полностью протестирован.

### Числа, которые стоит запомнить

- Четыре плоскости: управления, целевая, безопасности, наблюдательности.
- Приостановка по скорости сжигания: 2-кратное превышение ожидаемого суточного потребления бюджета.
- Регулярность: еженедельные канареечные эксперименты, ежемесячные игровые дни, ежеквартальный аудит.
- Пять экспериментов для LLM: память, сеть, провайдер, некорректный промпт, шторм KV-кеша.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|--------|------------|----------------------|
| SLI / SLO | «сервисные цели» | Индикатор + цель; обязательное предварительное условие |
| Радиус воздействия (blast radius) | «область охвата» | Множество сервисов / пользователей, затронутых экспериментом |
| Оповещение о скорости сжигания (burn-rate alert) | «контрольная точка бюджета» | Срабатывает, когда скорость потребления бюджета ошибок > 2-кратного ожидаемого |
| Игровой день (game day) | «ежемесячное учение» | Запланированное кросс-командное хаос-упражнение |
| LitmusChaos | «workflow CNCF» | Выпускной инструмент CNCF для хаос-инженерии в Kubernetes |
| Chaos Mesh | «CRD CNCF» | Инструмент CNCF в статусе «песочница», нативный для Kubernetes |
| Harness CE | «коммерческий с ИИ» | Chaos-инструмент Harness с рекомендациями на основе ИИ |
| Некорректный промпт (malformed prompt) | «бомба токенизатора» | Входные данные, блокирующие токенизацию |
| Шторм вытеснения KV-кеша (KV eviction storm) | «каскадная конкуренция» | Массовое вытеснение, вызывающее повторное заполнение |
