# Раздельное обслуживание (disaggregated serving) префила и декода — NVIDIA Dynamo и llm-d

> Префил (prefill) ограничен вычислениями; декод (decode) ограничен памятью. Запуск обоих на одном GPU приводит к перерасходу одного из ресурсов. Раздельное обслуживание (disaggregated serving) разносит их по отдельным пулам и передаёт KV-кэш (KV cache) между ними через NIXL (RDMA/InfiniBand или TCP). NVIDIA Dynamo (анонс GTC 2025, 1.0 GA) работает поверх vLLM/SGLang/TRT-LLM — его Planner Profiler и SLA Planner автоматически подбирают соотношение префил:декод для соблюдения целевых показателей уровня обслуживания (SLO). NVIDIA публикует данные о росте пропускной способности в этом диапазоне — developer.nvidia.com (2025-06) демонстрирует прирост примерно в 6 раз для DeepSeek-R1 MoE на GB200 NVL72 + Dynamo в среднелатентном режиме, а страница продукта Dynamo (developer.nvidia.com, без даты) заявляет до 50-кратного роста пропускной способности MoE на GB300 NVL72 + Dynamo по сравнению с Hopper. Цифра «30x» является агрегированным показателем сообщества по полным отчётам о стеках Blackwell + Dynamo + DeepSeek-R1; единый первичный источник, указывающий точно 30x, обнаружен не был, поэтому её следует рассматривать как ориентировочное направление. llm-d (Red Hat + AWS) построен на нативных возможностях Kubernetes: префил / декод / маршрутизатор представляют собой независимые сервисы с автоматическим масштабированием по ролям (HPA). llm-d 0.5 добавляет иерархическое выгрузка KV-кэша, маршрутизацию LoRA с учётом кэша, сетевой стек UCCL и масштабирование до нуля. Экономика: агрегированная внутренняя сводка по раскрытиям нескольких клиентов указывает на экономию 30–40% от расходов на инференс (inference) в размере $2 млн (то есть $600–800 тыс./год) при переходе от совместного (colocated) обслуживания к раздельному с Dynamo при неизменном SLO; конкретная цифра $2 млн → $600–800 тыс. является внутренним составным показателем, а не единственной опубликованной кейсовой историей — используйте её как ориентировочный порядок величины, а не как ссылку. Короткие промпты (<512 токенов, короткий вывод) не оправдывают стоимость передачи.

**Тип:** Обучение
**Языки:** Python (стандартная библиотека, учебный симулятор раздельного и совместного обслуживания)
**Предварительные требования:** Фаза 17 · 04 (Внутреннее устройство обслуживания vLLM), Фаза 17 · 08 (Метрики инференса)
**Время:** ~75 минут

## Цели обучения

- Объяснить, почему префил и декод имеют различные оптимальные распределения ресурсов GPU, и количественно оценить потери при совместном размещении.
- Составить схему архитектуры раздельного обслуживания: пул префилов, пул декодов, передача KV-кэша через NIXL, маршрутизатор.
- Назвать условие, при котором раздельное обслуживание не окупается (короткие промпты, короткие выводы).
- Различать NVIDIA Dynamo (оркестратор, работающий поверх стека) и llm-d (нативный для Kubernetes) и соотнести каждый с соответствующим операционным контекстом.

## Проблема

Вы запускаете Llama 3.3 70B на 8 H100. При смешанной нагрузке (длинные промпты + короткие выводы) GPU простаивают во время декода, поскольку основная вычислительная работа была выполнена на этапе префила. При другой нагрузке (короткие промпты + длинные выводы) происходит обратное. Совместное размещение префила и декода означает избыточное выделение ресурсов для обоих.

Влияние на бюджет: 20–40% времени GPU тратится не на тот ресурс. Вы приобретаете вычислительные мощности H100 для запуска ограниченного памятью декода или пропускную способность HBM H100 для запуска ограниченного вычислениями префила. Оба варианта — дорогостоящие потери.

Раздельное обслуживание разносит префил и декод по отдельным пулам, рассчитанным на узкое место каждого этапа. KV-кэш передаётся из пула префилов в пул декодов через высокоскоростную межсоединение.

## Концепция

### Почему узкие места различаются

**Префил** — запуск трансформера над всем входным промптом за один прямой проход. Доминируют матричные умножения; ограничен вычислениями. H100 FP8 обеспечивает около 2000 TFLOPS полезной пропускной способности. Эффективность батчинга высока — один прямой проход обрабатывает множество токенов.

**Декод** — генерация одного токена за итерацию с чтением всех весов на каждом шаге. Ограничен пропускной способностью памяти. HBM3 обеспечивает около 3 ТБ/с. Эффективность батчинга высока только при высокой параллельности — стоимость чтения весов распределяется на весь батч.

При совместном размещении: вы приобретаете GPU, оптимизированные для обоих задач. H100 хорош для обоих, но стоимость одинакова. В масштабе вы хотите пул префилов на H100 / вычислительно-ёмких GPU; пул декодов на H200 / память-ёмких GPU или с агрессивной квантизацией.

### Архитектура

```
            ┌──────────────┐
  Request → │    Router    │ ───────────────────────┐
            └──────┬───────┘                        │
                   │                                │
                   ▼ (только промпт)                │
            ┌──────────────┐    KV-кэш    ┌───────▼──────┐
            │   Пул       │ ─── NIXL ────► │   Пул       │
            │   префилов  │                │   декодов    │
            │ (вычисления)│                │  (память)    │
            └──────────────┘                └──────┬──────┘
                                                   │ токены
                                                   ▼
                                                 Клиент
```

NIXL — это межузловая транспортная подсистема NVIDIA. Использует RDMA/InfiniBand при наличии, в противном случае — TCP. Задержка передачи реальна — обычно 20–80 мс для KV-кэша промпта на 4K токена в 70B FP8. Именно поэтому короткие промпты не оправдывают раздельное обслуживание: стоимость передачи превышает экономию.

### Dynamo vs llm-d

**NVIDIA Dynamo** (анонс GTC 2025, 1.0 GA):
- Работает поверх vLLM, SGLang, TRT-LLM как оркестратор.
- Planner Profiler измеряет нагрузку, SLA Planner автоматически настраивает соотношения префил:декод.
- Ядро на Rust, расширяемость через Python.
- Рост пропускной способности: NVIDIA сообщает о 6-кратном приросте для DeepSeek-R1 MoE на GB200 NVL72 + Dynamo в среднелатентном режиме (developer.nvidia.com, 2025-06); сообщения сообщества о «до 30x» для полных стеков Blackwell + Dynamo + DeepSeek-R1 не имеют единого первичного источника и следует рассматривать как ориентировочные.
- GB300 NVL72 + Dynamo: до 50-кратного роста пропускной способности MoE по сравнению с Hopper, согласно странице продукта Dynamo (developer.nvidia.com, без даты).

**llm-d** (Red Hat + AWS, нативный для Kubernetes):
- Префил / декод / маршрутизатор в качестве независимых сервисов Kubernetes.
- Автоматическое масштабирование по ролям (HPA) с сигналами глубины очереди (префил) / загрузки KV-кэша (декод).
- `topologyConstraint packDomain: rack` — группировка префил+декод кластеров на одном стойке для высокоскоростной передачи KV-кэша.
- llm-d 0.5 (2026): иерархическая выгрузка KV-кэша, маршрутизация LoRA с учётом кэша, сетевой стек UCCL, масштабирование до нуля.

Используйте Dynamo, если вам нужен управляемый оркестратор, работающий поверх стека. Используйте llm-d, если вам нужны нативные примитивы Kubernetes и вы ориентированы на экосистему CNCF.

### Экономика

Агрегированная внутренняя сводка (не единственная опубликованная кейсовая история — ориентировочный порядок величины):

- $2 млн/год расходов на инференс при совместном обслуживании.
- Переход на раздельное обслуживание с Dynamo.
- Тот же объём запросов, тот же SLO по P99-задержке.
- Зафиксированная экономия: $600–800 тыс./год (сокращение на 30–40%).
- Без нового оборудования.

Эта цифра составлена на основе нескольких раскрытий клиентов, а не единственной кейсовой истории; ближайший опубликованный показатель — 2-кратное ускорение TTFT / 61% рост пропускной способности от Baseten при маршрутизации KV через Dynamo (baseten.co, 2025-10), а также прогноз VAST + CoreWeave о росте токенов/$ на 60–130% при частоте попаданий в KV-кэш 40–60% (vastdata.com, 2025-12). Экономия достигается за счёт оптимизации размера каждого пула; нагрузки с преобладанием префилов (RAG с префиксами 8K+) получают большую выгоду, чем сбалансированные.

### Когда не стоит разделять

- Промпты < 512 токенов и выводы < 200 токенов: стоимость передачи превышает выгоду.
- Маленький кластер (< 4 GPU): недостаточное разнообразие пулов.
- Команда не может эксплуатировать два пула GPU с масштабированием по ролям: Dynamo помогает, но не без усилий.
- Отсутствие RDMA-ткани: стоимость передачи по TCP значительно выше.

### Маршрутизатор интегрируется с Фазой 17 · 11

Раздельные маршрутизаторы учитывают KV-кэш (Фаза 17 · 11). Запрос попадает в пул декодов, содержащий его префикс — если совпадения нет, он проходит через префил → декод. Частота попаданий и раздельное обслуживание усиливают друг друга — маршрутизатор с учётом кэша определяет, нужен ли вообще новый префил.

### MoE на Blackwell — там реальные цифры

GB300 NVL72 + Dynamo демонстрирует 50-кратный рост пропускной способности MoE по сравнению с Hopper. Маршрутизация экспертов MoE требует высоких вычислений на этапе префила, но heavily памяти на этапе декода (кэши экспертов), поэтому раздельное обслуживание даёт двойную выгоду. Обслуживание моделей нового поколения в 2026 году преимущественно основано на MoE (DeepSeek-V3, будущие варианты GPT-5).

### Цифры, которые стоит запомнить

Бенчмарк-показатели меняются — NVIDIA и стек инференса публикуют обновлённые результаты каждый квартал. Перепроверяйте перед цитированием.

- DeepSeek-R1 на GB200 NVL72 + Dynamo: ~6-кратный рост пропускной способности по сравнению с базовым уровнем в среднелатентном режиме (developer.nvidia.com, 2025-06); утверждения сообщества «до 30x» для полных стеков Blackwell + Dynamo являются ориентировочными агрегатами без единого первичного источника.
- GB300 NVL72 + Dynamo: до 50-кратного роста пропускной способности MoE по сравнению с Hopper (developer.nvidia.com, без даты).
- Ориентир экономии (агрегированная внутренняя сводка, не единственная кейсовая история): $600–800 тыс./год от годовых расходов в $2 млн при неизменном SLO.
- Порог раздельного обслуживания: промпты >512 токенов + выводы >200 токенов.
- Передача KV через NIXL: 20–80 мс для KV-кэша промпта на 4K токена в 70B FP8.

## Ключевые термины

| Термин | Что говорят | Что это значит на самом деле |
|--------|------------|---------------------------|
| Раздельное обслуживание (disaggregated serving) | «разделение префил/декод» | Отдельные GPU-пулы для каждого этапа |
| NIXL | «транспорт NVIDIA» | Межузловая передача KV-кэша в Dynamo (RDMA/TCP) |
| NVIDIA Dynamo | «оркестратор» | Координатор, работающий поверх стека vLLM/SGLang/TRT-LLM |
| llm-d | «нативный Kubernetes» | Стек раздельного обслуживания от Red Hat + AWS на Kubernetes |
| Planner Profiler | «автоконфигурация Dynamo» | Измеряет нагрузку, настраивает соотношения пулов |
| SLA Planner | «политика Dynamo» | Автоматически подбирает соотношение префил:декод для соблюдения SLO |
| `packDomain: rack` | «топология llm-d» | Группировка префил+декод на одной стойке для быстрой передачи KV |
| UCCL | «единый коллективный» | Сетевой стек llm-d 0.5 для масштабирования до нуля |
| Маршрутизация экспертов MoE | «эксперт на токен» | Паттерн DeepSeek-V3; раздельное обслуживание помогает |
