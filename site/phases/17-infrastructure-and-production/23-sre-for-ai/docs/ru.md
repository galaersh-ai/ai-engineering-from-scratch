# Надёжность сервисов (SRE) для ИИ — Мультиагентное реагирование на инциденты, руководства по эксплуатации, предиктивное обнаружение

> SRE для ИИ использует LLM, привязанные к инфраструктурным данным (логи, руководства по эксплуатации, топология сервисов) через RAG для автоматизации этапов расследования, документирования и координации. Архитектурный паттерн 2026 года — мультиагентная оркестрация (multi-agent orchestration): специализированные агенты (логи, метрики, руководства) координируются супервайзером; ИИ предлагает гипотезы и запросы, человек одобряет решения. Datadog Bits AI и Azure SRE Agent выпускают это как управляемые продукты. Руководства по эксплуатации (runbooks) эволюционируют: NeuBird Hawkeye использует конкурирующую оценку (adversarial evaluation) — две модели анализируют один и тот же инцидент; согласие = уверенность, расхождение = неопределённость; оперативная память (operational memory) сохраняется при смене команды. Автоматическое устранение (auto-remediation) остаётся осторожным: ИИ предлагает, человек одобряет. Полностью автономное действие ограничено узким набором (перезапуск пода, откат конкретного деплоя) с жёсткими ограничениями — любой, кто продаёт «установил и забыл», переоценивает возможности. Зарождающийся рубеж: предикция до инцидента (pre-incident prediction). Исследование MIT сообщает, что LLM, обученный на исторических логах, температурах GPU и паттернах ошибок API, предсказал 89% сбоев за 10–15 минут раньше. Прогноз: к концу 2026 года 95% корпоративных LLM будут иметь автоматизированный фейловер.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, учебный симулятор мультиагентного триажа инцидентов)
**Предварительные знания:** Фаза 17 · 13 (Наблюдаемость (Observability)), Фаза 17 · 24 (Хаос-инжиниринг)
**Время:** ~60 минут

## Цели обучения

- Составить диаграмму архитектуры мультиагентного SRE для ИИ: координатор + специализированные агенты (логи, метрики, руководства) + врата человеческого одобрения.
- Объяснить, почему автоматическое устранение ограничено узким набором (перезапуск пода, откат деплоя), а не широким (перепроектирование сервиса).
- Назвать паттерн конкурирующей оценки (NeuBird Hawkeye): две модели согласны = уверенность; расхождение = эскалация.
- Привести результат MIT о 89% раннего обнаружения и операционное ограничение: предсказания без воздействия — это просто дашборды.

## Проблема

Инженер на дежурстве получает оповещение в 3 часа ночи: «Высокий уровень ошибок в checkout». Он проверяет Datadog, Loki, три руководства по эксплуатации, лог деплоя. Через 30 минут он понимает, что корневая причина — OOM в vLLM из-за всплеска KV-кэша. Он перезапускает под; ошибка исчезает.

В 2026 году первые 20 минут этого расследования автоматизируемы. Группировка логов по сервисам, корреляция с недавними деплоями, сопоставление с руководствами — всё это RAG + использование инструментов. Координирующий агент может провести первичный триаж и представить гипотезу ещё до того, как человек откроет Datadog.

Полностью автономное устранение — это другая проблема. Перезапуск пода: безопасно. Масштабирование пула GPU: безопасно, если политика позволяет. Перепроектирование сервиса: категорически нет. Дисциплина заключается в проведении чёткой границы.

## Концепция

### Мультиагентная архитектура

```
          Инцидент
             │
             ▼
        Координатор
        /    |    \
       ▼     ▼     ▼
  Агент логов  Агент метрик  Агент руководств
       │     │     │
       └─────┴─────┘
             │
             ▼
        Гипотеза + доказательства
             │
             ▼
        Человеческое одобрение
             │
             ▼
        Действие (ограниченный набор)
```

Координатор разбивает инцидент на подзапросы. Специализированные агенты имеют доступ к инструментам (поиск логов, PromQL, поиск документов). Координатор синтезирует данные, представляет гипотезу + доказательства человеку. Человек одобряет или перенаправляет.

### Область автоматического устранения

**Безопасно (узкий набор)**: перезапуск пода, откат конкретного деплоя, масштабирование пула в пределах заранее одобренных границ, включение заранее одобренного фича-флага.

**Небезопасно (широкий набор)**: изменение топологии сервиса, модификация лимитов ресурсов, деплой нового кода, изменение IAM, модификация баз данных.

Любой, кто продаёт «установил и забыл», переоценивает возможности. Набор безопасных действий расширяется по мере зрелости SRE для ИИ, но граница реальна.

### Конкурирующая оценка (NeuBird Hawkeye)

Две модели независимо анализируют один и тот же инцидент. Если они согласны в корневой причине — уверенность высока. Если расходятся — эскалируется человеку с видимыми обеими гипотезами. Простой паттерн, эффективный фильтр против галлюцинированных корневых причин.

### Оперативная память

Текучесть команды — тихий убийца традиционного SRE: уходит племенные знания. SRE для ИИ хранит руководства и постмортемы в векторной БД; агенты извлекают их при каждом новом инциденте. Когда новые инженеры присоединяются, ИИ имеет полную историю.

### Предикция до инцидента

Исследование MIT 2025 года: LLM, обученный на исторических логах, температурах GPU и паттернах ошибок API, предсказал 89% сбоев за 10–15 минут до их возникновения на тестовой выборке.

Проверка реальностью: предсказания без воздействия — это дашборды. Операционный вопрос: «когда мы предсказываем, что мы делаем?» Дренирование? Пейджер? Автомасштабирование? Ответ зависит от политики.

### Продукты 2026 года

- **Datadog Bits AI** — управляемый SRE-помощник внутри Datadog.
- **Azure SRE Agent** — нативный для Azure.
- **NeuBird Hawkeye** — конкурирующая оценка + оперативная память.
- **PagerDuty AIOps** — триаж + дедупликация.
- **Incident.io Autopilot** — командир инцидента + координация.

### Руководства как код

Руководства по эксплуатации эволюционируют из страниц Confluence в версионированные markdown со структурированными секциями (симптом, гипотеза, проверка, действие). Структурированные руководства обеспечивают лучшую выборку RAG. Начинайте внедрение AI-SRE с преобразования неструктурированных руководств в структурированные.
