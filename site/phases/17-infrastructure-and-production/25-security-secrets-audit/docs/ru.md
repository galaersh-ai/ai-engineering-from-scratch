# Безопасность — секреты (secrets), ротация API-ключей, журналы аудита (audit), защитные барьеры (guardrails)

> Устраните рассеивание секретов (secret sprawl) с помощью централизованных хранилищ (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault). Никогда не храните учётные данные в файлах конфигурации, файлах переменных окружения в системах контроля версий или электронных таблицах. Используйте IAM-роли вместо статических ключей; OIDC для CI/CD. Паттерн «AI-шлюз» (AI-gateway) — стандартное решение 2026 года: приложения → шлюз → провайдер модели, при этом шлюз извлекает учётные данные из хранилища во время выполнения запроса. Ротация (rotation) в хранилище — и все приложения получают новый ключ за минуты без переразвёртывания и без сообщений в Slack «у кого новый ключ». Политика ротации ≤ 90 дней; сканирование с помощью TruffleHog / GitGuardian / Gitleaks при каждом коммите. Нулевое доверие (zero trust): MFA, SSO, RBAC/ABAC, токены с коротким сроком жизни, состояние устройства (device posture). Очистка PII (PII scrubbing) использует распознавание сущностей для маскировки PHI/PII перед пересылкой; согласованная токенизация (consistent tokenization, подход «Mesh») отображает чувствительные значения в стабильные плейсхолдеры, чтобы LLM сохраняла семантику кода и связей. Сетевой исходящий трафик (egress): сервисы LLM в выделенной подсети VPC/VNet с разрешением только `api.openai.com`, `api.anthropic.com` и т. д.; блокировка всего остального исходящего трафика. Инцидент 2026 года: атака на цепочку поставок Vercel через скомпрометированные учётные данные CI/CD, в результате которой были украдены переменные окружения тысяч клиентских развёртываний.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, простой скрабер PII + запись в журнал аудита)
**Предварительные требования:** Фаза 17 · 19 (AI-шлюзы), Фаза 17 · 13 (Наблюдаемость)
**Время:** ~60 минут

## Цели обучения

- Перечислить четыре антипаттерна управления секретами (secret management): файлы конфигурации в VCS, захардкоженные переменные окружения, электронные таблицы, статические ключи — и назвать их замены.
- Объяснить паттерн «AI-шлюз извлекает из хранилища» как стандарт продакшена 2026 года.
- Реализовать скрабер PII с согласованной токенизацией (одно и то же значение → один и тот же плейсхолдер), чтобы семантика сохранялась.
- Назвать инцидент Vercel 2026 года и то, чему он научил в отношении гигиены учётных данных CI/CD.

## Проблема

Стажёр коммитит `.env` с API-ключами. Он быстро удаляет файл. Ключи уже находятся в истории git — сканирование GitGuardian это обнаруживает, а ваш процесс ротации заключается в «написать в команду, обновить 40 файлов конфигурации, переразвернуть все сервисы». Спустя 8 часов половина сервисов работает, а вторая половина ждёт окна для развёртывания.

Отдельно: запросы пользователей содержат «Мой номер SSN — 123-45-6789». Запрос отправляется в OpenAI. У вас есть BAA, но внутренняя политика требует маскировки PII перед пересылкой. Вы этого не сделали.

Отдельно: LLM-под (pod) в кластере EKS может обращаться к любому хосту в интернете. Кто-то вытягивает данные через DNS-запрос к домену, контролируемому атакующим. Ничего этого не заблокировало.

Безопасность сервисов LLM должна учитывать все три вектора. Учётные данные на основе хранилища. Очистка PII. Фильтрация исходящего сетевого трафика. Журналы аудита.

## Концепция

### Централизованное хранилище + извлечение через IAM-роль

**Хранилище (Vault)**: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager. Единый источник истины.

**IAM-роль**: приложение/шлюз аутентифицируется через свою IAM-учётную запись, а не через статический ключ. Хранилище возвращает секрет на время жизни токена.

**Паттерн «AI-шлюз»**: шлюз извлекает `OPENAI_API_KEY` из хранилища при каждом запросе. Ротация в хранилище — следующий запрос получает новый ключ. Без переразвёртываний.

### Политика ротации ≤ 90 дней

Все API-ключи, корневые токены хранилища, учётные данные CI/CD. Автоматизация ротации там, где это возможно. Ручная ротация логируется и отслеживается.

### Сканирование секретов

- **TruffleHog** — регулярные выражения + энтропия для коммитов.
- **GitGuardian** — коммерческое решение с высокой точностью.
- **Gitleaks** — решение с открытым исходным кодом, запускается в CI.

Запускайте при каждом коммите. Блокируйте PR при обнаружении нового секрета.

### Нулевое доверие (zero trust)

- MFA обязателен для всех учётных записей.
- SSO через SAML/OIDC.
- RBAC (на основе ролей) или ABAC (на основе атрибутов) для детализированного контроля доступа.
- Токены с коротким сроком жизни (часы, а не дни).
- Состояние устройства (device posture) — только корпоративные устройства с шифрованием диска.

### Очистка PII / PHI

Перед тем как запрос покинет вашу инфраструктуру:

1. Распознавание сущностей (NER) — spaCy NER, Presidio, коммерческие решения.
2. Маскировка найденных сущностей: `"Мой номер SSN — 123-45-6789"` → `"Мой номер SSN — [SSN_TOKEN_A3F]"`.
3. Согласованная токенизация (consistently tokenization, подход «Mesh»): одно и то же значение отображается в один и тот же плейсхолдер, чтобы LLM сохраняла связи.
4. Опциональное обратное отображение для ответа LLM.

Статические фильтры на регулярных выражениях ловят базовые шаблоны; NER ловит больше. Используйте оба метода.

### Защитные барьеры (guardrails) на входе и выходе

Вход: блокировка известных джейкбреков, запрещённых тем; ограничение частоты запросов для каждого пользователя.

Выход: фильтрация с помощью регулярных выражений для обнаружения утечек секретов (шаблоны API-ключей, шаблоны электронной почты в контексте отказа), классификатор для нарушений политик.

### Белый список исходящего сетевого трафика (egress whitelist)

Сервисы LLM в выделенной подсети:
- Белый список: `api.openai.com`, `api.anthropic.com`, конечные точки векторной БД, конечные точки хранилища.
- Всё остальное: отбрасывание.
- DNS через разрешитель с белым списком (для предотвращения exfiltration через DNS-туннелирование).

### Журнал аудита (audit log)

Неизменяемый журнал каждого вызова LLM, содержащий:
- Метку времени.
- Пользователя / тенанта.
- Хэш запроса (не сам запрос в целях конфиденциальности).
- Модель + версию.
- Количество токенов.
- Стоимость.
- Хэш ответа.
- Все срабатывания защитных барьеров.

Хранение в соответствии с нормативными требованиями (SOC 2 — 1 год, HIPAA — 6 лет).

### Инцидент Vercel 2026 года

Атака на цепочку поставок (supply-chain attack): скомпрометированные учётные данные CI/CD позволили вывести переменные окружения тысяч клиентских развёртываний. Урок: учётные данные CI/CD эквивалентны учётным данным продакшена. Храните их в хранилище. Ограничивайте область доступа. Проводите агрессивную ротацию.

### Числа, которые стоит запомнить

- Политика ротации: ≤ 90 дней.
- Сканирование при каждом коммите: TruffleHog / GitGuardian / Gitleaks.
- Vercel 2026: скомпрометированные учётные данные CI/CD → утечка переменных окружения тысяч клиентов.
- Срок хранения журнала аудита: SOC 2 = 1 год, HIPAA = 6 лет.

## Упражнения

1. Запустите `code/main.py`. Отправьте два запроса, содержащих один и тот же номер SSN. Убедитесь, что в обоих случаях используется один и тот же плейсхолдер.
2. Спроектируйте политику исходящего сетевого трафика для развёртывания vLLM на EKS с вызовом OpenAI + Anthropic + Weaviate.
3. Вы обнаружили ключ в истории git (2-летней давности). Как правильная реакция — ротировать ключ, очистить историю или и то и другое? Обоснуйте.
4. Ваш журнал аудита растёт на 10 ГБ в день. Спроектируйте уровни хранения: горячий — 30 дней, тёплый — 12 месяцев, холодный — 6 лет.
5. Обоснуйте, стоит ли использовать обратную токенизацию (подстановку реальных значений обратно в ответ LLM) с учётом сложности реализации по сравнению с отображением плейсхолдеров как есть.

## Ключевые термины

| Термин | Что говорят | Что это означает на самом деле |
|--------|-------------|-------------------------------|
| Vault (Хранилище) | «хранилище секретов» | Сервис централизованного управления учётными данными |
| IAM-роль | «аутентификация на основе идентичности» | Роль, присваиваемая приложению; возвращает учётные данные с коротким сроком жизни |
| OIDC для CI/CD | «токены, выданные облаком» | Нет статических ключей в CI — идентификация через OIDC |
| TruffleHog / GitGuardian / Gitleaks | «сканеры секретов» | Обнаружение секретов на этапе коммита |
| RBAC / ABAC | «контроль доступа» | На основе ролей vs. на основе атрибутов |
| Очистка PII (PII scrubbing) | «маскировка данных» | Удаление или токенизация чувствительных сущностей |
| Согласованная токенизация | «стабильные плейсхолдеры» | Одно и то же значение → один и тот же токен каждый раз |
| Подход «Mesh» | «Токенизация Mesh» | Паттерн токенизации с сохранением семантики |
| Белый список исходящего трафика (egress whitelist) | «список разрешённых исходящих адресов» | Доступны только разрешённые домены |
| Журнал аудита (audit log) | «неизменяемая история» | Запись в режиме «только добавление» для обеспечения соответствия (compliance) |
