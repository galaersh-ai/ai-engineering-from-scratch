# Выбор самостоятельного обслуживания (self-hosted serving) — llama.cpp, Ollama, TGI, vLLM, SGLang

> Четыре движка доминируют в самостоятельном инференсе (inference) в 2026 году. Выбирайте исходя из аппаратного обеспечения, масштаба и экосистемы. **llama.cpp** — самый быстрый на CPU: поддерживает наибольшее количество моделей, обеспечивает полный контроль над квантизацией (quantization) и многопоточностью. **Ollama** — одностроковая установка для разработчика на ноутбуке, на ~15–30% медленнее llama.cpp (Go + CGo + HTTP-сериализация), разрыв в пропускной способности до 3 раз при нагрузке, приближенной к продакшену. **TGI переведён в режим поддержки 11 декабря 2025 года** — исправляются только ошибки, сырая пропускная способность примерно на 10% ниже, чем у vLLM, но исторически обеспечивает лучшую наблюдаемость (observability) и интеграцию с экосистемой Hugging Face. Этот статус поддержки делает его рискованным долгосрочным выбором — SGLang или vLLM являются более безопасными вариантами по умолчанию для новых проектов. **vLLM** — универсальный движок для продакшена по умолчанию — версия v0.15.1 (февраль 2026) добавляет PyTorch 2.10, поддержку RTX Blackwell SM120 и оптимизацию для H200. **SGLang** — специалист по агентным (agentic) многотурным и префикс-ёмким сценариям — более 400 000 GPU в продакшене (xAI, LinkedIn, Cursor, Oracle, GCP, Azure, AWS). Аппаратные ограничения: только CPU → только llama.cpp. AMD / не NVIDIA → только vLLM (TRT-LLM привязан к NVIDIA). Паттерн пайплайна 2026 года: разработка = Ollama, промежуточные среды = llama.cpp, продакшен = vLLM или SGLang. Одни и те же GGUF/HF-веса на всех этапах.

## Проблема

Ваша команда начинает новый проект самостоятельного развёртывания LLM. Один инженер предлагает Ollama, другой — vLLM, третий говорит: «А разве TGI не работает из коробки?» Все трое правы в разных контекстах. Ни один из них не подходит для всех случаев.

В 2026 году дерево решений имеет значение: аппаратное обеспечение — в первую очередь, масштаб — во вторую, рабочая нагрузка — в третью. И одно конкретное событие 2025 года — переход TGI в режим поддержки 11 декабря — изменяет вариант по умолчанию для новых проектов.

## Теоретическая часть

### Пять движков

| Движок | Лучше всего подходит для | Примечания |
|--------|----------|-------|
| **llama.cpp** | CPU / Edge-устройства / минимальные зависимости / максимальная поддержка моделей | Самый быстрый на CPU, полный контроль |
| **Ollama** | Ноутбуки разработчиков, один пользователь, одностроковая установка | На 15–30% медленнее llama.cpp; разрыв пропускной способности в 3 раза в продакшене |
| **TGI** | Экосистема Hugging Face, регулируемые отрасли | **Переведён в режим поддержки 11 декабря 2025 года** |
| **vLLM** | Универсальный продакшен, 100+ пользователей | Основной вариант по умолчанию; v0.15.1, февраль 2026 |
| **SGLang** | Агентные многотурные сценарии, префикс-ёмкие нагрузки | Более 400 000 GPU в продакшене |

### Решение по аппаратному обеспечению

**Только CPU** → llama.cpp. Ollama тоже работает, но медленнее. Ни один другой движок не конкурирует на CPU.

**GPU AMD** → vLLM (поддержка AMD ROCm). SGLang тоже работает. TRT-LLM привязан к NVIDIA, поэтому не подходит.

**NVIDIA Hopper (H100 / H200)** → vLLM или SGLang или TRT-LLM. Все три — первоклассные варианты.

**NVIDIA Blackwell (B200 / GB200)** → TRT-LLM — лидер по пропускной способности (Фаза 17 · 07). vLLM и SGLang немного отстают.

**Apple Silicon (серия M)** → llama.cpp (Metal). Ollama оборачивает этот движок.

### Решение по масштабу

**1 пользователь / локальная разработка** → Ollama. Одна команда, первый токен — за секунды.

**10–100 пользователей / небольшая команда** → vLLM на одной GPU.

**100–10 000 пользователей / продакшен** → vLLM production-stack (Фаза 17 · 18) или SGLang.

**10 000+ пользователей / корпоративный сегмент** → vLLM production-stack + дезагрегация (Фаза 17 · 17) + LMCache (Фаза 17 · 18).

### Решение по рабочей нагрузке

**Общий чат / вопрос-ответ** → vLLM побеждает как универсальное решение по умолчанию.

**Агентные многотурные сценарии (инструменты, планирование, память)** → RadixAttention в SGLang (Фаза 17 · 06) доминирует.

**RAG с интенсивным повторным использованием префиксов** → SGLang.

**Генерация кода** → vLLM — хорош; SGLang немного лучше за счёт кэширования.

**Длинный контекст (128K+)** → vLLM + посегментный префилл (chunked prefill); SGLang + многоуровневое KV-кэширование.

### Ловушка режима поддержки TGI

Hugging Face TGI переведён в режим поддержки (maintenance mode) 11 декабря 2025 года — в дальнейшем исправляются только ошибки. Исторически: первоклассная наблюдаемость, лучшая интеграция с экосистемой HF (карточки моделей, инструменты безопасности), незначительное отставание от vLLM по сырым показателям пропускной способности.

Для новых проектов в 2026 году: по умолчанию не выбирайте TGI. Существующие развёртывания TGI могут продолжать работу, но со временем следует перейти на другой движок. SGLang и vLLM — более безопасные варианты по умолчанию.

### Паттерн пайплайна

Разработка (Ollama) → промежуточные среды (llama.cpp) → продакшен (vLLM). Одни и те же GGUF- или HF-веса на всех этапах. Инженеры быстро итерируют на ноутбуках; промежуточные среды воспроизводят квантизацию продакшена; продакшен — целевой сервер обслуживания.

### Оговорка по Ollama

Ollama отлично подходит для разработки. Она не подходит для совместного продакшена: HTTP-сериализация на Go создаёт накладные расходы, управление конкурентностью проще, чем в vLLM, поддержка OpenTelemetry отстаёт. Используйте Ollama там, где она хороша — один пользователь, одна команда — и переходите на vLLM для совместного использования.

### Самостоятельное развёртывание vs. управляемый сервис — отдельное решение

Фаза 17 · 01 (управляемые гиперскейлеры), · 02 (платформы инференса) охватывают управляемые решения. Данный урок предполагает, что вы уже приняли решение о самостоятельном развёртывании. Причины самостоятельного развёртывания: локализация данных, пользовательская дообученная модель, совокупные затраты на владение (TCO) при масштабировании, доменная модель, недоступная на хостинге.

### Числа, которые стоит запомнить

- Режим поддержки TGI: 11 декабря 2025 года.
- vLLM v0.15.1: февраль 2026; PyTorch 2.10; поддержка Blackwell SM120.
- Производственная база SGLang: более 400 000 GPU.
- Разрыв пропускной способности Ollama по сравнению с llama.cpp: на 15–30% медленнее; в 3 раза при продакшен-нагрузке.
