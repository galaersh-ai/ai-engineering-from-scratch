# Спекулятивный декодинг EAGLE-3 в продакшене

> Спекулятивный декодинг (speculative decoding) объединяет быструру модель-черновик с целевой моделью. Черновик предлагает K токенов; целевая модель проверяет их за один прямой проход (forward pass); принятые токены фактически бесплатны. В 2026 году EAGLE-3 является промышленным вариантом — он обучает голову черновика на скрытых состояниях целевой модели, а не на сырых токенах, что поднимает коэффициент принятия alpha в диапазон 0.6–0.8 для общего чата. Правильный вопрос не «насколько быстр черновик», а «каков alpha на моём трафике». Если alpha падает ниже ~0.55, спекулятивный декодинг при высокой параллельности приносит отрицательный эффект, поскольку каждый отклонённый черновик требует второго прохода целевой модели. Этот урок учит вас сначала измерять alpha, а затем включать опцию.

**Тип:** Учебный модуль
**Языки:** Python (стандартная библиотека, учебный симулятор коэффициента принятия)
**Предварительные знания:** Фаза 17 · 04 (внутренности серверного обслуживания vLLM), Фаза 10 · 18 (предсказание нескольких токенов)
**Время:** ~60 минут

## Цели обучения

- Назвать три поколения спекулятивного декодинга и объяснить, чем EAGLE-3 отличается от EAGLE-2 и от классической модели-черновика.
- Определить коэффициент принятия alpha, вычислить ожидаемое ускорение по alpha и K (длина черновика), а также определить порог alpha безубыточности для заданной параллельности.
- Объяснить, почему спекулятивный декодинг по умолчанию отключён (opt-in) в vLLM 2026 года и почему его включение без измерения alpha является антипаттерном в продакшене.
- Составить план измерений: какой бенчмарк, какое распределение промптов, какой уровень параллельности, какую метрику контролировать.

## Проблема

Декодирование (decode) ограничено пропускной способностью памяти. На H100 с Llama 3.3 70B FP8 каждый декодируемый токен считывает ~140 ГБ/с весов и выдаёт один токен. Вычислительные ресурсы GPU практически простаивают во время декодирования — узкое место — пропускная способность HBM, а не пропускная способность матричных умножений.

Спекулятивный декодинг использует этот разрыв. Генерируются K кандидатов токенов с помощью дешёвой модели-черновика, затем целевая модель проверяет все K за один проход. Каждый проверенный токен фактически бесплатен (амортизируется в пакете из K проходов, который целевая модель всё равно должна была бы выполнить).

Классический подход с моделью-черновиком использует меньшую модель из того же семейства (Llama 3.2 1B в качестве черновика для Llama 3.3 70B). Это работает, но коэффициент принятия посредственный — распределение меньшей модели расходится с целевой. EAGLE, затем EAGLE-2, затем EAGLE-3 обучают лёгкую голову черновика непосредственно на внутренних состояниях целевой модели, поэтому распределение черновика гораздо точнее отслеживает целевое. Именно поэтому alpha возрастает с 0.4 у модели-черновика до 0.6–0.8 у EAGLE-3.

Важный момент: EAGLE-3 по умолчанию отключён в vLLM 2026 года. Параметр `speculative_config` необходимо задавать явно. Без флага нет ускорения. Команды, включающие его без измерения alpha на реальном трафике, часто наблюдают ухудшение хвостовой латентности вместо улучшения.

## Концепция

### Что на самом деле даёт спекулятивный декодинг

Без спекулятивного декодинга стоимость на токен — один проход целевой модели. Со спекулятивным декодингом при длине черновика K и коэффициенте принятия alpha ожидаемое количество токенов на проход целевой модели составляет `1 + K * alpha`. Ускорение равно `(1 + K * alpha) / (1 + epsilon)`, где epsilon — накладные расходы черновика и верификации. При K=5, alpha=0.7: `(1 + 5*0.7) / (1 + 0.1) = 4.5 / 1.1 = 4.1x`. Реальные значения обычно составляют 2–3x, поскольку alpha редко бывает таким высоким на продакшенном трафике, а epsilon растёт при большом размере пакета.

### Почему alpha — единственная важная метрика

Отклонённые токены не исчезают — они вынуждают целевую модель выполнить повторный проход для первого отклонённого токена. На нагрузке, где alpha падает до 0.4, вы платите накладные расходы черновика плюс верификацию плюс повторную генерацию. При высокой параллельности (например, 256 одновременных запросов) декодирующий пакет уже достаточно велик, чтобы разрыв пропускной способности памяти между «только целевой моделью» и «целевой моделью с верификацией» сузился. При alpha ниже 0.55 на большинстве аппаратных средств 2026 года спекулятивный декодинг приносит отрицательный эффект.

Alpha зависит от нагрузки. На общем чате в стиле ShareGPT EAGLE-3, обученный на ShareGPT, достигает 0.6–0.8. На специализированном трафике (код, медицина, юриспруденция) голова черновика, обученная на общих данных, падает до 0.4–0.6. Обучение специализированной головы черновика восстанавливает alpha — это лёгкая и быстрая задача по обучению по сравнению с дообучением целевой модели.

### Поколения EAGLE кратко

- **Классическая модель-черновик**: малая модель из того же семейства. Alpha 0.3–0.5. Инфраструктура проста — загружены две модели, черновик выполняет K проходов на каждый проход целевой модели.
- **EAGLE-1 (2024)**: одна голова черновика, обученная на скрытых состояниях целевой модели (последний слой). Alpha ~0.5–0.6. Небольшие дополнительные параметры поверх целевой модели.
- **EAGLE-2 (2025)**: адаптивная длина черновика и древовидные черновики (проверка нескольких ветвей за один проход целевой модели). Alpha ~0.6–0.7. Более сложный планировщик черновика.
- **EAGLE-3 (2025–2026)**: голова черновика обучена на нескольких слоях целевой модели (не только последнем), лучшая согласованность. Alpha ~0.6–0.8 на общем чате.

### Рецепт для продакшена в 2026 году

1. Разворачиваете целевую модель в чистом виде. Измеряете базовые TTFT, ITL и пропускную способность при целевой параллельности.
2. Включаете черновик EAGLE-3 через `speculative_config` в vLLM. Перезапускаете бенчмарк.
3. Логируете коэффициент принятия alpha. vLLM V1 выводит его как `spec_decode_metrics.accepted_tokens_per_request`. Делите на запрошенную длину черновика, чтобы получить alpha.
4. Если alpha < 0.55 на распределении продакшенного трафика, отключайте спекулятивный декодинг или обучайте специализированную голову EAGLE-3.
5. При продакшенной параллельности перезапускаете. Убеждаетесь, что P99 ITL не ухудшилось.

### Продакшенная ловушка: хвост P99

Средний ITL снижается со спекулятивным декодингом. P99 может ухудшиться, если вы не настроите систему. Отклонённые черновики запускают двухпроходную последовательность (черновик + ошибка верификации + повторная генерация). При полном пакете эти два прохода сериализуются. Контролируйте P99 ITL, а не P50.

### Где EAGLE-3 уже развёрнут

Google развёрнул спекулятивный декодинг в AI Overviews в 2025 году (такое же качество, более быстрый ответ). vLLM V1 поставляет `speculative_config` как задокументированный интерфейс; N-gram GPU спекулятивный декодинг в V1 — это вариант, совместимый с фрагментированным префилом (chunked prefill). SGLang поддерживает EAGLE-3 в качестве рекомендуемого пути черновика для нагрузок с преобладанием префиксов.

### Формула безубыточности в одну строку

Ожидаемое ускорение: `S(alpha, K) = (1 + K*alpha) / (1 + verify_overhead)`. Приравнивая `S = 1`, получаем: `alpha_breakeven = verify_overhead / K`. При типичном `verify_overhead ~0.15` и K=5: `alpha_breakeven = 0.03`. Однако это чистая математика декодирования. При высокой параллельности накладные расходы верификации возрастают, а декодирующий пакет уже амортизирует чтения памяти по последовательностям, поэтому эффективный alpha_breakeven на практике повышается до ~0.45–0.55.

### Когда не следует использовать спекулятивный декодинг

- Пакетная офлайн-генерация (batch-1), где латентность не важна. Используйте обычную целевую модель.
- Очень короткие выходы (менее 50 токенов). Накладные расходы черновика и стоимость верификации преобладают.
- Специализированные домены без обученной на доменных данных головы черновика. Alpha слишком низкий.
- vLLM v0.18.0 + спекулятивный декодинг моделью-черновиком + `--enable-chunked-prefill`. Эта комбинация не компилируется. Задокументированное исключение — N-gram GPU спекулятивный декодинг в V1.

## Использование

`code/main.py` имитирует цикл декодирования со спекулятивным декодингом и без него для ряда значений alpha и длин черновика K. Скрипт выводит порог alpha безубыточки, измеренное ускорение и хвостовое поведение. Запустите его на нескольких комбинациях (alpha, K), чтобы точно определить, когда спекулятивный декодинг перестаёт давать выгоду.

## Ключевые термины

| Термин | Как говорят | Что это означает |
|--------|-------------|------------------|
| Спекулятивный декодинг (speculative decoding) | «черновик плюс верификация» | Предложение K токенов дешёвой моделью, проверка всех K за один проход целевой модели |
| Коэффициент принятия alpha (acceptance rate) | «spec accept rate» | Доля токенов черновика, принятых целевой моделью; единственная важная метрика |
| Длина черновика K (draft length) | «spec k» | Количество токенов, предлагаемых черновиком на каждый проход целевой модели; обычно 4–8 |
| Накладные расходы верификации epsilon (verify overhead) | «spec overhead» | Дополнительная стоимость верификации и повторной генерации по сравнению с обычным проходом целевой модели; растёт с размером пакета |
| EAGLE-3 | «последний EAGLE» | Вариант 2025–2026 годов; обучает голову черновика на нескольких слоях целевой модели; alpha 0.6–0.8 на общем чате |
| `speculative_config` | «конфигурация spec в vLLM» | Явный opt-in в vLLM V1; без умолчания нет ускорения |
| N-gram спекулятивный декодинг | «N-gram черновик» | GPU-черновик с использованием N-gram поисков в промпте; совместим с фрагментированным префилом |
| Альфа безубыточности (break-even alpha) | «no-op alpha» | Alpha, при которой спекулятивный декодинг даёт нулевое ускорение; контролируйте при продакшенной параллельности |
| Двухпроходный отказ черновика (rejected-draft two-pass) | «reroll cost» | Два прохода целевой модели при отклонении черновиков; определяет хвост P99 |
