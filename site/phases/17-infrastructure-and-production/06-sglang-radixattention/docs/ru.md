# SGLang и Radix-внимание (RadixAttention) для задач с тяжёлым префиксом

> SGLang рассматривает KV-кэш (KV cache) как первоклассный переиспользуемый ресурс, хранящийся в дереве radix. Тогда как vLLM планирует запросы по принципу FCFS (первым пришёл — первым обслужен), планировщик SGLang с учётом состояния кэша (cache-aware scheduler) отдаёт приоритет запросам с более длинными общими префиксами — по сути, это обход дерева в глубину, благодаря которому горячие ветви остаются в HBM. На Llama 3.1 8B с промптами ShareGPT длиной ~1K токенов SGLang достигает ~16 200 ток/с по сравнению с ~12 500 у vLLM, что даёт преимущества ~29%. На задачах с тяжёлым префиксом (RAG) преимущество достигает 6,4×. На задачах, связанных с клонированием голоса, частота попаданий в кэш префиксов превышает 86%. Развёрнут на более чем 400 000 GPU в 2026 году в xAI, LinkedIn, Cursor, Oracle, GCP, Azure, AWS. Важное ограничение: показатель 6,4× исчезает, когда порядок префиксов нестабилен — упорядочивание является ключевым рычагом управления для инженера.

**Тип:** Изучение
**Языки:** Python (стандартная библиотека, учебный radix-кэш + планировщик с учётом кэша)
**Предварительные требования:** Фаза 17 · 04 (Внутреннее устройство обслуживания vLLM), Фаза 14 (Агентный RAG)
**Время:** ~75 минут

## Цели обучения

- Составить схему Radix-внимания: как префиксы хранятся в дереве radix и как KV-блоки переиспользуются между последовательностями, корневыми для одной ветви.
- Объяснить планирование с учётом кэша и почему FCFS не подходит для трафика с тяжёлыми префиксами.
- Вычислить ожидаемое ускорение для задачи с заданной частотой попаданий в кэш префиксов и распределением длин промптов.
- Назвать принцип упорядочивания промптов, благодаря которому показатель 6,4× становится реальным, а не потенциальным.

## Проблема

Классическое обслуживание (serving) рассматривает промпт каждого запроса как нечто непрозрачное. Даже когда 5000 RAG-запросов начинаются с одного системного промпта на 2000 токенов и одинакового вводного фрагмента извлечения, vLLM выполняет префил (prefill) этого префикса на 2000 токенов 5000 раз. GPU выполняет одну и ту же работу снова и снова.

Наблюдение: промпты в агентных и RAG-задачах почти всегда имеют длинные общие префиксы. Системный промпт, схемы инструментов, few-shot примеры, вводные фрагменты извлечения, история диалога — всё это повторяется между запросами. Если сохранить KV-кэш для такого префикса один раз и переиспользовать его, его больше не придётся выполнять заново.

Radix-внимание делает именно это. Токены индексируются в дереве radix; каждый узел владеет KV-блоками для последовательности токенов на его пути от корня. Новый запрос проходит по дереву: любой узел, чей токен совпадает, переиспользует KV-блоки этого узла. Стоимость префила становится пропорциональной «новому» суффиксу, а не всему промпту.

Сложность состоит в планировании. Если два запроса разделяют префикс на 2000 токенов, а третий разделяет только 200 токенов того же префикса, вы хотите обслужить два длинных запроса вместе, чтобы длинный префикс оставался в HBM. FCFS делает обратное — он обслуживает тот запрос, который пришёл первым, потенциально вытесняя горячую ветвь до следующего запроса с длинным префиксом.

## Концепция

### Дерево radix как индекс KV-кэша

Дерево radix (компактное префиксное дерево / compact trie) хранит последовательности токенов. Каждый узел владеет диапазоном токенов и KV-блоками, вычисленными для этого диапазона. Дочерние узлы расширяют последовательность на один или несколько токенов.

```
root
 |- "You are a helpful assistant..."  (2,000 токенов, 124 KV-блока)
      |- "Context: <doc A>..."        (500 токенов, 31 блок)
           |- "Question: Alice..."    (80 токенов, 5 блоков)
           |- "Question: Bob..."      (95 токенов, 6 блоков)
      |- "Context: <doc B>..."        (520 токенов, 33 блока)
```

Поступает новый запрос с системным префиксом + "Context: <doc A>" + "Question: Carol". Планировщик проходит по дереву: системный префикс совпадает (124 блока переиспользуются), ветвь doc-A совпадает (31 блок переиспользуется), затем выделяются новые блоки только для "Question: Carol" (4 блока). Стоимость префила: 4 блока новых токенов. Без дерева: 160 блоков. Экономия ~40× на префиле.

### Планирование с учётом кэша (cache-aware scheduling)

Переиспользование на основе дерева radix бесполезно, если кэш постоянно перезаполняется. Два ключевых принципа:

1. **Раздача в глубину (Depth-first dispatch).** При выборе следующего запроса из очереди предпочтение отдаётся запросам, находящимся в той же ветви, что и текущий набор выполняемых задач. Это удерживает горячую ветвь в памяти.
2. **LRU на уровне ветвей, а не блоков.** Вытесняются целые ветви (начиная с наименее используемых листьев), а не отдельные блоки, так что структура кэша соответствует структуре radix.

FCFS нарушает оба принципа. Запрос, разделяющий 2000 токенов, оказывается за запросом, разделяющим только 50, после чего ветвь из 2000 токенов вытесняется ради ветви из 50 токенов.

### Цифры из бенчмарков, которые стоит запомнить

- Llama 3.1 8B, H100, промпты ShareGPT по 1K токенов: SGLang ~16 200 ток/с против vLLM ~12 500 (~29% преимущества).
- Задачи RAG с тяжёлым префиксом (одинаковый системный промпт + одинаковый документ, разные вопросы): до 6,4× на SGLang.
- Задачи клонирования голоса: частота попаданий в кэш префиксов 86,4%.
- Производственные частоты попаданий у клиентов SGLang: 50–99% в зависимости от дисциплины промптов.
- Развёрнут на более чем 400 000 GPU в 2026 году.

### Подводный камень порядка

Показатель 6,4× основан на последовательном порядке шаблонов промптов. Если ваш клиент формирует промпты как `[system, tools, context, history, question]` в одних запросах и `[system, context, tools, history, question]` в других, дерево не может найти общий префикс. То, что человеку кажется общим префиксом, для дерева radix является двумя различными последовательностями.

Рычаг управления инженера: ваш шаблон промпта — это ключ кэша. Зафиксируйте порядок. Сначала поместите всё неизменяемое (системный промпт, инструменты, схемы). Затем — контекст извлечения. Последним — вопрос пользователя. Не встраивайте динамическое содержимое в префикс.

Реальный случай из исследований: вынос динамического содержимого из кэшируемого префикса повысил частоту попаданий в кэш с 7% до 74% в одном изменении.

### Где Radix-внимание выигрывает и проигрывает

Выигрывает:
- RAG (одинаковый вводный фрагмент извлечения, разные вопросы).
- Агенты (одинаковые схемы инструментов, разные запросы).
- Чат с длинным системным промптом.
- Задачи распознавания голоса/изображений с повторяющимися вводными фрагментами.

Проигрывает (возвращается к пропускной способности уровня vLLM):
- Генерация одного ответа с уникальными промптами (дополнение кода, открытый чат без системного промпта).
- Динамические промпты, где каждый запрос встраивает уникальное содержимое в префикс.

### Почему это задача планирования, а не только оптимизации на уровне ядра

Переиспользование KV-кэша можно реализовать как приём на уровне ядра. Важное понимание SGLang заключается в том, что переиспользование окупается только тогда, когда планировщик удерживает горячую ветвь в памяти. Наивная политика «переиспользовать, если доступно» будет перезаполнять кэш при смешанной нагрузке. Планировщик с индексацией по дереву radix и превращает этот приём уровня ядра в 29% преимущество в продакшне.

### Взаимодействие с vLLM

Эти две системы не являются строгими конкурентами. В 2026 году vLLM добавил кэширование префиксов (`--enable-prefix-caching`) и маршрутизатор с учётом кэша (vLLM Router на Rust). Разрыв сократился, но не исчез полностью — весь стек SGLang построен на radix с нуля; vLLM добавил эту возможность поверх. Для задач, в которых доминирует переиспользование префиксов, SGLang по-прежнему является предпочтительным решением. Для универсального обслуживания без выраженных паттернов префиксов vLLM остаётся равнозначным или лучшим вариантом.

## Использование

`code/main.py` реализует учебный KV-кэш на основе дерева radix и планировщик с двумя политиками: FCFS и с учётом кэша. Запускает одну и ту же задачу через оба режима, выводит частоту попаданий в кэш префиксов и разницу пропускной способности. Затем запускает задачу со «случайным порядком», чтобы продемонстрировать падение показателя 6,4×.

## Применение

Этот урок создаёт `outputs/skill-radix-scheduler-advisor.md`. На основе описания задачи (структура шаблона промпта, паттерн извлечения, количество одновременных арендаторов) документ формирует рецепт упорядочивания промптов и рекомендацию о целесообразности внедрения SGLang (go/no-go).

## Упражнения

1. Запустите `code/main.py`. Сравните FCFS и планирование с учётом кэша на одной задаче. Откуда берётся разница — за счёт экономии на префиле, за счёт экономии на декодировании или за счёт задержек очереди?

3. Вычислите стоимость HBM для удержания системного промпта на 2000 токенов как одной ветви radix на Llama 3.1 8B. Сравните со стоимостью батча из 16 последовательностей без переиспользования префиксов.
4. Прочитайте статью SGLang RadixAttention. Объясните в трёх предложениях, почему LRU-вытеснение по форме дерева лучше LRU по блокам при нагрузке с тяжёлыми префиксами.
5. Клиент сообщает, что частота попаданий в кэш составляет всего 8%. Назовите три вероятные причины и диагностическую процедуру для каждой из них.

## Ключевые термины

| Термин | Что говорят | Что это означает на самом деле |
|--------|-------------|-------------------------------|
| Radix-внимание (RadixAttention) | «штука от SGLang» | KV-кэш, индексированный деревом radix, чтобы общие префиксы переиспользовали блоки |
| Дерево radix (Radix tree) | «компактное префиксное дерево» | Дерево, где каждый узел владеет диапазоном токенов и его KV-блоками |
| Планировщик с учётом кэша (Cache-aware scheduler) | «горячая ветвь в первую очередь» | Планировщик, отдающий приоритет запросам, разделяющим резидентную ветвь |
| Частота попаданий в кэш префиксов (Prefix-cache hit rate) | «сколько промпта было бесплатно» | Доля токенов промпта, обслуженных из переиспользованных KV-блоков |
| FCFS | «первым пришёл — первым обслужен» | Стандартное планирование, нарушающее локальность префиксов |
| LRU на уровне ветвей (Branch-level LRU) | «вытеснение листа» | Политика вытеснения, соответствующая структуре radix |
| Порядок шаблонов промптов (Prompt template ordering) | «ключ кэша» | Порядок компонентов промпта определяет то, что дерево может переиспользовать |
| Фиксация системного промпта (System prompt pinning) | «резидентный префикс» | Удержание неизменяемой системной части для предотвращения нежелательного вытеснения |
