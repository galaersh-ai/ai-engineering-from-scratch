# Выпускной проект 09 — Агент миграции кода (обновление языка/среды выполнения на уровне репозитория)

> MigrationBench от Amazon (Java 8 → 17) и мигратор Google App Engine с Python 2 на Python 3 задают планку 2026 года. OpenRewrite от Moderne выполняет детерминированные преобразования AST в промышленном масштабе. Grit решает ту же задачу с помощью DSL в стиле codemod. Промышленный паттерн совмещает оба подхода: детерминированный слой для безопасных преобразований плюс агентный слой для неоднозначных случаев, песочница для сборки каждой ветки и тестовый каркас, который становится зелёным до открытия PR. Выпускной проект (capstone) заключается в миграции 50 реальных репозиториев и публикации процента успешных миграций с таксономией ошибок.

**Тип:** Выпускной проект
**Языки:** Python (агент), Java / Python (целевые проекты), TypeScript (панель управления)
**Предварительные требования:** Фаза 5 (NLP), Фаза 7 (трансформеры), Фаза 11 (инженерия LLM), Фаза 13 (инструменты), Фаза 14 (агенты), Фаза 15 (автономность), Фаза 17 (инфраструктура)
**Задействованные фазы:** P5 · P7 · P11 · P13 · P14 · P15 · P17
**Время:** 30 часов

## Проблема

Миграция кода (migration) в крупном масштабе — одна из наиболее чистых промышленных применений кодовых агентов 2026 года. Истина очевидна: проходит ли тестовый набор после миграции? Награда реальна: миграция Java 8 кластера — проект масштаба целого штата. А бенчмарки общедоступны: подмножество из 50 репозиториев MigrationBench. OpenRewrite от Moderne обрабатывает детерминированную часть. Агентный слой (agent layer) берёт на себя всё, что не под силу рецептам OpenRewrite: неоднозначные преобразования, рассогласование систем сборки, редкая синтаксическая задний план, поломки транзитивных зависимостей.

Вы создадите агента, который принимает репозиторий Java 8 (или Python 2) и формирует ветку с прохождением всех тестов после миграции. Вы измерите процент успешных миграций, сохранение покрытия тестами, стоимость миграции одного репозитория и построите таксономию ошибок. Сравнение с детерминированным базовым подходом покажет, где действительно кроется ценность агента.

## Концепция

Конвейер состоит из двух слоёв. **Детерминированный слой** (OpenRewrite для Java, libcst для Python) выполняет основную массу механических преобразований безопасно: импорты, сигнатуры методов, редактирование null-безопасности, try-with-resources, замена устаревших API. Он быстр и генерирует аудитируемые diffs. **Агентный слой** (OpenAI Agents SDK или LangGraph поверх Claude Opus 4.7 и GPT-5.4-Codex) обрабатывает случаи, которые не поддаются рецептам: обновление файлов сборки (Maven/Gradle/pyproject), конфликты транзитивных зависимостей, случайные падения тестов, пользовательские аннотации.

Каждый репозиторий получает песочницу (sandbox) Daytona с предустановленной целевой средой выполнения. Агент итеративно: запускает сборку, классифицирует ошибки, применяет исправление, перезапускает. Жёсткие ограничения: 30 минут на репозиторий, $8 на репозиторий, 20 ходов агента. Если все тесты пройдены и дельта покрытия не отрицательна, открывается PR. Если нет — репозиторий классифицируется по классу ошибки с приложенным доказательством.

Таксономия ошибок является основным результатом. На 50 репозиториях — что сломалось? Транзитивные зависимости? Пользовательские аннотации? Версия инструмента сборки? Случайные падения тестов, не связанные с миграцией? Каждый класс получает количество и образцовый diff. Авторы будущих рецептов смогут целенаправленно работать с тремя наиболее частыми классами.

```
      |

   (безопасно, быстро, аудируемо, ~70-80% исправлений)
      |

      |

цикл агента (Claude Opus 4.7 / GPT-5.4-Codex):
   - запуск сборки -> фиксация ошибок
   - классификация ошибок (сборка, тесты, линтер)
   - применение исправления (патч или повторный запуск рецепта)
   - повторный запуск
   - бюджет: 30 мин, $8, 20 ходов
      |
      v
врата тестов и дельты покрытия
      |
      v (пройдено)
открыть PR
      |

```

## Технологический стек

- Детерминированный слой: OpenRewrite (Java) или libcst (Python)
- Агент: OpenAI Agents SDK или LangGraph поверх Claude Opus 4.7 + GPT-5.4-Codex
- Песочница: Daytona devcontainers для каждой ветки, предустановленная целевая среда выполнения (Java 17 / Python 3.12)
- Системы сборки: Maven, Gradle, uv (Python)
- Бенчмарки: подмножество Amazon MigrationBench из 50 репозиториев (Java 8 → 17), репозитории Google App Engine с миграцией Python 2 → Python 3
- Тестовый каркас: параллельный запуск, покрытие через Jacoco (Java) или coverage.py (Python)
- Наблюдаемость: Langfuse + пакет трассировок для каждого репозитория с каждым фрагментом diff
- Панель управления: таксономия ошибок с количеством по классам и образцовыми diff

## Реализация

1. **Проход рецептов.** Запустите рецепты OpenRewrite (Java) или libcst (Python). Захватите 70-80% миграций, являющихся механическими. Зафиксируйте как коммит «recipe».

2. **Пробная сборка.** Песочница Daytona: установите целевую среду выполнения, запустите сборку. Если зелёная — переходите к тестам. Если красная — передайте агенту.

3. **Цикл агента.** LangGraph с инструментами: `run_build`, `read_file`, `edit_file`, `run_test`, `git_diff`. Агент классифицирует ошибку (зависимость, синтаксис, тест, инструмент сборки) и применяет целевое исправление. Перезапуск.

4. **Бюджетные ограничения.** 30 минут реального времени на репозиторий, $8, 20 ходов агента. При превышении — остановка и классификация как «budget_exhausted» с текущим diff.

5. **Врата тестов и покрытия.** После зелёной сборки запустите тестовый набор. Сравните покрытие с базовым репозиторием. Если покрытие упало более чем на 2%, классифицируйте как «coverage_regression».

6. **Открытие PR.** При успехе отправьте ветку и откройте PR с diff и описанием того, какие рецепты были применены и какие коммиты создал агент.

7. **Таксономия ошибок.** Для каждого неудачного репозитория присвойте класс: `dep_upgrade_required`, `build_tool_drift`, `custom_annotation`, `test_flake`, `syntax_edge_case`, `budget_exhausted`. Создайте панель управления.

8. **Запуск на 50 репозиториях.** Выполните на подмножестве MigrationBench. Отчитайтесь о процентах успешных миграций по классам, стоимости за репозиторий, сохранении покрытия и сравнении с чисто детерминированным базовым подходом.

## Упражнения

1. Запустите конвейер миграции используя только OpenRewrite (без агента). Сравните процент успешных миграций с полным конвейером. Определите случаи, в которых именно агент является решающим фактором.

2. Реализуйте проверку «чистоты линтера»: после миграции запустите линтер стиля (spotless для Java, ruff для Python). Считайте PR непрошедшим, если появились новые ошибки линтера. Измерьте частоту сохранения покрытия при ухудшении стиля.

3. Добавьте оптимизатор «минимального diff»: после прохождения тестов веткой агента устраните лишние изменения вторым проходом. Отчитайтесь об уменьшении размера diff.

4. Расширьте на третью миграцию: Node 18 → Node 22. Переиспользуйте обёртку песочницы; замените слой рецептов на пользовательский codemod.

5. Измерьте время до первой успешной сборки (TTFGB) как метрику удобства использования. Цель: p50 менее 10 минут.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|-----------------|------------------------|
| Детерминированный слой | «Движок рецептов» | OpenRewrite / libcst: декларативные преобразования AST с гарантиями безопасности |
| Codemod | «Программа, изменяющая код» | Правило преобразования, механически изменяющее исходный код |
| Рассогласование сборки | «Несоответствие версий инструментов» | Тонкие различия поведения Maven / Gradle / uv между мажорными версиями |
| Класс ошибки | «Категория таксономии» | Маркированная причина, по которой репозиторий не был мигрирован: зависимость, синтаксис, тест, инструмент сборки, бюджет |
| Дельта покрытия | «Сохранение покрытия» | Изменение процента покрытия тестами от базовой ветки к мигрированной |
| Ход агента | «Раунд вызова инструментов» | Один цикл план → действие → наблюдение в цикле агента |
| Исчерпание бюджета | «Достигнут лимит» | Репозиторий израсходовал лимит 30 мин / $8 / 20 ходов без прохождения |
