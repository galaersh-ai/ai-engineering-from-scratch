# Выпускной проект 11 — Панель наблюдаемости (observability dashboard) и оценки LLM

> Langfuse перешёл на модель open-core. Arize Phoenix опубликовал маппинги семантических соглашений (semconv) для GenAI 2026 года. Helicone и Braintrust значительно расширили возможности атрибуции затрат по пользователям. OpenLLMetry от Traceloop стала де-факто стандартом для инструментирования SDK. В продакшене используется ClickHouse для трассов (tracing), Postgres для метаданных, Next.js для UI и небольшой набор задач оценки (DeepEval, RAGAS, LLM-судья), работающих по отобранным трассам. Создайте самостоятельно размещённую систему, принимающую данные как минимум из четырёх семейств SDK, и продемонстрируйте обнаружение искусственно внесённой регрессии менее чем за пять минут.

**Тип:** Выпускной проект (capstone)
**Языки:** TypeScript (UI), Python / TypeScript (приём + оценки), SQL (ClickHouse)
**Предварительные требования:** Фаза 11 (инженерия LLM), Фаза 13 (инструменты), Фаза 17 (инфраструктура), Фаза 18 (безопасность)
**Затронутые фазы:** P11 · P13 · P17 · P18
**Время:** 25 часов

## Проблема

Каждая AI-команда, обрабатывающая продакшен-трафик в 2026 году, ведёт плоскость наблюдаемости (observability) наряду с моделью. Атрибуция затрат. Обнаружение галлюцинаций. Мониторинг дрифта. Сигналы джейлбрейка. Панели SLO. Оповещения об утечках PII. Открытые решения — Langfuse, Phoenix, OpenLLMetry — пришли к семантическим соглашениям OpenTelemetry GenAI как к схеме приёма данных. Теперь можно инструментировать OpenAI, Anthropic, Google, LangChain, LlamaIndex и vLLM одним SDK и отправлять совместимые спаны.

Вы создадите самостоятельно размещённую панель (dashboard), принимающую данные как минимум из четырёх семейств SDK, выполняющую набор задач оценки по отобранным трассам, обнаруживающую дрифт и отправляющую оповещения. Критерий измерения: при целенаправленно внесённой регрессии (промпт, начинающий выдавать PII) панель должна обнаружить её и отправить оповещение менее чем за пять минут.

## Концепция

Приём данных осуществляется по OTLP HTTP. SDK генерируют спаны с семантическими соглашениями GenAI: `gen_ai.system`, `gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.response.id`, `llm.prompts`, `llm.completions`. Спаны хранятся в ClickHouse для столбцовой аналитики; метаданные (пользователи, сессии, приложения) — в Postgres.

Задачи оценки выполняются пакетно по отобранным трассам. DeepEval оценивает добросовестность, токсичность и релевантность ответа. RAGAS вычисляет метрики извлечения, когда трасса содержит контекст извлечения. Пользовательские LLM-судьи выполняют проверки специфические для предметной области (утечка PII, ответы вне политики). Результаты оценок записываются в тот же ClickHouse в виде спанов оценки, связанных с родительской трассой.

Обнаружение дрифта отслеживает распределения в пространстве эмбеддингов (индекс стабильности популяции PSI или расхождение KL по эмбеддингам промптов) и тенденции оценок. Оповещения направляются в Prometheus Alertmanager, а оттуда — в Slack / PagerDuty. Интерфейс построен на Next.js 15 с Recharts.

## Архитектура

```
продакшен-приложения:
  OpenAI SDK  +  Anthropic SDK  +  Google GenAI SDK
  LangChain + LlamaIndex + vLLM
       |
       v
  OpenTelemetry SDK с семантическими соглашениями GenAI
       |
       v  OTLP HTTP
  коллектор (приём, выборка, маршрутизация)
       |
       +-------------+-----------+
       v             v           v
   ClickHouse    Postgres    S3-архив
   (спаны)       (метаданные)  (сырые события)
       |
       +---> задачи оценки (DeepEval, RAGAS, LLM-судья)
       |     по отбору или по всей трассе
       |     записывают спаны оценки обратно
       |
       +--> детектор дрифта (PSI / KL по эмбеддингам промптов)
       |
       +---> метрики Prometheus -> Alertmanager -> Slack / PagerDuty
       |
       v
   Next.js 15 панель (Recharts)
```

## Технологический стек

- Приём: SDK OpenTelemetry + семантические соглашения GenAI; транспорт OTLP HTTP
- Коллектор: OpenTelemetry Collector с процессором хвостовой выборки (tail-sampling) для контроля затрат
- Хранилище: ClickHouse для спанов, Postgres для метаданных, S3 для архива сырых событий
- Оценки: DeepEval, RAGAS 0.2, пакет оценки Arize Phoenix, пользовательский LLM-судья
- Дрифт: PSI / KL по совокупным эмбеддингам промптов (sentence-transformers), еженедельно
- Оповещения: Prometheus Alertmanager -> Slack / PagerDuty
- UI: Next.js 15 App Router + Recharts + server actions
- Поддерживаемые SDK из коробки: OpenAI, Anthropic, Google GenAI, LangChain, LlamaIndex, vLLM

## Реализация

1. **Конфигурация коллектора.** OpenTelemetry Collector с приёмником OTLP HTTP, хвостовым сэмплером, сохраняющим 100% ошибочных трасс и 10% успешных, и экспортерами в ClickHouse и S3.

2. **Схема ClickHouse.** Таблица `spans` со столбцами, соответствующими семантическим соглашениям GenAI: `gen_ai_system`, `gen_ai_request_model`, `input_tokens`, `output_tokens`, `latency_ms`, `prompt_hash`, `trace_id`, `parent_span_id`, а также JSON-столбец для длинных полезных нагрузок. Добавьте вторичные индексы по `user_id` и `app_id`.

3. **Тест покрытия SDK.** Напишите небольшое клиентское приложение, использующее каждый SDK (OpenAI, Anthropic, Google, LangChain, LlamaIndex, vLLM) с автоматической инструментацией OpenLLMetry. Убедитесь, что каждый генерирует канонические GenAI-спаны, попадающие в ClickHouse.

4. **Задачи оценки.** Периодическая задача читает отобранные трассы за последние 15 минут и запускает DeepEval для оценки добросовестности, токсичности и релевантности ответа. Результаты — спаны оценки, связанные с родительской трассой.

5. **Пользовательский LLM-судья.** Судья по утечке PII: по полученному ответу вызывается защитная LLM для оценки вероятности утечки PII. Ответы с высокими баллами попадают в очередь разбора.

6. **Обнаружение дрифта.** Еженедельная задача вычисляет PSI между совокупными эмбеддингами промптов текущей недели и бейзлайном за предыдущие 4 недели. Если PSI превышает порог — генерируется оповещение.

7. **Панель.** Next.js 15 со страницами: обзор (спанов/сек, стоимость/пользователь, p95 латентность), трассы (поиск + водопад), оценки (тренд добросовестности, токсичность), дрифт (PSI во времени), оповещения.

8. **Цепочка оповещений.** Экспортер Prometheus считывает агрегированные оценки и перцентили латентности; Alertmanager маршрутизирует предупреждения в Slack, а критические нарушения — в PagerDuty.

9. **Зонд регрессии.** Внесите ошибку: оцениваемый чат-бот начинает выдавать поддельные SSN в 1% случаев. Измерьте MTTR: от момента внесения ошибки до оповещения в Slack.

## Упражнения

1. Добавьте пользовательскую инструментацию для фреймворка Haystack. Убедитесь, что канонические спаны с корректными атрибутами `gen_ai.*` попадают в ClickHouse.

2. Замените DeepEval на оценщики Phoenix для тех же трасс. Измерьте разброс оценок между двумя системами оценки.

3. Уточните детектор дрифта: вычисляйте PSI по каждому `app-id` вместо глобального значения. Отобразите дрифт-трейлы по приложениям.

4. Добавьте страницу «Влияние на пользователя»: стоимость на пользователя и частота ошибок на пользователя с графирами.

5. Разработайте политику хвостовой выборки, сохраняющую 100% трасс с токсичностью > 0,5 плюс 10% стратифицированную выборку остальных. Измерьте внесённое смещение выборки.

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|------|-----------------|------------------------|
| GenAI semconv | «Атрибуты OTel для LLM» | Спецификация OpenTelemetry 2025 для атрибутов спанов LLM (система, модель, токены) |
| Хвостовая выборка (Tail sampling) | «Выборка после трассы» | Коллектор решает сохранить или отбросить трассу после её завершения (может увидеть ошибки) |
| PSI | «Индекс стабильности популяции» | Метрика дрифта, сравнивающая два распределения; значение > 0,2 обычно означает значимый дрифт |
| LLM-судья | «Оценка в виде модели» | LLM, оценивающая вывод другой LLM по критериям (добросовестность, токсичность, PII) |
| Политика хвостовой выборки (Tail-sampling policy) | «Правило сохранения» | Правило, определяющее, какие трассы сохранять, а какие отбрасывать; ошибочные + коэффициент выборки |
| Спан оценки (Eval span) | «Связанный трасс оценки» | Дочерний спан с результатом оценки, связанный с исходным спаном вызова LLM |
| Стоимость на пользователя (Cost per user) | «Юнит-экономика» | Стоимость в долларах, атрибутированная по `user_id` за период; ключевая продуктовая метрика |
