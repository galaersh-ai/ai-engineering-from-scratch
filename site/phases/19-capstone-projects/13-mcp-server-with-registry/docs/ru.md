# Выпускной проект 13 — MCP-сервер с реестром и управлением доступа

> Протокол Model Context Protocol перестал быть технологией будущего и стал стандартным спецификацией для использования инструментов в 2026 году. Anthropic, OpenAI, Google и все крупные IDE поставляют MCP-клиенты. Pinterest опубликовал свою внутреннюю экосистему MCP-серверов. Реестр (registry) AAIF формализовал метаданные возможностей в `.well-known`. AWS ECS опубликовал эталонный беспостоянный (stateless) вариант развёртывания. Агент goose от Block внедрил тот же протокол в hosted-ассистента. Продакшн-архитектура 2026 года: транспорт StreamableHTTP, области видимости OAuth 2.1, контроль политик OPA и реестр (registry), позволяющий командам платформы обнаруживать, проверять и подключать серверы. Реализуйте всё это с нуля.

**Тип:** Выпускной проект
**Языки:** Python (сервер, через FastMCP) или TypeScript (@modelcontextprotocol/sdk), Go (сервис реестра)
**Предварительные требования:** Фаза 11 (инженерия LLM), Фаза 13 (инструменты и MCP), Фаза 14 (агенты), Фаза 17 (инфраструктура), Фаза 18 (безопасность)
**Задействованные фазы:** P11 · P13 · P14 · P17 · P18
**Время:** 25 часов

## Проблема

MCP стал lingua franca для использования инструментов. Claude Code, Cursor 3, Amp, OpenCode, Gemini CLI и все управляемые агенты теперь потребляют MCP-серверы. Продакшн-задачи состоят не в написании серверов (FastMCP делает это простым), а в их развёртывании в масштабе с учётом корпоративных требований: области видимости (scopes) OAuth для каждого клиента, политики OPA для деструктивных инструментов, беспостоянный (stateless) масштабирование через StreamableHTTP, реестр для обнаружения, журналы аудита для каждого вызова инструмента. Внутренняя экосистема MCP от Pinterest и спецификация реестра AAIF задают планку на 2026 год.

Вы создадите MCP-сервер, предоставляющий 10 внутренних инструментов (только чтение из Postgres, список объектов S3, Jira, Linear, Datadog и др.), интерфейс реестра (registry UI) для обнаружения на уровне платформы и механизм одобрения человеком для деструктивных инструментов. Нагрузочное тестирование продемонстрирует горизонтальное масштабирование StreamableHTTP. Журнал аудита удовлетворит требованиям корпоративной проверки безопасности.

## Концепция

Ревизия MCP 2026 года устанавливает StreamableHTTP как транспорт по умолчанию. В отличие от предыдущего варианта с stdio и SSE, StreamableHTTP по умолчанию беспостоянный (stateless): единственный HTTP-эндпоинт принимает запросы JSON-RPC, передаёт ответы потоком и поддерживает долгоживущие соединения для уведомлений. Беспостоянность означает возможность горизонтального масштабирования за балансировщиком нагрузки.

Авторизация основана на OAuth 2.1 с областями видимости для каждого инструмента. Токен несёт области вида `jira:read`, `s3:list`, `postgres:query:readonly`. MCP-сервер проверяет области видимости при вызове инструмента, а не только при начале сеанса. Для высокорисковых инструментов сервер отклоняет любой вызов, чья область видимость не была повышена до `approved:by:human` за последние N минут — повышение осуществляется через карточку одобрения в Slack.

Реестр (registry) — это отдельный сервис. Каждый MCP-сервер предоставляет документ `.well-known/mcp-capabilities` со своим манифестом инструментов, URL транспорта, требованиями авторизации. Реестр (registry) опрашивает, проверяет и индексирует серверы. Команды платформы используют интерфейс реестра для просмотра доступных инструментов, необходимых областей видимости и владельцев.

## Архитектура

```
MCP-клиент (Claude Code, Cursor 3, ...)
          |
          v
StreamableHTTP поверх HTTPS (JSON-RPC + потоковая передача)
          |
          v
MCP-сервер (FastMCP) за балансировщиком нагрузки
          |
   +------+------+---------+----------+------------+
   v             v         v          v            v
Postgres    Список S3   Jira       Linear     Datadog
(только чтение) (постранично) (чтение)  (чтение)   (запрос)
          |
   +------+-------------+
   v                    v
Политика OPA       Деструктивный инструмент MCP (отдельный сервер)
                        |
                        v
                   Одобрение человека через Slack
                        |
                        v
                   Журнал аудита (только добавление, на клиента)

  Сервис реестра (registry)
     |
     v  GET /.well-known/mcp-capabilities от каждого сервера
     v
     Интерфейс: поиск / проверка / включение-отключение / владелец
```

## Технологический стек

- Фреймворк сервера: FastMCP (Python) или `@modelcontextprotocol/sdk` (TypeScript)
- Транспорт: StreamableHTTP поверх HTTPS (беспостоянный)
- Авторизация: OAuth 2.1 с идентификацией рабочей нагрузки через SPIFFE / SPIRE
- Политики: правила OPA / Rego для каждого инструмента; служба принятия решений по политикам для каждого запроса
- Реестр (registry): self-hosted, потребляет манифесты `.well-known/mcp-capabilities`
- Одобрение человека: интерактивное сообщение Slack для деструктивных инструментов
- Развёртывание: AWS ECS Fargate или Fly.io, один сервер на клиента или общий с разделением по клиентам
- Аудит: структурированный JSONL в бакете на клиента с линией вызовов

## Реализация

1. **Интерфейс инструментов.** Предоставьте 10 внутренних инструментов: запрос только для чтения в Postgres, список объектов S3, поиск/получение в Jira, поиск/получение в Linear, запрос метрик Datadog, справочник дежурных PagerDuty, GitHub только для чтения, поиск в Notion, поиск в Slack, чтение из Salesforce. Каждый инструмент имеет типизированную схему и метку области видимости.

2. **Сервер FastMCP.** Подключите инструменты. Настройте транспорт StreamableHTTP. Добавьте промежуточный слой для интроспекции токенов OAuth и контроля областей видимости.

3. **Политика OPA.** Правило Rego для каждого инструмента: какие области видимости допускают вызов, какая маскировка PII применяется, какие ограничения на размер полезной нагрузки установлены. Служба принятия решений вызывается при каждом вызове инструмента.

4. **Сервис реестра (registry).** Отдельный сервис на Go или TS, который опрашивает `.well-known/mcp-capabilities` у зарегистрированных серверов, проверяет с помощью JSON Schema и предоставляет интерфейс списка / поиска / проверки / включения-отключения.

5. **Манифест возможностей.** Каждый сервер предоставляет `.well-known/mcp-capabilities` с: списком инструментов, требованиями авторизации, URL транспорта, командой-владельцем, SLO.

6. **Изоляция деструктивных инструментов.** Инструменты, изменяющие состояние (создание в Jira, создание в Linear, запись в Postgres), размещаются на втором MCP-сервере со строгим процессом авторизации: токены должны иметь область видимости `approved:by:human`, повышённую через карточку Slack в течение 15 минут.

7. **Журнал аудита.** Дополнение только в конец формата JSONL на клиента: `{timestamp, user, tool, args_redacted, response_redacted, outcome}`. Маскировка PII через Presidio перед записью.

8. **Нагрузочное тестирование.** 100 одновременных клиентов через StreamableHTTP. Продемонстрируйте горизонтальное масштабирование, добавив второй реплику; покажите перераспределение нагрузки балансировщиком без привязки к сеансу.

9. **Тесты на соответствие.** Запустите официальный набор тестов на соответствие MCP для обоих серверов. Пройдите все обязательные разделы.

## Использование

```
$ curl -H "Authorization: Bearer ***" \
       -X POST https://mcp.internal.example.com/ \
       -d '{"jsonrpc":"2.0","method":"tools/call",

[policy]    scope postgres:query:readonly present; allowed

```

## Упражнения

1. Добавьте новый инструмент (поиск в Confluence). Внедрите его через процесс проверки реестра без изменения основного сервера.

2. Напишите политику OPA, маскирующую результаты запросов Postgres, содержащие столбцы с именами `email`, `ssn` или `phone`. Проверьте с помощью тестового запроса.

3. Сравните производительность StreamableHTTP и stdio по локальной задержке. Приведите медианное (p50) и 95-процентное (p95) значения для каждого вызова.

4. Реализуйте квоту на клиента: максимальное количество N вызовов в минуту для каждого инструмента на клиента. Ограничьте с помощью второго правила OPA.

5. Запустите набор тестов на соответствие MCP из [mcp-conformance-tests](https://github.com/modelcontextprotocol/conformance) и исправьте каждую ошибку.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|-----------------|------------------------|
| StreamableHTTP | «Транспорт MCP 2026» | Беспостоянный HTTP + потоковая передача; заменяет SSE + stdio для сетевых серверов |
| Манифест возможностей | «Документ well-known» | `.well-known/mcp-capabilities` со списком инструментов, авторизацией, URL транспорта |
| OPA / Rego | «Движок политик» | Open Policy Agent для авторизации вызовов инструментов на основе внешних правил |
| Повышение области видимости | «Одобрено человеком» | Краткосрочная область видимости, предоставляемая через одобрение в Slack, обязательная для деструктивных инструментов |
| Реестр (registry) | «Обнаружение инструментов» | Сервис, индексирующий MCP-серверы на основе их манифестов возможностей |
| Идентификация рабочей нагрузки | «SPIFFE / SPIRE» | Криптографическая идентификация сервиса для выдачи токенов OAuth |
| Набор тестов на соответствие | «Тесты спецификации» | Официальный набор тестов MCP для проверки корректности StreamableHTTP и манифестов инструментов |
