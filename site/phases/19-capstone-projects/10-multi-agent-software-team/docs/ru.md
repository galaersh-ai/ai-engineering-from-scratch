# Выпускной проект 10 — Мультиагентная команда разработки ПО

> Архитектура «фабрики» SWE-AF, ролевое промптирование MetaGPT, типизированный граф акторов AutoGen 0.4, Devin от Cognition и Droids от Factory — все они к 2026 году сошлись к единой форме: архитектор планирует, N программистов работают параллельно в рабочих деревьях (worktree), ревьюер проверяет, тестировщик верифицирует. Параллельные рабочие деревья превращают реальное время в пропускную способность. Общее состояние и протоколы передачи задач становятся поверхностью отказов. Выпускной проект (capstone) — построить такую команду, оценить на SWE-bench Pro и опубликовать отчёт о том, какие передачи ломаются и как часто.

**Тип:** Выпускной проект
**Языки:** Python / TypeScript (агенты), Shell (скрипты рабочих деревьев)
**Предварительные знания:** Фаза 11 (инженерия LLM), Фаза 13 (инструменты), Фаза 14 (агенты), Фаза 15 (автономия), Фаза 16 (мультиагентность), Фаза 17 (инфраструктура)
**Задействованные фазы:** P11 · P13 · P14 · P15 · P16 · P17
**Время:** 40 часов

## Проблема

Одноагентные среды разработки упираются в потолок при работе над крупными задачами. Не потому, что какой-то отдельный агент слаб, а потому, что контекст в 200 тысяч токенов не может вместить план архитектуры, четыре параллельных среза кодовой базы, комментарии ревьюера и результаты тестов. Мультиагентные (multi-agent) «фабрики» разделяют задачу: архитектор владеет планом, программисты — реализацией в параллельных рабочих деревьях, ревьюер проверяет, тестировщик верифицирует. «Фабричная» архитектура SWE-AF, ролевая модель MetaGPT, типизированный граф акторов AutoGen — все три описывают одну и ту же форму.

Поверхностью отказов является передача задач. Архитектор планирует то, что программисты не могут реализовать. Программисты создают конфликтующие патчи. Ревьюер одобряет галлюцинированное исправление. Тестировщик обгоняет ещё не завершившего работу программиста. Вы построите одну из таких команд, запустите её на 50 задачах из SWE-bench Pro, отслежите каждую передачу и опубликуете разбор ошибок.

## Концепция

Роли — это типизированные агенты. **Архитектор** (Claude Opus 4.7) читает задачу, составляет план и разбивает его на подзадачи с явными интерфейсами. **Программисты** (Claude Sonnet 4.7, N параллельных экземпляров, каждый в `git worktree` + песочнице Daytona) реализуют подзадачи независимо. **Ревьюер** (GPT-5.4) читает объединённый дифф и либо одобряет, либо запрашивает конкретные исправления. **Тестировщик** (Gemini 2.5 Pro) запускает набор тестов в изолированной среде и сообщает о прохождении/непрохождении с артефактами.

Связь осуществляется через общую доску задач (на файле или Redis). Каждая роль берёт задачи, которые ей разрешено обрабатывать. Передачи — это типизированные сообщения по протоколу A2A (agent-to-agent). Проблемы координации: разрешение конфликтов слияний (роль координатора или автоматическое трёхстороннее слияние), синхронизация общего состояния (план замораживается, когда программисты начинают работу; перепланирование — отдельное событие) и проверка ревьюера (ревьюер не может одобрить собственные изменения или изменения, которые он сам предложил).

Усиление расхода токенов (token amplification) — это скрытая стоимость. На каждой границе ролей добавляются промпты-резюме и контекст передачи. Запуск из 40 ходов в одноагентном режиме превращается в 160 ходов суммарно на четыре роли. Критерии оценки специально взвешивают эффективность по токенам относительно одноагентного варианта, поскольку вопрос стоит не «работает ли мультиагентный подход», а «побеждает ли он по соотношению цена/качество».

## Архитектура

```
URL задачи в GitHub
      |
      v
Архитектор (Opus 4.7)
   читает задачу, формирует план с подзадачами и интерфейсами
      |
      v
Доска задач (файл / Redis)
      |
   +-- подзадача 1 ---+-- подзадача 2 ---+-- подзадача 3 ---+-- подзадача 4 ---+
   v                  v                  v                  v                  v
Программист A      Программист B      Программист C      Программист D     (4 параллельных)
 (Sonnet)           (Sonnet)           (Sonnet)           (Sonnet)
 рабочее дерево A   рабочее дерево B   рабочее дерево C   рабочее дерево D
 Daytona            Daytona            Daytona            Daytona
      |                  |                  |                  |
      +--------+---------+---------+--------+
               v
          Координатор слияний  (трёхстороннее слияние + разрешение конфликтов)
               |
               v
          Ревьюер (GPT-5.4)
               |
               v
          Тестировщик  (Gemini 2.5 Pro)  -> проходит? -> открываем PR
                                          -> не проходит? -> возвращаем программисту
```

## Технологический стек

- Оркестрация (orchestration): LangGraph с общим состоянием и подграфами для каждого агента
- Обмен сообщениями: протокол A2A (Google, 2025) для типизированных межагентных сообщений
- Модели: Opus 4.7 (архитектор), Sonnet 4.7 (программисты), GPT-5.4 (ревьюер), Gemini 2.5 Pro (тестировщик)
- Изоляция рабочих деревьев: `git worktree add` для каждого программиста + песочница Daytona
- Координатор слияний: собственное трёхстороннее слияние + разрешение конфликтов при помощи LLM
- Оценка: SWE-bench Pro (50 задач), сценарии SWE-AF, HumanEval++ для модульных тестов
- Наблюдаемость: Langfuse с разметкой по ролям, учёт токенов для каждого агента
- Развёртывание: K8s с отдельным Deployment для каждой роли и HPA по размеру очереди

## Порядок реализации

1. **Доска задач.** Файловая реализация на JSONL с типизированными сообщениями: `plan_request`, `subtask`, `diff_ready`, `review_needed`, `test_needed`, `approved`, `rejected`, `replan_needed`. Агенты подписываются на теги.

2. **Архитектор.** Читает задачу из GitHub, запускает Opus 4.7 с шаблоном плана, требующим явных интерфейсов подзадач (затрагиваемые файлы, публичные функции, влияние на тесты). Выпускает одну `plan_request` с DAG подзадач.

3. **Программисты.** N параллельных воркеров, каждый берёт одну подзадачу с доски. Каждый создаёт новую ветку через `git worktree add` и песочницу Daytona. Реализует подзадачу. Выпускает `diff_ready` с патчем и набором изменённых тестов.

4. **Координатор слияний.** По завершении всех программистов выполняет трёхстороннее слияние N веток в промежуточную ветку. Разрешение конфликтов при помощи LLM — только при пересечении файлов на уровне файловой системы.

5. **Ревьюер.** GPT-5.4 читает объединённый дифф. Не может одобрять диффы, которые он сам написал. Выпускает `approved` (бездействие) или `review_feedback` с запросами на конкретные изменения, направляемыми обратно соответствующему программисту.

6. **Тестировщик.** Gemini 2.5 Pro запускает набор тестов в чистой песочнице. Сохраняет артефакты. Выпускает `test_passed` или `test_failed` с трассировками стека. Непрошедшие тесты возвращаются программисту, ответственному за соответствующую подзадачу.

7. **Учёт передач.** Каждое сообщение, пересекающее границу ролей, получает спан в Langfuse с размером полезной нагрузки и использованной моделью. Вычисляется усиление расхода токенов на подзадачу (программист_токены + ревьюер_токены + тестировщик_токены + доля_архитектора / программист_токены).

8. **Оценка.** Запуск на 50 задачах из SWE-bench Pro. Сравнение pass@1 и стоимости на решённую задачу с одноагентным вариантом (один Sonnet 4.7 в одном рабочем дереве).

9. **Разбор ошибок.** Для каждой нерешённой задачи определяется, какая передача сломалась (план слишком расплывчатый, конфликт слияния, ложное одобрение ревьюером, случайный сбой тестировщика). Составляется гистограмма отказов передач.

## Готовый результат

`outputs/skill-multi-agent-team.md` — это конечный артефакт. При указании URL задачи и уровня параллелизма команда создаёт PR, готовый к слиянию, с разбивкой расхода токенов по ролям.

| Вес | Критерий | Как измеряется |
|:-:|---|---|

| 20 | Параллельное ускорение | Реальное время относительно одноагентного варианта |
| 20 | Качество ревью | Частота ложных одобрений при внедрённых ошибках |
| 20 | Эффективность токенов | Совокупные токены на решённую задачу относительно одноагентного варианта |
| 15 | Инженерия координации | Разрешение конфликтов слияний, гистограмма отказов передач |
| **100** | | |

## Упражнения

1. Внедрите очевидную ошибку в дифф во время выполнения (лишний `return None` перед основным телом). Измерьте частоту ложных одобрений ревьюером. Настройте промпт ревьюера так, чтобы частота ложных одобрений не превышала 5%.

2. Сократите до двух программистов (архитектор + программист + ревьюер + тестировщик, программист выполняет две подзадачи последовательно). Сравните реальное время и частоту прохождения.

3. Замените координатора слияний на ограничение «один автор на запись» (подзадачи затрагивают непересекающиеся наборы файлов). Измерьте нагрузку на планирование со стороны архитектора.

4. Замените ревьюера с GPT-5.4 на Claude Opus 4.7. Измерьте частоту ложных одобрений и изменение расхода токенов.

5. Добавьте пятую роль: документатор (Haiku 4.5). После ревью он формирует запись в журнале изменений. Измерьте, оправдывает ли качество документации дополнительный расход токенов.

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|------|---|---|
| Параллельное рабочее дерево | «Изолированная ветка» | `git worktree add`, создающий свежее рабочее дерево для каждого программиста |
| Доска задач | «Общая шина сообщений» | Файловое или Redis-хранилище типизированных сообщений, на которые подписываются агенты |
| Передача | «Граница ролей» | Любое сообщение, пересекающее контекст одной роли и попадающее в контекст другой |
| Усиление расхода токенов | «Накладные расходы мультиагентности» | Совокупные токены на все роли / токены одноагентного варианта для той же задачи |
| Протокол A2A | «Агент-к-агенту» | Спецификация Google 2025 года для типизированных межагентных сообщений |
| Координатор слияний | «Интегратор» | Компонент, выполняющий трёхстороннее слияние и урегулирование конфликтов |
| Ложное одобрение | «Галлюцинация ревьюера» | Ревьюер одобряет дифф, содержащий известные ошибки |
