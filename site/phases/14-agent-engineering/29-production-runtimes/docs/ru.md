# Среды выполнения (Production Runtimes): очередь, событие, cron

> Промышленные агенты (Production agents) работают в шести формах сред выполнения (runtime shapes): запрос-ответ (request-response), потоковая передача (streaming), устойчивое выполнение (durable execution), фоновая обработка на основе очередей (queue-based background), событийно-ориентированная (event-driven) и по расписанию (scheduled). Выберите форму среды выполнения до выбора фреймворка. Наблюдаемость (Observability) является неотъемлемой частью любой формы.

**Тип:** Теория
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 13 (LangGraph), Фаза 14 · 22 (Голос / Voice)
**Время:** ~60 минут

## Цели обучения

- Назвать шесть форм сред выполнения в продакшене и сопоставить каждую с конкретным фреймворком / продуктом.
- Объяснить, почему устойчивое выполнение (durable execution, LangGraph) важно для задач с длинным временным горизонтом (long-horizon).
- Описать событийно-ориентированную среду выполнения и указать, когда подходит Claude Managed Agents.
- Обосновать утверждение о том, что наблюдаемость (observability) является неотъемлемой частью многошаговых агентов.

## Проблема

Промышленные агенты (Production agents) выходят из строя способами, которые Jupyter-ноутбук не обнаруживает: сетевые тайм-ауты на 37-м шаге, пользователь кладёт трубку во время голосового вызова, задание cron погибает при перезагрузке машины, фоновый воркер (worker) исчерпывает память. Форма среды выполнения определяет, какие сбои являются восстанавливаемыми.

## Концепция

### Запрос-ответ (Request-response)

- Синхронный HTTP. Пользователь ждёт завершения.
- Применимо только для коротких задач (<30 с).
- Стеки (Stacks): Agno (Python + FastAPI), Mastra (TypeScript + Express/Hono/Fastify/Koa).
- Наблюдаемость (Observability): стандартные HTTP-логи доступа + спаны OTel.

### Потоковая передача (Streaming)

- SSE или WebSocket для постепенного вывода результатов.
- LiveKit расширяет это до WebRTC для голоса/видео (Урок 22).
- Стеки: любой фреймворк с поддержкой потоковой передачи + фронтенд, обрабатывающий SSE/WS.
- Наблюдаемость: время обработки каждого фрагмента (chunk), задержка первого токена (first-token latency), хвостовая задержка (tail latency).

### Устойчивое выполнение (Durable execution)

- Состояние сохраняется в чекпоинтах (checkpointed) после каждого шага; автоматически возобновляется при сбое.
- Модель акторов (actor model) AutoGen v0.4 изолирует сбои в пределах одного агента (Урок 14).
- Ключевое отличие LangGraph (Урок 13).
- Необходимо, когда количество шагов неизвестно, а стоимость восстановления высока.

### На основе очередей / фоновая обработка (Queue-based / background)

- Задача поступает в очередь (queue), воркеры (workers) её обрабатывают, результаты возвращаются через вебхуки (webhooks) или публикацию/подписку (pub/sub).
- Необходимо для долгосрочных агентов (long-horizon agents) (десятки–сотни шагов на задачу, согласно объявлению Anthropic о компьютерном использовании).
- Стеки: Celery (Python), BullMQ (Node), SQS + Lambda (AWS), собственные решения.
- Наблюдаемость: глубина очереди, распределение задержек по задачам, размер очереди мёртвых писем (DLQ).

### Событийно-ориентированная (Event-driven)

- Агенты подписываются на триггеры: новое электронное письмо, открытие PR, срабатывание cron.
- Claude Managed Agents покрывает этот сценарий из коробки (Урок 17).
- CrewAI Flows (Урок 15) структурирует детерминированные событийно-ориентированные воркфлоу (workflows).
- Наблюдаемость: источник триггера, задержка от события до начала выполнения, задержка агента.

### По расписанию (Scheduled)

- Агенты в формате cron, выполняющиеся периодически.
- Комбинируются с устойчивым выполнением, чтобы неудачный ночной запуск возобновился при следующем тике.
- Стеки: Kubernetes CronJob + фреймворк устойчивого выполнения; управляемые решения (Render cron, Vercel cron).

### Паттерны развёртывания 2026 года

- **CrewAI Flows** для событийно-ориентированного промышленного использования.
- **Agno** — не сохраняющее состояние (stateless) FastAPI для Python-микросервисов.
- **Mastra** — серверные адаптеры (Express, Hono, Fastify, Koa) для встраивания.
- **Pipecat Cloud / LiveKit Cloud** для управляемого голосового взаимодействия (Урок 22).
- **Claude Managed Agents** для управляемых долгосрочных асинхронных задач.

### Наблюдаемость — неотъемлемая часть (Observability is load-bearing)

Без спанов OpenTelemetry GenAI (Урок 23) и бэкенда Langfuse/Phoenix/Opik (Урок 24) невозможно отладить многошагового агента, вышедшего из строя на 40-м шаге. В промышленном использовании это не является опцией. Разница между «мы отлаживаем быстро» и «мы воспроизводим всё с нуля с дополнительным логированием».

### Где среды выполнения в продакшене выходят из строя

- **Неверный выбор формы.** Выбор запрос-ответа для задачи длительностью 5 минут. Пользователи кладут трубку; воркеры накапливаются; повторные попытки множатся.
- **Отсутствие DLQ.** Воркеры очередей без очереди мёртвых писем (dead-letter queue). Неудачные задачи бесследно исчезают.
- **Непрозрачная фоновая работа.** Фоновый агент работает без экспорта трассировки. Сбои невидимы, пока пользователь не сообщит о них.
- **Пропуск устойчивого состояния.** Любое выполнение длительностью > 30 секунд, которое нельзя позволить себе перезапустить, требует устойчивого выполнения (durable execution).

## Реализация

`code/main.py` — демонстрация нескольких форм сред выполнения на stdlib:

- Эндпоинт запрос-ответа (простая функция).
- Обработчик потоковой передачи (генератор).
- Воркер на основе очередей с DLQ.
- Реестр событийных триггеров.
- Планировщик в формате cron.

Запуск:

```bash
python3 code/main.py
```

Вывод: пять трассировок, демонстрирующих поведение каждой формы на одной задаче. Одна и та же логика агента, разные оболочки. Устойчивое выполнение (шестая форма) целенаправленно рассматривается в Уроке 13 с чекпоинтингом (checkpointing) LangGraph.

## Применение

- **Запрос-ответ (Request-response)** для чат-интерфейса.
- **Потоковая передача (Streaming)** для постепенного вывода результатов.
- **Устойчивое (Durable)** для задач с длинным временным горизонтом (long-horizon).
- **Очередь (Queue)** для пакетной / асинхронной / долгосрочной обработки.
- **Событийное (Event)** для реактивности агентов.
- **Cron** для вспомогательных задач (консолидация памяти, оценки (evals), отчёты о затратах).

## Ключевые термины

| Термин | Что говорят | Что это означает |
|--------|-------------|------------------|
| Запрос-ответ (Request-response) | «Синхронный» | Пользователь ждёт; только для коротких задач |
| Потоковая передача (Streaming) | «SSE / WS» | Постепенный вывод; лучший UX; задержка наблюдаема по фрагментам |
| Устойчивое выполнение (Durable execution) | «Возобновление после сбоя» | Чекпоинтированное состояние; перезапуск с последнего шага |
| На основе очередей (Queue-based) | «Фоновые задачи» | Продюсер / пул воркеров / DLQ |
| Событийно-ориентированный (Event-driven) | «На основе триггеров» | Агент реагирует на внешние события |
| DLQ | «Очередь мёртвых писем» (Dead-letter queue) | Место хранения неудачных задач |
| Claude Managed Agents | «Управляемая среда» | Управляемый Anthropic долгосрочный асинхронный сервис с кэшированием и компактификацией |
