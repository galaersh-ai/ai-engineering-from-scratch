# Использование инструментов и вызов функций (Tool Use and Function Calling)

> Toolformer (Schick et al., 2023) начал автоматическую аннотацию инструментов методом обучения без учителя (self-supervised tool annotation). Berkeley Function Calling Leaderboard V4 (Patil et al., 2025) задаёт планку на 2026 год: 40% — агентные сценарии, 30% — многотурнировые, 10% — живые данные, 10% — синтетические, 10% — галлюцинации (hallucination). Однотурнировый вызов решён. Память, динамическое принятие решений и цепочки инструментов на длинном временном горизонте — нет.

**Тип:** Сборка (Build)
**Языки:** Python (стандартная библиотека)
**Предусловия:** Фаза 14 · 01 (Цикл агента), Фаза 13 · 01 (Вызов функций — подробный разбор)
**Время:** ~60 минут

## Цели обучения

- Объяснить сигнал обучения без учителя в Toolformer: сохранять аннотации инструментов только в том случае, если выполнение снижает функцию потерь (loss) на следующем токене (token).
- Назвать пять категорий оценки в BFCL V4 и то, что измеряет каждая из них.
- Реализовать реестр инструментов (tool registry) на стандартной библиотеке с валидацией (validation) схемы, приведением аргументов и песочницей выполнения (sandboxing).
- Диагностировать три нерешённых проблемы 2026 года: цепочки инструментов на длинном временном горизонте, динамическое принятие решений и память.

## Проблема

Ранние системы использования инструментов задавали вопрос: может ли модель предсказать корректный вызов функции (function)? Современные системы спрашивают: может ли модель связать инструменты в цепочку на 40 шагов, с памятью, с частичной наблюдаемостью, с восстановлением после сбоев инструментов, без галлюцинаций о несуществующих инструментах?

Toolformer установил базовый уровень: модели могут обучаться, когда вызывать инструменты, методом обучения без учителя. BFCL V4 определяет цель оценки на 2026 год. Промежуток между ними — это пространство, в котором живут production-агенты.

## Концепция

### Toolformer (Schick et al., NeurIPS 2023)

Идея: позволить модели аннотировать собственный корпус предобучения кандидатами API-вызовов. Каждый кандидат выполняется. Аннотация сохраняется только в том случае, если включение результата инструмента снижает функцию потерь на следующем токене. Дообучение (fine-tuning) проводится на отфильтрованном корпусе.

Задействованные инструменты: калькулятор, система вопросов и ответов, поисковые системы, переводчик, календарь. Сигнал обучения без учителя основан исключительно на том, помогает ли инструмент предсказать текст — никаких меток от человека.

Результат масштабирования (scaling): использование инструментов проявляется при увеличении масштаба. Маленькие модели страдают от аннотаций инструментов; большие — выигрывают. Именно поэтому передовые модели (frontier model) 2026 года имеют встроенные возможности работы с инструментами, тогда как большинству моделей с 7 млрд параметров необходим специальный файнтюнинг для надёжного использования инструментов.

### Berkeley Function Calling Leaderboard V4 (Patil et al., ICML 2025)

BFCL — де-факто стандарт оценки на 2026 год. Состав V4:

- **Агентные сценарии (40%)** — полные траектории агента (trajectory): память, многотурнировое взаимодействие, динамические решения.
- **Многотурнировые (30%)** — интерактивные диалоги с цепочками инструментов.
- **Живые данные (10%)** — реальные запросы пользователей (более сложное распределение).
- **Синтетические (10%)** — сгенерированные тестовые примеры.
- **Галлюцинации (10%)** — обнаружение ситуаций, когда инструмент вызывать не следует.

V4 ввёл оценку на основе состояния: после последовательности вызовов инструментов проверяется реальное состояние API (например, «создан ли файл?»), а не сопоставляется абстрактное синтаксическое дерево (AST) вызовов. V4 добавил категории веб-поиска, памяти и чувствительности к формату.

Ключевой вывод 2026 года: однотурнировый вызов функций практически решён. Ошибки сосредоточены в памяти (перенос контекста между турами), динамическом принятии решений (выбор инструментов на основе предыдущих результатов), цепочках на длинном временном горизонте (дрейф (drift) после 20+ шагов) и обнаружении галлюцинаций (отказ от вызова, когда ни один инструмент не подходит).

### Схема инструмента (Tool Schema)

У каждого провайдера есть схема. Они различаются в деталях, но имеют одинаковую структуру:

```
name: string
description: string (что делает, когда использовать)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropic использует `input_schema` напрямую. OpenAI использует `function.parameters`. Оба принимают JSON Schema. Описания играют ключевую роль — модель читает их для выбора подходящего инструмента. Плохие описания инструментов — основная причина ошибок с выбором неправильного инструмента.

### Валидация аргументов (Argument Validation)

Нельзя доверять ни одному вызову инструмента без проверки. Необходимо валидировать:

1. **Приведение типов (Type Coercion).** Модель может вернуть строку `"5"` там, где схема требует целое число (int). Приведение допустимо, если однозначно; в остальных случаях — отклонение.
2. **Валидация перечислений (Enum Validation).** Если схема указывает `status in {"open", "closed"}`, а модель выдаёт `"in_progress"` — отклонить с описанием ошибки.
3. **Обязательные поля (Required Fields).** Отсутствующее обязательное поле → немедленное наблюдение (observation) об ошибке модели, а не сбой.
4. **Валидация формата (Format Validation).** Даты, адреса электронной почты, URL — проверять конкретными парсерами, а не регулярными выражениями.

Каждая ошибка валидации должна возвращать структурированное наблюдение, чтобы модель могла повторить попытку с правильной структурой.

### Параллельные вызовы инструментов (Parallel Tool Calls)

Современные провайдеры поддерживают параллельные вызовы инструментов в одном ходе ассистента. Цикл (loop):

1. Модель генерирует 3 вызова инструментов с различными `tool_use_id`.
2. Среда выполнения выполняет их (параллельно, если они независимы).
3. Каждый результат возвращается как блок `tool_result`, коррелированный по `tool_use_id`.

Инженерное правило: идентификаторы корреляции — нагрузкоопределяющие. При их замене возникает маршрутизация неправильного результата к неправильному инструменту.

### Песочница выполнения (Sandboxing)

Выполнение инструмента — это граница песочницы. Подробности см. в Уроке 9. Кратко: каждый инструмент должен указывать область чтения/записи, сетевой доступ, тайм-аут и ограничение памяти. Универсальный `run_shell(cmd)` — тревожный сигнал; конкретный `git_status()` безопаснее.

## Реализация

`code/main.py` реализует реестр инструментов production-уровня:

- Валидатор подмножества JSON Schema (только стандартная библиотека).
- Регистрация инструментов с описанием, входной схемой, тайм-аутом и исполнителем.
- Приведение аргументов и валидация перечислений.
- Параллельная диспетчеризация инструментов с идентификаторами корреляции.
- Ошибки как структурированные строки наблюдений.

Запуск:

```
python3 code/main.py
```

Трассировка показывает мини-агента, вызывающего три инструмента за один ход, с одним намеренно некорректным вызовом, который отклоняется с описанием ошибки, на основе которой модель может действовать.

## Применение

У каждого провайдера своя схема инструментов — Anthropic, OpenAI, Gemini, Bedrock. Используйте слой трансляции (OpenAI Agents SDK, Vercel AI SDK, LangChain tool adapter), если нужна поддержка нескольких провайдеров. BFCL — эталонный бенчмарк: запустите его для своего агента перед выпуском, если использование инструментов является центральной частью продукта.

## Выпуск

`outputs/skill-tool-registry.md` генерирует каталог инструментов, схему и реестр для заданной предметной области. Включает проверку качества описаний (описывает ли каждый инструмент модели, когда его использовать?).

## Упражнения

1. Добавить инструмент «бездействие» (no-op), позволяющий модели явно отказаться от использования любых других инструментов. Измерить на тесте на галлюцинации по аналогии с BFCL.
2. Реализовать приведение аргументов для строковых значений int и float. На каком этапе приведение начинает скрывать реальные ошибки?
3. Добавить тайм-аут на каждый инструмент и автоматический выключатель (circuit breaker) — отказ от инструмента на 60 секунд после 3 последовательных неудач. Как это влияет на восстановление модели?
4. Ознакомиться с описанием BFCL V4. Выбрать одну категорию (например, «многотурнировую») и запустить 10 примеров запросов через своего агента. Сообщить долю успешных прохождений.
5. Перенести валидатор на стандартной библиотеке на Pydantic или Zod. Какие ошибки уловил Pydantic/Zod, которых не замечена простая реализация?

## Ключевые термины

| Термин | Как говорят | Что это на самом деле |
|--------|-------------|----------------------|

| Toolformer | «Автоматическая аннотация инструментов» | Schick 2023 — сохранение тех вызовов инструментов, результаты которых снижают функцию потерь на следующем токене |
| BFCL | «Berkeley Function Calling Leaderboard» | Бенчмарк 2026 года: 40% агентных, 30% многотурнировых, 10% живых, 10% синтетических, 10% галлюцинаций |
