# OpenAI Agents SDK: передачи (handoffs), ограничители (guardrails), трассировка (tracing)

> OpenAI Agents SDK — лёгкий мультиагентный фреймворк (framework), построенный на Responses API. Пять примитивов: Agent, Handoff, Guardrail, Session, Tracing. Передачи (handoffs) — это инструменты (tools) с именами вида `transfer_to_<agent>`. Ограничители (guardrails) срабатывают на входе или выходе. Трассировка (tracing) включена по умолчанию.

**Тип:** Теория + практика
**Языки:** Python (стандартная библиотека)
**Предварительные знания:** Фаза 14 · 01 (Agent Loop), Фаза 14 · 06 (Tool Use)
**Время:** ~75 минут

## Цели обучения

- Назвать пять примитивов OpenAI Agents SDK.
- Объяснить передачи (handoffs): почему они смоделированы как инструменты (tools), какую форму имени видит модель и как происходит передача контекста.
- Различать входные ограничители (input guardrails), выходные ограничители (output guardrails) и ограничители инструментов (tool guardrails); объяснить режим `run_in_parallel` по сравнению с блокирующим режимом.
- Реализовать среду выполнения на стандартной библиотеке с передачами (handoffs) + ограничителями (guardrails) + трассировкой в стиле спанов (span-style tracing).

## Проблема

Агенты (agents), которые не умеют корректно делегировать задачи, загружают всё в один промпт. Агенты без ограничителей (guardrails) могут передавать персональные данные (PII), нарушать политики или зацикливаться. SDK от OpenAI формализует три примитива, которые делают мультиагентную работу (multi-agent work) управляемой.

## Концепция

### Пять примитивов

1. **Agent.** LLM + инструкции + инструменты (tools) + передачи (handoffs).
2. **Handoff.** Делегирование другому агенту. Представляется модели как инструмент (tool) с именем `transfer_to_<agent_name>`.
3. **Guardrail.** Валидация на входе (только для первого агента), на выходе (только для последнего агента) или при вызове инструмента (для каждого функционального инструмента).
4. **Session.** Автоматическая история диалогов между ходами.
5. **Tracing.** Встроенные спаны (spans) для генераций LLM, вызовов инструментов, передач и ограничителей.

### Передачи как инструменты

Модель видит `transfer_to_billing_agent` в списке своих инструментов (tools). Вызов этого инструмента сигнализирует среде выполнения (runtime):

1. Скопировать контекст диалога (или свернуть его через бета-функцию `nest_handoff_history`).
2. Инициализировать целевого агента с его инструкциями.
3. Продолжить выполнение с целевым агентом.

Это паттерн супервайзера (supervisor pattern) (Урок 13 / Урок 28), доведённый до продукта.

### Ограничители (Guardrails)

Три варианта:

- **Входные ограничители (input guardrails).** Запускаются на входе первого агента. Отклоняют небезопасные или выходящие за рамки запросы до вызова LLM.
- **Выходные ограничители (output guardrails).** Запускаются на выходе последнего агента. Перехватывают утечки PII, нарушения политик, некорректные ответы.
- **Ограничители инструментов (tool guardrails).** Запускаются для каждого функционального инструмента. Валидируют аргументы, проверяют права доступа, аудитируют выполнение.

Режимы:

- **Параллельный (по умолчанию).** LLM-ограничитель работает параллельно с основной LLM. Более низкая хвостовая латентность. При срабатывании работа основной LLM отбрасывается (потеря токенов).
- **Блокирующий (`run_in_parallel=False`).** LLM-ограничитель выполняется первым. При срабатывании токены не тратятся на основной вызов.

При срабатывании ограничителя выбрасывается исключение `InputGuardrailTripwireTriggered` / `OutputGuardrailTripwireTriggered`.

### Трассировка (Tracing)

Включена по умолчанию. Каждая генерация LLM, вызов инструмента, передача и ограничитель порождают спан (span). Переменная среды `OPENAI_AGENTS_DISABLE_TRACING=1` отключает трассировку. Функция `add_trace_processor(processor)` направляет спаны на собственный бэкенд параллельно с хранилищем OpenAI.

### Сессии (Sessions)

`Session` хранит историю диалогов в бэкенде (SQLite, Redis, пользовательский). `Runner.run(agent, input, session=session)` автоматически загружает и дополняет историю.

### Где этот паттерн может сломаться

- **Дрейф передач (Handoff drift).** Агент A передаёт агенту B, который передаёт обратно агенту A. Добавьте счётчик переходов.
- **Обход ограничителя (Guardrail bypass).** Ограничители инструментов (tool guardrails) срабатывают только для функциональных инструментов; встроенные инструменты (чтение файлов, запросы к вебу) требуют отдельной политики.
- **Избыточная трассировка (Over-tracing).** Чувствительные данные в спанах. Используйте правила захвата контента OTel GenAI (Урок 23) — храните во внешнем хранилище, ссылайтесь по ID.

## Реализация

`code/main.py` реализует архитектуру SDK на стандартной библиотеке:

- `Agent`, `FunctionTool`, `Handoff` (как функциональный инструмент с семантикой передачи).
- `Runner` с входными/выходными/инструментальными ограничителями, диспетчеризацией передач и счётчиком переходов.
- Простой эмиттер спанов для демонстрации структуры трассировки.
- Агент-триаж, который передаёт в отдел биллинга или поддержки в зависимости от запроса пользователя; ограничитель срабатывает на одном входе.

Запуск:

```
python3 code/main.py
```

Трассировка показывает две успешные передачи, одно срабатывание входного ограничителя и дерево спанов, отражающее то, что выдаёт настоящий SDK.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|--------|-------------|----------------------|
| Agent (Агент) | «LLM + инструкции» | Тип Agent в SDK; владеет инструментами (tools) и передачами (handoffs) |
| Handoff (Передача) | «Трансфер» | Инструмент (tool), который вызывает модель для делегирования другому агенту |
| Guardrail (Ограничитель) | «Проверка политики» | Валидация на входе / выходе / вызове инструмента |
| Tripwire (Сработавший ограничитель) | «Срабатывание ограничителя» | Исключение, выбрасываемое при отклонении ограничителем |
| Session (Сессия) | «Хранилище истории» | Память диалогов, сохраняемая между запусками |
| Tracing (Трассировка) | «Спаны» | Встроенная наблюдаемость (observability) для LLM + инструментов + передач + ограничителей |
| Blocking guardrail (Блокирующий ограничитель) | «Последовательная проверка» | Ограничитель выполняется первым; токены не тратятся при срабатывании |
| Parallel guardrail (Параллельный ограничитель) | «Параллельная проверка» | Ограничитель работает параллельно; более низкая латентность, токены тратятся при срабатывании |
