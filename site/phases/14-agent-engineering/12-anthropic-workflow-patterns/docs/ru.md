# Рабочие процессы Anthropic: простое лучше сложного

> Шлунц и Чжан (Anthropic, декабрь 2024) проводят различие между рабочими процессами (workflow, предопределённые пути) и агентами (agent, динамическое использование инструментов). Пять паттернов рабочих процессов покрывают большинство случаев. Начинайте с прямых вызовов API. Добавляйте агентов только тогда, когда шаги невозможно предсказать.

**Тип:** Изучение + практическая работа
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Фаза 14 · 01 (Цикл агента / Agent Loop)
**Время:** ~60 минут

## Цели обучения

- Назвать пять паттернов рабочих процессов Anthropic: цепочка промптов (prompt chaining), маршрутизация (routing), параллелизация (parallelization), оркестратор-рабочие агенты (orchestrator-workers), оценщик-оптимизатор (evaluator-optimizer).
- Объяснить различие между агентом и рабочим процессом, а также инженерные издержки каждого подхода.
- Определить, когда следует выбрать рабочий процесс вместо агента (и наоборот).
- Реализовать все пять паттернов с использованием стандартной библиотеки Python на основе scripted LLM.

## Проблема

Команды прибегают к фреймворкам мультиагентных систем для задач, которые требуют одного вызова функции. Цена реальна: фреймворки добавляют уровни абстракции, которые маскируют промпты, скрывают управление потоком и порождают преждевременную сложность. Публикация Шлунца и Чжана от декабря 2024 года — наиболее цитируемое индустриальное возражение: начинайте с простого, добавляйте сложность только тогда, когда она оправдана.

## Концепция

### Рабочие процессы и агенты

- **Рабочий процесс (Workflow).** LLM и инструменты оркестрированы через предопределённые кодовые пути. Инженер управляет графом.
- **Агент (Agent).** LLM динамически управляет собственными инструментами и определяет собственные шаги. Модель управляет графом.

Оба подхода имеют свои преимущества. Рабочие процессы дешевле, быстрее и проще в отладке. Агенты открывают возможности для задач с открытым исходом, но делают режимы отказов сложнее для анализа.

### Расширенная LLM

Основа всех пяти паттернов: одна LLM с тремя подключёнными возможностями — поиск (извлечение / retrieval), инструменты (tools, действия), память (memory, сохранение состояния). Любой вызов API может использовать эти возможности.

### Пять паттернов

1. **Цепочка промптов (Prompt chaining).** Выход вызова 1 является входом для вызова 2. Используйте, когда задача допускает чёткую линейную декомпозицию.etween шагами можно добавить программные проверки.

2. **Маршрутизация (Routing).** Классифицирующая LLM выбирает, какой последующий LLM или инструмент вызвать. Используйте, когда категорически различные входные данные требуютdifferent обработки (поддержка первого уровня vs возврат средств vs ошибки vs продажи).

3. **Параллелизация (Parallelization).** Запуск N вызовов LLM параллельно с агрегацией результатов. Два варианта: разбиение на сегменты (sectioning, разные блоки) и голосование (voting, один промпт, N запусков, большинство/синтез).

4. **Оркестратор-рабочие агенты (Orchestrator-workers).** Оркестрирующая LLM динамически определяет, какие рабочие агенты (worker, также LLM) запустить, и синтезирует их вывод. Аналогично циклам агентов (agent loop), но оркестратор не зацикливается бесконечно.

5. **Оценщик-оптимизатор (Evaluator-optimizer).** Одна LLM предлагает ответ, другая LLM оценивает его. Итерации продолжаются до тех пор, пока оценщик не одобрит результат. Это обобщённый Self-Refine (Урок 05).

### Где рабочие процессы лучше агентов

- **Предсказуемые задачи.** Если вы можете перечислить шаги — перечисляйте их.
- **Задачи с ограниченным бюджетом.** Рабочие процессы имеют ограниченное количество шагов; агенты могут уйти в бесконечный цикл.
- **Задачи с требованиями комплаенса.** Аудиторам нужно читать граф, а не выводить его из траекторий.

### Где агенты лучше рабочих процессов

- **Исследования с открытым исходом.** Когда следующий шаг зависит от результата предыдущего.
- **Задачи переменной длительности.** Работа от минут до часов, когда количество шагов заранее неизвестно.
- **Новые области.** Когда вы ещё не знаете правильный рабочий процесс — сначала исследуйте, затем кодифицируйте.

### Спутник: инженерия контекста

«Effective context engineering for AI agents» (Anthropic 2025) формализует смежную дисциплину: окно в 200k токенов — это бюджет, а не контейнер. Что включать, когда сжимать контекст, когда позволять ему расти. Подробно рассмотрено на фазе 14 в уроке о сжатии контекста (ранее урок 06 в этой учебной программе до перенумерации).

## Практическая реализация

Файл `code/main.py` реализует все пять паттернов рабочих процессов на основе `ScriptedLLM`:

- `prompt_chain(input, steps)` — последовательный.
- `route(input, classifier, handlers)` — классификация + диспетчеризация.
- `parallel_vote(prompt, n, aggregator)` — N запусков, агрегация.
- `orchestrator_workers(task, workers)` — оркестратор выбирает рабочих агентов.
- `evaluator_optimizer(task, proposer, evaluator, max_iter)` — цикл до одобрения.

Запуск:

```
python3 code/main.py
```

Каждый паттерн выводит свой трасс. Общее количество строк кода на каждый паттерн составляет ~10–15; стоимость фреймворка исчисляется тысячами строк.

## Применение

- Прямые вызовы API для большинства задач.
- Фреймворк только тогда, когда паттерн действительно требует состояния, сохраняющегося между вызовами (LangGraph), конкурентности по модели акторов (actor-model, AutoGen v0.4) или шаблонизации ролей (CrewAI).
- Прибегайте к Claude Agent SDK, когда вам нужна архитектура Claude Code без необходимости пересобирать её с нуля.

## Публикация

Файл `outputs/skill-workflow-picker.md` выбирает подходящий паттернов для заданного описания задачи, включая обоснование решения и путь рефакторинга к агенту, если рабочие процессы окажутся недостаточны.

## Ключевые термины

| Термин | Как часто говорят | Что это на самом деле |
|--------|-------------------|----------------------|
| Workflow (рабочий процесс) | «Предопределённый поток» | Граф вызовов LLM и инструментов, управляемый инженером |
| Agent (агент) | «Автономный AI» | Граф, управляемый моделью; динамическое управление инструментами |
| Augmented LLM (расширенная LLM) | «LLM с инструментами» | LLM + поиск + инструменты + память; атомарная единица |
| Prompt chaining (цепочка промптов) | «Последовательные вызовы» | Выход вызова N является входом для вызова N+1 |
| Routing (маршрутизация) | «Диспетчеризация классификатором» | Выбор цепочки/модели для обработки входных данных |
| Parallelization (параллелизация) | «Развёртывание» | N параллельных вызовов; агрегация по сегментам или голосованию |
| Orchestrator-workers (оркестратор-рабочие агенты) | «Диспетчер-агент» | Оркестрирующая LLM динамически выбирает специализированные LLM |
| Evaluator-optimizer (оценщик-оптимизатор) | «Предлагающий + судящий» | Итерации до одобрения оценщиком; обобщённый Self-Refine |
