# AutoGen v0.4: модель акторов (Actor Model) и фреймворк агентов

> AutoGen v0.4 (Microsoft Research, январь 2025) переработал оркестрацию агентов вокруг модели акторов (actor model). Асинхронный обмен сообщениями, событийно-управляемые агенты (event-driven agents), изоляция отказов, естественная конкурентность. Фреймворк находится в режиме поддержки, а Microsoft Agent Framework (публичная предварительная версия, октябрь 2025) становится его преемником.

**Тип:** изучение + практическое построение
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 14 · 01 (Цикл агента), Фаза 14 · 12 (Шаблоны воркфлоу)
**Время:** ~75 минут

## Цели обучения

- Описать модель акторов (actor model): агенты как акторы (actors), сообщения как единственный механизм межпроцессного взаимодействия (IPC), изоляция отказов (fault isolation) для каждого актора.
- Назвать три уровня API AutoGen v0.4 — Core, AgentChat, Extensions — и назначение каждого из них.
- Объяснить, почему разделение доставки сообщений и их обработки обеспечивает изоляцию отказов и естественную конкурентность (natural concurrency).
- Реализовать среду выполнения (runtime) акторов на stdlib в Python и перенести на неё двухагентный процесс ревью кода.

## Проблема

Большинство фреймворков агентов работают синхронно: один агент производит данные, другой потребляет — всё в пределах одного стека вызовов. Сбои обрушивают весь стек. Конкурентность добавляется искусственно. Для распределённого исполнения требуется переписывание кода.

Ответ AutoGen v0.4: модель акторов (actor model). Каждый агент — это актор с приватным входящим ящиком (inbox). Сообщения — единственный способ взаимодействия. Среда выполнения (runtime) разделяет доставку и обработку. Отказы изолируются в пределах одного актора. Конкурентность является встроенной. Распределённость — это просто другой транспорт.

## Концепция

### Акторы (Actors)

Актор обладает:

- Приватным состоянием (к нему никогда не обращаются напрямую извне).
- Входящим ящиком (очередь сообщений).
- Обработчиком (handler): `receive(message) -> effects`, где эффекты могут быть: «ответить», «отправить другому актору», «создать нового актора», «обновить состояние», «остановить себя».

Два актора не могут разделять память. Они могут только обмениваться сообщениями.

### Три уровня API в AutoGen v0.4

1. **Core.** Низкоуровневый фреймворк акторов. `AgentRuntime`, `Agent`, `Message`, `Topic`. Асинхронный обмен сообщениями, событийный подход.
2. **AgentChat.** Высокоуровневое API для решения задач (замена ConversableAgent из v0.2). `AssistantAgent`, `UserProxyAgent`, `RoundRobinGroupChat`, `SelectorGroupChat`.
3. **Extensions.** Интеграции — OpenAI, Anthropic, Azure, инструменты, память.

### Почему разделение имеет значение

В модели v0.2 вызов `agent_a.chat(agent_b)` синхронно блокирует agent_a до тех пор, пока agent_b не вернёт результат. В v0.4 вызов `send(agent_b, msg)` помещает сообщение во входящий ящик agent_b и возвращает управление. Среда выполнения доставляет сообщение позже. Три следствия:

- **Изоляция отказов (Fault isolation).** Выход из строя агента B не приводит к падению агента A — среда выполнения перехватывает ошибку в обработчике B и принимает решение (логировать, повторить, поместить в очередь недоставленных сообщений / dead-letter queue).
- **Естественная конкурентность (Natural concurrency).** Одновременно находится множество сообщений в процессе передачи; акторы обрабатывают свои входящие ящики параллельно.
- **Готовность к распределённому исполнению.** Абстракция «входящий ящик + транспорт» одинакова вне зависимости от того, находится ли актор в том же процессе или на другом хосте.

### Топологии (Topologies)

- **RoundRobinGroupChat.** Агенты по очереди сменяют друг друга в фиксированном порядке.
- **SelectorGroupChat.** Выбирающий агент (selector) определяет, кто выступит следующим, на основе контекста разговора.
- **Magentic-One.** Эталонная мультиагентная команда для веб-просмотра, выполнения кода, работы с файлами. Построена на AgentChat.

### Наблюдаемость (Observability)

Встроенная поддержка OpenTelemetry. Каждое сообщение генерирует спан (span); вызовы инструментов несут атрибуты `gen_ai.*` в соответствии с семантическими соглашениями OTel GenAI 2026 года (Урок 23).

### Статус: режим поддержки

Начало 2026 года: AutoGen v0.7.x стабилен для исследований и прототипирования. Microsoft переключила активную разработку на Microsoft Agent Framework (публичная предварительная версия 1 октября 2025; стабильный релиз 1.0 планируется к концу первого квартала 2026). Паттерны AutoGen хорошо переносятся — модель акторов является долговечной идеей.

## Реализация

`code/main.py` реализует среду выполнения акторов на stdlib:

- `Message` — типизированная полезная нагрузка с полями `sender`, `recipient`, `topic`, `body`.
- `Actor` — абстрактный класс с методом `receive(message, runtime)`.
- `Runtime` — цикл событий (event loop) с общей очередью (queue), доставкой сообщений, изоляцией отказов.
- Демонстрация с двумя акторами: `ReviewerAgent` выполняет ревью кода, `ChecklistAgent` проверяет чек-лист (checklist); они обмениваются сообщениями до достижения консенсуса (consensus).

Запуск:

```
python3 code/main.py
```

Трассировка показывает доставку сообщений, моделируемый сбой одного актора, который не приводит к падению другого, и сходимость к общему заключению.

## Применение

- **AutoGen v0.4/v0.7** (режим поддержки) — стабилен для исследований, прототипирования, мультиагентных паттернов.
- **Microsoft Agent Framework** (публичная предварительная версия) — перспективное направление; те же идеи модели акторов в обновлённом API.
- **Топология роя LangGraph (swarm topology)** (Урок 13) — аналогичный паттерн через передачу (handoff) с использованием общих инструментов.
- **Пользовательская среда выполнения акторов (custom actor runtime)** — когда необходим определённый транспорт (NATS, RabbitMQ, gRPC).

## Упражнения

1. Добавьте очередь недоставленных сообщений (dead-letter queue): при возникновении исключения в обработчике сохраняйте проблемное сообщение для последующего анализа вручную. Как часто срабатывает DLQ в вашей тестовой системе?
2. Реализуйте `SelectorGroupChat`: выбирающий актор определяет, кто обработает следующее сообщение, на основе состояния разговора.
3. Добавьте распределённый транспорт: замените внутрпроцессную очередь HTTP-сервером с обменом сообщениями в формате JSON, чтобы акторы могли работать в отдельных процессах.
4. Подключите спан (span) OpenTelemetry к каждому сообщению (или заглушку без реальной работы). Эмитируйте `gen_ai.agent.name`, `gen_ai.operation.name` в соответствии с Уроком 23.
5. Прочитайте статью об архитектуре AutoGen v0.4. Перенесите свою тестовую реализацию на реальный API `autogen_core`. Что вы опустили, но что важно для продакшена?

## Ключевые термины

| Термин | Как обычно говорят | Что это означает на самом деле |
|--------|--------------------|--------------------------------|
| Актор (Actor) | «Агент» | Приватное состояние + входящий ящик + обработчик; нет разделяемой памяти |
| Сообщение (Message) | «Событие» | Типизированная полезная нагрузка; единственный способ взаимодействия акторов |
| Входящий ящик (Inbox) | «Почтовый ящик» | Очередь ожидающих сообщений для каждого актора |
| Среда выполнения (Runtime) | «Хост агента» | Цикл событий, маршрутизирующий сообщения и изолирующий отказы |
| Тема (Topic) | «Канал» | Именованный маршрут публикации-подписки (publish-subscribe) между акторами |
| Изоляция отказов (Fault isolation) | «Пусть падает» | Выход из строя одного актора не приводит к падению других |
| RoundRobinGroupChat | «Команда с фиксированной ротацией» | Агенты по очереди сменяют друг друга |
| SelectorGroupChat | «Команда с контекстной маршрутизацией» | Выбирающий агент определяет следующего |
| Magentic-One | «Эталонная команда» | Мультиагентный отряд для веба + кода + файлов |
