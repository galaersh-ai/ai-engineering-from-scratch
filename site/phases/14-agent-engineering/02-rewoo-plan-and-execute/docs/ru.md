# ReWOO и Plan-and-Execute: разделение планирования и выполнения

> ReAct чередует мысль и действие в одном потоке. ReWOO разделяет их: сначала создаётся единый план, затем выполняется. На 500% меньше токенов, +4% точности на HotpotQA, а планировщик можно дистиллировать в модель размером 7B. Plan-and-Execute обобщил этот подход; Plan-and-Act масштабировал его для навигации по вебу.

**Тип:** Практическая работа
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Фаза 14 · 01 (цикл агента)
**Время:** ~60 минут

## Цели обучения

- Объяснить, почему разделение ReWOO на Planner / Worker / Solver (планировщик / рабочий агент / решатель) экономит токены и повышает устойчивость по сравнению с чередующимся циклом ReAct.
- Реализовать план в виде DAG, исполнитель с порядком зависимостей и решатель, который компонует результаты рабочих агентов — всё на стандартной библиотеке.
- Определять, когда задачу следует выполнять по схеме «сначала план, потом выполнение», а когда — через чередующийся ReAct, используя фреймворк «пять рабочих процессов» (Anthropic, 2026).
- Распознавать, когда для долгосрочных задач с вебом или мобильными устройствами необходимы синтетические данные о планах из Plan-and-Act.

## Проблема

Чередующийся цикл мысль-действие-наблюдение в ReAct прост и гибок, однако каждый вызов инструмента должен нести полный предшествующий контекст — включая каждую предыдущую мысль. Расход токенов растёт пропорционально квадрату глубины. Хуже того: когда инструмент завершается ошибкой на середине цикла, модель должна заново вывести весь план из наблюдения об ошибке.

ReWOO (Xu и др., arXiv:2305.18323, май 2023) заметил это и выдвинул гипотезу: составьте весь план заранее, извлекайте доказательства параллельно, собирайте ответ в конце. Один вызов LLM для планирования, N вызовов инструментов для извлечения доказательств (параллельно), один вызов LLM для решения. Торговля: меньшая гибкость (план статичен) в обмен на гораздо лучшую эффективность по токенам и более понятные режимы сбоев.

## Концепция

### Три роли

```
Planner:  user_question -> [plan_dag]
Workers:  [plan_dag]     -> [evidence]        (вызовы инструментов, возможно параллельные)
Solver:   user_question, plan_dag, evidence -> final_answer
```

Planner (планировщик) создаёт DAG. Каждый узел определяет инструмент, его аргументы и зависимости от предыдущих узлов (ссылки вида `#E1`, `#E2`). Worker (рабочие агенты) выполняют узлы в топологическом порядке. Solver (решатель) связывает всё воедино.

### Почему токенов на 500% меньше

ReAct увеличивает длину промпта линейно с ростом числа шагов. На шаге 10 промпт содержит мысль 1, действие 1, наблюдение 1, мысль 2, действие 2, наблюдение 2 и так далее. Каждый промежуточный шаг также избыточно включает исходный промпт.

ReWOO использует один промпт планировщика (большой), N маленьких промптов рабочих агентов (каждый — только вызов инструмента, без цепочки) и один промпт решателя. На HotpotQA статья измеряет примерно в 5 раз меньше токенов при повышении точности на +4%.

### Почему это устойчивее

Если рабочий агент 3 завершается ошибкой в ReAct, цикл должен выйти из ошибки на ходу. В ReWOO рабочий агент 3 возвращает строку с ошибкой; решатель видит её в контексте исходного плана и может корректно деградировать. Локализация сбоев происходит на уровне узла, а не шага.

### Дистилляция планировщика

Второй результат статьи: поскольку планировщик не видит наблюдений, можно дообучить модель размером 7B на выходах планировщика от учителя размером 175B. Маленькая модель выполняет планирование; большая модель не нужна при инференсе. Это стало стандартом — многие продакшн-агенты 2026 года используют маленький планировщик и большой исполнитель или наоборот.

### Plan-and-Execute (LangChain, 2023)

Команда LangChain в августе 2023 года обобщила ReWOO в именованный паттерн: Plan-and-Execute. Заранее созданный планировщик генерирует список шагов, исполнитель выполняет каждый шаг, опциональный перепланировщик может скорректировать план после наблюдения за результатами. Это ближе к ReAct, чем к ReWOO (перепланировщик возвращает наблюдения в планирование), но сохраняет экономию токенов.

### Plan-and-Act (Erdogan и др., arXiv:2503.09572, ICML 2025)

Plan-and-Act масштабирует паттерн на долгосрочные агенты для веба и мобильных устройств. Ключевой вклад — синтетические данные о планах: генератор размеченных траекторий создаёт обучающие данные, в которых план явно выражен. Используется для файнтюнинга моделей планировщиков, которые сохраняют работоспособность за 30–50 шагов на задачах типа WebArena, где одна траектория ReAct теряет связность.

### Когда что выбирать

| Паттерн | Когда использовать |
|---------|-------------------|
| ReAct | Короткие задачи, неизвестная среда, нужна реактивная обработка исключений |
| ReWOO | Структурированные задачи с известными инструментами, чувствительность к токенам, параллелизуемые доказательства |
| Plan-and-Execute | Как ReWOO, но с перепланированием после частичного выполнения |
| Plan-and-Act | Долгосрочные задачи (>30 шагов), веб/мобильные/использование компьютера |
| Дерево мыслей (Tree of Thoughts) | Стоит платить за поиск (Урок 04) |

Рекомендация Anthropic от декабря 2024 года: начинайте с простейшего. Если задача — один вызов инструмента и сводка, не стройте ReWOO. Если задача — 40-шаговое исследование, не ограничивайтесь одним ReAct.

## Реализация

`code/main.py` реализует учебный ReWOO:

- `Planner` — запрограммированная политика, которая создаёт план DAG из промпта.
- `Worker` — диспетчеризует вызов инструмента для каждого узла через реестр.
- `Solver` — запрограммированная композиция, которая читает доказательства и формирует окончательный ответ.
- Разрешение зависимостей — ссылки вида `#E1` подставляются результатами предыдущих рабочих агентов.

Демо отвечает на вопрос «Какова численность населения столицы Франции, округлённая до миллионов?» с помощью двухэтапного плана: (1) найти столицу, (2) найти численность населения, затем решить.

Запуск:

```
python3 code/main.py
```

Трассировка показывает полный план, затем результаты рабочих агентов, затем композицию решателя. Сравните количество токенов (мы выводим приблизительный подсчёт символов) с чередующимся запуском в стиле ReAct — ReWOO выигрывает на таких структурированных задачах.

## Применение

LangGraph поставляет Plan-and-Execute как рецепт (`create_react_agent` для ReAct, пользовательские графы для plan-execute). Flows от CrewAI кодируют паттерн напрямую: вы определяете задачи заранее, а граф Flow выполняет их. Подход с синтетическими данными от Plan-and-Act по-прежнему в основном является исследованием; паттерн среды выполнения (явный план DAG) поставляется в продакшне через LangGraph и CrewAI Flows.

## Поставка

`outputs/skill-rewoo-planner.md` генерирует план DAG в формате ReWOO из пользовательского запроса с учётом каталога инструментов. Он валидирует план (отсутствие циклов, все ссылки разрешены, все инструменты существуют) перед передачей исполнителю.

## Упражнения

1. Параллелизуйте выполнение рабочих агентов для независимых узлов плана. Что это даёт на DAG с 6 узлами и 2 параллельными группами?
2. Добавьте узел перепланировщика, который срабатывает при ошибке любого рабочего агента. Какое минимальное изменение ReWOO превращает его в Plan-and-Execute?
3. Замените `Planner` маленькой моделью (класс 7B), а `Solver` оставьте на передовой модели. Сравните качество от конца до конца — где расщепление не справляется?
4. Прочитайте раздел 4 статьи ReWOO о дистилляции планировщика. Воспроизведите концептуально результат 175B -> 7B: какие обучающие данные нужны и как оценивать качество плана?
5. Перенесите учебную программу в формат траектории Plan-and-Act: план — последовательность, а не DAG. Какие компромиссы меняются?

## Ключевые термины

| Термин | Что говорят | Что это значит на самом деле |
|--------|-------------|------------------------------|
| ReWOO | «Рассуждение без наблюдений» | План, затем извлечение доказательств параллельно, затем решение — без наблюдений в промпте планировщика |
| Plan-and-Execute | «Паттерн plan-execute от LangChain» | ReWOO с опциональным узлом перепланировщика после выполнения |
| Plan-and-Act | «Масштабированный plan-execute» | Явное разделение планировщик/исполнитель с синтетическими обучающими данными о планах для долгосрочных задач |
| Ссылка на доказательство | «#E1, #E2, ...» | Заполнитель узла плана, подставляемый результатом предыдущего рабочего агента при диспетчеризации |
| Дистилляция планировщика | «Маленький планировщик, большой исполнитель» | файнтюнинг маленькой модели на трассировках планировщика от большого учителя |
| Эффективность по токенам | «Меньше проходов» | В 5 раз меньше токенов на HotpotQA по сравнению с ReAct в статье |
| DAG-исполнитель | «Топологический диспетчер» | Выполняет узлы плана в порядке зависимостей; параллельно на каждом уровне |
