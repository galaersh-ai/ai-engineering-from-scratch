# CrewAI: ролевые экипажи и флоу

> CrewAI — ролевой мультиагентный (multi-agent) фреймворк 2026 года. Четыре примитива: агент (Agent), задание (Task), экипаж (Crew), процесс (Process). Две основные формы: экипажи (автономное, ролевое взаимодействие) и флоу (событийно-ориентированные, детерминированные). Документация говорит прямо: «для любого production-ready приложения начните с флоу».

**Тип:** Теория + практика
**Языки:** Python (стандартная библиотека)
**Предварительные знания:** Фаза 14 · 12 (шаблоны воркфлоу), Фаза 14 · 14 (модель акторов)
**Время:** ~75 минут

## Цели обучения

- Назвать четыре примитива CrewAI (Agent, Task, Crew, Process) и то, за что каждый отвечает.
- Различать последовательный (Sequential), иерархический (Hierarchical) и планируемый консенсусный (Consensus) процессы; выбирать один для каждой нагрузки.
- Различать экипажи (автономные, ролевые) и флоу (событийно-ориентированные, детерминированные) и объяснить рекомендацию документации для продакшена.
- Подключать инструменты с помощью декоратора `@tool` и подкласса `BaseTool`; анализировать различие между структурированными выходами и свободным текстом.
- Назвать четыре типа памяти (memory) CrewAI и указать, когда каждый из них полезен.
- Реализовать трёхагентный (three-agent) экипаж на стандартной библиотеке (исследователь, писатель, редактор), создающий бриф.
- Распознать три режима сбоя CrewAI: раздувание промптов, накладные расходы — агент-управляющий + LLM, хрупкие передачи.

## Проблема

Команды, внедряющие мультиагентные (multi-agent) фреймворки, упираются в одну и ту же стену. «Автономное взаимодействие» звучит отлично в демо. Затем клиент подаёт баг-репорт, и вам нужен детерминированный повтор (replay). Или финотдел спрашивает, во сколько обходится каждый запуск экипажа с маршрутизацией через LLM. Или дежурному в 3 часа ночи нужно узнать, какой агент застрял.

Свободные экипажи с маршрутизацией через LLM не дают ответов ни на один из этих вопросов. Чистые DAG (ориентированные ациклические графы) отвечают на все, но теряют форму исследования, необходимую агенту для брейншторминга.

Разделение CrewAI честно описывает компромисс. Экипажи — для совместной, ролевой, исследовательской работы. Флоу — для событийно-ориентированного, кодоуправляемого, аудитируемого продакшена. Один фреймворк, две формы, выбор зависит от области применения.

## Концепция

### Четыре примитива

API CrewAI мал. Запомните это — остальное сводится к конфигурации.

- **Агент (Agent).** `role + goal + backstory + tools + (опционально) llm`. Предыстория (backstory) играет ключевую роль. Она определяет тон, суждения и условия остановки агента. Инструменты (tools) — функции, которые агент может вызывать (подробнее ниже).
- **Задание (Task).** `description + expected_output + agent + (опционально) context + (опционально) output_pydantic`. Переиспользуемая единица работы. `expected_output` — контракт. `context` перечисляет предшествующие задания, чьи выходы передаются далее. `output_pydantic` навязывает структурированную форму.
- **Экипаж (Crew).** Контейнер. Управляет списком `agents`, списком `tasks`, `process`, а также необязательными настройками `memory`, `verbose` и `manager_llm`.
- **Процесс (Process).** Стратегия выполнения. Sequential (последовательный), Hierarchical (иерархический), Consensus (консенсусный, планируемый). Определяет форму запуска.

Агенты не видят друг друга напрямую. Задания ссылаются на агентов. Экипаж упорядочивает задания. Процесс решает, кто берёт следующее задание. Это и есть вся ментальная модель.

> **Проверено на** CrewAI 0.86 (2026-05). В более новых версиях типы процессов могут быть переименованы или объединены; сверьтесь с [документацией CrewAI Processes](https://docs.crewai.com/concepts/processes) прежде чем опираться на конкретную форму.

### Последовательный vs иерархический vs консенсусный

- **Последовательный (Sequential).** Задания выполняются в порядке объявления. Выход задания N доступен как `context` для задания N+1. Минимальная стоимость. Наиболее предсказуемый. Используйте, когда порядок фиксирован.
- **Иерархический (Hierarchical).** Менеджер-агент (отдельный вызов LLM) маршрутизирует между специалистами. CrewAI создаёт менеджера либо из вашей конфигурации `manager_llm`, либо из значения по умолчанию. Менеджер выбирает следующее задание на каждом раунде и может отклонить или перенаправить. Используйте, когда у вас четыре или более специалиста и порядок действительно зависит от предыдущего результата.
- **Консенсусный (Consensus).** Планируется, но в настоящее время не реализован в публичном API. Документация зарезервировала это название для будущего процесса на основе голосования. Не используйте его сегодня.

Иерархический добавляет вызов LLM менеджера на каждом раунде поверх каждого вызова специалиста. Стоимость токенов может утроиться при пятишаговом запуске. Платите за это только тогда, когда вам нужна маршрутизация.

### Экипажи vs флоу

Это основное противопоставление, которое документация выносит вперёд в 2026 году.

- **Экипаж (Crew).** Автономия, управляемая LLM. Фреймворк определяет форму во время выполнения. Подходит для: исследований, брейншторминга, первых черновиков — везде, где сам путь является частью ответа. Сложно воспроизвести. Сложно тестировать. Дёшево для прототипирования.
- **Флоу (Flow).** Событийно-ориентированный граф, который вы контролируете. `@start` отмечает точку входа. `@listen(topic)` отмечает шаг, который запускается при отправке соответствующей темы другим шагом. Каждый шаг — чистый Python (может вызывать экипаж внутри). Подходит для: продакшена. Наблюдаемый. Тестируемый. Детерминированный.

Рекомендация документации для продакшена в 2026 году: начните с флоу. Встраивайте экипажи как вызовы `Crew.kickoff()` внутри шагов флоу, когда автономия оправдывает свои затраты. Флоу обеспечивает аудитируемый след, а экипаж — исследование. Комбинируйте, а не выбирайте.

### Подключение инструментов

Три способа дать агенту инструмент. Выберите самый простой подходящий.

1. **Декоратор `@tool`.** Чистые функции становятся инструментами. Сигнатура — это схема; docstring — описание, которое видит LLM. Лучший вариант для разовых вспомогательных функций.

   ```python
   from crewai.tools import tool

   @tool("Search the web")
   def search(query: str) -> str:
       """Return top results for the query."""
       return run_search(query)
   ```

2. **Подкласс `BaseTool`.** Инструмент на основе класса с явной схемой аргументов, поддержкой async и повторами (retries). Используйте, когда инструмент имеет состояние (клиент, кэш) или требует структурированных аргументов.

   ```python
   from crewai.tools import BaseTool
   from pydantic import BaseModel

   class SearchArgs(BaseModel):
       query: str
       limit: int = 10

   class SearchTool(BaseTool):
       name = "web_search"
       description = "Search the web and return top results."
       args_schema = SearchArgs

       def _run(self, query: str, limit: int = 10) -> str:
           return self.client.search(query, limit=limit)
   ```

3. **Встроенные наборы инструментов (toolkits).** CrewAI поставляет адаптеры первого-party: `SerperDevTool`, `FileReadTool`, `DirectoryReadTool`, `CodeInterpreterTool`, `RagTool`, `WebsiteSearchTool`. Подключаются одним импортом.

Структурированные выходы используют Pydantic. Передайте `output_pydantic=MyModel` в задании. CrewAI проверяет ответ LLM по модели и либо приводит к нужному формату, либо повторяет попытку. Сочетайте это со строгой строкой `expected_output`. Свободный текст подходит для черновиков; структурированные выходы — то, что могут потреблять последующие флоу.

### Механизмы памяти

CrewAI поставляет четыре типа памяти из коробки. Они комбинируются: экипаж может включить все четыре одновременно.

> **Проверено на** CrewAI 0.86 (2026-05). В последних версиях всё проходит через единую систему `Memory`, объединяющую эти четыре хранилища. Концептуальная модель ниже по-прежнему верна, но публичный классовый интерфейс в более новых версиях может быть сведён к единственной точке входа `Memory`; сверьтесь с [документацией CrewAI memory](https://docs.crewai.com/concepts/memory) для актуального API.

- **Краткосрочная (Short-term).** Буфер разговора в рамках одного запуска. Очищается по завершении.
- **Долгосрочная (Long-term).** Сохраняется между запусками. Хранится в векторной БД (по умолчанию Chroma, заменяемая). Извлекается по сходству с текущим заданием.
- **Сущностная (Entity).** Факты по сущностям. «Клиент X находится на корпоративном тарифе». Ключируется по сущности, а не по сходству. Сохраняется между запусками.
- **Контекстуальная (Contextual).** Извлечение по мере сборки. Подтягивает релевантную память в момент, когда агенту она нужна, а не загружает заранее.

Включается в экипаже через `memory=True` или конфигурацию по типам. Опирается на провайдер эмбеддингов (embedding), который вы настраиваете (по умолчанию OpenAI, заменяется на локальный). Память — одна из областей, где CrewAI оправдывает себя по сравнению с более лёгкими фреймворками; чистый LangGraph требует настроить каждую из этих возможностей вручную.

### Когда CrewAI подходит

- От трёх до шести агентов с именованными ролями и совместным воркфлоу. Черновики, рецензии, планирование, брейншторминг.
- Маршрутизация, где суждение LLM о следующем шаге является частью ценности (иерархический).
- Везде, где команде проще читать `role + goal + backstory`, чем определение графа.

### Когда CrewAI не подходит

- Детерминированные DAG с строгим упорядочением. Используйте LangGraph (Урок 13). Форма графа — правильная абстракция; ролевая рамка CrewAI становится помехой.
- Бюджет латентности менее секунды. Иерархический добавляет круговые запросы. Даже последовательный сериализует промпты, содержащие предыстории и предыдущие выходы.
- Одноагентные циклы. Пропустите фреймворк; цикл агента (Урок 1) и реестр инструментов короче.

Урок 17 (Tradeoffs агентных фреймворков) сравнивает их в матрице. Кратко: CrewAI находится в углу «совместный ролевой».

### Зависимости

Независим от LangChain. Python 3.10–3.13. Использует `uv`. Количество звёзд: см. [crewAIInc/crewAI](https://github.com/crewAIInc/crewAI) (снимок на 2026-05). Интеграция с AWS Bedrock задокументирована; поставщики сообщают о существенном ускорении по сравнению с LangGraph на нагрузках QA, но методология (датасет, оборудование, метрика оценки) не опубликована, поэтому цифры от поставщика фреймворка следует рассматривать как ориентировочные.

### Где этот шаблон даёт сбои

- **Раздувание промптов из-за предысторий.** Предыстория в 2000 слов на агента и экипаж из пяти агентов исчерпывают бюджет контекста до первого вызова инструмента. Держите предыстории в пределах 200 слов. Переиспользуйте формулировки между агентами; не повторяйте домашний стиль пять раз.
- **Накладные расходы токенов менеджера-LLM.** Иерархический процесс добавляет вызов LLM-менеджера перед каждым вызовом специалиста. На экипаже из пяти заданий это шесть вместо пяти вызовов LLM, причём вызов менеджера несёт полный список заданий плюс предыдущие выходы. Переключайтесь на последовательный, если маршрутизация не зависит от результата.
- **Хрупкие передачи (handoffs).** `expected_output` задания N — «очеред». Задание N+1 читает его как `context` и пытается разобрать три раздела. LLM сгенерировал четыре. Нижестоящий агент импровизирует. Исправьте через `output_pydantic` в задании N, чтобы задание N+1 читало типизированный объект, а не свободный текст.
- **Экипаж как продакшен.** Свободный экипаж развёрнут в продакшен без обёртки флоу. Вариативность выходов высока; повтор невозможен; дежурный не может сравнить неудачный запуск с удачным. Оберните флоу.

## Реализация

`code/main.py` реализует версии обеих форм на стандартной библиотеке плюс трёхагентный экипаж.

Структура:

- Датаклассы `Agent`, `Task`, соответствующие API CrewAI.
- `SequentialCrew.kickoff(inputs)` запускает задания в порядке объявления, передавая выходы как `context`.
- `HierarchicalCrew.kickoff(topic)` добавляет агента-менеджера, выбирающего следующего специалиста на каждом раунде; останавливается при «done».
- `Flow` с декораторами `@start` и `@listen(topic)`, простой event-loop и трассировка.
- Декоратор `tool(name)`, повторяющий форму `@tool` из CrewAI.
- `Memory` с хранилищами `short_term`, `long_term`, `entity`; имитация сходства через numpy.
- Мок-ответы LLM — захардкоженные строки, ключируемые по роли и префиксу входа. Без сети. Детерминированный.

Конкретная демонстрация: экипаж исследователя, писателя и редактора, создающий бриф по теме «агентная инженерия 2026». Исследователь извлекает (мокированные) источники. Писатель составляет черновик. Редактор ужимает текст. Тот же экипаж проходит через флоу, демонстрируя детерминированную форму.

Запуск:

```bash
python3 code/main.py
```

Трассировка включает: последовательный экипаж, передающий выходы через `context`; иерархический экипаж с выбором менеджера (исследователь, писатель, редактор, затем «done»); флоу, выполняющий те же три шага с явными темами (`researched`, `drafted`, `edited`); вызовы инструментов через `@tool`; долгосрочную память, сохраняющуюся между двумя запусками (kickoffs).

Трассировка экипажа подвижна; менеджер теоретически может переставить порядок. Трассировка флоу фиксирована. Этот выбор и есть урок.

## Применение

- **CrewAI Flow** для продакшена. Даже когда флоу состоит из одного шага, вызывающего `Crew.kickoff()`. Флоу обеспечивает границу аудита.
- **CrewAI Crew (Sequential)** для совместной работы с чётким порядком, особенно черновики и циклы рецензирования.
- **CrewAI Crew (Hierarchical)** когда маршрутизация зависит от результата и у вас четыре или более специалиста.
- **LangGraph** (Урок 13) для явных конечных автоматов, устойчивого возобновления, строгого упорядочения.
- **AutoGen v0.4** (Урок 14) для параллелизма по модели акторов и изоляции сбоев.
- **OpenAI Agents SDK** (Урок 16) для продуктов на базе OpenAI с передачами и ограничениями (guardrails).
- **Claude Agent SDK** (Урок 17) для продуктов на базе Claude с подагентами и хранилищем сессий.

## Выпуск

`outputs/skill-crew-or-flow.md` выбирает экипаж или флоу для задачи и создаёт минимальную реализацию. Жёсткий отказ от: экипажа без предыстории, флоу без явных тем, иерархического с менее чем тремя специалистами.

## Ловушки

- **Предыстория как украшение.** Она формирует выходы. Протестируйте три варианта на агента; вариативность реальна. Выберите один и заморозьте.
- **Пропуск `expected_output`.** Без контракта на каждое задание последующие задачи получают любой результат, который выдал LLM. Экипаж работает; аудит падает.
- **Память всегда включена.** Долгосрочная память пишет при каждом запуске. Векторная БД растёт. Извлечение становится шумным. Ограничьте запись заданиями, где факт является устойчивым.
- **Дрейф промпта менеджера.** Промпт менеджера в иерархическом режиме неявный. Если маршрутизация ведёт себя странно, выведите его в verbose-режиме и прочитайте.
- **Побочные эффекты инструментов в экипажах.** Экипаж может вызвать инструмент больше раз, чем ожидалось. POST, DELETE, платёжные операции должны быть в шаге флоу, никогда не в инструменте экипажа.

| Agent | «Персона» | Роль + цель + предыстория + инструменты |
| Task | «Единица работы» | Описание + ожидаемый выход + исполнитель + необязательный структурированный выход |
| Crew | «Команда агентов» | Контейнер для агентов + заданий + процесса |
| Process | «Стратегия выполнения» | Sequential / Hierarchical / Consensus (планируется) |
| Flow | «Детерминированный воркфлоу» | Событийно-ориентированный, управляемый кодом, тестируемый |
| Backstory | «Промпт персоны» | Формирователь тона и суждений агента |
| `@tool` | «Функциональный инструмент» | Декоратор, превращающий функцию в инструмент, доступный агенту |
| `BaseTool` | «Классовый инструмент» | Инструмент на основе класса с схемой аргументов, повторами и поддержкой async |
| Entity memory | «Факты по сущностям» | Память, привязанная к клиенту / аккаунту / проблеме |
| Long-term memory | «Межзапусковая память» | Память на основе векторного хранилища, сохраняющаяся между запусками (kickoffs) |
| Contextual memory | «Извлечение по мере необходимости» | Память, извлекаемая в момент, когда агенту она нужна |
| Manager LLM | «Маршрутизирующий агент» | Дополнительный LLM в иерархическом процессе, выбирающий следующее задание |
| `expected_output` | «Контракт задания» | Строка, указывающая агенту (и аудиту) желаемую форму выхода |
