# Оценочно-ориентированная разработка агентов (Eval-Driven Agent Development)

> Рекомендация Anthropic: «Начните с простых промптов, оптимизируйте их с помощью всесторонней оценки (evaluation) и добавляйте многоэтапные агентные системы только при необходимости.» Оценка — не последний шаг. Это внешний цикл (loop), который управляет каждым решением в Фазе 14.

**Тип:** Изучение + Практическая реализация
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Все уроки Фазы 14.
**Время:** ~60 минут

## Цели обучения

- Назвать три уровня оценки — статические эталоны (static benchmarks), пользовательские офлайн-оценки (custom offline evals), онлайн-оценки в продакшне (online production) — и назначение каждого.
- Описать тесный цикл «оценщик-оптимизатор» (evaluator-optimizer).
- Описать лучшую практику 2026 года: оценки живут рядом с кодом, запускаются в CI, блокируют PR.
- Связать каждый урок Фазы 14 с тем случаем оценки (eval case), который он порождает.

## Проблема

Агенты проходят демо. В продакшне они выходят из строя способами, которые демо не может предсказать. Бенчмарки (benchmarks) отвечают на вопрос «способна ли эта модель в целом?», а не на вопрос «отправляет ли этот агент правильные патчи для моего продукта?» Решение: оценка на трёх уровнях, запускаемая непрерывно, где каждый ограничитель (guardrail) и каждое выученное правило сопоставлены с конкретным случаем оценки.

## Концепция

### Три уровня оценки

1. **Статические эталоны (Static benchmarks)** — SWE-bench Verified для кода (Урок 19), WebArena/OSWorld для браузинга / рабочего стола (Урок 20), GAIA для универсальных задач (Урок 19), BFCL V4 для использования инструментов (Урок 06). Используются для межмодельного сравнения и блокировки регрессий. Проблема заражения данных (contamination) реальна: SWE-bench+ обнаружил утечку 32,67% решений. Всегда указывайте баллы Verified / с аудитом +-.

2. **Пользовательские офлайн-оценки (Custom offline evals)** — под форму вашего продукта:
   - LLM-в-роли-судьи (LLM-as-judge) (Langfuse, Phoenix, Opik — Урок 24).
   - На основе выполнения (execution-based) — запуск патча, проверка тестов.
   - На основе траектории (trajectory-based) — сравнение последовательностей действий с эталонным; OSWorld-Human показывает, что лучшие агенты превосходят эталон в 1,4–2,7 раза.

3. **Онлайн-оценки (Online evals)** — в продакшне:
   - Воспроизведение сессий (session replays) (Langfuse).
   - Оповещения, срабатываемые ограничителями (guardrail-triggered alerts) (Урок 16, 21).
   - Отслеживание стоимости / задержки на каждом шаге (per-step cost / latency tracking) (Урок 23, спаны OTel).

### Цикл «оценщик-оптимизатор» (Evaluator-optimizer, Anthropic)

Тесный цикл:

1. Генератор (proposer) создаёт выходные данные.
2. Оценщик (evaluator) проверяет.
3. Уточнение (refine), пока оценщик не одобрит.

Это обобщение подхода Self-Refine (Урок 05). Любой агентный процесс, который вам важен, можно обернуть в цикл «оценщик-оптимизатор» для повышения надёжности.

### Лучшая практика 2026 года

- Оценки живут рядом с кодом.
- Запускаются в CI при каждом PR.
- Мерж блокируется по результатам оценок (например, «нет регрессии > 5% относительно main»).
- Каждый ограничитель (guardrail) сопоставлен с случаем оценки.
- Каждое выученное правило (Reflexion, выученное правило в воркфлоу) сопоставлено с случаем сбоя.

### Где подход оценочно-ориентированной разработки даёт сбой

- **Нет базовой линии.** Оценки без известного последнего хорошего результата (last-known-good) нечитаемы. Сохраняйте базовые линии (baselines).
- **LLM-судья без привязки к внешнему контексту.** Судьи тоже галлюцинируют. Паттерн CRITIC (Урок 05) — судья опирается на внешние инструменты.
- **Переобучение на оценках (overfitting to evals).** Оптимизация под оценки отдаляет от реальной полезности в продакшне. Чередуйте случаи.
- **Нестабильные оценки (flaky evals).** Недетерминированные случаи вызывают ложные срабатывания. Фиксируйте зерена (seeds), делайте снимки состояния (snapshot state).

## Реализация

`code/main.py` — средство оценки (eval harness) на стандартной библиотеке Python:

- Реестр случаев с категориями (бенчмарк, пользовательский, онлайн).
- Тестируемый агент с заранее заданным поведением.
- Цикл «оценщик-оптимизатор»: генерация, проверка, уточнение до прохождения или максимального числа раундов.
- Ворота CI (CI gate): агрегированная доля прохождений + регрессия относительно базовой линии.

Запуск:

```
python3 code/main.py
```

Вывод: прохождение/непрохождение для каждого случая, флаг регрессии, вердикт ворот CI.

## Применение

- Пишите случаи оценки в том же репозитории, что и код агента.
- Запускайте их при каждом PR через CI.
- Проваливайте сборку при регрессии.
- Отслеживайте долю прохождений (pass rate) во времени.
- Привязывайте каждый случай сбоя в продакшне к новому случаю оценки.

## Отправка

`outputs/skill-eval-suite.md` формирует трёхуровневый набор оценок для агентного продукта с воротами CI и отслеживанием регрессии.

## Упражнения

1. Возьмите один из ваших сбоев в продакшне. Напишите случай оценки, который его воспроизводит. Проходит ли ваш агент его сейчас?
2. Создайте рубрику для LLM-судьи для вашей предметной области с тремя измерениями (фактическая точность, тон, охват). Оцените 50 сессий.
3. Подключите набор оценок к CI. Проваливайте сборку при регрессии >=5%.
4. Добавьте метрику эффективности траектории: сколько шагов совершил агент по сравнению с эталонной траекторией?
5. Сопоставьте каждый урок Фазы 14 с случаем оценки в вашем наборе. Есть ли пропуски? Это брешь, которую нужно закрыть.

## Ключевые термины

| Термин | Как обычно говорят | Что это означает |
|--------|-------------------|------------------|
| Статический эталон (Static benchmark) | «Готовая оценка» | SWE-bench, GAIA, AgentBench, WebArena, OSWorld |
| Пользовательская офлайн-оценка (Custom offline eval) | «Оценка предметной области» | LLM-в-роли-судьи / выполнение / траектория на данных вашего продукта |
| Онлайн-оценка (Online eval) | «Оценка в продакшне» | Воспроизведение сессий, оповещения ограничителей, отслеживание стоимости/задержки |
| Цикл «оценщик-оптимизатор» (Evaluator-optimizer) | «Генерация-проверка-уточнение» | Итерации до одобрения судьёй |
| Ворота CI (CI gate) | «Блокер мержа» | Провал сборки при регрессии оценок |
| Базовая линия (Baseline) | «Последний известный хороший результат» | Эталонный балл для обнаружения регрессии |
| Эффективность траектории (Trajectory efficiency) | «Шаги относительно эталона» | Количество шагов агента, делённое на минимум человеческого эксперта |
