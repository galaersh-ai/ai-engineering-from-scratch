# Семантические соглашения OpenTelemetry для GenAI

> GenAI SIG (Special Interest Group) OpenTelemetry, запущенный в апреле 2024 года, определяет стандартную схему телеметрии агентов. Имена спанов, атрибуты и правила захвата контента унифицированы между поставщиками, поэтому трейсы агентов означают одно и то же в Datadog, Grafana, Jaeger и Honeycomb.

**Тип:** Теория + практика
**Языки:** Python (стандартная библиотека)
**Предварительные требования:** Фаза 14 · Урок 13 (LangGraph), Фаза 14 · Урок 24 (платформы наблюдаемости)
**Время:** ~60 минут

## Цели обучения

- Называть категории спанов GenAI: модель/клиент (model/client), агент (agent), инструмент (tool).
- Различать спаны `invoke_agent` типа CLIENT и INTERNAL и понимать, когда какой применяется.
- Перечислять основные атрибуты GenAI: имя поставщика (provider name), модель запроса (request model), идентификатор источника данных (data-source ID).
- Объяснять контракт захвата контента (content-capture): подключение по требованию (opt-in), `OTEL_SEMCONV_STABILITY_OPT_IN`, рекомендация по внешним ссылкам.

## Проблема

Каждый поставщик изобретает собственные имена спанов. Команды эксплуатации вынуждены создавать панели мониторинга для каждого фреймворка отдельно. GenAI SIG OpenTelemetry решает эту проблему, определяя единый стандарт, на который ориентируется вся экосистема.

## Концепция

### Категории спанов

1. **Спаны модели / клиента (model / client spans).** Охватывают сырые вызовы LLM. Генерируются SDK поставщиков (Anthropic, OpenAI, Bedrock) и адаптерами моделей фреймворков.
2. **Спаны агентов (agent spans).** `create_agent` (при создании агента) и `invoke_agent` (при его выполнении).
3. **Спаны инструментов (tool spans).** Один на каждый вызов инструмента; связаны со спаном агента отношением родитель-потомок.

### Именование спанов агентов

- Имя спана: `invoke_agent {gen_ai.agent.name}`, если имя задано; в противном случае — `invoke_agent`.
- Тип спана (Span kind):
  - **CLIENT** — для удалённых сервисов агентов (OpenAI Assistants API, Bedrock Agents).
  - **INTERNAL** — для внутрипроцессных фреймворков агентов (LangChain, CrewAI, локальный ReAct).

### Основные атрибуты

- `gen_ai.provider.name` — `anthropic`, `openai`, `aws.bedrock`, `google.vertex`.
- `gen_ai.request.model` — идентификатор модели.
- `gen_ai.response.model` — выбранная модель (может отличаться от запроса из-за маршрутизации).
- `gen_ai.agent.name` — идентификатор агента.
- `gen_ai.operation.name` — `chat`, `completion`, `invoke_agent`, `tool_call`.
- `gen_ai.data_source.id` — для RAG: какой корпус или хранилище было запрошено.

Существуют технологически специфичные соглашения для Anthropic, Azure AI Inference, AWS Bedrock, OpenAI.

### Захват контента

Правило по умолчанию: инструментарий НЕ ДОЛЖЕН захватывать входы/выходы по умолчанию. Захват осуществляется по требованию (opt-in) через:

- `gen_ai.system_instructions`
- `gen_ai.input.messages`
- `gen_ai.output.messages`

Рекомендуемый производственный паттерн: хранить контент во внешнем хранилище (S3, хранилище логов), записывая на спаны ссылки (идентификаторы указателей, а не текст). Это механизм защиты от отравления контента из Урока 27, интегрированный в наблюдаемость (observability).

### Стабильность

По состоянию на март 2026 года большинство соглашений являются экспериментальными. Подключение к стабильной предварительной версии (stable preview):

```
OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental
```

Datadog v1.37+ нативно сопоставляет атрибуты GenAI со своей схемой LLM Observability. Другие бэкенды (Grafana, Honeycomb, Jaeger) поддерживают сырые атрибуты.

### Типичные ошибки

- **Захват полных промптов в спанах.** PII, секреты и данные клиентов попадают в трейсы, доступные операторам. Храните во внешнем хранилище.
- **Отсутствие `gen_ai.provider.name`.** Панели мониторинга для нескольких поставщиков ломаются, когда невозможно определить источник.
- **Спаны без связей с родителями.** Осиротевшие спаны инструментов. Всегда передавайте контекст (context propagation).
- **Отсутствие установки стабильного opt-in.** Атрибуты могут быть переименованы при обновлении бэкенда.

## Практическая часть

Файл `code/main.py` реализует эмиттер спанов на стандартной библиотеке Python, соответствующий соглашениям GenAI:

- `Span` со схемой атрибутов GenAI.
- `Tracer` с `start_span`, вложенными контекстами.
- Сценарий запуска агента, генерирующий: `create_agent`, `invoke_agent` (INTERNAL), спаны для каждого инструмента, спаны `chat` для вызовов LLM.
- Режим захвата контента, сохраняющий промпты во внешнем хранилище и записывающий идентификаторы на спаны.

Запуск:

```
python3 code/main.py
```

Вывод: дерево спанов со всеми необходимыми атрибутами GenAI и демонстрацией внешнего хранилища со ссылками на захваченный контент по требованию (opt-in content references).

## Применение

- **Datadog LLM Observability** (v1.37+) нативно сопоставляет атрибуты.
- **Langfuse / Phoenix / Opik** (Урок 24) — автоматическая инструментация экосистемы.
- **Jaeger / Honeycomb / Grafana Tempo** — сырые трейсы OTel; создание панелей мониторинга на основе атрибутов GenAI.
- **Собственный хостинг** — запуск OTel Collector с процессором GenAI.

## Отправка

Файл `outputs/skill-otel-genai.md` интегрирует спаны OTel GenAI в существующего агента с настройками захвата контента по умолчанию и хранением внешних ссылок.

## Упражнения

1. Инструментируйте свой цикл (loop) ReAct из Урока 01 спанами `invoke_agent` (INTERNAL) + спанами для каждого инструмента. Отправьте данные в экземпляр (instance) Jaeger.
2. Добавьте захват контента в режиме «только ссылки»: промпты в SQLite, атрибуты спанов содержат только идентификаторы строк.
3. Изучите спецификацию для `gen_ai.data_source.id`. Интегрируйте его в поиск Mem0 из Урока 09.
4. Установите `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` и убедитесь, что атрибуты не переименовываются коллектором.
5. Создайте панель мониторинга: «какие ошибки инструментов коррелируют с какими моделями» на основе только атрибутов GenAI.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|--------|-------------|----------------------|
| GenAI SIG | «Группа OpenTelemetry по GenAI» | Рабочая группа OTel, определяющая схему |
| invoke_agent | «Спан агента» | Имя спана, представляющего выполнение агента |
| CLIENT span | «Удалённый вызов» | Спан для вызова удалённого сервиса агента |
| INTERNAL span | «Внутрипроцессный» | Спан для внутрипроцессного выполнения агента |
| gen_ai.provider.name | «Поставщик» | anthropic / openai / aws.bedrock / google.vertex |
| gen_ai.data_source.id | «Источник RAG» | Какой корпус/хранилище дал результат поиска |
| Content capture | «Логирование промптов» | Захват сообщений по требованию; в продакшене — во внешнее хранилище |
| Stability opt-in | «Режим предварительного просмотра» | Переменная окружения для фиксации экспериментальных соглашений |
