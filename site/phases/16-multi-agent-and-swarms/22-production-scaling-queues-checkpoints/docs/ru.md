# Продакшен-масштабирование — очереди, чекпоинты, надёжность

> Масштабирование мультиагентных систем до тысяч параллельных запусков требует **надёжного выполнения**. Рантайм LangGraph записывает чекпоинт после каждого супер-шага, ключеванного по `thread_id` (Postgres по умолчанию); при аварии работника аренда снимается и другой работник возобновляет. Агенты могут спать бесконечно, ожидая ввода человека. **MegaAgent** (arXiv:2408.09955) использовал очередь производитель-потребитель на агента с тремя состояниями (Idle / Processing / Response) и двухуровневой координацией (внутригрупповой чат + межгрупповой чат администратора). **Fiber/async** опережает потоки на задачу для стриминга LLM: потоки бездействуют 99% времени, ожидая токены, тогда как файберы кооперативно уступают при I/O. Контраргумент: «Scaling Agentic Software» от Ashpreet Bedi выступает за **FastAPI + Postgres + ничего больше**, пока нагрузка не докажет обратное — простые архитектуры работают лучше, чем ожидалось. Этот урок построит надёжный лог чекпоинтов, очередь задач на агента с переходами состояний, демонстрацию async-vs-thread и закрепит прагматическое правило «начинай просто».

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib, `asyncio`, `sqlite3`)
**Предварительные требования:** Фаза 16 · 09 (Параллельные сети роёв), Фаза 16 · 13 (Общая память)
**Время:** ~75 минут

## Проблема

Прототип мультиагентной системы работает на одном ноутбуке с тремя агентами в in-memory event loop. Вы переходите в продакшен:

- Агенты иногда работают часами (длительное исследование, ожидание человека в цикле).
- Рабочие процессы падают. Перезапуск теряет состояние.
- Пиковая нагрузка в 10× средней; нужно горизонтальное масштабирование.
- Пользователи платят за запуск агента; нужна семантика «ровно один раз» для биллинга.

In-memory event loop ничего из этого не обеспечивает. Нужен слой надёжного выполнения. Каноничные варианты 2026 года:

1. Движок рабочих процессов с чекпоинтами (Temporal, рантайм LangGraph).
2. Очередь сообщений с хранилищем состояний (Postgres + SQS/RabbitMQ).
3. Фреймворки модели акторов (producer-consumer на агента в MegaAgent).
4. Самодельный FastAPI + Postgres (аргумент Bedi).

Этот урок построит миниатюру каждого варианта.

## Концепция

### Надёжное выполнение — паттерн

Движок надёжного выполнения сохраняет полное состояние программы после каждого «шага» (супер-шага на языке LangGraph). При аварии:

```
работник падает на полпути
  -> тайм-аут аренды
  -> другой работник подхватывает thread_id
  -> возобновляет с последнего чекпоинта
  -> без дублирования побочных эффектов
```

Требования для работы:

- **Сериализуемое состояние.** Всё состояние агента должно быть сохраняемым. Замыкания функций с активными соединениями с БД не выживают.
- **Детерминированное возобновление.** При одинаковом состоянии и одинаковых входах агент воспроизводит те же действия (или делегирует внешнему детерминистическому оракулу для LLM-вызовов).
- **Идемпотентные побочные эффекты.** Внешние вызовы (вызовы инструментов, платежи) должны быть идемпотентными или использовать ключ дедупликации.

LangGraph записывает чекпоинт после каждого супер-шага; Temporal — после каждого activity; Restate использует event-sourced журналы. Все три реализуют один и тот же паттерн.

### Рантайм LangGraph

Каждый агент имеет `thread_id`; состояние — типизированный dict; каждый супер-шаг записывает строку в таблицу чекпоинтов. При возобновлении рантайм воспроизводит с последнего чекпоинта, а не с нуля. Агенты могут вызвать `interrupt()` для ожидания ввода человека; рантайм сохраняет и освобождает работника. Когда ввод поступает, любой работник может возобновить.

Это референсный продакшен-дизайн на апрель 2026 года.

### Очередь на агента в MegaAgent

arXiv:2408.09955 описывает эксперимент по масштабированию: тысячи параллельных агентов в одном кластере. Архитектура:

```
агент i:
  состояние ∈ {Idle, Processing, Response}
  in_queue   <- сообщения адресованные агенту i
  out_queue  -> ответы + побочные эффекты

координаторы:
  внутригрупповой чат  (агенты в одной группе)
  межгрупповой чат администратора  (высокоуровневая маршрутизация)
```

Двухуровневая координация позволяет внутригрупповому общению происходить плотно, тогда как межгрупповое остаётся разреженным — паттерн, используемый для сохранения линейной стоимости при тысячах агентов.

### Async vs поток на задачу

Вызовы LLM — I/O-bound. Поток, ожидающий следующий токен, бездействует 99% времени. Потоки стоят ~1MB RAM каждый; при 10 000 параллельных вызовах это 10GB только на стеки.

Файберы (Python `asyncio`, Go goroutines, Rust `tokio`) кооперативно уступают при I/O. Те же 10 000 вызовов помещаются в процесс комфортно. В масштабе LLM-агентов async — не оптимизация, а архитектура.

Исключение: CPU-bound пост-обработка (эмбеддинги, трюки токенизатора) всё ещё требует потоков или процессов. Отделяйте I/O-слой от CPU-слоя.

### Контраргумент Bedi

«Scaling Agentic Software» (Ashpreet Bedi, 2026) утверждает, что большинство команд перекомплицируют до измерения нагрузки. Прагматичный дефолт:

- FastAPI + Postgres.
- Каждый запуск агента — строка; состояние обновляется на месте с оптимистичной блокировкой.
- Фоновые задачи через `pg_notify` или простой Celery-воркер.
- Политика повторов в коде приложения.

При нагрузке до ~100 параллельных запусков агентов на поддающихся управлению задачах этого обычно достаточно. Улучшайте, когда измерите проблему.

Правило: внедряйте движки надёжного выполнения, когда упрётесь в конкретную проблему, которую простые архитектуры не решают. Внедрение на ранней стадии сжигает время на церемонии, которые не окупаются.

### Семантика «ровно один раз»

Для платных запусков агентов нужна «эффективно ровно один раз» (at-least-once доставка + идемпотентный потребитель). Инженерные приёмы:

- **Ключ дедупликации на запуск.** Включайте его в каждый вызов побочного эффекта.
- **Паттерн outbox.** Побочные эффекты сначала пишутся в таблицу, затем отдельный процесс их выполняет. Оба шага идемпотентны.
- **Компенсирующие транзакции.** Когда побочный эффект успешен, но запись отслеживания падает, запланируйте компенсацию.

Это паттерны проектирования БД, а не специфика LLM. Налог LLM только в том, что вызовы LLM медленные; всё остальное — стандартные распределённые системы.

### Радужное развёртывание

Мультиагентная исследовательская система Anthropic использует «радужные развёртывания»: несколько версий рантайма агента работают параллельно, чтобы долгоживущие агенты не убивались при каждом деплое кода. Канарейки новых версий на части трафика; старые версии выводятся по мере завершения агентов.

Это стандарт для долгоживущих состоящих систем; адаптация 2026 года в том, что агенты могут жить часами, поэтому циклы деплоя должны это учитывать.

### Каноничный продакшен-чеклист

- Надёжное состояние (чекпоинты, снимки или outbox + воспроизводимый лог).
- Идемпотентные побочные эффекты.
- Async I/O-слой для LLM-вызовов.
- At-least-once доставка с дедупликацией.
- Радужный/канареечный деплой для состоящих нагрузок.
- Наблюдаемость: трассировка на агента, аудит супер-шагов, счётчик повторов.

## Реализуем

`code/main.py` реализует:

- `CheckpointStore` — лог чекпоинтов на SQLite с ключами thread-id. Каждый супер-шаг добавляет строку.
- `run_with_checkpoint(agent, thread_id)` — имитирует аварию на полпути; второй работник возобновляет с последнего чекпоинта.
- `AgentQueue` — автомат состояний Idle / Processing / Response на агента с небольшой очередью задач.
- `demo_async_vs_threads()` — запускает 500 параллельных имитированных «вызовов LLM» через asyncio и через потоки; выводит реальное время и пиковую память (приближённо).

Запуск:

```
python3 code/main.py
```

Ожидаемый вывод: возобновление с чекпоинта успешно после имитированной аварии; async-версия обрабатывает 500 параллельных вызовов за < 1с; потоковая версия занимает несколько секунд и потребляет на порядки больше памяти на параллельную единицу.

## Результат

Каноничное продакшен-упрощение:

- **Начинай просто (правило Bedi).** FastAPI + Postgres, пока не измерите проблему.
- **Инструментируй всё до оптимизации.** Гистограмма латентности на запуск, время на шаг, счётчик повторов, категоризация сбоев.
- **Паттерн outbox для побочных эффектов.** Особенно платежи и внешние API.
- **Радужные деплои.** Никогда не убивайте запущенные агенты во время деплоя.
- **Внедряйте движки надёжного выполнения (Temporal / LangGraph / Restate), когда** упрётесь в конкретные проблемы: часовые ожидания человека, кросс-регионная координация, сложные политики повторов/компенсаций.
- **Async для I/O-слоя.** Потоки только для CPU-bound пост-обработки.

## Упражнения

1. Запустите `code/main.py`. Подтвердите, что возобновление с чекпоинта работает; измерьте разницу между async и потоковой параллельностью.
2. Реализуйте таблицу **outbox**: каждый вызов инструмента сначала пишется в outbox, затем отдельная goroutine/задача выполняет. Проверьте идемпотентность, запустив вызов дважды.
3. Смоделируйте **радужный деплой**: две параллельные версии рантайма; маршрутизируйте половину новых thread_id на каждую; подтвердите, что запущенные потоки на старой версии не прерываются.
4. Прочитайте документацию рантайма LangGraph (ссылка ниже). Определите, какие функции рантайма заняли бы больше всего времени для воспроизведения в самодельной версии FastAPI + Postgres. Это аргумент за внедрение или можно отложить?
5. Прочитайте MegaAgent (arXiv:2408.09955), раздел 3. Двухуровневая координация (внутригрупповая + межгрупповой чат администратора) явно описана. Набросайте, как вы бы отобразили это на очередь сообщений с двумя семействами очередей.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|----------------|------------------------|
| Надёжное выполнение | «Сохранение состояния программы» | Движок записывает состояние после каждого супер-шага; восстановление после аварии детерминировано. |
| Супер-шаг | «Транзакционная граница» | Единица работы между чекпоинтами. Термин LangGraph. |
| thread_id | «Идентификатор запуска агента» | Ключ, связывающий чекпоинты и логику возобновления. |
| Идемпотентность | «Безопасно повторять» | Повтор побочного эффекта даёт тот же результат, что и первая попытка. |
| Паттерн outbox | «Отделение побочных эффектов» | Намерение пишется в таблицу; отдельный исполнитель выполняет и помечает как выполненное. |
| At-least-once доставка | «Возможны дубликаты» | Семантика очереди сообщений; ключ дедупликации делает потребителя effectively-once. |
| Радужный деплой | «Перекрывающиеся версии» | Несколько версий рантайма параллельно во время долгоживущих нагрузок. |
| Async-файбер | «Кооперативная передача» | Конкурентность пользовательского уровня; дешевле потоков для I/O-bound нагрузок. |
| Чекпоинт | «Снимок состояния» | Сериализованное состояние на границе супер-шага; ключ для возобновления. |
