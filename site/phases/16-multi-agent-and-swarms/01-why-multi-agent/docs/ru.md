# Почему мультиагентная система?

> Одному агенту попался на пути стенка. Умный ход не в том, чтобы сделать его больше — а в том, чтобы использовать несколько агентов.

**Тип:** Изучение
**Языки:** TypeScript
**Предварительные требования:** Фаза 14 (Инженерия агента)
**Время:** ~60 минут

## Цели обучения

- Выявить потолок одиночного агента (переполнение контекста, смешанная экспертиза, последовательный узкое место) и объяснить когда разбиение на несколько агентов является правильным шагом
- Сравнить шаблоны оркестрации (пайплайн, параллельное распределение, супервайзер, иерархия) и выбрать подходящий для заданной структуры задачи
- Проектировать мультиагентную систему с четкими границами ролей, общей состояния и контрактом коммуникации
- Анализировать tradeoffs мультиагентной сложности (задержка, стоимость, сложность отладки) по сравнению с простотой одиночного агента

## Проблема

Вы создали одиночного агента в фазе 14. Он работает. Он может читать файлы, выполнять команды, вызывать API и размышлять о результатах. Затем вы направили его на реальный код: 200 файлов, три языка, тесты, зависящие от инфраструктуры, и требование исследовать внешние API перед написанием кода.

Агент застрял. Не потому что LLM глуп, а потому что задача превышает то, что может справиться один цикл агента. Контекстное окно заполняется содержимым файлов. Агент забывает, что прочитал 40 вызовов инструментов назад. Он пытается быть исследователем, программистом и ревьюером одновременно, и делает все это плохо.

Это потолок одиночного агента. Вы его достигаете каждый раз, когда задача требует:

- **Больше контекста, чем помещается в одно окно** - чтение 50 файлов превышает 200k токенов
- **Разные экспертизы на разных этапах** - исследование требует другого запроса, чем генерация кода
- **Работа, которая может выполняться параллельно** - почему читать три файла последовательно, когда можно это сделать одновременно?

## Концепция

### Потолок одиночного агента

Одиночный агент — один цикл, одно контекстное окно, одна система запроса. Представьте себе:

```
┌─────────────────────────────────────────┐
│            SINGLE AGENT                 │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │         Context Window            │  │
│  │                                   │  │
│  │  research notes                   │  │
│  │  + code files                     │  │
│  │  + test output                    │  │
│  │  + review feedback                │  │
│  │  + API docs                       │  │
│  │  + ...                            │  │
│  │                                   │  │
│  │  ██████████████████████ FULL ███  │  │
│  └───────────────────────────────────┘  │
│                                         │
│  One system prompt tries to cover       │
│  research + coding + review + testing   │
│                                         │
│  Result: mediocre at everything         │
└─────────────────────────────────────────┘
```

Три вещи сломались:

1. **Переполнение контекста** - результаты инструментов накапливаются. На третьей итерации агент уже съел 150k токенов содержимого файлов, выводов команд и предыдущих размышлений. Критические детали из пятой итерации потерялись.

2. **Смешение ролей** - система запроса "ты исследователь, программист, ревьюер и тестировщик" выдает агента, который лишь наполовину исследует, лишь наполовину пишет код и никогда не завершает ревью.

3. **Последовательный узкое место** - агент читает файл A, затем файл B, затем файл C. Три последовательных вызова LLM. Три последовательных выполнения инструментов. Нет параллелизма.

### Многокорпусное решение

Разделите работу. Дайте каждому агенту одну задачу, одно контекстное окно и системный запрос, настроенный под эту задачу:

```
┌──────────────────────────────────────────────────────────┐
│                    ORCHESTRATOR                          │
│                                                          │
│  "Build a REST API for user management"                  │
│                                                          │
│         ┌──────────┬──────────┬──────────┐               │
│         │          │          │          │               │
│         ▼          ▼          ▼          ▼               │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│   │RESEARCHER│ │  CODER   │ │ REVIEWER │ │  TESTER  │  │
│   │          │ │          │ │          │ │          │  │
│   │ Reads    │ │ Writes   │ │ Checks   │ │ Runs     │  │
│   │ docs,    │ │ code     │ │ code     │ │ tests,   │  │
│   │ finds    │ │ based on │ │ quality, │ │ reports  │  │
│   │ patterns │ │ research │ │ finds    │ │ results  │  │
│   │          │ │ + spec   │ │ bugs     │ │          │  │
│   └─────┬────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  │
│         │           │            │             │         │
│         └───────────┴────────────┴─────────────┘         │
│                          │                               │
│                     Merge results                        │
└──────────────────────────────────────────────────────────┘
```

Каждый агент имеет:
- Фокусированный системный запрос ("Вы кодовый ревьюер. Ваша единственная задача — находить баги.")
- Свое собственное контекстное окно (не загрязненное работой других агентов)
- Ясную схему ввода/вывода (получает заметки исследований, выдает код)

### Реальные системы, которые это делают

**Claude Code субагенты** - когда Claude Code создает субагент с инструментом `Task`, он создает дочерний агент с ограниченной задачей. Родитель сохраняет свой контекст чистым. Дочерний выполняет фокусированную работу и возвращает сводку.

**Devin** - запускает планировщик-агент, программист-агент и браузер-агент. Планировщик разбивает работу на шаги. Программист пишет код. Браузер исследует документацию. Каждый имеет отдельный контекст.

**Мультиагентные команды программистов (SWE-bench)** - лучшие системы на SWE-bench используют исследователя, читающего код, планировщика, который проектирует исправление, и программиста, который реализует его. Одноагентные системы получают меньший балл.

**ChatGPT Глубокое Исследование** - запускает несколько поисковых агентов параллельно, каждый исследуя разные направления, затем синтезирует результаты.

### Спектр

Мультиагентность не бинарна. Это спектр:

```
SIMPLE ──────────────────────────────────────────── COMPLEX

 Single        Sub-         Pipeline      Team         Swarm
 Agent         agents

 ┌───┐       ┌───┐        ┌───┐───┐    ┌───┐───┐    ┌─┐┌─┐┌─┐
 │ A │       │ A │        │ A │ B │    │ A │ B │    │ ││ ││ │
 └───┘       └─┬─┘        └───┘─┬─┘    └─┬─┘─┬─┘    └┬┘└┬┘└┬┘
               │                │        │   │       ┌┴──┴──┴┐
             ┌─┴─┐          ┌───┘───┐    │   │       │shared │
             │ a │          │ C │ D │  ┌─┴───┴─┐    │ state │
             └───┘          └───┘───┘  │  msg   │    └───────┘
                                       │  bus   │
 1 loop      Parent +      Stage by    │       │    N peers,
 1 context   child tasks   stage       └───────┘    emergent
                                       Explicit      behavior
                                       roles
```

**Одиночный агент** - один цикл, один запрос. Хорош для простых задач.

**Субагенты** - родитель создает дочерние агенты для фокусированных подзадач. Родитель сохраняет план. Дочерние отчитываются о результатах. Это то, что делает Claude Code.

**Пайплайн** - агенты выполняют последовательно. Выход одного агента становится входом другого. Хорош для стадийных процессов: исследование -> код -> ревью -> тестирование.

**Команда** - агенты выполняют параллельно с общим шиной сообщений. Каждый имеет роль. Оркестратор coординирует. Хорошо, когда нужны разные навыки одновременно.

**Рой (Swarm)** — множество идентичных или очень похожих агентов с общим состоянием. Нет фиксированного оркестратора. Агенты берут работу из очереди. Хорош для параллельных задач высокой пропускной способности.

### Четыре шаблона мультиагентной системы

#### Шаблон 1: Пайплайн

```
Input ──▶ Agent A ──▶ Agent B ──▶ Agent C ──▶ Output
          (research)  (code)      (review)
```

Каждый агент преобразует данные и передает их дальше. Просто в терминах рассуждений. Ошибка на одном этапе блокирует остальные.

#### Шаблон 2: Распределение / Сбор

```
                ┌──▶ Agent A ──┐
                │              │
Input ──▶ Split ├──▶ Agent B ──├──▶ Merge ──▶ Output
                │              │
                └──▶ Agent C ──┘
```

Распределяйте работу между параллельными агентами, затем объединяйте результаты. Хорошо для задач, которые разлагаются на независимые подзадачи.

#### Шаблон 3: Оркестратор-работник

```
                    ┌──────────┐
                    │  Orch.   │
                    └──┬───┬───┘
                  task │   │ task
                 ┌─────┘   └─────┐
                 ▼               ▼
           ┌──────────┐   ┌──────────┐
           │ Worker A │   │ Worker B │
           └──────────┘   └──────────┘
```

Умный оркестратор решает, что делать, делегирует работу работникам и синтезирует результаты. Оркестратор сам по себе агент с инструментами для запуска работников.

#### Шаблон 4: Плоский рой

```
         ┌───┐ ◄──── msg ────▶ ┌───┐
         │ A │                  │ B │
         └─┬─┘                  └─┬─┘
           │                      │
      msg  │    ┌───────────┐     │ msg
           └───▶│  Shared   │◄────┘
                │  State    │
           ┌───▶│  / Queue  │◄────┐
           │    └───────────┘     │
      msg  │                      │ msg
         ┌─┴─┐                  ┌─┴─┐
         │ C │ ◄──── msg ────▶ │ D │
         └───┘                  └───┘
```

Нет фиксированного оркестратора. Агенты общаются напрямую друг с другом. Поведение возникает из взаимодействия. Труднее отлаживать, но масштабируется до многих агентов.

### Когда НЕ использовать мультиагентную систему

Мультиагентность добавляет сложность. Каждое сообщение между агентами — потенциальная точка сбоя. Отладка переходит от "прочтите одну беседу" к "следуйте сообщениям через пять агентов".

**Оставайтесь одиночным агентом, когда:**
- Задача помещается в одно контекстное окно (менее ~100k токенов рабочих данных)
- Не требуется разные системные запросы для разных этапов
- Последовательное выполнение достаточно быстро
- Задача простая настолько, что ее разделение добавляет больше издержек, чем пользы

**Стоимость сложности:**
- Каждый граница агента — это убыточный шаг сжатия: контекст агента A суммируется в сообщении для агента B
- Логика координации (кто что делает, когда и в каком порядке) является источником ошибок
- Задержка увеличивается: N агентов означают N последовательных вызовов LLM минимум, больше если они должны обмениваться сообщениями друг с другом
- Стоимость умножается: каждый агент сжигает токены независимо

Правило пальца: если задача требует менее 20 вызовов инструментов и помещается в 100k токенов, оставьте ее одиночным агентом.

## Строим его

### Шаг 1: Перегруженный одиночный агент

Вот одиночный агент, который пытается делать все. У него один большой системный запрос и одно контекстное окно с исследованием, кодом и ревью:

```typescript
type AgentResult = {
  content: string;
  tokensUsed: number;
  toolCalls: number;
};

async function singleAgentApproach(task: string): Promise<AgentResult> {
  const systemPrompt = `You are a full-stack developer. You must:
1. Research the requirements
2. Write the code
3. Review the code for bugs
4. Write tests
Do ALL of these in a single conversation.`;

  const contextWindow: string[] = [];
  let totalTokens = 0;
  let totalToolCalls = 0;

  const research = await fakeLLMCall(systemPrompt, `Research: ${task}`);
  contextWindow.push(research.output);
  totalTokens += research.tokens;
  totalToolCalls += research.calls;

  const code = await fakeLLMCall(
    systemPrompt,
    `Given this research:\n${contextWindow.join("\n")}\n\nNow write code for: ${task}`
  );
  contextWindow.push(code.output);
  totalTokens += code.tokens;
  totalToolCalls += code.calls;

  const review = await fakeLLMCall(
    systemPrompt,
    `Given all previous context:\n${contextWindow.join("\n")}\n\nReview the code.`
  );
  contextWindow.push(review.output);
  totalTokens += review.tokens;
  totalToolCalls += review.calls;

  return {
    content: contextWindow.join("\n---\n"),
    tokensUsed: totalTokens,
    toolCalls: totalToolCalls,
  };
}
```

Проблемы с этим подходом:
- Контекстное окно растет на каждом этапе. На этапе ревью оно содержит заметки исследований, код и предыдущие размышления.
- Системный запрос универсален. Он не может быть настроен для каждого этапа.
- Ничто не выполняется параллельно.

### Шаг 2: Специализированные агенты

Теперь разделите его. Каждый агент получает одну задачу:

```typescript
type SpecialistAgent = {
  name: string;
  systemPrompt: string;
  run: (input: string) => Promise<AgentResult>;
};

function createSpecialist(name: string, systemPrompt: string): SpecialistAgent {
  return {
    name,
    systemPrompt,
    run: async (input: string) => {
      const result = await fakeLLMCall(systemPrompt, input);
      return {
        content: result.output,
        tokensUsed: result.tokens,
        toolCalls: result.calls,
      };
    },
  };
}

const researcher = createSpecialist(
  "researcher",
  "You are a technical researcher. Read documentation, find patterns, and summarize findings. Output only the facts needed for implementation."
);

const coder = createSpecialist(
  "coder",
  "You are a senior TypeScript developer. Given requirements and research notes, write clean, tested code. Nothing else."
);

const reviewer = createSpecialist(
  "reviewer",
  "You are a code reviewer. Find bugs, security issues, and logic errors. Be specific. Cite line numbers."
);
```

Каждый специалист имеет фокусированный запрос. Каждый получает чистое контекстное окно с только необходимыми данными.

### Шаг 3: Координация через сообщения

Соедините специалистов с явным обменом сообщениями:

```typescript
type AgentMessage = {
  from: string;
  to: string;
  content: string;
  timestamp: number;
};

async function multiAgentApproach(task: string): Promise<AgentResult> {
  const messages: AgentMessage[] = [];
  let totalTokens = 0;
  let totalToolCalls = 0;

  const researchResult = await researcher.run(task);
  messages.push({
    from: "researcher",
    to: "coder",
    content: researchResult.content,
    timestamp: Date.now(),
  });
  totalTokens += researchResult.tokensUsed;
  totalToolCalls += researchResult.toolCalls;

  const coderInput = messages
    .filter((m) => m.to === "coder")
    .map((m) => `[From ${m.from}]: ${m.content}`)
    .join("\n");

  const codeResult = await coder.run(coderInput);
  messages.push({
    from: "coder",
    to: "reviewer",
    content: codeResult.content,
    timestamp: Date.now(),
  });
  totalTokens += codeResult.tokensUsed;
  totalToolCalls += codeResult.toolCalls;

  const reviewerInput = messages
    .filter((m) => m.to === "reviewer")
    .map((m) => `[From ${m.from}]: ${m.content}`)
    .join("\n");

  const reviewResult = await reviewer.run(reviewerInput);
  messages.push({
    from: "reviewer",
    to: "orchestrator",
    content: reviewResult.content,
    timestamp: Date.now(),
  });
  totalTokens += reviewResult.tokensUsed;
  totalToolCalls += reviewResult.toolCalls;

  return {
    content: messages.map((m) => `[${m.from} -> ${m.to}]: ${m.content}`).join("\n\n"),
    tokensUsed: totalTokens,
    toolCalls: totalToolCalls,
  };
}
```

Каждый агент получает только сообщения, адресованные ему. Нет загрязнения контекста. Заметки исследователя с 50k токенами документации никогда не попадают в контекст ревьюера.

### Шаг 4: Сравнение

```typescript
async function compare() {
  const task = "Build a rate limiter middleware for an Express.js API";

  console.log("=== Single Agent ===");
  const single = await singleAgentApproach(task);
  console.log(`Tokens: ${single.tokensUsed}`);
  console.log(`Tool calls: ${single.toolCalls}`);

  console.log("\n=== Multi-Agent ===");
  const multi = await multiAgentApproach(task);
  console.log(`Tokens: ${multi.tokensUsed}`);
  console.log(`Tool calls: ${multi.toolCalls}`);
}
```

Мультиагентный вариант использует больше общего количества токенов (три агента, три отдельных вызова LLM), но контексты каждого агента остаются чистыми. Качество каждого этапа улучшается благодаря специализированному системному запросу.

## Используйте его

Этот урок создает повторяемый запрос для принятия решения о том, когда следует использовать мультиагентную систему. См. `outputs/prompt-multi-agent-decision.md`.

## Упражнения

1. Добавьте четвертого специалиста: "тестер" агент, который получает код от программиста и отзывы ревьюера, затем пишет тесты
2. Измените пайплайн так, чтобы ревьюер мог отправлять обратную связь к программисту для цикла обновлений (максимум 2 раунда)
3. Преобразуйте последовательный пайплайн в распределение: запустите исследователя и "анализатор требований" параллельно, затем объедините их результаты перед передачей к программисту

## Ключевые термины

| Термин | Что говорят люди | Что на самом деле означает |
|------|----------------|----------------------|
| Рой (Swarm) | "Многоагентная система" | Сеть равноправных агентов с общим состоянием и без фиксированного лидера. Поведение возникает из локального взаимодействия. |
| Оркестратор | "Босс-агент" | Агент, у которого есть инструменты для запуска и управления другими агентами. Он планирует и делегирует работу, но может не выполнять ее сам. |
| Координатор | "Трафикопроводник" | Нераспределенная компонента (часто просто код, а не LLM), которая направляет сообщения между агентами на основе правил. |
| Консенсус | "Агенты согласились" | Протокол, при котором несколько агентов должны достичь согласия перед продолжением. Используется, когда конфликтующие результаты требуют решения. |
| Эмерджентное поведение | "Агенты разобрались сами" | Системные паттерны, возникающие из взаимодействий агентов, но не были явно программированы. Может быть полезным или вредным. |
| Распределение / сбор | "Map-reduce для агентов" | Разделение задачи между параллельными агентами (распределение), затем объединение их результатов (сбор). |
