# Передачи и рутины — оркестрация без состояния

> Swarm от OpenAI (октябрь 2024) дистиллировал мультиагентную оркестрацию в два примитива: **рутины** (инструкции + инструменты как системный промпт) и **передачи (handoffs)** (инструмент, возвращающий другого агента). Без автомата состояний, без DSL ветвления — LLM маршрутизирует, вызывая нужный инструмент передачи. OpenAI Agents SDK (март 2025) — продакшен-преемник. Сам Swarm остаётся самым чистым концептуальным референсом — весь его исходный код помещается в несколько сотен строк. Паттерн вирусный, потому что API примерно «агент = промпт + инструменты; передача = функция, возвращающая агента». Ограничение: без состояния, поэтому память — проблема вызывающего.

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib)
**Предварительные требования:** Фаза 16 · 04 (Примитивная модель)
**Время:** ~60 минут

## Проблема

Каждый мультиагентный фреймворк хочет, чтобы вы изучили его DSL: узлы и рёбра LangGraph, команды и задачи CrewAI, GroupChat и менеджеры AutoGen. DSL — реальные абстракции, но они делают вещь тяжелее, чем нужно.

Swarm идёт в противоположном направлении: используйте возможность вызова инструментов, которая уже есть у модели. Передачи становятся вызовами инструментов. Оркестратор — это текущий агент, ведущий разговор. Автомат состояний неявно задан в системных промптах агентов.

## Концепция

### Два примитива

**Рутина.** Системный промпт, определяющий роль агента и доступные инструменты. Представьте набор инструкций: «вы — агент сортировки; если пользователь спрашивает о возврате, передайте агенту возвратов».

**Передача (handoff).** Инструмент, который агент может вызвать и который возвращает новый объект Agent. Рантайм Swarm обнаруживает возвращаемое значение Agent и переключает активного агента на следующий ход.

Это и есть вся абстракция.

```python
def transfer_to_refunds():
    return refund_agent  # Swarm видит возврат Agent → переключает активного агента

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

Системный промпт агента сортировки заставляет его выбирать правильную передачу на основе сообщения пользователя. Вызов инструментов LLM выполняет маршрутизацию.

### Почему это вирусно

- **Маленький API.** Два концепта для изучения.
- **Использует то, что модель уже делает.** Вызов инструментов уже продакшен-уровня у всех провайдеров.
- **Нет обременения автоматом состояний.** Вы не описываете граф; промпты агентов описывают, кому они передают.

### Компромисс без состояния

Swarm явно не сохраняет состояние между запусками. Фреймворк хранит историю сообщений во время запуска, но ничего не сохраняет на постоянной основе. Память, непрерывность, долгосрочные задачи — все это проблема вызывающего.

В продакшене (OpenAI Agents SDK, март 2025) это была одна из главных изменений: SDK добавляет встроенное управление сессиями, ограничения (guardrails) и трассировку, сохраняя примитив передачи.

### Когда Swarm/передачи подходят

- **Паттерны сортировки.** Фронтовый агент направляет пользователя к специалисту.
- **Передачи на основе навыков. «Если задача требует код, вызовите кодера; если исследование — исследователя».**
- **Короткие, ограниченные разговоры.** Поддержка клиентов, FAQ-to-тикет, простые рабочие процессы.

### Когда Swarm затрудняется

- **Длинные сессии с разделяемой памятью.** Передачи сбрасывают состояние разговора в промпт нового агента плюс историю. Нет постоянного состояния между агентами без памяти на стороне вызывающего.
- **Параллельное выполнение.** Передача — по одному за раз; активный агент переключается. Параллелизм требует от вызывающего оркестрации нескольких запусков Swarm.
- **Аудит и воспроизведение.** Запуски без состояния трудно воспроизвести точно; выбор передачи LLM недетерминирован.

### OpenAI Agents SDK (март 2025)

Продакшен-преемник добавляет:

- **Состояние сессии.** Постоянный поток между запусками.
- **Ограничения (guardrails).** Хуки валидации ввода/вывода.
- **Трассировка.** Каждый вызов инструмента и передача логируются.
- **Фильтры передач.** Контроль контекста при передаче.

Примитив передачи выживает; продакшен-эргономика добавляется вокруг него.

### Swarm vs GroupChat

Оба используют маршрутизацию через LLM, но различаются в том, **кто выбирает следующего**:

- GroupChat: выборщик (функция или LLM) выбирает следующего выступающего извне.
- Swarm: текущий агент выбирает преемника, вызывая инструмент передачи.

Swarm — «агент решает, что дальше»; GroupChat — «менеджер решает, что дальше». Решение Swarm живёт в вызове инструмента активного агента; решение GroupChat — в `GroupChatManager`.

## Реализуем

`code/main.py` реализует Swarm с нуля: dataclass агента, механизм передачи (инструмент возвращает Agent) и цикл запуска, обнаруживающий переключения агентов.

Запуск:

```
python3 code/main.py
```

## Результат

Чеклист:

- **Логирование передач.** Каждая передача записывает событие трассировки с исходящим агентом, целевым агентом, снимком контекста.
- **Правила передачи контекста.** Решите, что перемещается при передаче: полная история (дорого), последние N сообщений или резюме.
- **Ограничение на передачу.** Передача специалисту с другими правами на инструменты должна быть аутентифицирована — иначе prompt injection может принудить нежелательные передачи.
- **Обнаружение циклов.** Два агента, передающие друг другу — типичный сбой; обнаруживайте простой проверкой последних K передач.
- **Агент fallback.** Если цель передачи не существует, возвращайтесь к безопасному дефолту.

## Упражнения

1. Запустите `code/main.py`, перенаправьте к агенту возвратов. Подтвердите, что на втором ходу активный агент — возвраты.
2. Добавьте правило обнаружения циклов: если одни и те же два агента передавали друг другу 3 раза подряд, принудительно завершите. Спроектируйте fallback.
3. Прочитайте документацию OpenAI Agents SDK по фильтрам передач. Реализуйте версию «резюмирование при передаче»: исходящий агент сжимает контекст в краткий список перед входом нового агента.
4. Сравните передачу Swarm с выборщиком GroupChatManager. Какой паттерн усугубляет prompt injection и почему?
5. Прочитайте кулинарную книгу Swarm (https://developers.openai.com/cookbook/examples/orchestrating_agents). Определите одно явное решение проектирования, которое Swarm принимает, а OpenAI Agents SDK изменил или сохранил.

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|----------------|------------------------|
| Рутина | «Промпт агента» | Системный промпт + список инструментов. Определяет роль и доступные передачи. |
| Передача (handoff) | «Переключение на другого агента» | Инструмент, который активный агент может вызвать и который возвращает нового Agent. Рантайм переключает активного агента. |
| Без состояния | «Нет памяти между запусками» | Swarm ничего не сохраняет; память — ответственность вызывающего. |
| Активный агент | «Кто говорит сейчас» | Агент, ведущий текущий разговор. Передача меняет его. |
| Передача контекста | «Что перемещается при передаче» | Политика того, какую историю видит входящий агент: полную, последние N или резюме. |
| Цикл передач | «Агенты пинг-понга» | Режим отказа, когда два агента постоянно передают друг другу. |
| OpenAI Agents SDK | «Продакшен Swarm» | Преемник марта 2025; добавляет сессии, ограничения, трассировку поверх примитива передачи. |
| Фильтр передач | «Шлюз при передаче» | Функция SDK для проверки и изменения контекста на границе передачи. |
