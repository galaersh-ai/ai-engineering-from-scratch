# Параллельные / роевые / сетевые архитектуры

> В отличие от супервайзера: нет центрального решающего. Агенты читают разделяемую шину событий, подбирают работу асинхронно, записывают результаты обратно. LangGraph явно поддерживает «роевую архитектуру» для децентрализованных динамических сред. Matrix (arXiv:2511.21686) представляет и управление потоком, и поток данных как сериализованные сообщения, передаваемые через распределённые очереди для устранения узкого места оркестратора. Компромисс явный: детерминизм и прослеживаемость в обмен на масштабируемость. Рой подходит для задач со множеством независимых подзадач; не подходит для задач, требующих единого согласованного плана.

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib, `threading`, `queue`)
**Предварительные требования:** Фаза 16 · 05 (Паттерн супервайзера), Фаза 16 · 04 (Примитивная модель)
**Время:** ~75 минут

## Проблема

Супервайзер масштабируется до нескольких работников. А если до сотен? Сам супервайзер становится узким местом: каждое решение о том, кто что делает, проходит через одного агента. Один медленный шаг планирования останавливает всю систему.

Роевые архитектуры переворачивают дизайн. Вместо центрального планировщика, распределяющего работу, работники подбирают работу из разделяемой очереди. «Координация» встроена в семантику шины событий. Нет оркестратора; система масштабируется до тех пор, пока масштабируется очередь.

## Концепция

### Форма

```
                ┌──── разделяемая очередь ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Работник  Работник  Работник   Работник
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            пул результатов
```

Нет оркестратора. Каждый работник повторяет: взять задачу, обработать, записать результат (и опционально поставить в очередь последующие задачи).

### Когда рой подходит

- **Много независимых задач.** Парсинг, трансформация, классификация. Задачи не зависят друг от друга.
- **Работа с переменной длительностью.** Если одни задачи занимают 100мс, а другие 10с, рой автоматически балансирует нагрузку — быстрые работники берут следующие задачи. Супервайзер должен предугадывать длительность.
- **Пропускная способность важнее детерминизма.** Вам важно общее время завершения, а не строгий порядок.

### Когда рой не работает

- **Упорядоченные рабочие процессы.** Если шаг 3 требует выходные данные шага 2, рой рискует запустить шаг 3 до завершения шага 2.
- **Задачи с глобальным планом.** Сложные исследовательские вопросы выигрывают от планировщика. Рой исследователей производит независимые факты, а не согласованный отчёт.
- **Отладка.** Без центрального лога и асинхронной работы воспроизведение бага дорогое.

### Matrix (arXiv:2511.21686)

Matrix — статья 2025 года, которая доводит рой до его логического завершения: и управление потоком, и поток данных — сериализованные сообщения на распределённых очередях. Нет центрального координатора. Устойчивость к сбоям обеспечивается долговечностью сообщений. Масштабируемость — проблема брокера сообщений, а не системы.

Вклад: модель программирования, где мультиагентная координация — это «на какую тему сообщений подписан этот агент?» вместо «какого агента выберет супервайзер следующим?». Это делает систему похожей на pub/sub event mesh.

### Роевая архитектура LangGraph

Документация LangGraph 2025 года явно описывает «роевую архитектуру» как один из мультиагентных паттернов: агенты — узлы, но рёбра образуют ориентированный граф с циклами, и любой узел может быть активирован из пула. Работник берёт из доступных задач по условию, а не по назначению супервайзера.

### Режим отказа: голодание и горячие точки

Если все работники берут самую быструю доступную задачу, долгие задачи никогда не берутся, пока не останутся единственными. Классическое голодание очереди.

Смягчения:
- Очереди с приоритетом и явным старением (увеличение приоритета со временем ожидания).
- Специализация работников: некоторые работники берут только «долгие» задачи.
- Обратное давление: ограничьте количество быстрых задач в очереди.

### Связь с маршрутизацией по содержанию

Рой естественно сочетается с маршрутизацией по содержанию (Урок 22). Вместо универсальной очереди — одна очередь на тип сообщения. Специализированные работники подписаны только на свой тип. Это основа архитектур шин сообщений, масштабируемых до тысяч агентов.

## Реализуем

`code/main.py` реализует рой из 4 рабочих потоков, берущих задачи из разделяемой `queue.Queue`. Задачи имеют переменную длительность (некоторые быстрые, некоторые медленные). Демо сравнивает:

- **Последовательный бейзлайн:** один работник обрабатывает все задачи последовательно.
- **Фиксированное назначение:** каждая задача назначена конкретному работнику (в стиле супервайзера).
- **Рой:** работники берут из разделяемой очереди.

Рой автоматически балансирует нагрузку; при фиксированном назначении быстрые работники бездельничают, пока их назначенная задача медленная.

Запуск:

```
python3 code/main.py
```

Вывод показывает количество задач на работника (рой распределяет неравномерно, но оптимально) и время выполнения.

## Применение

`outputs/skill-swarm-fit.md` оценивает,ควร ли задача использовать рой или супервайзер. Входы: независимость задач, дисперсия длительности, требования порядка, потребности в отладке.

## Результат

Чеклист:

- **Очередь с приоритетом и старением.** Предотвращает голодание долгих задач.
- **Идемпотентность работника.** Задача может быть взята повторно, если работник падает во время выполнения. Работники должны быть идемпотентными.
- **Долговечная очередь.** Используйте Kafka, Redis Streams или очередь на базе БД для продакшена. `queue.Queue` — только in-memory.
- **Наблюдаемость на задачу.** У каждой задачи есть trace ID; каждый работник логирует начало/конец с ним.
- **Обратное давление.** Если очередь растёт быстрее, чем работники её разбирают, замедлите производителя.

## Упражнения

1. Запустите `code/main.py`. Насколько рой быстрее последовательного на workload с переменной длительностью? Насколько быстрее фиксированного назначения?
2. Добавьте вариант с очередью приоритетов (используйте `queue.PriorityQueue`). Назначьте приоритет по полю «важность» задачи. Наблюдайте, не голодают ли задачи с низким приоритетом при постоянной нагрузке.
3. Реализуйте детектор горячих точек: логируйте когда любой работник обрабатывает в 3× больше задач, чем самый медленный. Что это говорит о распределении длительности задач?
4. Прочитайте аннотацию и раздел 3 статьи Matrix (arXiv:2511.21686). Определите один конкретный компромисс, который Matrix принимает (прирост масштабируемости) и один, который отдаёт (прослеживаемость, детерминизм).
5. Конвертируйте демо роя на использование `queue.Queue` кортежей (тип_задачи, данные), с подпиской работников только на определённые типы. Какие правила маршрутизации имеют смысл при гетерогенных задачах?

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|----------------|------------------------|
| Роевая архитектура | «Децентрализованные агенты» | Работники берут из разделяемой очереди; нет центрального оркестратора. |
| Шина событий | «Агенты подписаны на темы» | Брокер сообщений, маршрутизирующий задачи работникам по типу или содержанию. |
| Голодание | «Задача никогда не выполняется» | Задача с низким приоритетом никогда не берётся, потому что более приоритетная работа поступает непрерывно. |
| Горячие точки | «Один работник тонет» | Дисбаланс нагрузки, когда один работник получает большинство задач. |
| Обратное давление | «Замедлите производителя» | Механизм, сигнализирующий выше по потоку прекратить производство, когда очередь заполняется. |
| Идемпотентный работник | «Безопасно перезапускать» | Задача, обработанная дважды, даёт тот же результат. Требуется потому что работники могут падать во время выполнения. |
| Долговечная очередь | «Переживает сбои» | Очередь на базе диска или реплицированного хранилища; задачи не теряются при падении работника. |
| Фреймворк Matrix | «Полноценный рой с передачей сообщений» | И данные, и управление потоком — сериализованные сообщения на распределённых очередях. |
