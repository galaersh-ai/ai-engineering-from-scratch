# Общая память и паттерны доски объявлений

> В мультиагентных системах 2026 года сосуществуют два подхода: **пул сообщений** (все видят сообщения всех, как в AutoGen GroupChat или MetaGPT) и **доска объявлений с подпиской** (агенты подписаны на релевантные события, как в Context-Aware MCP или фреймворке Matrix). Оба являются единственной состоящей частью мультиагентной системы — а значит, оба там, где живут интересные баги. Референсный режим отказа — **отравление памяти**: один агент галлюцинирует «факт», другие агенты считают его верифицированным, и точность постепенно деградирует способом, который гораздо сложнее отлаживать, чем немедленный крах. Этот урок строит обе структуры на stdlib, инжектирует атаку отравления и показывает три смягчения, которые реально работают в продакшене.

**Тип:** Изучение + Практическая работа
**Языки:** Python (stdlib, `threading`)
**Предварительные требования:** Фаза 16 · 04 (Примитивная модель), Фаза 16 · 09 (Параллельные сети роёв)
**Время:** ~75 минут

## Проблема

Мультиагентным системам нужно место, где агенты разделяют факты. Буквальный вариант — «передавать всё в сообщениях» — но это изобретает разделяемое состояние с лишним копированием. Другой — «дать всем глобальный лог» — но глобальные логи растут бесконечно и легко отравляются. Третий — «спроецировать обзор для каждого агента» — масштабируется, но требует схему.

Когда один из агентов галлюцинирует и записывает галлюцинацию в разделяемое состояние, каждый нижестоящий агент, читающий это состояние, принимает галлюцинацию за факт. К тому моменту, когда человек замечает, цепочка рассуждений ушла на пять шагов вглубь, а первопричина — в третьем сообщении, когда-либо написанном. Отладка деградации точности мультиагентной системы сложнее отладки краха.

Это отравление памяти. Это второе по документированности семейство отказов в таксономии MAST (Cemri et al., arXiv:2503.13657), и оно структурное: любой дизайн общей памяти без прослеживаемости и неверифицируемого верификатора рано или поздно это продемонстрирует.

## Концепция

### Две основные топологии

**Полный пул сообщений.** Каждый агент читает каждое сообщение. AutoGen GroupChat и MetaGPT используют это. Просто, прозрачно,inspectable, но не масштабируется за ~10 агентов, потому что контекст каждого агента заполняется работой других.

```
агент-A ──запись──▶ ┌────────────────┐ ◀──чтение── агент-D
                   │ пул сообщений  │
агент-B ──запись──▶ │                │ ◀──чтение── агент-E
                   │ (глобальный лог)│
агент-C ──запись──▶ └────────────────┘ ◀──чтение── агент-F
```

**Доска объявлений с подпиской.** Агенты объявляют интерес к темам; субстрат маршрутизирует только релевантные сообщения. CA-MCP (arXiv:2601.11595) и децентрализованный фреймворк Matrix (arXiv:2511.21686) используют это. Масштабируется дальше, но требует предварительного проектирования схемы, чтобы подписки были осмысленными.

```
                   ┌─ тема: цены ──┐
агент-A ──pub────▶ │                  │ ──▶ агент-D (подписан)
                   ├─ тема: заказы ──┤
агент-B ──pub────▶ │                  │ ──▶ агент-E (подписан)
                   ├─ тема: оповещения ──┤
агент-C ──pub────▶ │                  │ ──▶ агент-F (подписан)
                   └──────────────────┘
```

### Когда побеждает каждый

- **Полный пул** побеждает когда агентов мало (< 10), они гетерогенны и разговор короткий. Рассуждать о том, кто что сказал, тривиально, когда все видят всё.
- **Доска объявлений** побеждает когда агентов много, они гомогенны по роли, но многочисленны по экземпляру (рои), и разговор долгосрочный. Маршрутизация экономит стоимость токенов и загрязнение контекста.

Продакшен-системы часто комбинируют: маленький полный пул наверху (слой планирования), доски объявлений внизу (слой работников).

### Отравление памяти — один сценарий

Три агента работают над исследовательской задачей. Агент A — агент извлечения. Агент B — резюматор. Агент C — аналитик.

1. A получает страницу и записывает сообщение в разделяемое состояние: «Исследование сообщает об улучшении точности на 42%».
2. Полученная страница на самом деле говорила «улучшение на 4.2%». A галлюцинировал десятичную точку.
3. B, читая разделяемое состояние, пишет: «Сообщается о большом приросте точности на 42% (источник: A)».
4. C, читая разделяемое состояние, пишет: «Рекомендуется внедрение — рост на 42% трансформационный».
5. Финальный отчёт цитирует число 42%, которого никогда не существовало.

Ни один агент не упал. Ни один тест не провалился. Система «работала». Галлюцинация перешла из контекста одного агента в рассуждения каждого нижестоящего агента через разделяемое состояние.

### Почему это структурно

Без разделяемого состояния галлюцинация агента A остаётся в контексте A. Нижестоящие агенты повторно извлекли бы или повторно вывели и могли бы поймать ошибку. При наивном разделяемом состоянии контекст A становится контекстом всех, и галлюцинация отмывается в факт.

Проблема не в разделяемом состоянии как таковом — а в разделяемом состоянии **без прослеживаемости и без независимого верификатора**. Три смягчения решают это:

1. **Атрибуция прослеживаемости при каждой записи.** Каждая запись в разделяемом состоянии фиксирует кто записал, когда, при каком промпте и (если применимо) какой источник цитировал агент. Нижестоящие агенты читают с скептицизмом, привязанным к прослеживаемости.
2. **Версионируйте записи; рассматривайте их как только добавляемые.** Коррекция — новая запись, которая заменяет старую, а не обновление на месте. Аудит-след сохраняется.
3. **Сохраняйте как минимум одного агента, который не может писать в разделяемое состояние.** Верификатор, доступный только для чтения, сэмплирует записи, повторно извлекает источники и отмечает несоответствия. Поскольку он не может писать в пул, его невозможно отравить пулом.

### Прецедент доски объявлений (Hayes-Roth, 1985)

Паттерн доски объявлений на четыре десятилетия старше агентов LLM. Hayes-Roth (1985, «A Blackboard Architecture for Control») описал специализированные Источники Знаний, которые наблюдают глобальную доску, вносят частичные решения и запускают другие источники. Доска объявлений 2026 года (CA-MCP, Matrix) — тот же паттерн с агентами LLM в роли Источников Знаний и JSON-объектами в качестве частичных решений. Старая литература документирует решения для борьбы за запись, Opportunistic Control и согласованности, которые современные системы переоткрывают.

### Проекция vs полный обзор

Чистая доска объявлений даёт каждому подписчику одну проекцию (ограниченную темой). Более агрессивный дизайн — **проекция на агента**: каждый агент получает обзор, настроенный под его роль. Редьюсеры состояния LangGraph — каноничная реализация 2026 года: функция редьюсера сворачивает глобальное состояние в ролевой срез.

Проекция на агента масштабируется дальше, но требует схему. Без неё вы пересоздаёте ad-hoc проекцию в промпте каждого агента.

### Паттерны борьбы за запись

Одновременная запись несколькими агентами — задача параллелизма, а не только LLM. Три паттерна работают:

- **Последовательный писатель (один производитель).** Все записи проходят через одного координатора, который сериализует. Просто, но узкое место.
- **Оптимистичная блокировка с версионированием.** У каждой записи есть версия; писатели падают при несовпадении версий и повторяют. Классический приём БД.
- **Разбиение по темам.** Разные агенты владеют разными темами. Нет межтемовой борьбы. Требует спроектированных границ разбиения.

Большинство фреймворков 2026 года по умолчанию используют последовательного писателя, потому что вызовы LLM достаточно медленные, чтобы борьба была редкой и узкое место не мешало.

### Неверифицируемый верификатор

Самое нагруженное смягчение — верификатор, доступный только для чтения. Правила реализации:

- Верификатор разделяет состояние с командой (читает доску или пул).
- Верификатор не имеет дескриптора записи в разделяемое состояние — только в отдельный канал верификации.
- Верификатор независимо извлекает источники, цитируемые в записях. Отмечает несоответствия.
- Собственные выходы верификатора маршрутизируются человеку или отдельному агенту решений, никогда обратно в пул.

Без этого разделения выходы верификатора становятся новыми записями в пуле, что означает: отравленный пул отравляет верификатор, который отравляет свои верификации.

## Реализуем

`code/main.py` реализует обе топологии на stdlib Python плюс игрушечная атака отравления и три смягчения.

- `MessagePool` — потокобезопасный лог только добавления с полным чтением.
- `Blackboard` — pub/sub с ключами по темам и подписками на агента.
- `ProvenanceEntry` — каждая запись фиксирует (автор, временная метка, хэш промпта, URI источника).
- `PoisoningScenario` — запускает исследовательскую задачу из трёх агентов, где агент A галлюцинирует десятичную точку. Печатает финальный отчёт.
- `Verifier` — агент, доступный только для чтения, который повторно извлекает источники и отмечает несоответствия. Запускает тот же сценарий с верификатором.

Запуск:

```
python3 code/main.py
```

Ожидаемый вывод:
- Запуск 1 (без верификатора): галлюцинированные 42% распространяются в финальный отчёт.
- Запуск 2 (с верификатором): верификатор отмечает несоответствие, пул помечен как «отмеченный», финальный отчёт содержит опровержение.

## Результат

Для любого дизайна общей памяти:

- Записывайте прослеживаемость при каждой записи: `(автор, временная метка, хэш_промпта, цитируемые_вызовы_инструментов, URI_источника)`.
- Делайте лог только добавления. Коррекции — новые записи, ссылающиеся на заменяемую.
- Разверните как минимум одного верификатора, доступного только для чтения, с независимым доступом к источникам.
- Маршрутизируйте выход верификатора в отдельный канал, а не обратно в разделяемый пул.
- Логируйте соотношение записей, которые являются заменами — растущее соотношение — ранний признак паттернов галлюцинаций.

## Упражнения

1. Запустите `code/main.py`. Подтвердите, что запуск 1 распространяет галлюцинацию, а запуск 2 её ловит.
2. Добавьте вторую галлюцинацию: агент B изобретает размер датасета. Верификатор должен поймать обе без ручной настройки под каждую.
3. Переключите полный пул на доску объявлений с разбиением по темам (`цены`, `резюме`, `анализы`). Какие сценарии отравления разбиение по темам делает сложнее, а какие не помогает?
4. Прочитайте Hayes-Roth (1985, «A Blackboard Architecture for Control»). Определите два паттерна управления из статьи, не обсуждённых в этом уроке, от которых системы 2026 года выиграли бы.
5. Прочитайте CA-MCP (arXiv:2601.11595). Отобразите его Shared Context Store на класс MessagePool или Blackboard из `code/main.py`. Какие примитивы CA-MCP добавляет поверх?

## Ключевые термины

| Термин | Что говорят | Что это на самом деле |
|------|----------------|------------------------|
| Пул сообщений | «Разделяемая история чата» | Лог только добавления, который читает каждый агент. Полная прозрачность, плохая масштабируемость. |
| Доска объявлений | «Разделяемое рабочее пространство» | Pub/sub с ключами по темам. Агенты подписаны на релевантные темы. Масштабируется дальше. |
| Прослеживаемость | «Кто что записал» | Метаданные при каждой записи: автор, временная метка, промпт, источники. |
| Отравление памяти | «Галлюцинации распространяются» | Ошибка одного агента попадает в разделяемое состояние, нижестоящие агенты принимают её за факт. |
| Только добавление | «Нет обновлений на месте» | Коррекции — новые записи, которые заменяют. Сохраняет аудит-след. |
| Неверифицируемый верификатор | «Независимый аудитор» | Агент, доступный только для чтения, который повторно извлекает источники и отмечает несоответствия. |
| Проекция | «Ограниченный обзор» | Обзор на агента, вычисляемый из глобального состояния. Редьюсеры LangGraph — каноничный случай. |
| Источник знаний | «Специализированный агент» | Термин Hayes-Roth 1985 года для участника доски объявлений. |
